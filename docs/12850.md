# 当乌云落下时，不要被烧伤

> 原文:[https://www . site point . com/don-get-burn-when-the-cloud-going-down/](https://www.sitepoint.com/dont-get-burned-when-the-cloud-goes-down/)

![](../Images/fd06ec0c70540ecf7ede3274b2a5c067.png "aws-logo")昨天，亚马逊的 S3 云存储服务[今年第二次关闭](http://www.centernetworks.com/amazon-s3-down-july-2008)，导致一系列 web 2.0 网站随之关闭。SmugMug、Twhirl、Twitter、SlideShare 甚至 SitePoint 等网站和应用都受到了宕机的影响。正如 WebWorkerDaily 所指出的，亚马逊的 SLA 保证了 99.9%的正常运行时间——或者说每月不超过 45 分钟的停机时间。因此，可以肯定的是，亚马逊将因最近长达数小时的宕机而欠很多网站退款。

然而，对于亚马逊或其他云存储服务的许多用户来说，宕机是不可接受的，它给企业带来的损失不仅仅是错过托管的价格。旅游指南初创公司 [Planaroo](http://planaroo.com/) ，也受到了 S3 中断的负面影响，今天在他们的公司博客上写了一篇文章，概述了[他们从亚马逊的问题中了解到的](http://webspeed.typepad.com/planaroo/2008/07/rethinking-what.html)。以下是他们的观点:

*   备份是一件好事。有一个 B 计划是必要的。花额外的钱备份您的关键任务文件。Planaroo 对他们在 S3 上保存的所有照片都有备份，所以当亚马逊的服务中断时，他们只需替换照片数据库中的一个 URL，就可以轻松切换到备份。
*   修好它。不要抱最好的希望。然而，Planaroo 做错了一件事，那就是认为亚马逊会迅速改变现状。两个小时后，他们决定放弃等待亚马逊，转而使用他们的备份。教训:当事情开始出错时，不要以为你的主人会很快解决它们。你的应急计划是有原因的，所以使用它。
*   **保持备份最新并随时可用。**当您的主主机出现故障，您需要启动备份时，请确保它是最新的。如果你不得不浪费时间上传文件到你的备份服务器，那么你只是在浪费时间。
*   其他服务也可能下降。不仅仅是你的宿主。在 API 和托管库的世界里，许多初创公司依赖多个第三方服务来保持运转。例如，Planaroo 使用[谷歌的 AJAX 库](http://code.google.com/apis/ajaxlibs/)。谷歌可能会遇到与亚马逊类似的问题(事实上，谷歌的应用引擎在一个月前出现了一些[延长的停机时间](http://www.techcrunch.com/2008/06/17/google-app-engine-goes-down-and-stays-down/)，所以为你依赖的每项服务制定一个应急计划是值得的。

最后一点强调了这个列表的重要之处:它不仅仅适用于 S3 或云存储服务。这些都是适用于任何托管环境的好经验。

要获得 Planaroo 列出的完整课程列表，请查看他们的博客文章。
