# 公共网站管理工具:各个击破

> 原文：<https://www.sitepoint.com/public-website-admin-tools-divide-and-conquer/>

设计 web 应用程序的第一步是弄清楚如何管理 beast 以及这些管理工具应该放在哪里。这通常归结为一个关键问题:公共网站本身是否也应该是管理工具？虽然没有一种方法在每种情况下都有效，但我通常发现将管理工具分开比将工具嵌入面向公众的网站更有意义。以这种方式解决这个问题的主要原因是:

**不同性质的野兽**

面向公众的网站和网站管理工具本质上是完全不同的应用程序。你的面向公众的网站需要对任何数量的浏览器、平台和用户都是高度可访问的；它应该是高度符合标准和谷歌友好。从开发的角度来看，可伸缩性是一个大问题。这意味着完全禁用视图状态和有争议的会话状态，或者避免使用像 ASP.NET AJAX UpdatePanels 这样有用的工具，而不是更精简的脚本。

另一方面，您的管理工具可能针对可控平台上的一小部分用户。带宽使用可能不是一个主要问题，也不是像标准合规性，可访问性和搜索引擎优化这样的细节。你可以更自由地让 ASP.NET 成为 ASP.NET，不用担心大的视图状态大小，使用 UpdatePanels，拥有大量的会话变量，或者假装你是一个桌面开发人员。它有趣且令人振奋，更不用说有助于加速开发过程。

更有效的方法是预先分别对待每个应用程序，不必担心限制您的工具集以保持管理工具的美观和面向公众，也不必担心损害站点的目标以保持管理工具开发的合理性。

**项目管理**

生活中的一个事实是，对管理人员来说很重要的问题，比如“这个网站看起来怎么样”往往会被拖延，超出所有可能的信念。而世俗的问题，如“这个网站将如何工作”往往在过程的早期就得到解决。现在，如果你的逻辑工具和工作流程都依赖于那些管理层的问题，那么你可能会陷入困境，因为你不能真正开始重要的建设，直到大人物真正关注他们要求的项目。但是，如果你已经把东西分开了，你就可以在任何人做出使用哪种粉红色的艰难决定之前拥有一个有效的管理工具。

另一个好处是，如果你有资源，在你的团队的不同成员之间分配工作变得容易得多，因为他们不需要为相同的代码浪费时间。最后，三年后，你可以给公共网站改头换面，而不用浪费更多的开发资源。

**安全**

使用完全独立的管理应用程序提供了许多安全选项。首先，从 IIS 的角度来看，在 web 服务器级别强制管理应用程序在 SSL 下运行要容易得多。其次，对于我们这些锡箔帽团队的成员来说，您可以通过限制对管理工具的访问来进一步增强传输层安全性。根据您的部署情况，可以将它们放在 VPN 后面，也可以将它们放在企业防火墙内。

一个主要的潜在安全问题是特权升级。现在，如果你在你的公共网站上使用“更特殊”但面向公众的用户来更新内容，你就离向公众开放你的管理工具的重要部分只差一个 web.config 打字错误了。然而，如果您使用单独的应用程序，那么这样的场景就没有风险。您甚至可以在管理工具中使用完全独立的身份验证方法，这对于人们习惯使用 Active Directory 进行身份验证的企业应用程序来说非常方便。

最后一个安全优势是在数据库层。独立的应用程序使使用独立的数据库帐户变得更干净、更容易，允许用户从功能上将公共权限与私有权限一直分离到核心。这意味着，即使事情变得非常糟糕，并且基于 url 的 Sql 注入漏洞进入了大范围，您仍然可以稍微松一口气，因为您知道您的公共站点的数据库用户已经被非常非常地锁定了。

我不建议采用这种方法，除非已经采用了可靠的三层设计，其中核心业务逻辑嵌入在第三个组件中。使用 SqlDataSource 以这种方式构建网站是一种愚蠢的做法。但是，对于一个结构合理的项目来说，这是一个非常成功的方法。
