# 深入探究 QMail

> 原文：<https://www.sitepoint.com/digging-deeper-into-qmail/>

去年，我在开源的几篇文章中加入了 QMail。经常的讨论转移到为什么 QMail 不像其他同类产品那样，不进行改进，而是让管理员一点一点地修补，以达到 Postfix、Sendmail 和其他同类产品的水平。

作为一个 Sendmail 的长期用户，我在 2003 年末和 2004 年为一些客户开始使用 QMail 我也经历了一个学习曲线，适应构建和调整它以满足需求(在某种程度上现在仍然如此！).

我已经开始喜欢 QMail 了——尤其是使用 tcpserver 来选择性地允许、禁止拒绝消息或者干脆直接断开与服务器的连接。我还喜欢在 SMTP 之前使用 POP 的便利性，以及控制 IP 地址为 SMTP 中继缓存多长时间的能力。

我的一位同事，布朗米互联网公司总裁奥班·兰比是一位经验丰富的系统管理员，他来自 sendmail，他对手工打补丁的看法与我不同。

他在给我的信中写道:“就像生活中大多数美好的事情一样，当入门门槛提高时，如果你花时间越过这个门槛，回报似乎总是非常值得的。”。

“不可否认，Qmail 比 Sendmail 更难安装，但 Qmail 的灵活性总是让我的工作变得容易得多。Qmail 庞大的用户群、面向客户端(Qmailadmin)和管理员(Vqadmin)的基于 web 的电子邮件帐户管理、第三方软件集成的简易性(Spamassassin)以及配置文件和目录的智能布局，使我在配置邮件服务器时可以轻松做出选择。”

绝对是 QMail 的粉丝，他是我今天使用它的主要原因之一。然而——出于好奇，为什么 QMail 没有简化(至少在一些评论者和我自己看来)一个内置了许多手动补丁以方便部署和配置的构建，我去了源代码并向 D.J. Bernstein 询问了其中的一些问题。

我不得不假设一个月后没有回应是因为:

*   他想——“这家伙到底是谁？”；>),
*   伯恩斯坦似乎参与了许多不同的项目，可能会收到数十亿封这种性质的电子邮件，
*   他的哲学根本不符合我们的愿望。

嗯——这很好——没人需要别人告诉你如何构建你的应用程序。然而，当你去开源的时候——期望得到强的和弱的输入。我们正在寻找一些便利，Bernstein 正试图构建健壮和安全的软件。当这两者交织在一起时，有时会导致问题，就像我们在其他大大小小的软件中看到的那样。

我的问题相当简单，没有挑战性:

1)在“标准”类型的许可证下获得许可——源代码可以“不加修改”地与作为补丁提供的任何改进共享(因此是我们当前的讨论)——那些最受欢迎的补丁是否有可能被审查并成为金牌补丁——作为源代码的一部分包含在未来的发行版中？

2)由于他“似乎”是 QMail 代码库的唯一所有者，当他自愿或不自愿地不再管理源代码时，是否有继任计划？

3)2001 年，Bernstein 声称全球有超过 70 万个系统在运行 QMail。我们现在在哪里？

作为一个 Qmail 粉丝，显然还有很多其他粉丝，我希望在某个时候能得到这些问题的答案。也许一些更接近 QMail 开发或该项目的开发者社区的读者会分享一些见解。
