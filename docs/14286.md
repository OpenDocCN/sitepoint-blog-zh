# 为灾难做准备

> 原文：<https://www.sitepoint.com/planning-for-disaster/>

布莱恩·琼斯写的一篇关于 Linux.com 的文章让我开始快速回顾我为自己和客户的系统制定的灾难恢复计划。

我来自金融服务行业的企业级 IT 部门，有丰富的规划经验，所以即使在我现在工作的中小型环境中，灾难恢复也总是成为客户对话的一部分。

和大多数人一样，我从操作的 web 端(数据库、配置文件、数据等)执行传统备份。)在本地作为热备份，在远程每天使用自动化。每月一次，将这些备份制作成 CD 或 DVD，并存放在银行保险箱中。

在内部(非 web)发生相同的基本过程，每天在本地热拷贝中备份，每周远程备份，以及每月归档到 CD 或 DVD 一次。

在许多情况下，我也开始在客户机上使用小型 RAID 阵列。例如，一个客户每天处理音频和视频。它们运行两个 160 GB 驱动器的外部 firewire RAID 1 阵列。在 RAID 1 中，驱动器 A 由驱动器 b 镜像。我们还有一个驱动器 C，它每周换出一次并被带离现场。在这个场景中，我们选择了热插拔驱动器，这样我们就可以根据需要拉出和更换镜像驱动器。

最近，我还测试了第三方备份服务的使用，这些服务通过宽带或更高的互联网连接提供压缩数据传输和存储。这些供应商利用加密来保证流程的安全，并无限期地或通过服务初始化时设置的参数来存储数据。我发现了一个特殊的服务，提供 1 GB 的压缩存储，每月 17.50 美元(备份 2 GB 的数据)。

回到开头——我还没有做过的一件事是调查像 SUSE 中那样的剖析软件。SCPM(系统配置文件管理)看起来是这样一个附加到我的计划的候选人。SCPM 分析和存储关于(例如)Apache、防火墙和其他守护程序配置文件的信息，并将它们存储在一个文本文件中。只要这些文件包含在您的备份计划中，从服务器灾难中恢复可能会更容易。

基本上，重新安装带有所需守护程序的基本操作系统后，将恢复配置文件，然后 SCPM 应该为这些关键应用程序重新部署存储的配置数据。你不仅从灾难中恢复过来，而且可能还快了几个小时。

我期待听到您的灾难恢复计划、目标或问题。
