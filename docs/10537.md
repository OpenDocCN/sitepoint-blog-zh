# 你应该关闭 PHP 标签吗？争论还在继续…

> 原文:[https://www.sitepoint.com/closing-php-tags-debate/](https://www.sitepoint.com/closing-php-tags-debate/)

我对我最近的文章 [**的回应感到惊讶，你应该关闭你的 PHP 代码标签吗？**](https://www.sitepoint.com/should-you-close-your-php-code-tags/) 没想到会有争议或者在开发者中引起这么大的争论！

概括地说，结束 PHP 标签(？>)是完全可选的，如果你正在编写一个 PHP 专用的文件并且没有混合 HTML 的话。包含单个类定义的文件是很好的选择。

省略结束标记还有另一个好处:不可能意外地在文件末尾添加空白。因此，在修改 HTTP 页头之前，不能将输出发送到浏览器。

许多回答者认为这是一种黑客行为，并表示 PHP 程序员应该简单地修改他们的代码。对于其他人来说，省略结束标记可以防止出现错误的可能性——即使根本不存在错误。这对于许多参与者使用不同 ide 的项目特别有用。

受访者大多属于三个群体之一:

1.  PHP 开发人员，可能在更大的项目或团队中工作，哪里省略了结尾？>是有用的、推荐的或记录在案的编码标准。
2.  编码纯粹主义者，可能有 XHTML 或 XML 背景，他们认为缺少标签是错误的。
3.  其他学科的开发人员认为这个讨论是荒谬的！

根据 SitePoint 的民意调查，三分之一的 PHP 开发人员没有使用结束标记，三分之一的人认为这是一种不好的做法，还有三分之一的人没有意识到它可以被省略。

为了记录，我经常省略结束标记。这对 PHP 解释器或浏览器来说没什么区别，所以没有理由包含它。然而，我可以理解为什么有些开发人员不喜欢这种编码风格。

提出了一些有趣的想法。第一种是解释器可以默认采用 PHP 代码，并允许转义成 HTML，例如

```
 // PHP file
DoSomeStuff();
<?html
<p>output HTML</p>
?>
DoMoreStuff(); 
```

我怀疑这个想法太革命了。它可以作为 php.ini 选项添加，但是它会破坏现有的应用程序并导致混乱。

然而，纯 PHP 文件是一个更有潜力的想法。本质上，您可以指定一个文件只包含 PHP 代码。这可以通过以下方式实现:

*   使用特殊的文件扩展名，例如*文件名*T2。pphp ，或
*   使用修改后的包含/要求函数，例如

```
 include('filename.php', PHP_ONLY); 
```

如果实现了这一点，您的 PHP 代码就不需要开始或结束标记。此外，可以在不影响现有 PHP 应用程序的情况下添加功能——它将是向后兼容的。

但是，嘿，这只是一个想法，可能永远不会比这篇文章更进一步。PHP 编码风格的争论将会在…
