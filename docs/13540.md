# 防弹 HTML:完善标记的 37 个步骤

> 原文:[https://www.sitepoint.com/html-37-steps-perfect-markup/](https://www.sitepoint.com/html-37-steps-perfect-markup/)

这篇文章强调并回答了一些关于 HTML 的常见问题。HTML 是 Web 的基础，开发者和设计者都需要理解它。

##### 1.什么是 HTML？

HTML，即超文本标记语言，是一种主要用于 Web 文档的标记语言。任何用标记语言编写的文档都散布着标记，这些标记表明了某些段落的含义。从 2.0 版本开始，HTML 已经成为一种更通用的标记语言的应用:SGML(标准通用标记语言)。

HTML 定义了许多元素类型。元素类型为其内容赋予一些语义含义。例如，`em`元素类型将其内容的重点放在周围的文本上。元素是元素类型的具体实例。一个元素通常由一个开始标签(`<em>`)、一些内容和一个结束标签(`</em>`)组成。

```
This HTML stuff is really, <em>really</em> nifty!
```

HTML 允许省略一些结束标签(甚至一些开始标签)。不要混淆标签和元素；即使省略了`<body>`和`</body>`标签，一个`body`元素也会存在。某些元素类型不能有结束标记。一个例子是`br`，它表示换行符。

```
Baa baa black sheep, have you any wool?<br> 

Yes sir, yes sir, three bags full
```

开始标记可以包含属性，包括属性名、等号(=)和属性值。例如，我们可以使用 lang 属性来指定元素内容的语言。

```
Jean-Claude often exclaimed <em lang="fr"> bon sang</em> despite the fact that no-one understood him.
```

在某些情况下，属性值必须用引号括起来，所以最好总是用引号括起所有属性值。一些布尔属性允许在 HTML 中最小化，这意味着名称和等号被省略(例如用`selected`代替`selected="selected"`)。有些属性对于某些元素类型是必需的，例如，`img`元素中的`alt`属性。

```
<img src="/images/sitepoint.gif" alt="SitePoint">
```

初学者经常使用“alt tag”这样的短语，但这是不正确的命名法；alt 是属性，不是标签。标签被`<...>`包围。

##### 2.HTML 有哪些不同的版本？

HTML 的第一个版本(1989 年)没有版本号；它只是“HTML”。由互联网工程任务组(IETF)于 1995 年发布的 HTML 的第一个“标准化”版本被称为 HTML 2.0。

然后[万维网联盟](http://www.w3.org) (W3C)就形成了。它在 1997 年推出了第一个“标准”版本:HTML 3.2。它的继任者 HTML 4.0 于 1998 年问世，很快在 1999 年被 HTML 4.01 取代。这是 HTML 的最新版本。W3C 已经宣布，它将不会创建 HTML 的进一步版本。建议使用 HTML 4.01 创建 HTML 文档。

然而，[网络超文本应用技术工作组](http://www.whatwg.org) (WHATWG)正在致力于被称为 [HTML5](http://whatwg.org/specs/web-apps/current-work/#html5) 的工作，希望它最终能被接受为 W3C 推荐标准。

##### 3.XHTML 呢？

在 HTML 4.01 成为最终推荐标准的几个月后，W3C 发布了 XHTML 1.0。这被视为“HTML 的下一个版本”，但这种看法并不完全正确。正如规范所说，XHTML 1.0 是“作为 XML 1.0 应用的 HTML 4 的重新表述”。换句话说，它是具有一组预定义的元素类型和属性(以及语义)的 XML，对应于 HTML 4.01 的元素类型和属性。它甚至有和 HTML 一样的三种风格。

许多设计人员和开发人员接受了 XHTML，因为它被视为前进的方向。很少有人理解 XHTML 和 HTML 之间的深刻差异，因为它们看起来如此相似。现实情况是，最常用的浏览器 Internet Explorer 不支持 XHTML 的任何形式。更现代的浏览器，如 Opera、Firefox 和 Safari 确实支持 XHTML，但它们的市场份额太低，以至于它们的支持对公众可访问的网站没有任何显著影响。

通过遵守 XHTML 1.0 规范中著名的[附录 C](https://www.w3.org/TR/xhtml1/#guidelines) 中提出的许多准则，可以将 XHTML 文档作为 HTML 提供。这种方法允许只支持 HTML 的浏览器能够“处理”文档，但实际上，它只不过是 HTML。以这种方式提供服务时，我们不能使用 XHTML 的任何特性，因为我们根本没有真正使用 XHTML——我们只是假装在使用。

##### 4.HTML 区分大小写吗？

不是，但是 XHTML 是。在 XHTML 中，所有的标签和属性都必须是小写的。传统上，HTML 元素名称和标记是用大写字母书写的，但是随着 XHTML 的出现，这种约定慢慢地让位于小写元素名称的 XHTML 标准。

##### 5.`DOCTYPE`声明是做什么的？

必须在文档中任何其他标记之前的`DOCTYPE`声明通常如下所示:

```
<!DOCTYPE html PUBLIC "-//W3C//DTD HTML 4.01//EN" 

 "https://www.w3.org/TR/html4/strict.dtd">
```

它指定了文档根元素(`HTML`)的元素类型、公共标识符和系统标识符。

公共标识符(`-//W3C//DTD HTML 4.01//EN`)显示谁发布了文档类型定义，或 DTD，(`W3C`)；DTD 的名称(`DTD HTML 4.01`)；以及编写 DTD 的语言(`EN`，用于英语)。请注意，它没有说任何关于网页本身的语言；这里指定的是 DTD 的语言。

系统标识符(`https://www.w3.org/TR/html4/strict.dtd`)是实际 DTD 的 URI(统一资源标识符，或“网址”)。

`DOCTYPE`声明告诉验证器(一个检查网页语法有效性的程序)应该根据哪个 DTD 来测试页面的兼容性。浏览器过去并不关心`DOCTYPE`声明，但是现代浏览器使用它来决定页面是“现代的”(并且可能期望 W3C HTML 文档中详细描述的行为)还是老派的(并且期望浏览器呈现带有旧浏览器所展示的所有错误和怪癖的页面)。文档的`DOCTYPE`会影响 Internet Explorer、Opera、Firefox(和其他基于 Mozilla 的浏览器)、Safari 和大多数其他现代 web 浏览器使用的渲染模式。一个完整的`DOCTYPE`声明——包括系统标识符——告诉浏览器这是一个现代文档。如果缺少系统标识符，或者根本没有 DOCTYPE 声明，浏览器会认为这是一个旧文档，并以“古怪模式”呈现它。

##### 6.什么是 DTD？

DTD 或文档类型定义指定了我们可以在 web 页面中使用的元素类型和属性。它还定义了我们如何一起使用这些元素的规则——这是我们的标记语言的规范。DTD 也可以声明我们可以使用的字符实体；稍后将详细介绍这些内容。

验证器将测试 web 页面是否符合 DOCTYPE 声明中指定的 DTD，要么通过系统标识符显式地测试，要么使用公共标识符隐式地测试。浏览器使用非验证解析器，并不真正读取 DTD。它们有关于各种元素类型的内置知识，通常还有一个硬编码的字符实体列表。

对于 HTML 4.01，这是最新和最伟大的版本，有三种不同的 dtd:严格，过渡和框架集。

##### 7.严格、过渡和框架集 dtd 有什么区别？

这些 dtd 之间的区别包括它们声明的元素类型和属性，以及它们如何允许或要求元素类型嵌套。

*   HTML 4.01 严格的 DTD 强调内容与表现和行为的分离。这是 W3C 为所有新文档推荐的 DTD。
*   [HTML 4.01 过渡 DTD](https://www.w3.org/TR/html4/loose.dtd) 旨在将老式(HTML 4 之前)文档转换为现代标记时过渡使用。它不是用来创建新文档的。它包含 11 种表示元素类型和过多的表示属性，这些属性在严格的 DTD 中是不赞成使用的。对于位于框架集中的页面来说，过渡 DTD 通常也是必需的，因为它声明了在另一个框架中打开链接所需的目标属性。
*   [HTML 4.01 框架集 DTD](https://www.w3.org/TR/html4/frameset.dtd) 用于框架集页面。W3C 不赞成使用框架。对于现代网站，使用服务器端脚本技术通常被认为是更好的解决方案。

##### 8.我应该使用哪种 DOCTYPE？

如果我们正在创建一个新的网页，W3C 推荐我们使用 HTML 4.01 Strict。

如果我们试图将旧的 HTML 2.0 或 HTML 3.2 文档转换到现代世界，我们可以使用 HTML 4.01 Transitional，直到我们成功地将所有表示问题转换到 CSS，并将所有行为问题转换到 JavaScript。

##### 9.为什么我应该验证我的标记？

为什么我们要在将文本发布到网上之前对其进行拼写检查？因为错误和差错会使读者困惑，并从重要信息中转移注意力。标记也是如此。无效标记会使浏览器、搜索引擎和其他用户代理感到困惑。结果可能是不正确的渲染，功能失调的页面，搜索引擎没有索引的页面，程序崩溃或者我们所知道的宇宙末日！

如果我们的页面没有按照我们预期的方式显示，我们应该在开始寻找其他问题(或者在 SitePoint 上寻求帮助)之前验证我们的标记。对于无效的标记，没有任何保证。

使用 W3C 的 [HTML 验证器来检查你的页面的兼容性。不要忘记包含一个 DOCTYPE 声明，这样验证器就知道应该根据哪些标准来检查您的文档。](http://validator.w3.org)

HTML Tidy 是一个免费的工具，可以帮助我们整理杂乱的标记，很好地格式化它，使它更容易阅读。

##### 10.为什么 HTML 允许草率编码？

它没有，但是它建议用户代理处理并尝试恢复标记错误。

有时据称 HTML 允许像`<b><i>foo</b></i>`这样不恰当的嵌套元素。那不是真的；验证器会抱怨不正确的嵌套标签，因为它们不构成有效的 HTML。然而，浏览器通常会猜测作者的意思，所以错误可能不会被发现。

有些人不喜欢 HTML 允许省略某些(但不是全部！)结束标签。这对浏览器来说不是问题，因为有效的标记总是可以被明确地解析。在早期，省略某些结束标签是很常见的，(例如`</p>`和`</li>`)。如今，对所有元素使用显式结束标记通常被认为是一种好的做法，除了那些被禁止使用的元素，比如`br`和`img`。

##### 11.为什么验证器抱怨我的`<embed>`标签？

从未成为任何 HTML 推荐的一部分。这是一个非标准的扩展，尽管大多数浏览器都支持，但它不是 HTML 的一部分。

在 20 世纪 90 年代末的“浏览器大战”中，像微软和网景这样的浏览器供应商通过在 HTML 中添加许多“酷”的特性来进行竞争，使网页样式化成为可能。这些附加功能的问题在于它们不是标准化的，而且在不同浏览器之间大多不兼容。

还有其他曾经相当普遍的元素(`marquee`，有人吗？)，但从未包含在 HTML 推荐中。如果可以避免的话，不要使用它们。

许多其他属性在 20 世纪 90 年代非常普遍，但是从来没有被包括在官方的 HTML 推荐中。`marginwidth`就是一个例子。

##### 12.字符编码(`charset`)是什么意思？

计算机只能处理数字。我们在屏幕上看到的字母或图像是以数字代码的形式通过互联网和计算机的各个部分传输的，计算机将其视为一组二进制数字(1 和 0)。

为了理解这些数字，我们需要定义一个能够传递某种信息的最小单位。当我们处理文本时，这个单位叫做字符。这是一个相当抽象的概念。称为“大写 A”的字符没有定义的视觉外观；更像是“A 的想法”。

接下来，我们需要建立一组我们将使用的抽象字符。这就是所谓的字符集。字符集是我们所拥有的抽象字符的集合。对于 HTML 来说，标准字符集是 ISO 10646，它实际上和 [Unicode](http://www.unicode.org) 是一回事。它是一个由数万个字符组成的集合，代表了地球上大多数的书面语言。

字符的视觉外观称为字形。一组特定的字形被称为字体。“大写字母 A”的字形在不同字体之间会有所不同，但这不会改变抽象字符的潜在含义。

现在，由于计算机只处理数字，我们必须有一种方法用数字代码来表示每个字符。字符集中的每个字符都有一个代码位置或代码点。码位是字符集中字符的数字表示(索引)。Unicode 中的码位通常以十六进制表示(例如，0x0041 表示“大写 A”)。

最后，编码——不幸的是，有时也称为“字符集”或“charset ”,尽管我们在这里坚持使用正确的术语“字符编码”——是一种表示这些码位的机制，通常使用八位字节，八位字节是由 8 个二进制数字组成的组(因此能够表示 0 到 255 之间的数字，包括 0 和 255)。

在计算机通信的早期，人们使用小型字符集，只包含特定语言的基本需求。最广为人知的一套可能是 ASCII (ISO 646)，它只包含 128 个字符，其中 33 个是不可打印的“控制代码”。ASCII 字符集有 128 个代码点，从 0 到 127 依次编号。编码是简单的一对一:“大写 A”的码位是 65 (0x41)，编码为 65(二进制的 1000001)。

ASCII 在英语世界之外用处不大，因为它只包含字母 A-Z、数字 0-9 和一些基本的标点符号。国际标准化组织(ISO)发布了一套名为 ISO 8859 的标准，用其他语言所需的字符扩充了 ASCII 字符集。在西方世界，最常见的是 ISO 8859-1，被称为拉丁-1。它包含书写大多数西欧语言所需的字符。ISO 8859 系列既是字符集又是字符编码。每个字符集包含 256 个字符，可以使用 8 个二进制数字进行编码。他们每个人都使用 ASCII 字符集作为子集，即前 128 个代码点是相同的。

但是即使 256 个字符也不足以编写一些语言。例如，中文需要成千上万的字符。设计了几种互不兼容的中文编码，但对于那些想要跨越语言和文化障碍交流信息的人来说，仍然存在很大的问题。

从这一点来看，很容易为每个字符创建一个使用 16 位甚至 32 位二进制数字的字符编码。然而，使用 32 位编码会导致大多数文档比它们需要的大四倍。

解决方案是一种叫做 UTF-8 的可变长度编码。它使用 8 到 48 位对每个码位进行编码，并且可以处理整个 Unicode(或 ISO 10646)字符集。前 128 个代码点编码为 8 位，与 ASCII 中的相应代码点相同。大多数西欧语言可以用单个八位字节进行编码，偶尔会用 16 位字符来表示带有变音符号的字母(例如,)。

作为 web 文档的作者，这对我们有什么影响？如果我们使用代码点在 ASCII 范围之外的字符，编码就变得非常重要。指定错误的编码，页面将很难阅读，甚至无法阅读。

那么，我们如何着手指定编码呢？正确的做法是在内容类型的 HTTP 头中发送此信息:

```
Content-Type: text/html; charset=utf-8
```

HTTP 头是由我们的 web 服务器发送的，所以我们必须调整服务器来改变编码信息。我们如何做到这一点将取决于我们使用的网络服务器。对于 Apache，可以在全局配置文件(httpd.conf)中指定，也可以在本地指定。htaccess 文件。但是如果我们使用共享主机，我们可能没有足够的权限来调整配置。在这种情况下，我们需要一个服务器端脚本语言来发送我们自己的 HTTP 头；这里有一个 PHP 的例子:

```
header('Content-Type: text/html; charset=utf-8');
```

我们还可以在 META 元素中使用 HTTP 等价物来指定编码:

```
<meta http-equiv="Content-Type" 

 content="text/html; charset=utf-8">
```

如果真正的 HTTP 头包含编码信息，这个`meta`元素将被忽略。尽管如此，它还是很有用的，因为如果访问者将我们的页面保存到硬盘上并在本地查看，就会用到它。在这种情况下，没有 web 服务器来发送 HTTP 头，所以将使用`meta`元素。

HTML 没有默认的编码，所以我们应该确保指定它。

Microsoft Windows 下的常见编码是 Windows-1252。它与 ISO 8859-1 非常相似，但也有不同之处。在 ISO 8859-1，0x80 和 0x9F 之间的代码点范围是为控制字符保留的。在 Windows-1252 中，该范围用于 ISO 编码中缺少的一些有用的字符(例如，印刷正确的引号)。我不建议在网络上使用这种编码，因为它是 Windows 特有的。但是，它是 Windows 下许多文本编辑器的默认编码。

##### 13.什么是 BOM？

BOM 或字节顺序标记用于一些使用 8 位以上编码码位的编码(如 UTF-8 和 UTF-16)。计算机处理器(CPU)可以采用两种不同的方案来存储大整数:“大端”和“小端”。BOM 由 16 位组成，写在文件的最开始，告诉浏览器正在使用哪个方案。

不幸的是，许多旧浏览器不能处理这些信息，所以它们将这些位显示为字符数据。如果您在页面顶部看到几个奇怪的字符，原因可能是浏览器没有处理 BOM(或错误指定的编码)。

解决这个问题的唯一方法是避免使用 BOM。可以将文档保存为 UTF-8 的编辑器通常允许我们选择是否包含 BOM。

##### 14.我应该声明什么编码？

非常非常简单:我们必须指定保存源文件时使用的编码！如果我们将文件保存为`ISO 8859-1`，就必须指定编码为 iso-8859-1；如果我们保存为 UTF-8，我们指定它为 utf-8。这里唯一的问题是，我们可能并不总是知道我们的编辑器使用什么编码来保存文件。不过，任何称职的编辑器都应该给我们指定编码的选项。

如果我们用英语写作，我们选择哪种编码并不重要，因为我们通常会使用在大多数编码中编码相同的字符。美国-ASCII 码，UTF ISO 8859-1-8…随你挑。对于我们这些用其他语言编写代码的人来说，选择变得更加重要。我的母语——瑞典语——比英语字母表多使用了三个字母。这些在 ISO 8859-1 都有，所以我可以在它和 UTF-8 编码之间选择。浏览器对 UTF-16 的支持很差，所以应该避免在公共网站上使用。

我的建议是尽可能使用 UTF-8 编码，没有 BOM。它本身可以表示 Unicode 字符集中的任何字符。

避免在公共网页上使用 Windows-1252，因为它是一种特定于 Windows 的编码。用 ISO 8859-1 代替(或者 ISO 8859-15，如果你需要欧元符号的话)。

##### 15.如何在编码范围之外插入字符？

如果我们使用 ISO 8859-1 编码，并希望在内容中包含欧元符号，该怎么办？字符集中没有欧元符号，因此无法对其进行编码，尽管它出现在 ISO 10646 并可以在网页上使用。

我们有两种选择:命名实体或数字引用。

欧元符号的命名实体是`&euro;`。实体以&符号(`&`)开始，以分号(；).在某些情况下，我们可以省略分号，但总是把它放进去绝对是个好习惯。实体名称区分大小写。

数字引用可以是十进制(`&#8212;`)或十六进制(`&#x2014;`)，但通常使用十进制表示法更安全，因为一些旧浏览器不能处理十六进制版本。请注意，该数值引用了 ISO 10646 的代码点；这与我们为文档指定的编码无关。

引用(十进制)总是有效的。命名实体可能会在较旧的浏览器中引起问题，因为有些浏览器只支持 HTML 实体的子集。

##### 16.为什么我需要写`&amp;`而不是只写`&`？

某些字符在 HTML 中有特殊的含义:`<`(小于)、`>`(大于)、`&`(与号)、`"`(引号)、`'`(撇号)。在某些情况下，当我们想在普通文本中使用这些字符时，我们需要用 HTML 实体来替换它们。

前四个字符的实体如下:

*   `&lt;`(小于)
*   `&gt;`(大于)
*   `&amp;`(与号)
*   `&quot;`(引号)

XML 为撇号(`&apos;`)定义了一个实体，但是 HTML 不包括这个实体。撇号只能使用数字引用(`&#39;`)进行转义。

因为,“与”符号用于这些实体，所以几乎总是要对它进行转义，包括在属性值内部使用它的情况，比如链接的 href 属性。不幸的是，在 URIs,“与”是一个非常常见的参数分隔符，这意味着在 URIs 遇到“与”是很常见的。

在 HTML 中，大多数时候，未转义的&符号不会破坏任何东西(尽管 XHTML 是另一回事)。浏览器中的错误处理例程从错误中恢复，并且一切正常。但是，如果我们碰巧有一个查询参数的名称与 HTML 中预定义的命名实体之一相匹配…

##### 17.标题元素应该如何使用？

HTML 标题元素类型有`h1`、`h2`、`h3`、`h4`、`h5`和`h6`。数字表示标题的结构层次，这意味着我们应该像在学校里学的那些大纲一样对待标题(毕业后很快就忘记了)。

页面上的顶级标题必须是一个`h1`。它应该描述页面是关于什么的。大多数页面会有一个`h1`标题，但是处理几个不同主题的非常复杂的文档可能需要不止一个。

标题将标记下一个结构级别。其下的任何子级别都将是`h3`，依此类推。当我们在层次结构中向下移动时，我们永远不能跳过一个标题级别。一个`h4`不要跟着一个`h2`；中间应该有一个`h3`。(验证器不会抱怨这一点，但这是很好的实践。)

用`Hn`元素类型标记标题很重要。屏幕阅读器等辅助技术可以利用适当的标题层次来呈现文档的轮廓。如果我们使用`<font size="7">...</font>`，他们不能。

##### 18.什么是块级和内联元素？

HTML 使用两种主要的元素类型:块级元素和行内元素。两者的区别主要是语义和语法上的。

块级元素通常是其他元素的“容器”。块级元素的例子有`div`、`p`、`form`和`table`。一些块级元素(如`p`)只能包含文本和行内元素。其他的(例如`form`)只能包含块级元素(在严格的 DTD 中)。还有一些，像`div`，可以包含文本、行内元素和块级元素。默认情况下，块级元素在呈现时前后都有一个隐式换行符；换句话说，我们不能只使用严格的 HTML 就让两个块级元素并排存在。(这样做需要 CSS。)

内联元素是可以“内联”存在于文本中的元素。例子包括`a`、`em`、`q`和`span`。内联元素只能包含文本和其他内联元素。内联元素不能包含块级元素，有一个例外:`object`(它被称为被替换的内联元素，与`img`相同)。呈现时，行内元素前后没有任何隐含的换行符。

在某些情况下，子元素类型会受到额外的限制。例如，锚链接(`a`)可以包含文本和行内元素，但不能包含其他`a`元素；您不能嵌套链接。

严格 dtd 和过渡 dtd 之间的规则有些不同。在严格的 DTD 中，一些块级元素，包括`body`、`blockquote`、`form`，只能有块级的子元素。在过渡 DTD 中，它们还可以包含文本和内联元素作为直接子元素。

##### 19.我可以用 CSS 做一个块级的内联元素吗？

不。这是一个常见的误解。初学者有时认为，通过将 display: block 声明应用于一个`a`元素，他们将能够在链接内部放置一个块级的`h1`。事实并非如此。

HTML 有块级和行内元素。CSS 有块和行内框(还有一些其他的)。这些是非常不同的事情。HTML 中的区别与语义和语法有关，而 CSS 中的区别与呈现和表示有关。默认情况下，块级元素生成块框，行内元素生成行内框(这是一个非常简单的解释，但通常是正确的)。display 属性可以改变生成的框的类型，但是 CSS 不能改变 HTML 的语法或句法规则。

##### 20.为什么外部 CSS 和 JavaScript 文件是个好主意？

从维护的角度来看，内容、表现和行为的完全分离是值得努力的。然后，如果我们想改变我们网站的颜色，我们可以简单地编辑一个样式表，而不是更新可能成千上万的 HTML 文档。如果我们使用样式属性并编写内联 CSS，那么在重新设计我们的站点时，我们将不得不编辑所有这些 HTML 文档，而不是简单地编辑单个样式表文件。

还有另一个问题:CSS 和 JavaScript 通常都包含在 HTML 中有特殊含义的字符。如果 CSS 代码或 JavaScript 代码嵌入到 HTML 文档中，这些字符需要进行转义。如果我们已经嵌入了 JavaScript，并且使用了在 SGML 注释(`<!--...-->`)中“隐藏”脚本代码的古老做法，我们就不能使用减量操作符(`--`)，因为双连字符将终止注释。

##### 21.应该用`p`还是`br`？

元素标记了一段文本。段落是一个或多个表达一个想法的句子。

换行符(`br`)主要是一种表示工具，应该由 CSS 而不是 HTML 来处理。然而，在一些情况下，换行符可以说是有语义意义的，例如在诗歌、歌词、邮政地址和计算机代码样本中。这些可以构成对`br`的合法使用，但是使用`br`来分隔段落绝对是对`br`元素的滥用。

另一方面，`p`有非常明确的语义:它表示一个段落。有时，web 作者倾向于将`p`视为通用的块级容器元素，但这是不正确的。看到一个`label`和一个`input`字段包装在一个`form`中的一个`p`中并不罕见，但是我认为这在语义上是错误的。标签和输入字段不构成“段落”。

##### 22.“语义”是什么意思？

<q>**se-man-TIC【si-' man-tik】**
形容词的，与单词或其他符号的不同含义有关的，或由此产生的。</q>

*(定义来自[dictionary.com](http://www.dictionary.com))*

当我们谈到“语义标记”时，我们指的是正确使用元素类型——基于它们的含义——来标记内容。相反的是“表示标记”或“标签汤”，作者选择元素类型是因为它们的默认呈现，而不是它们的语义。

举个例子:这是一种语义上正确的标记网页顶层标题的方法:

```
<h1>Heading Text</h1>
```

这是一种不自然的方式:

```
<br><font size="7"><b>Heading Text</b></font><br>
```

HTML 的语义丰富性相当有限。HTML 最初是物理学家用来交换科学文档的，这在可用的元素类型集中表现得非常清楚。如果 HTML 是由会计师或图书馆员发明的，它可能会有一组非常不同的元素类型。

HTML 也有两种语义中立的元素类型:块级`div`和内联级`span`。这两者都不暗示其内容有任何特定的语义；`div`只是一个“文档的划分”，而`span`是一个“字符的跨度”。另一方面，我们有语义定义明确的元素类型:`p`(文本段落)、`table`(表格数据)、`ul`(无序列表)，等等。

HTML 的目的是标记文档的语义，并在某种程度上显示其内容的结构。HTML 与该文档在浏览器中的外观没有任何关系(尽管浏览器对每种元素类型都有默认的样式)。

##### 23.是不是应该用`strong`和`em`代替`b`和`i`？

除非你真的想强调什么。这些符号不能互换。

在过去糟糕的日子里，作者会用`b`和`i`来强调单词。

在同样糟糕的现代，作者使用`strong`和`em`来使文本变成粗体或斜体。

`em`表示语义强调。当大声朗读时(更大声，更慢)，应用它的内容应该有某种强调。`strong`表示更强的强调，但是现在经常被认为是多余的(你可以嵌套`em`元素来表示增强的强调)。一些专家建议`strong`仅用于某些应该被明确指出的页面元素(如“当前页面”指示器)，而不要在正文中标记单词或短语。

`b`和`i`没有语义；它们只表示粗体或斜体。它们有助于遵守在 HTML 中没有语义正确的元素类型的印刷约定。例如，船名传统上用斜体书写，但是在 HTML 中没有`ship`元素类型。因此我们可以使用`<i>Titanic</i>`。

##### 24.为什么布局表格被认为是有害的？

*   将非表格信息标记为`table`在语义上是错误的。
*   它们可能会导致可访问性或可用性问题(尤其是对于一些辅助技术)，尤其是当嵌套层次很深时。
*   它们将表示问题与内容混为一谈，使得很难或不可能实现替代样式和输出设备独立性。
*   它们用大量不必要的 HTML 标记来膨胀文档标记，这对低带宽用户(使用拨号连接或移动设备的用户)以及 web 服务器的负载和带宽都是有害的。

##### 25.我应该使用`div` s 而不是布局表格吗？

不，我们应该尽可能使用语义正确的元素类型，只有在没有其他选择的情况下才恢复到`div` s。

滥用`div` s 并不比滥用`table` s 好。我们可以在几乎任何元素类型上设置`id`和`class`属性。我们可以将 CSS 规则分配给几乎任何元素类型，而不仅仅是 T4

##### 26.表格是否被否决？

一点也不。`table`是用于标记表格数据的合适的、语义上正确的元素类型:在两个或多个维度上有关系的信息。不推荐使用表格，但是布局表格是一个问题。

##### 27.`address`元素类型的正确用法是什么？

`address`用于标记页面(或页面的一部分)的联系信息。这可以是邮政地址、电子邮件地址、电话号码或几乎任何联系方式。`address`是块级元素，只能包含文本和行内元素。在大多数浏览器中，默认呈现是斜体，但是使用 CSS 可以很容易地改变这一点。

一个常见的误解是,`address`只用于标记邮政地址，但事实并非如此。

##### 28.`dfn`元素类型的正确用法是什么？

`dfn`用来标记一个术语的“定义实例”。这是一种印刷惯例，在科学文献中尤其常见，当一个新术语第一次出现在文本中时，用斜体表示，读者可能不熟悉这个术语。因此，`dfn`的默认呈现是斜体。

一个常见的误解是`dfn`意味着“定义”，许多作者使用它的方式与使用`abbr`或`acronym`相同(通过使用 title 属性来提供术语的解释)。某个术语在一个文档中只应该用`dfn`标记一次(第一次使用和解释的地方)。

##### 29.`var`元素类型的正确用法是什么？

`var`用于标记一些文本的可变或可替换部分。将这些变量用斜体表示是一种印刷惯例，在现实生活中，这些变量将被实际数据所取代。例如，在电话系统手册中，将来电转接到另一个分机的说明可能是这样的:

```
<kbd>* 21 * <var>extension</var> #</kbd>
```

这里使用了一个`var`元素来标记“extension”(默认为斜体)。有人试图对电话系统进行编程，将他的来电转接到分机 942，他会键入“`*21*942#`”。因此,`var`元素表示您不应该实际键入“e-x-t-e-n-s-i-o-n ”,而是输入实际的分机号码。“扩展”这个词是一个变量。

一个常见的误解是`var`应该用于标记编程代码样本中的变量。

##### 30.我应该在`q`元素中使用引号吗？

不，规范明确说明了用户代理的责任是给内联报价添加引号。不幸的是，一些较旧的浏览器(如 Internet Explorer 6)不符合该规范，不会添加引号。一种选择是用 JavaScript 插入引号，并使用 CSS 的一些特殊样式为禁用 JavaScript 的 IE 用户插入引号。有人提出了一些只支持 CSS 的解决方案，但它们在 Lynx 等非 CSS 浏览器中会失败。

##### 31.`abbr`和`acronym`有什么区别？

似乎没有人真正知道这个问题的答案！甚至 HTML 规范在这一点上也自相矛盾。

是“浏览器大战”期间网景对 HTML 的扩展。`acronym`是微软的扩展。两者或多或少意味着同一件事。这两种元素类型都包含在 HTML 规范中，具有不同的语义。问题是，似乎没有人能够解释这些语义是什么。

让我们看几个字典定义，然后:

<q>**ab-bre-VI-A-tion【uh-Bree-vee-' ey-shuhn】**
一个词或短语的缩写形式，用来代表整体。
**AC-ro-nym【AK-ruh-nim】**
由一个固定短语或一系列单词中的首字母或几组字母组成的单词。</q>

缩略词的定义是它是一个单词，也就是说，它可以发音。因此，“NATO”将是一个缩写词，由短语“北大西洋公约组织”的首字母组成。然而，根据字典的定义，“FBI”不会是一个缩写词，因为它不是作为一个单词发音，而是拼写出来的(eff bee eye)。这就是问题的开始。“FBI”在技术上被认为是一个首字母缩写词，关于这个词，字典有如下解释:

<q>**in-I-TiAl-ism【I-' nish-uh-Liz-uhm】**
n . 1 .由一组词的首字母组成的名字或术语，作为一个单独的词发音。
2。代表一个名字、组织等的一组首字母，每个字母单独发音。</q>

第一个定义几乎与缩略词相同，但第二个定义更宽松。然而，在 HTML 中没有`initialism`元素类型，并且这种混淆由于这样一个事实而加剧，即在正常的美国说法中“acronym”被用作“initialism”的同义词。

HTML 规范提供了以下定义:

<q>`abbr`:表示缩写形式(例如，WWW，HTTP，URI，马萨诸塞州。等。).
`acronym`:表示缩写(如 WAC、radar 等。).</q>

到目前为止，看起来规范符合字典的定义，这意味着“FBI”应该用`abbr`标记，因为它不能作为一个单词发音。然而，再往下几段，说明书说，

西方语言大量使用缩写词，如“GmbH”、“NATO”和“F.B.I .”，以及缩写词“m .”、“Inc .”、“et al .”、“等”

你困惑了吗？我是。那么安全的做法应该是始终使用`abbr`，因为所有的缩写都是缩写，但反之则不然。然而，这种方法有一个小问题。当 W3C 决定使用`abbr`而不是`acronym`作为缩写和首字母缩写时，微软非常恼火，以至于他们拒绝支持`abbr`！(不过，他们已经开始在 Internet Explorer 7 中支持`abbr`。)

那么，一个可怜的网络作者该怎么办呢？我们为什么要这么麻烦呢？让一个元素附加一个标题属性可能很好，但是我们可以使用 SPAN 来实现。据称，这个想法是标记缩写和首字母缩略词对辅助技术有益；尤其是屏幕阅读器。但是屏幕阅读器往往会忽略`abbr`和`acronym`，因为没有人知道如何正确使用它们，而且微软也不支持`abbr`。这是一个第二十二条军规。

这个常见问题的答案是:不知道！就我个人而言，我使用`abbr`来表示像“Inc .”这样明显的缩写和像“FBI”这样的首字母缩写，我使用`acronym`来表示像“GIF”这样可以发音为单词的东西。但是由于规范的模糊性，我不能指责任何人将“FBI”标记为缩写(尽管“Inc .”肯定不是缩写)。而“SQL”呢，有人拼出，有人念成“续集”？(我会用`abbr`。)

##### 32.为什么`<feature X>`被弃用？

初学者最常问的“特性”是链接的目标属性。这个特性在 HTML 4.01 Strict 中被否决了，但是在 HTML 4.01 Transitional 中仍然有效。Transitional 中允许的许多其他元素类型和属性从 Strict 中移除。

放弃这些项目的原因是 W3C 希望促进内容(HTML)、表示(CSS)和行为(JavaScript)之间的分离。在视口内居中制作元素是一个表示问题；因此它应该由 CSS 而不是一个`center`元素来处理。打开新的浏览器窗口是一个行为问题；因此它应该由 JavaScript 而不是一个`target`属性来处理。

通常，被否决的功能是在 20 世纪 90 年代末的浏览器战争时期出现的，当时浏览器供应商通过向 HTML 添加各种扩展来进行竞争，以使其成为某种页面布局语言。这些特性包含在 HTML 3.2 中是为了给混乱带来某种秩序，但这并不是 HTML 的初衷。当 HTML 4 发布时，作者试图通过贬低 HTML 3.2 中他们认为“有害”的部分来“回收网络”，至少在严格的 DTD 中是这样。

换句话说，东西被弃用是有原因的。除非万不得已，否则不要使用这些功能。

##### 33.每个图片都必须有一个 alt 属性吗？

是的，`img`元素类型需要`alt`属性。为什么？不是所有的用户都能看到图像，也不是所有的用户代理都能理解或显示图像。例如:

*   盲人或视力非常低的人看不到图像。屏幕阅读器无法描述图像。
*   网速较慢的用户(拨号上网或手机上网)有时会禁用图片以加快上网速度。
*   像 Lynx 这样的文本浏览器不支持图像。
*   搜索引擎机器人不能理解图像。

因此，我们必须使用`alt`属性为每个图像提供一个文本等价物。此文本等效不应描述图像；它应该传达相同的信息。编写好的文本对等物并不容易，需要大量的练习。请记住，显示的是等效文本，而不是图像。

那么对于给定的图像，什么是好的文本等价物呢？这取决于使用图像的上下文！并不是每张图片都有一个“完美”的文本。让我们看一个例子:假设我们有一幅吃草的牛的图像。这头特殊的牛恰好是一头阿伯丁安格斯牛。然后让我们考虑这个图像的几个用例。

*   在第一个例子中，这个图像被用作一篇关于苏格兰肉牛养殖的文章的通用插图。真正的奶牛与文章没有密切关系；它只是一个插图，一个吸引读者眼球的装饰性设计元素，缓解文字的单调。在这种情况下，图像没有传达任何相关信息。因此它应该有一个空的等价文本:`alt=""`。
*   在第二种情况下，图像被用在一个关于农场动物的儿童网站上。该网页展示了各种动物的图片:马、羊、猪、牛等。每张图片旁边都有一段文字，介绍每个物种的一些事实。在这种情况下，`alt="Cow:"`可能是合适的。它是一只阿伯丁安格斯并不重要；这张图片代表了牛的四足动物。
*   在第三种情况下，图像被用在一个关于不同品种的牛的网站上。在这里，它是用来说明什么样的阿伯丁安格斯看起来像，以及它是如何不同于其他品种。该页面包含许多图片，每张图片都有一个标识该品种的标题，但没有其他文本信息。在这种情况下，对应的文字应该描述阿伯丁安格斯牛特有的属性和特征:强健的体格、厚实的胸部、相对较短的腿、头后像水牛一样的驼峰等。
*   在第四种情况下，图像用于摄影师的作品集页面。这是几幅图像中的一幅，主题非常不同。这是少数情况之一，其中的`alt`属性实际上可能包括图像本身的描述，例如，“一只黑色的阿伯丁安格斯在阳光下吃草，背景是本尼维斯。”

正如我们所看到的，适当的文本对等依赖于上下文。有时(实际上经常是)它应该是空的，因为图像不传达任何在伴随文本中不存在的信息。有些人声称这种图像应该是通过 CSS 指定的背景图像，但在许多情况下这是不切实际的，图像实际上是内容的一部分——即使它没有向那些看不到它的人传达任何有用的信息。

对于包含文本的图像，文本等价物当然应该复制图像中的文本。对于饼状图之类的东西，文本对等物应该传达关于百分比的信息——与图像传达的信息相同。

替代文本不应太长。有些浏览器不自动换行，也不能以任何方式格式化。如果我们需要一个更长的文本等价物，我们应该把它放在别的地方，并通过`longdesc`属性链接到它。

当用户将鼠标指针悬停在图像上时，Internet Explorer 和旧的 Netscape 浏览器会在工具提示中显示`alt`属性。这构成了对文本等效数据的不正确使用。我们应该为“工具提示”信息使用`title`属性。为了隐藏工具提示中出现的替代文本，我们可以使用一个空标题:`title=""`。

##### 34.`class`和`id`有什么区别？

一个`id`唯一地标识一个 HTML 文档中的特定元素。它就像一个社会安全号，为该元素提供了一个唯一的句柄。正如两个人不能有相同的社会保险号一样，文档中的两个元素也不能有相同的`id`。`id` s 在页面中必须是唯一的。

一个类表示一个元素有一些(可能)与其他元素共有的特征。一个元素可以属于多个类。一个类比可以是职业:一个人可以既是木匠又是护士，有许多木匠和护士。(不过，他们都有独特的社会安全号码。)

和类主要用于 CSS 和/或 JavaScript。在 CSS 中，一个`id`比一个类具有更高的特异性，这使得为特定元素指定特殊规则变得容易。使用 JavaScript，我们可以使用元素的`id` ( `document.getElementById()`)来查找元素。

我们给每页最多出现一次的页面元素分配`id`,比如导航菜单、页脚、侧边栏等。我们还可以将`id`分配给在整个网站中只出现一次的特定元素，比如一个特定的图像，如果我们想要为它制定特定的 CSS 规则，或者用 JavaScript 来操作它。

我们将类分配给具有一些共同特征的元素，通常通过 CSS 规则显示属性。

s 和类名应该尽可能“语义化”。他们应该描述某样东西是什么，而不是它看起来像什么。由此可见，`id="menu"`比`id="left"`强多了；尤其是如果我们重新设计并把菜单移到右边。

s 和类名是区分大小写的，即使在 HTML 中也是如此。但是，我们不应该依赖于区分大小写(也就是说，我们不应该使用只有大小写不同的名字)。

##### 35.为什么`id="123"`不起作用？

`id`、`name`和`class`属性的值必须以字母开头(a-z 或 A-Z)。

##### 36.为什么`<a href=My Cool Page.html>`不起作用？

这种情况下有两个原因。

包含除字母、数字和少数其他字符以外的字符的属性值必须用引号括起来(双引号或单引号)。例如，任何需要包含空格的属性值都必须用引号括起来。最简单和最安全的解决方案是始终引用属性值。要在引用的值中包含引号，我们有两种选择。我们可以使用“其他”引号将值括起来(`alt='My "new" car'`、`alt="Jane's car"`)，或者使用实体或引用(`alt="My &quot;new&quot; car"`、`alt='Jane&#39;s car'`)。(注意,`&apos;`实体不能用于 HTML。)

第二个原因是 URI 有空间。这些需要编码，如下所示:

```
<a href="My%20Cool%20Page.html">
```

“`%20`”表示“代码点为 0x20 的字符”。0x20 是空格字符的码位。这仅适用于 URIs，不适用于一般的属性值。

##### 37.如何将一个 HTML 页面包含在另一个 HTML 页面中？

对于严格的 DTD，只有一个有效选项:对象元素类型:

```
<object type="text/html" href="http://example.com/foo.html">  

  Alternate content here for browsers that don't support OBJECT.  

</object>
```

不幸的是，对`object`的支持在 Internet Explorer 中几乎不存在。

对于过渡 DTD，我们也可以使用`iframe`元素类型:

```
<iframe src="http://example.com/foo.html">  

  Alternate content here for browsers that don't support IFRAME.  

</iframe>
```

更好的方法是在服务器端处理包含。使用服务器端包含(SSI)是将文件包含到另一个文件中的最简单方法，只要它们来自同一个域:

```
<!--#include virtual="/foo.shtml"-->
```

注意，这种技术不能用于将一个完整的 HTML 文档包含在另一个文档中；它只能用于 HTML 的片段。

其他服务器端技术允许我们执行更高级的任务。当然，您的 web 服务器必须支持这些技术。通常，免费托管的共享服务器不提供任何这样的技术——甚至 SSI 也不提供。
