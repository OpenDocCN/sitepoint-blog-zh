# WCAG2 隐藏的金块:信息关系

> 原文：<https://www.sitepoint.com/the-hidden-nuggets-of-wcag2-information-relationships/>

信息和关系成功标准是一个非常复杂的要求。与其他成功标准不同，仅遵循一两个列出的技术是不够的，重要的是遵循所有相关的技术。这意味着这个成功标准涵盖了广泛的需求——事实上，它涵盖了许多 WCAG1 检查点，如表头、标题、编码列表等。

## 标题

标题是人们浏览内容的一种有价值的方式——无论是视觉上的还是通过屏幕阅读器的音频反馈。当处理一百页的 PDF 时，用户可以通过标题导航是绝对必要的。屏幕阅读器可以拉出标题列表，这是屏幕阅读器用户确定特定页面内容的标准方式，而不必通读整个文本。Adobe Reader 缺乏对这一要求的支持是澳大利亚政府信息管理办公室决定 pdf 不可访问的原因之一——因为两种最常见的屏幕阅读器(JAWS 和 WindowEyes)不能一致地识别标题(有关更多信息，请参见我关于[调查](http://www.gianwild.com/2010/12/31/between-a-rock-and-a-pdf/)结果的博文)。

关于标题有一些争议。一些易访问性专家说每页应该只有一个 H1，其他人说多个 h1 是可以接受的。在我看来，这真的不重要，只要你在整个网站上保持一致。然而，我确实认为每一页都应该以 H1 开头，尽管有些人认为在导航中使用隐藏标题(例如，用来指示子部分的开始)会自动排除在页面开头使用 H1。

导航中的隐藏标题是一个非常重要的问题，我将在以后关于跳转链接的文章中讨论这个问题。

## 字段标签

信息和关系下的另一个重要技术是对字段标签的要求。当涉及到字段和字段标签时，有许多子句，我一直看到这种技术被错误地实现。我一次又一次地看到没有字段标签的字段。人们经常提到 H65 技术，它允许使用 TITLE 元素，但是只有当 label 元素不能使用时才允许。信息和关系确实允许隐藏标签，但是还有另一个成功标准:标签或说明，它要求标签是可见的。这个规则有一些例外，我将在以后关于 WCAG2 和 forms 的文章中讨论。

## 字段集

使用 FIELDSET 元素的要求是 WCAG1 级 AA 检查点。在 WCAG2 中，这是一个 A 级要求。辅助技术在解释字段集元素方面有了很大的改进，现在可以一致地依靠它来提供某些信息，而以前标题是分组字段元素的首选方法。

FIELDSET 元素应始终用于单选按钮和复选框，其中每一项仅在涉及总体问题时才有意义，例如:

你最喜欢什么颜色？

*   蓝色
*   格林（姓氏）；绿色的
*   紫色

每一个特定的单选按钮字段标签(“绿色”、“蓝色”或“紫色”)都没有意义，除非在阅读时提出一个最重要的问题:“你最喜欢的颜色是什么？”在这种情况下，三个单选按钮应该包含在一个 FIELDSET 元素中，并且该元素应该有一个图例“你最喜欢的颜色是什么？”请记住，如果您使用字段集，您必须始终有一个适当的图例。

## 表格的结构化标记

当我在 1998 年第一次进入可访问性行业时，周围还没有可以读取表头的屏幕阅读器。这就是要求在 WCAG1 中以线性方式读取内容时内容有意义的原因。自那以后，屏幕阅读器得到了改进，因此现在必须对数据表进行编码。所有数据表都必须使用 TH 来标识表头，使用 TD 来标识数据单元格。然后，您可以自己决定是使用 SCOPE 还是 TH ID / TD 头。对于一个简单的数据表，通常只使用 TH 和 TD 元素就足够了。

然而，WCAG2 超越了表头。WCAG2 还要求对数据表使用 SUMMARY 和 CAPTION 元素。这两个元素经常互换使用，但它们实际上意味着非常不同的东西。标题元素应该只有几个词，本质上是表格的标题，例如“按行业划分的可访问网站的数量”。SUMMARY 元素提供了内容的高度概括，可以是几行，例如“政府网站是最常访问的，54%的测试网站是可访问的。小企业网站是最不容易访问的，只有 3%的测试网站是可访问的。这些数字是有根据的估计，但我要说，根据我的经验，这些百分比是正确的。

## 链接列表

在 W3C 工作组时，我们确实努力复制人们在现实世界中使用的可访问性技术。我们知道 WCAG1 已经过期，许多易访问性专家正在提出他们自己的需求，当我们认为这些技术有价值时，我们在 WCAG2 中包含了这些技术。使用 OL、UL 或 DL 对一组链接(如导航)进行编码就是这些技术之一。

## q 和批量报价

隐藏在技术 G115:使用语义元素来标记结构，和技术 H49:使用语义标记来标记强调的或特殊的文本，是使用 Q 和 BLOCKQUOTE 元素来编码语音的要求。在 WCAG1 中，这是 AA 级要求。技术 G115 和 H49 是 A 级技术。我发现很多客户都在为这个需求而挣扎，因为他们的网站上经常有很多语音，找到并编码这些语音很费时间。另一个问题是浏览器如何呈现 Q 元素——有时浏览器会添加引号，有时不会，所以这确实需要一些花哨的 CSS 步法。

当然，WCAG1 不使用 BLOCKQUOTE 缩进文本的要求仍然有效。令人惊讶的是，我仍然经常遇到这个问题。

## 还有什么？

有了信息和关系的成功标准，你需要小心不要走极端。例如，这个成功标准的一个非常具体的失败是为布局表格使用结构化标记(TH、标题、摘要等)。

另一个需要注意的失败是使用 JavaScript 来模拟链接。允许使用 ONCLICK(尽管我将在以后撰写关于 JavaScript 和可访问性的文章)，但只能与 HREF 和 AREA 元素结合使用。

这些是我对信息和关系成功标准的解释，但我很高兴听到你自己的解释！
