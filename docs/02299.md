## 你的问题

### 问:在确定什么样的框架、库、工具包(等等)时，你推荐的第一步是什么？)应该用于 CSS 布局和样式？

Miriam Suzanne: 在工具包的速度/熟悉度和附加的依赖包袱之间总是有一个权衡。如果你正在做一个个人项目，那就熟悉一些。犒劳自己。否则，从最小和最抽象的工具包开始，然后逐步深入。 [Bootstrap](https://getbootstrap.com) 是一个很大的承诺。[服饰秤](http://oddbird.net/accoutrement/)不是。我有几个工具包，在任何大于一页的项目开始时，我都会加载这些工具包——然后从那里开始我的工作。如果你知道你会需要它:马上添加它。如果你认为你可能不需要它，等到你需要的时候。

这总是一个平衡的行为，没有办法每次都“做对”。做好以后改变主意的准备。重构不是一个错误——它是迭代过程的全部。

### 问:为什么是苏西？

**Miriam Suzanne:** 在 2009 年的时候，在网格上创建流畅的布局要困难得多。有更多的浏览器不一致，我们只有浮动工作，我们没有`border-box`大小。我使用的技术是从娜塔莉·唐恩的 [CSS 系统](http://blog.natbat.net/post/46614243624/css-systems)演讲中学来的。这是一个很棒的演讲，如果你想知道我所有的灵感来自哪里。

她的系统很棒，但是需要大量的数学运算。数学并不复杂，但非常重复——如果没有 Sass，结果看起来就像无意义的数字。我不喜欢这样，所以我在我们的一次工作务虚会上抱怨，有人向我展示了无礼——这在当时刚刚开始获得一些小的流行。那天晚上我给苏西写了信。它很小，只是帮助我做重复的数学。我还内置了几个助手来处理浏览器 bug。这就是我想要的。

当时的其他开源选项是带有预定网格的巨大 CSS 库，我对此不感兴趣。据我所知，Susy 是第一个以原生方式将网格系统移植到 Sass 的人。没有办法用纯 CSS 来构建 Susy。

老实说，我已经不怎么用 Susy 了。它不能解决我经常面临的一个问题。但我希望它对处理浮动网格和旧浏览器的人仍然有用。

### 问:使用 Susy 与边框尺寸和 flexbox 有什么相似和不同之处？

Miriam Suzanne: 这三样你都可以一起用！这些实际上解决了稍微不同的问题:

*   Flexbox 是我所说的“布局技术”(类似于浮动、内嵌块、表格等)。这是浏览器理解在页面上创建区域并管理它们如何流动的一种方式。所有这些技术都有优点和缺点。但是 flexbox 是唯一一个被设计来处理布局的。目前，这是我最喜欢的，尽管有问题。很快，我们也将获得 CSS 中的网格布局，所以这将是另一种选择。我建议学习它们是如何工作的，这样你就可以在最有意义的地方使用它们。一个工具永远不够！

*   [盒子大小](https://developer.mozilla.org/en-US/docs/Web/CSS/box-sizing)帮助你以新的方式描述单个元素的大小。是否要定义内容，并让填充展开？如果是这样，使用内容框。如果你想描述外部尺寸，并让内容调整，使用边框。拥有这种灵活性是巨大的，与七年前相比是一次巨大的升级。同样，了解这两个选项是如何工作的，这样你可以更加灵活。

*   Susy 只是一个计算器。如果你发现自己做了大量的数学运算来得到你想要的网格，那么使用 Susy。或者，在许多情况下，我认为说“也许你的网格实际上不需要那么复杂”会更好。

### 问:你能分享一下 Susy 在幕后做的一些计算吗？

**Miriam Suzanne:** 主要的 Susy 计算是这样的:
`$target-width / $context-width = $multiplier`
最简单的，这等同于使用`percentage(1/5)`从 5 列中得到 1 列。当你加上排水沟时，事情就变得更复杂了。总数实际上不是 5，而是更大的数字。

如果你对基础感兴趣，可以看看 Susy 3ish 分支,在那里我们将 Susy 精简到最基本的东西。

### 问:有没有一个工具包是你现在主要从事的？

**Miriam Suzanne:** 我已经将我的关注点向后移了一步，减少了对一致的*输出*(我们想要生成的风格指南)的关注，更多地关注一致的*输入*。我已经慢慢建立了一套 Sass 助手，像 Susy 一样，完全不固执己见。我们的[装备](http://oddbird.net/accoutrement/)工具是如此抽象，它们大部分根本没有输出。但是它们有助于我们定义代码中的模式，在一个地方跟踪这些模式，并根据这些信息自动生成风格指南。

### 问:在开始制作工具包时，你首先应该考虑什么？

米丽娅姆·苏珊娜:我喜欢这件。

> 我喜欢别人对为自己构建工具包感兴趣。我认为构建工具包是学习代码的好方法。不管有多少人解决了这个问题，自己再次解决它是一个有用的练习。

我从这里开始:

*   我的代码库中有什么没有意义？什么东西对于没有写它的人来说是不可读的？
*   有哪些问题是我要反复解决的？
*   我在哪里重复我自己，在代码中或者在我的过程中？
*   我在哪里感到沮丧或无聊？

工具包应该总是用来使你的代码更可读，更有意义。使用 Susy 的一个重要原因是因为`12.456%`没有任何意义，但是`span(1 of 6)`有。如果你有没有意义的代码，找出是什么赋予它意义。`clearfix`是另一个很好的例子。如果不把它藏在 mixin 里面，你的`clearfix`只是一堆奇怪的无意义的黑客。当您调用一个名副其实的 mixin 时，您就清楚地了解了该代码存在的原因。

我也认为在你的工具设计中迭代是很重要的。你的第一个解决方案可以是草率的和过度设计的。这是一个很好的起点！然后你试着使用它，找出哪些部分妨碍了你，并修复它们。我觉得这些混血儿妨碍了苏西，所以我打算做点什么。随着我们添加和删除不像我们预期的那样有帮助的功能，装备都经历了不同的阶段。

先把它搞错！一会儿清理干净！
