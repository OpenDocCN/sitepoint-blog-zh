# JavaScript 并不邪恶

> 原文:[https://www.sitepoint.com/javascript-isnt-evil/](https://www.sitepoint.com/javascript-isnt-evil/)

以下内容转载自《科技时报》第 159 期。

继我的[避免邪恶的 JavaScript 社论最后一期](https://www.sitepoint.com/avoiding-evil-javascript/)之后，我得到了大量矛盾的反馈。这并不奇怪，因为人们倾向于对可访问性及其在网络上的重要性有强烈的看法。

然而，在我向您展示一些简单实用的方法来编写更好的 JavaScript 之前，我想澄清一下我在一些反馈中发现的一个常见误解。

JavaScript 本身并不邪恶，使用 JavaScript 提供流畅、增强的用户体验的网站和应用程序也不邪恶。我所说的“邪恶”是 JavaScript 的使用方式，它不必要地阻止了一些用户访问网站或应用程序。

在标准 HTML 和 CSS 的基础上构建流畅的、基于 JavaScript 的用户体验不仅是可能的，而且非常实用。这个基础使您不仅能够向禁用 JavaScript 的用户提供站点内容，还能够向搜索引擎等自动化系统提供内容。

只要稍加思考，您甚至可以让您的 JavaScript 功能与辅助技术配合得很好，并且对于只使用键盘的用户也很好。

但是，当 Ajax 出现时，事情就开始分崩离析了，用户开始叫嚣着要开发它所带来的类似桌面的应用程序。通常，静态 HTML/CSS 不能胜任为这些应用程序提供有用基础的任务，构建非 Ajax 替代方案将是一个完全独立的项目——可能是一个开发和维护成本不合理的项目。在极端的情况下，您的应用程序所做的事情可能无法转化为基于页面的普通 HTML 模型。

在我看来，解决这一困境的方法是将这些类型的应用程序从当前基于页面的 Web 中分离出来，并将它们转移到一个“应用程序 Web ”,这个 Web 就像现在的 Web 一样普遍可用，但它是从根本上以应用程序为中心设计的，并且解决了当前我们试图将类似桌面的应用程序硬塞进一个为提供页面信息而设计的系统中所导致的所有问题。

这个“应用程序网络”可以像一个新的 URL 协议一样简单(hatp://代表超应用程序传输协议？)或可被浏览器识别的 MIME 类型，事实上，许多供应商已经尝试(或正计划尝试)这样做:

*   [Java Web Start](http://java.sun.com/products/javawebstart/) (Java 网络启动协议)
*   XML 用户界面语言 (XUL)
*   [Macromedia Central](http://www.adobe.com/products/central/)
*   即将推出: [Adobe Apollo](http://labs.adobe.com/wiki/index.php/Apollo)
*   即将推出:[微软 WPF](http://wpf.netfx3.com/) (XAML)

同时，W3C 也在通过 Web 应用程序格式工作组研究这个问题。

到目前为止，所有这些举措都没有得到广泛采用，因为除了网络浏览器之外，还需要安装一些特定的软件(在 XUL，它需要一个特定的网络浏览器)。除非它们能像 HTML、CSS 和 JavaScript 一样无处不在，否则伪装成网页的类似桌面的应用程序仍将是更受欢迎的选择——也是有问题的选择。

那么，在缺乏一个适用于类似桌面的应用程序的无处不在的平台的情况下，基于 Ajax 的类似桌面的 web 应用程序是否属于“邪恶的 JavaScript”的范畴呢我个人认为是这样的，但是根据你的具体情况，它们可能是不可避免的邪恶。

web 技术的发展在继续，但是作为开发人员，我们需要尽我们所能利用今天我们所掌握的技术和资源。在我看来，最重要的一点是，在你做出可能阻止某些用户访问你的网站的决定之前，你要确保自己消息灵通。
