## 谈话要点

> 人们来到像我们这样的位置上，希望我们知道一切，这是一个很大的问题。我们甚至不能承认我们不知道事情，这有时是一个非常奇怪的处境。—【3 点 40 分】

* * *

> 我认为我们需要很多同理心，在我们的市场上有很多不同的工作方式。我品尝了所有的食物。我认为每个人都应该为一个机构工作一段时间，在那里你必须不断地使用各种技术。—【6 点 30 分】

* * *

> 所以，我想，<q>我们在为谁建造这个？我们建造这些是为了在会议上做主题演讲，还是为了解决问题？</q>我认为这部分是因为人们仍然不理解我们的工作，仍然不重视我们作为专家的价值。——【7 点 30 分】

* * *

> JavaScript 未关闭。JavaScript 就是不靠谱。无法加载 JavaScript。JavaScript 可能会很慢。JavaScript 可能会受到干扰。—【14:00】

* * *

> 我认为把 JavaScript 妖魔化是错误的方法。我们需要明白 JavaScript 是已知的。到处都是。每个浏览器都有。会有的。有些情况下它不可用并且会出错，但是这些都是边缘情况，不值得关注。
> 
> 我们应该更加关注这样一个事实，如果我们说 JavaScript 是已知的，我们也有责任让 JavaScript 工作。现在，我们有太多的 JavaScript 解决方案，它们只是将框架用于所有的事情，或者有一个成功的案例，却从来没有定义一个错误的案例。—【15:00】

* * *

> 我觉得不是一次性的状态，渐进式的增强。根据定义，它是进步的。我们现在有一个很好的机会开始拥有我们的界面。这完全是关于谁对界面负责。如果我们说我们是用 JavaScript 来做的，那么我们也要对最终用户看到的东西负 100%的责任。如果我们说我们不依赖 JavaScript，我们实际上可能会为每个人提供一个非常糟糕的解决方案。—【18:40】

* * *

> 我雇佣比我优秀的人。我希望人们在求职面试中不同意我的观点，并提出自己的观点。我希望人们比我更好，因为这样我可以做其他事情，他们可以做我认为我不想再覆盖的技术部分，我想再改变。—【21:40】

* * *

> 我们被视为非常反社会的人。我认为这是我们需要打破的。如果我们被视为角落里编写代码的极客，我们给他们很多钱，让他们闭嘴，这对我们的市场没有任何帮助。我们需要被视为普通人，应该参与每一次会议和每一项决策，而不仅仅是产品的最终推出。—【25:30】

*【咯咯笑】*

David:

说到多年来一直是开发人员的传道者，因为这是版本控制节目，我们喜欢以一个哲学问题开始我们的节目给我们的客人。你今天的哲学问题是:在你的职业生涯中，你是什么版本，为什么？

Christian:

我想在我的职业生涯中，我是 1.0 版，也许已经是 2.0 版了。原因是因为我改造了自己几次。我最初是一名网页开发人员…嗯，实际上是 3 岁。我最初是一名电台记者，后来我发现了网络，成为了一名网络开发人员。当我还是一名网站开发人员时，我很快就意识到，在我们的结构和公司中，你只能晋升到一定的级别，然后才能不再从事技术工作，而是进入管理层。

我当时不想这么做，因为我太年轻，不成熟，所以我发明了开发人员福音传道的想法，这是开发人员和公司其他人之间的过渡角色。这是一个翻译的角色。我写了一本关于它的手册，然后给自己起了这个职称。后来，人们意识到，从文书工作的角度来看，不给我这个职位会很尴尬，所以他们不得不这么做。这就是我在 Mozilla 的经历，也是我上一份工作的经历。我是那里的主要布道者。

现在在我现在工作的公司，微软，我们已经有了一个福音传道的结构。我感觉我现在正在进入一个不同的世界，在那里我有许多同龄人可以从我的流氓尝试中受益，做他们多年来一直在做的同样的事情。

我认为对于我们的工作来说，哲学最重要的一点就是你要灵活。不要认为你会在接下来的 20 年里做一件你现在非常兴奋的事情。你可以改变很多，而且你会从中得到乐趣。改变并不是说你失去了什么。这实际上让你更感兴趣了。我从未听说过像我们这样的工作，每年都不一样，而且我们做得很开心。

David:

这很有趣。我们和很多人谈过，我们发现大多数人并没有被培养成工程师和他们所扮演的角色。有时候，你所受的教育与你最终做什么没有太大关系。当然，在我们这个行业，事情变化非常快。

Christian:

是真的。前几天我和 Rob Conery 聊了聊，他刚刚出版了一本名为 [The Imposter's Handbook](https://bigmachine.io/products/the-imposters-handbook/) 的书，这本书讲述的是像我们这样的人，他们在 IT 公司身居高位，但从未接受过正式培训，也从未获得过计算机科学学位。

它给了你我们错过的所有东西，但我们应该知道，人们只是希望我们知道，因为我们已经在某个水平上了。人们来到像我们这样的位置上，希望我们知道一切，这是一个很大的问题。我们甚至不能承认我们不知道事情，这有时是一个非常奇怪的处境。这是一次很好的谈话，他谈到了太出名的危险，人们只是希望你能理解他们所说的一切。

有时候人们会向我走来。我发现了一些 JavaScript 模式。他们走过来对我说:哦，你是这种模式的发明者。太神奇了。它节省了我很多能量，我喜欢用它来做这做那。我只是站在那里，就像，<q>哦，原来如此。有意思。我不知道你可以这样用它。</q>

不要太看得起我们市场上的人。如你所说，我们都用水做饭。我们都假装过得很好。在一个像我们这样时刻变化的世界里，这是你必须要做的。

Tim[4:17]:

说到像你之前提到的三次改变你的职业生涯，你最初是如何进入 web 开发领域的？

Christian:

当我在一家电台工作时，我开始发现了网络，我意识到，哦，我的上帝。在我的空闲时间，我做了一些关于准将 64 和阿米加、[演示和类似的事情。](https://en.wikipedia.org/wiki/Demoscene)

其中一些小组开始有 FTP 服务器，我们把我们的东西放在那里。后来，我们有了邮件列表和 IRC。然后我们有了网站，我想，哦，太酷了。我的公司有互联网接入，所以我基本上是在那个互联网接入上开始看这些网站的。

我想，“嘿，我可以用汇编和 BASIC 编程。我可能也会做网站。我开始查看源代码，下载当时的 HTML 规范，就像 zip 文件一样，然后去了医院，因为我必须做手术。

在一台笔记本电脑上，我写了我的第一个网站，并把它发给了我家乡的几家公司。一家大型汽车公司找到我，对我说:哦！你知道 HTML。不如我们给你一大笔钱来建我们的内部网。我说，<q>好吧，听起来不错。</q>那是我开始做网页开发的时候。

我在那家公司开始工作。在我的简历上有了这样一个大名字后，很多代理公司开始对我感兴趣，我开始在代理市场工作了几年。我去了雅虎，成为我的大公司之一。从那里到 Mozilla，再从那里到现在的微软。

我经历了整个代理领域，经历了为小客户做建筑的整个过程。我做过很多内容管理系统，像企业级的内容管理系统。我认为这也让我对我现在所做的事情有了一个认识——理解开发者必须处理的事情，因为不是每个 web 开发都是一样的。人们有不同的问题。去找一个不得不使用 60，000 美元的内容管理系统的人来指责他们的 HTML 的结果是不公平的，因为他们甚至不能控制它。

我认为我们需要很多同理心，在我们的市场上有很多不同的工作方式。我品尝了所有的食物。我认为每个人都应该为一个机构工作一段时间，在那里你必须不断地使用各种技术。你总是需要两个月的时间来学习它，然后你不得不放弃它，因为下一个项目是在另一个内容管理系统或后端等类似的东西上工作。

在那里你会很快了解到，我们工作中最大的任务是快速学习、实施、快速抛弃，并找到新的工作——这听起来像是浪费，但奇怪的是，这就是我们市场的运作方式。

David:

感觉上，你必须发展的技能是获得新事物的能力，然后能够舒适地放弃它，并意识到事物在变化，你需要采用新的东西…这样你就不会与一种技术结婚，成为做这一件事的人，但你不知道如何做其他任何事情。

Christian:

我有点担心我们在市场上变得多么自私。我们发明了大量的东西来赢得一周的黑客新闻，然后…我的工作很忙，所以我在黑客新闻上看到一些东西，每个人都对此非常兴奋。我想，好吧，下周我有时间的时候，我会去看一看。下周当我有时间去看的时候，第一篇博文都是那个，<q>这个用起来很危险。你不应该再用它了。</q>

所以，我想，<q>我们在为谁建造这个？我们建造这些是为了在会议上做主题演讲，还是为了解决问题？</q>我认为这部分是因为人们仍然不理解我们的工作，仍然不重视我们作为专家的价值。他们只是把我们当成他们不了解的人来看待。

对于一个你认为很糟糕的项目，得到表扬是非常令人沮丧的，但这种事情在我们身上经常发生，因为我们被视为公司的交付部分。但是我们仍然不像数据库架构师那样容易理解，也不像那些技术已经以适当的教育方式存在了 30 年的人那样容易理解。

对于 web 开发来说，仍然是这样的事情，他们一直在使用新的东西。如果我们雇佣新员工，他们完全抛弃这个人以前做的事情，从头开始，这是可以的。我就像，<q>说真的，这就是我们在做的？</q>

似乎我们越来越多地进入一个我们只是互相炫耀的世界。这比建造真正的项目更让我们愤怒。我认为我们应该回到这样一个阶段，我们应该考虑我们向最终用户提供了什么，以及我们是如何实现的，而不是我们开发它有多容易。

Tim[8:36]:

你觉得我们如何回到那个阶段？听起来这列火车正以如此不可思议的速度行驶，业内的每个人都在试图推出下一个令人兴奋并引起人们注意的东西。

你认为这个行业可以做些什么来让我们回到你所说的正轨？

Christian:

我认为部分原因是我们自己不那么贪婪了。理解这不是关于我们有多酷，而是我们为我们的终端用户做了什么。这有时意味着你走出你的舒适区，说，<q>你知道吗，我不打算写下一个很酷的节点包管理系统。我要去找一份我在银行工作的合同，或者我在一个我讨厌的网站工作的合同，我想改变它。我想和从事这些工作的开发人员谈谈。</q>

例如，我几乎每周都参加会议，参加在美国中西部，或者在罗马尼亚，或者在你通常不去的国家的会议，但不是很酷的会议，在那里你会遇到更有趣的开发人员，因为他们必须提供日常工作。

当你的谈话中有一些东西激励他们时，几周后你会收到电子邮件，内容是:<q>看，我们实施了它，这就是我们节省了多少钱。</q>我想，这比看<q>酷多了哇！你让我印象深刻，但我比你强。还是看看我的东西吧。</q>

所以我认为我们也应该改变步伐，有时说不，有时说不，这还没有准备好。我不打算用它。我们不想用旧技术来使用它，但这个用起来很酷。

我在雅虎和其他公司做的也是挤出时间来创新和做事。我们在雅虎有内部黑客日。我们有内部演示，轻松的谈话。我认为重要的是，你要把你的日常交付和你的<q>分开，你在哪里玩你要做的下一个版本？</q>

我们不能在实时产品中这样做——这对我们的最终用户不公平。但是我们并没有要求我们的公司给我们额外的 20%的时间来做更好或更坏的事情，因为我们的最终产品不应该是最终用户的测试。

现在，我们处在一个奇怪的位置，人们竭尽全力雇佣我们。没有其他市场有这个问题。我去我家，一半人都在担心自己的工作。我们会这样，嗯，我们担心一天会收到十份工作邀请。这是一个非常奇怪的世界。

我也为此写了一篇博文。如果你想改变这个世界，那么我们现在就应该利用这一优势，让我们成为公司的交付者，同时在我们的发展中发挥创造力。更重要的是，在求职面试中，要问这些公司的人力成长是怎样的。

每个人都说，是的，我们有很高的津贴和诸如此类的东西。但是要问这样的问题，<q>嘿，如果我成为一个爸爸呢？我可以一周工作 4 天吗？你在这家公司的产假计划是什么？你如何让少数民族进入公司？我们该如何应对这其中的多样性？</q>

我们会就这些问题发表主题演讲，但真正重要的是当人们试图雇用我们时，因为我们目前占据上风，所以让我们利用这一优势，使我们的工作环境不像现在这样充满竞争——我们可以发挥创造力，但也允许我们在其中变老，因为现在它让我们筋疲力尽。我们就像 80 年代的投资银行家。

David:

那绝对是真的。当然，我已经看到了很多公司和招聘人员联系我的工作列表，所有这些都在谈论，你有机会使用这项新技术。你可以用这个东西做东西。

通常是一些未经测试的东西，或者是一些新的和潮流的东西，因为他们知道这将吸引寻求刺激的工程师。我认为这不是人们在寻找这些工作时真正应该寻找的。

Christian[12:08]:

公司这么做很好。没有什么比被困在一个过时的环境中更糟糕的了，除非你是一个朝九晚五的开发人员，并且你完全可以这样做，然后回家和家人一起生活。没关系。

对我来说，未经测试的环境和工作描述中随意列出的酷东西总是一个危险信号。我最喜欢的是当他们只是错了，他们就像，你可以要求在一项一年前的技术上获得五年的经验。不幸的是，这也发生了。

再说一次，我认为我们不应该对此自大。我们应该指出这些。当我们收到这样的工作邀请时，我们基本上应该说，我不感兴趣，因为你的工作描述的措辞。

如果你在推特上拿这件事开玩笑，那是没有用的。我们真的应该指出，这是不合适的报价给我们。这样我们也能提供更好的工作机会。

Tim:

Chris，我想听听你对渐进式改进的看法。你对它的看法。你认为开发人员在构建软件和 web 应用程序时应该做什么。你对此有什么想法？

Christian:

我已经写了很多关于这个的东西——大概从 2003 年开始。甚至更早，比如 2002 年。我是渐进增强的坚定支持者，认为这是构建东西的最佳方式。这甚至不是一个技术问题，而是一个方法问题。

如果你要跳进一条河里，了解这条河足够深并且里面没有鳄鱼可能是个好主意。相反，我们只是跳进河里说，好吧，鳄鱼可能没有那么危险，只有怪物才会把鳄鱼从东西上弄下来。

遗憾的是，现在我们的最佳实践是我们在 2002 年和 2003 年讨论过的。渐进式改进在当时的意思是说，<q>根本不要依赖 JavaScript。JavaScript 将被关闭，这是你无法预料的。</q>

我认为这不再是事实。JavaScript 未关闭。JavaScript 就是不靠谱。无法加载 JavaScript。JavaScript 可能会很慢。JavaScript 可能会受到干扰。它的本质意味着它不像 HTML 和 CSS 那样是容错的。这两者，犯了一个错误，没有关系，最终用户仍然得到一些东西。在 JavaScript 中，一个错误就意味着一切都完了。这是很多人在学习这门语言时遇到的问题。

我想，我试图记住的事情是，不要把这归咎于任何特定的技术。这只是防御性的编码。当我谈到渐进式改进时，这是我试图记住的事情。这不是说，你不用这项技术。讲的是代码防守。以这样一种方式编码，当你在所有 web 开发中最不稳定的环境中工作时，在某些情况下有一个后备可能是一个好主意。

所以对我们来说，你点击按钮，它发出一个 Ajax 请求，也许把它变成一个表单，如果 Ajax 请求不起作用，表单提交。这与依赖 JavaScript 的想法没有任何关系，而是让您的应用程序更具防御性——在一个越来越不稳定的环境中。

David:

越来越不稳定的环境确实是一个问题，因为现在，当你用 JavaScript 发布东西时，你依赖于这么多层技术。你有了框架，你有了运输工具。所有这些事情都在幕后发生，无论这些事情是否是动态生成的。

如今，为了能够防御性地编码，你必须从上到下了解你的栈。那是一个非常困难的命题。我无法告诉你我遇到这种情况的频率……我能不假思索地想到一些拥有大型高端产品的大型出版商的情况。

例如，我试图点击一个下载预览一些软件的一家大公司，使照片处理软件，不得命名。他们网站的用户界面依赖于按钮被设置为`disabled`的事实。他们在 HTML 中使用了`disabled`参数。我可以进去，我可以关闭`disabled`参数并下载预览，因为他们没有检查。

这可能是因为他们在一个为他们动态生成某些内容的环境中操作，错误检查是通过他们使用的框架进行的，他们不知道检查是否有人进入并做出了更改，有人是否能够绕过这一点。

Tim[35:42]:

为了更好地说明这一点，因为我从很多批评家那里听到——我相信你和大卫一样——谈论不写任何 JavaScript 是没有意义的。在我的日常工作中，98%的时间我都在写 JavaScript。

这真的不是一个回避特定技术的问题。它与检查东西有关，测试真的很难。当你制作一个网络应用程序，并在手机上打开它，打开另外六个标签和三个应用程序。让你的 CPU 忙起来。看看会发生什么。试着打破它。

当你成功地破坏它时——对我来说，大多数时候是在我试图测试它的时候:第一次运行代码时，它就被破坏了！一旦发生这种情况，我就问自己，我怎样才能做得更好？我怎样才能确保，即使我非常努力地尝试在这里造成一个错误，我怎样才能让这个东西仍然工作？

我认为这是渐进式改进的核心思想。

David:

我认为你是绝对正确的，因为它是关于看你正在做的事情的弹性，并在不同的环境中尝试它。这是克里斯与这些公司谈论的一部分，这些公司试图让编码人员尽快扭转局面。
