# PNG32 hacks 和 IE6。值得吗？

> 原文：<https://www.sitepoint.com/png32-hacks-and-ie6-is-it-worth-it/>

去年你可能还记得我浏览了一下我认为被低估的[的质量，这是一种似乎在雷达下溜走的图像文件格式——png 8](https://www.sitepoint.com/png8-the-clear-winner/)。

即:

1.  小文件大小
2.  在所有当前的浏览器中分级透明
3.  不需要需要 JavaScript、ActiveX 和/或其他专有技术的潜在脆弱黑客。

虽然，不可否认的是，你不得不准备稍微降低你对 IE6 渲染的期望，但我们认为这种权衡总的来说是值得的——特别是当你开始磨练自己的技能以生成在 IE6 中降级良好的 PNG8 图像时。就我个人而言，我不排除在某些情况下使用 PNG32，但 PNG8 无疑是我的首选武器。

记住这一点，很高兴在新的 Cozi Tech 博客上读到乔治·赖利关于 PNG32 hacks 和 IE6 的一篇非常有趣的文章。简单介绍一下背景，George 在微软从事 IIS 工作七年，因此有两件事我们大多数人都不能声称:

1.  一系列来之不易的微软忍者调试技能
2.  一本微软联系人的黑皮书，用来测试无意义的 IE6 巫毒教

然而，在一个项目中部署了历史悠久的 IE6 PNG32 hack 之后，他们在接下来的几个月里一直在与不一致的浏览器挂起作斗争。

> 这个问题消失了几个星期，但在 10 月下旬，它又卷土重来，每天重复出现多次。有些机器似乎免疫，有些机器几乎可以 100%成功地重现它。没有明显的共性；它对 IE6 的几个版本进行了重新评价。必须解决这个问题:我们不能发布一个有时会严重挂起 IE6 的 web 应用程序。

这几乎是最终的调试噩梦:一个严重到足以破坏应用程序的问题——也就是说，足以让用户强行杀死他们的浏览器——但这个问题似乎通过茶叶或每日星座专栏来选择受害者。

即使是微软自己尝试的解决方案——通过 JavaScript 预加载图像——在他们特定的应用环境中也没有用。最后，乔治和他的团队决定适可而止。

> 我们已经决定不再花大力气让我们的 web 应用程序在 IE6 上看起来像在现代浏览器上一样好。除了我们在透明 PNGs 事件中忍受的巨大痛苦之外，与 IE6 的斗争对我们来说是一个巨大的时间消耗，特别是当涉及到解决其 CSS 错误时。

当你想到我们中的许多人可能已经在不知道的情况下将这种问题构建到了站点中——嘿，一个错误报告出现了，你耸耸肩并解释说这可能是他们安装的东西——这必须让你在使用 PNG32 时三思，而 IE6 仍然是一个考虑因素。
