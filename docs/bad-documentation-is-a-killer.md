# 糟糕的文档是黑仔

> 原文：<https://www.sitepoint.com/bad-documentation-is-a-killer/>

几乎每个开发人员都遇到过一些糟糕的文档。我甚至不记得有多少次我不得不与一个不满的技术支持人员谈论一个本应很容易发现的问题。我工作过的两家公司甚至因为糟糕的文档而停止使用原本很好的软件。

文档的挑战在于大多数开发人员真的不想写文档。对于我们这些从事网络工作的人来说，这很容易，因为文档对于许多网络项目来说并不重要。我的大多数客户不需要我指导他们如何浏览网站或使用我提供的基本管理工具。然而，事情一直在变化。

随着网站和基于网络的工具变得越来越先进，我发现自己在从事更多需要文档的项目。我为外部 API 和服务编写了文档，为团队中的其他开发人员编写了说明，为最终用户编写了教程。我早期的大部分尝试都不成功，最终我成了一名心怀不满的技术支持人员。防止这种事发生在你身上；这里有一些想法，可以帮助您更好地编写文档，并在编写过程中节省一些时间。

**了解你的受众**

就像任何其他项目一样，你需要知道你在为谁写作。如果你是为精通技术的终端用户或其他开发者写的，你通常可以删掉一些介绍性的内容。同样，你也不想让普通的网上冲浪者对你如何处理复杂的数据处理或内存分配的细节感到困惑。

如果你发现你有不同的用户组，不要害怕写多组文档。乍一看，这似乎令人望而生畏，但在许多情况下，为特定用户群编写更短、更简洁的文档比试图让每个人都满意花费的时间要少。一个很好的例子是 CMS 供应商，他们中的大多数为开发人员、IT 管理员和最终用户提供单独的文档。

**利用自动化工具**

大多数语言都有在某种程度上自动化文档的系统。通常这些系统不能为最终用户创造出足够好的东西，但是对于其他开发人员来说，文档通常是正确的。在大多数语言中，这些工具基于代码中的注释生成文档。集成它们只是改变您的注释风格以适应文档生成器的期望。实际上，我发现这提供了额外的好处，让我写更一致的评论。

**让文档成为社区工作**

一些对编程最有用的参考包括一群人。一个完全由用户驱动的系统，比如 wiki，或者一个文档系统，比如 MSDN 图书馆，允许用户将内容附加到现有的文章中，既可以提高文档的有效性，又可以减少维护文档所需的工作量。Wikis 也是内部准备文档的一个很好的工具。随着开发的进行，很容易向 wiki 添加快速注释，然后在发布时将内容编译成文档。即使它是一个只能由员工编辑的系统，它也为解决方法、错误和常见缺陷提供了一个中心参考。

**留出写文档的时间**

我见过的大部分灾难，都是匆忙走完流程的结果。削减一个功能或者推迟一个截止日期来为文档腾出时间是很困难的，但是这是值得的。无论你是以一场技术支持噩梦告终，还是花费数周时间试图让新开发人员跟上速度，你都会后悔没有花时间把事情写下来。

随着 web 应用程序变得越来越复杂，web 服务和 API 变得越来越普及，编写文档对于开发人员来说将变得越来越重要。希望这些建议能帮助你成为一个更高效的文档作者。
