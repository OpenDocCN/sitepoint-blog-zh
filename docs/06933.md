# 选择框架时要考虑的 6 件事

> 原文:[https://www . site point . com/6-things-to-consider-when-choose-a-framework/](https://www.sitepoint.com/6-things-to-consider-when-choosing-a-framework/)

您已经决定在编写下一个新应用程序时使用框架是有意义的，如果您已经熟悉了某个特定的框架，那么您可能会在开始时倾向于使用该框架。但你确定这真的是最适合手头任务的吗？

以尽职调查的名义，在选择一个特定的框架之前，这里有一些你应该问自己的问题，以确保你不是在“逆着颗粒”编程，也确保它能够满足你现在和长期的需求。

## 我需要从框架中得到什么？

熟悉很重要，但功能更重要。因此，考虑一下你真正需要框架的什么功能。例如，如果您只需要路由功能，那么使用全栈框架就没有意义。一旦您确定了您的情况需要什么，您就可以开始明智地比较各种可用框架，减少您的选择，从而更容易选择最终候选框架。

## 我期望框架帮助管理一致性吗？

在大型开发团队中保持适当的沟通是很困难的，尤其是如果团队是分布式的。在格式化代码、命名对象等方面，个人会有自己的偏好。并且可以重新实现代码库中其他地方已经可用的代码。框架可以在这方面有所帮助，但是要注意过分依赖它来保持一致性。框架不是编码标准、代码评审和其他内部控制策略的替代品。

## 有好的文档吗？

我相信我们都经历过理解我们写的代码是多么困难，搁置六个月，然后再回来。尽管是我们写的，但我们好像是第一次读它。有了框架，你总是在阅读和使用别人的代码。选择一个具有提供良好文档和培训历史的框架，将会使理解代码和使用框架发挥其最大潜力变得更加容易。

## 这个框架是积极开发的吗？它有活跃的用户基础吗？

编写与底层框架不紧密耦合的代码是非常耗时的，因此框架成为应用程序不可或缺的一部分是很常见的。因此，如果你所依赖的框架最终失败了，你将面临两个选择:继续使用框架并自己承担维护工作，或者重新编写代码以使用新的框架。两者都不是非常诱人的主张！当务之急是确保你不会被定时炸弹卡住，所以当你还在计划阶段的时候，花时间研究一下框架的历史和社区，这样你就可以避免这个陷阱。

## 我在生产环境中运行的框架能工作吗？

PHP 程序员通常可以编写在已知环境中运行的应用程序，而 JavaScript 程序员必须处理不同的浏览器和平台组合，但是 PHP 环境仍然可以改变。操作系统可以升级；PHP 版本号可以递增；等等。该框架不需要使用 PHP 每个新版本的最新和最好的语言特性，但要确保它没有使用过时的特性。客户不关心实现的选择(这是理所当然的)，所以由框架触发的日志文件中的错误和警告会对您的应用程序、您的开发人员和您的组织产生不良影响，而不是对框架的开发人员。

## 哪些业务因素影响了我的决策？

如果你是一个小企业，想在谈判中给一个特别大的企业留下深刻印象，你可能会被迫选择一个他们的开发团队更喜欢的框架。这种情况会导致选择经常使用某个特定的框架(我实际上见过一个 PHP 应用程序使用了不止一个而是三个不同的框架，正是因为这个原因)。随后的后果可能是好的或坏的，可能是你无法控制的，但却是值得铭记的。

并不是每个应用程序都需要使用框架来编写。但是如果您已经决定您的需要，那么将您的需要与各种框架产品的特性和优点进行比较是有益的。它可能是一个你已经熟悉的框架，也可能是一个新的框架，但是只有通过客观的分析，你才能确定它真的是最适合的。
