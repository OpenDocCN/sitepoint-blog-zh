# 管理云中的变化

> 原文：<https://www.sitepoint.com/managing-change-in-the-cloud/>

让我们想象一个我们很多人都暗暗希望实现的场景。你是一个聪明的开发者，有一个好主意。你创建了一个基于云的应用程序，允许你的用户互相发送消息，并每天跟踪对方的行踪。我说过每天吗？不，你真的很聪明，现在你可以一直实时跟踪他们。很快你就有几十个人…不，几百个…等等！现在每天都有成千上万的用户注册。

你知道定期与你的用户交流是很重要的，你每天都在了解他们。你调整他们使用应用程序的方式，很快他们就会与你分享他们生活的方方面面。你已经创建了一个人类行为测试实验室，你在幕后操纵！你能够发现他们在买什么，在哪里买，甚至他们渴望什么的趋势。这一刻！

为了改进应用程序，你推出了一个新版本。但是，你的用户社区非但没有感谢你“帮助”他们的努力，反而反抗并开始在他们的社区帖子和博客中发表贬低性的言论。那些从来没有给过你一分钱的免费订阅者认为他们可以驱动你的应用程序的设计和感觉！

好吧，我显然是在说脸书。他们最近推出的新功能使他们的应用程序保持新鲜，引起了用户社区的许多投诉和评论，使他们成为新闻。关于负面新闻比没有新闻好的格言可能适用于名人——但硅谷不是好莱坞。虽然很痛苦，但脸书的变革管理试验并不是独一无二的，应该受到所有基于云的应用程序设计者和开发者的密切关注。我们需要吸取教训，这些教训已经在 IT 行业传授了很多年。

# 变更管理

变更管理的定义因您的视角而异。技术开发人员通常将变更管理定义为管理技术对象(即代码、数据、配置参数)变更的规程。当然，这是一个至关重要的过程，也是我们所有人在使用应用程序时都要处理的事情。但脸书和其他云解决方案提供商需要关注的变化管理重点是管理客户的体验，以及他们如何将这种变化融入他们的业务。

适应变化对每个人来说一般都很难。预计会有一些中断，但你必须知道如何在转变过程中管理预期。每当你觉得一个新的愤怒的小鸟可能不会给你的客户带来太多的关注时，就偷偷放进去。但是在购物车结帐过程中添加一个新页面，你可能会失去很多买家。在月末或季度处理过程中更换会计应用程序，你可能会失去很多用户。

从客户的角度来看，基于云的应用的主要优势之一是使用新应用的入门成本较低。你的目标是建立一个“有粘性”的应用程序，让客户永远不想离开——但糟糕的变更管理肯定会失去粘性，让客户转向竞争对手的解决方案。

# 云中的变更管理

在云出现之前，解决方案提供商通过允许其客户群运行不同版本的产品来促进变更管理，当客户管理自己的服务器并且每个客户都有自己的应用程序实例时，这很容易做到。客户有时间规划升级，并为业务流程变更、测试和培训做好准备。客户在其时间表允许的情况下部署产品升级，并支付实施团队的费用以支持他们完成升级。

您的客户现在将这些升级活动和责任推回给您，即云解决方案提供商。云降低了客户的总拥有成本，也减少了他们对应用程序的控制，这让他们很恼火。除非您对升级时间表和计划非常透明(很少有提供商做到这一点)，否则您的客户必须对变化做出反应，而不是主动做好准备。

该行业缺乏对客户变更管理的关注，这说明了云计算行业的成熟度有多低。甚至行业领导者也犯有错误指导变革管理的错误。我经常在我的公司和客户网站上使用谷歌应用程序。我很惊讶经常会有新的特性或变化在我不知道的情况下发布。我不止一次遇到过带有公式的电子表格意外停止工作，然后发现发布了一个新功能。我为对他们的挑剔向谷歌道歉(我不想我的搜索引擎优化被关闭！)但这表明，即使是该行业的巨头也还没有搞清楚这一点。

# 将变革管理作为竞争优势

我假设你的应用不与谷歌竞争(还没有！)这可能是对你有利的一点。围绕您的客户群及其时间表规划增强和升级将是一项与众不同的功能，有助于您在竞争中脱颖而出。

从业务或客户的角度来看，变更管理是管理向应用程序或流程的用户引入新功能或流程变更的规程。客户和用户必须在部署之前做好使用这些新功能的准备。与您的客户沟通变更的类型，让他们知道为什么要进行变更，并培训他们如何使用变更，这些是变更管理的首要步骤。

典型的云应用或服务将所有客户放在同一个实例上。引入新功能意味着所有客户同时获得这些功能。在向所有客户推广之前，没有早期采用者客户试用测试版。至少这是我的经历。但一定要这样吗？

云走向成熟的下一步可能是一种更加规范的软件部署和变更管理方法。我意识到成本是一个考虑因素，为所有用户维护一个应用程序或服务的实例比多个版本更便宜。但还是有选择的。

将您的客户划分为早期采用者，并对您的应用给予精选的少数几个价格折扣，承认他们在您的变革管理过程中发挥的重要作用。让他们觉得自己很特别，并奖励他们的反馈。在试用期间，将他们发送到一个备用 URL(假设您可以成功移植他们的数据)，并监控他们的使用情况和任何服务问题。这个小组是变革管理的晴雨表。成功部署并评估他们的体验将有助于预测新版本对您的整个客户群的影响。

另一个选择是在不同的 URL 运行多个版本的应用程序。您可能已经有了可用作客户群的基本版和高级版。通过不同的 URL 将每个段发送到您的应用程序，一次仅部署到一个段。这让您有机会见证您的部署的影响，并确定它是否已准备就绪。当然，如果您的测试过程是防弹的，并且您非常了解您的客户，那么这些都是不必要的…对吗？

我知道支持您的应用程序的多个版本的成本可能是一个问题。但是，难道你不想早点发现你的改变是否会让客户张开双臂迎接你的新应用，或者逃向你的竞争对手？如果出现问题，为了防止客户流失，这种投资难道不值得吗？

在整个开发周期中与客户沟通并说服他们改变的需要不仅是改变管理的重要部分，也是客户关系管理的重要部分。这不全是关于你有多了解你的客户。这也关乎他们对你的了解和信任。变更管理是一种你用来建立和展示这种信任的技术。

我的经验表明，变革管理通常被认为是只有“大”公司才会做的事情，对小公司来说没有必要或不可行。但经验也告诉我，无论公司规模大小，这一点经常被忽视。对吧，脸书？如果小公司想成长为大公司，服务中断和失去信任是更重要的。

在当今的环境下，您如何升级和部署您的应用程序？它被认为是涉及整个团队的技术活动还是商业活动？我有兴趣了解您如何利用您的应用解决变更管理问题，以及部署应用的设备是否会影响您的战略。欢迎在下面发邮件或评论。至于我，我要去网上了解这个时间线的东西是什么。
