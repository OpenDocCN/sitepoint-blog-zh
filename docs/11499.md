# 为什么谷歌不应该给 Chrome 开绿灯

> 原文:[https://www.sitepoint.com/google-chrome-go-plugin/](https://www.sitepoint.com/google-chrome-go-plugin/)

![Google Go Gopher mascot](../Images/7e1fa051153a617cb475a61373c333b6.png) *(不好意思——我忍不住在标题中又用了一个‘去’的双关语！)*

最近看了一下[谷歌的 Go 编程语言](https://www.sitepoint.com/its-all-go-for-google/)。乍一看，这似乎是桌面或服务器端 web 开发的一个好选择，但似乎 Google 有更雄心勃勃的计划。

根据源代码评论和对开发团队的采访，Go 可能会集成在 Google Native Client (NaCl)中。NaCl 是一个开源插件，它允许本机 32 位 x86 代码直接在 web 浏览器中运行。代码经过沙箱保护、验证和限制，以确保它不会对浏览器、其他应用程序或底层操作系统造成任何损害。虽然 NaCl 是实验性的，但它已经包含在 Chrome 网络浏览器中(但被禁用), Quake 已经被转换为演示该技术。

潜在地，Go 可能是 NaCl 的合适选择。开发人员可以部署编译后的可执行文件，甚至是原始源代码，它们可以在浏览器中快速运行。复杂的游戏和处理器密集型应用将成为可能。

## 我们需要更多的插件吗？

你安装了多少插件？大多数人会有 Flash，Java，Adobe Reader，也许还有 Silverlight 或者 Google Gears。对每个提示都点击“是”的新手用户大概有几十个。

网络的主要吸引力在于平台独立性和即时部署。然而，业界对将所有桌面应用程序转移到网上的痴迷引发了插件开发，这抵消了它的优势:

1.  依赖插件违反了平台独立性。创建一个基于 Go 的客户端应用程序几乎肯定会把你和 Google Chrome 捆绑在一起，因为 NaCl 永远不会适用于所有的操作系统和浏览器组合。许多网络应用程序仍然依赖 IE，因为使用了 ActiveX——尽管 Chrome 和 Firefox 都有 ActiveX 实现。
2.  NaCl 和 Go 将提供原始速度，因此游戏、图形和视频应用程序是合理的选择。然而，复杂的应用程序可能有数百兆字节大小，即时部署是不可能的。虽然我承认 NaCl 非常聪明，但处理器密集型应用在浏览器之外肯定会运行得更好。

Mozilla 在最近对谷歌 Chrome 框架的批评中表达了类似的担忧。虽然 Chrome Frame 是一个已知问题的利基解决方案，但 Mozilla 担心，如果公司回避网络标准，转而支持基于插件的解决方案，网络可能会变得支离破碎。

我并不是说所有的插件都是不好的，在某些情况下，它们提供了最终会成为浏览器标准的工具。例如，Flash 允许我们今天观看视频，而不是等待几年才能看到 HTML5 的`video`标签。然而，web 和桌面是不同的平台，各有优缺点。虽然界限越来越模糊，但是用插件把一个桌面应用硬塞到浏览器里是明智的吗？

如果你发现自己越来越依赖插件，也许你应该考虑重新设计，让你的应用程序利用网络技术。或者，在必要时发布一个利用 web 连接的桌面应用程序。

Google——和 NaCl 玩得开心，但是请不要把 web 变成基于 Go 的二进制应用程序的分发平台！

你怎么想呢?NaCl 和 Go 是个好主意吗？是我们的插件太多了还是它们改善了浏览器体验？如果 web 浏览器持续需要定制的改进，它们应该被用作应用程序平台吗？
