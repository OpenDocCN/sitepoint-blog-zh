# 谁在利用艾瑞亚。

> 原文：<https://www.sitepoint.com/whos-using-aria/>

![A square peg in a round hole](img/b20e5e9a6ff68e001200bc6972a07638.png)

我不是指开发人员，我是指*真实的人*——实际的用户，他们使用辅助设备浏览网页，使用 [<abbr title="Accessible Rich Internet Application">ARIA</abbr>](https://www.w3.org/WAI/intro/aria) 启用的应用程序？

看，当我在 2006 年第一次为 JavaScript 选集研究可访问 Ajax 的问题时，还没有像<abbr title="Accessible Rich Internet Application">ARIA</abbr>这样的东西；只有最终成为 it 的早期想法草案，主要基于 IBM 所做的工作。

我不为所动。不是因为任何具体的细节，而是背后的整个概念——认为所有的 RIA 都需要让它们对屏幕阅读器可用是**更多信息**的想法似乎是天真的。

现在我们有了一个正式的规范，浏览器和屏幕阅读器都有可用的实现，我问同样的问题——T2 RIA 真的适合辅助设备吗，如果它们有角色和状态信息的话，或者我们是在试图把一个方钉塞进一个圆孔？我们是否只是在与目标设备根本不兼容的范例之上添加了信息层？

我的问题实际上是，任何基于 Ajax 的应用程序，如果需要一个 <abbr title="Application Programming Interface">API</abbr> 来为消费设备提供足够的信息来理解它，是否可能比我们以前拥有的更好——你提交一个表单，然后你得到一个新的页面作为响应——这是几代设备都能够很好理解的。

我不想再重提我的[停止使用 Ajax](http://dev.opera.com/articles/view/stop-using-ajax/) 的论点——一点也不，我接受 Ajax 已经存在，并欢迎它成为程序员工具箱中有价值的补充。但是炒作还没有平息，Ajax 仍然被过度使用。关于【Ajax 应用程序在多大程度上被期望从静态应用程序逐步增强的问题仍然是一个很大的问题，因为在[Craig 最近的一篇帖子](https://www.sitepoint.com/5-reasons-to-avoid-supported-browser-lists/ "SitePoint blog post: 5 Reasons Why You Should Not Publish Supported Browser Lists, by Craig Buckler")中的大部分评论都过于尖锐地突出了这一点。

那么 <abbr title="Accessible Rich Internet Application">ARIA</abbr> 的用户在哪里呢？而*他们*是怎么想的？

这是一个诚实的问题，因为我真的不知道。有什么信息吗？是否开展了任何研究，令人信服地回答了这样一个问题:对于使用辅助技术的人来说，<abbr title="Accessible Rich Internet Application">RIA</abbr>支持的应用程序是否比传统的发布和响应风格的应用程序更好的用户体验，否则<abbr title="Rich Internet Applications">RIA</abbr>应该会降低用户体验？

ARIA 真的是前进的方向吗，或者只是一个寻找问题的解决方案？
