## 谈话要点

> 基本上，在一天结束时，它是关于假设你的设计没有完美的视图——我们可以创建这些没有理想宽度和高度的布局；无论你的屏幕有多小或多宽，我们都可以为网络创造出非常漂亮的、不受设备限制的布局。

* * *

> 我开始在客户合同中收到越来越多的建立 iPhone 网站的请求，这有点奇怪。这让我开始思考，我们下一个要建的网站是什么？在某个时候，其他一些新的很酷的东西将会出现在移动网络上。这不是特别好的扩展。

* * *

> 我赶上了出版的最后期限，我以为那会是它的结束。人们真的接受了这个想法。他们开始重新设计他们的网站，使它们更具响应性。我开始看到代理机构积极响应，然后将解决方案带给他们的客户，最终我认为这只是一点一滴地往上爬。

* * *

> 他们开始谈论他们从中看到的所有业务优势，并发布这些指标和数据。这让媒体和出版界的许多人刮目相看，这真的很令人兴奋。

* * *

> 该设计合作伙伴从未向维珍美国的客户展示过静态比较。基本上，从他们第一次见面开始，他们就有了一个响应式的原型——在非常早期的阶段，但他们有了一个每个人都可以在不同的浏览器、不同的设备上加载的原型。这个想法真的很有变革性。

* * *

> comps 不能告诉你什么，首先布局要做什么。有一个网站的图片，但是如果浏览器不支持 web 字体，或者如果出现网络错误，一些关键资源无法下载，或者如果浏览器不理解 JavaScript，或者延迟太长而无法执行，或者诸如此类的情况，会发生什么呢？

* * *

> 与其构建所有这些媒体查询的老鼠窝，来处理我必须响应性地计划的所有这些不同的布局条件情况，如果我可以有这样的东西，好的，给定这个容器的宽度，我想将这个设计规则应用到这个特定的模块，那将是巨大的，并且我认为它可能会取代我今天编写的媒体查询的很大一部分。

* * *

> 我觉得响应式设计只是我道歉的事情！

Tim:

非常酷，非常酷。每个人都给出了不同的答案，这真是令人惊讶。

Ethan:

啊，很好，很好。

Tim:

你是发明<q>响应式设计</q>概念的人——或者至少给它起了个名字，对吗？

Ethan:

是的，我喜欢说我创造了这个术语，因为我真正做的是我把几种不同的技术，我认为，放在一把雨伞下——<q>柔性网格</q>，这就像是反应式设计的主力。我想这是我更直接参与塑造的一个，但其他两个，像灵活的网格和媒体查询，显然就像我的工作之前的那些。我喜欢认为响应式设计基本上就像一个战斗口号，把我们带到这个更灵活，更设备无关的设计的想法。

Tim:

我想很多人都听说过这个术语，它被抛来抛去，应用于很多事情，但是对于那些可能不太适应这个概念的人，或者可能不熟悉这个术语或者没有听说过这个术语的人，你会如何为典型的 web 开发人员受众定义响应式设计呢——就此而言，包括 web 开发人员和设计人员？

Ethan:

这是一个很好的问题。我做了很多咨询和研讨会，其中一些是和我的好朋友兼同事凯伦·麦克格兰一起做的。当我们走进室内的某个地方时，总是有很多关于响应式设计这个术语的兴奋，但有时对它的含义有很多不同的定义。所以说到底，最简单的，响应式布局是一种基于网络的布局，使用灵活的布局——基本上是以百分比或比例定义的布局，而不是僵硬、不灵活的像素，然后他们在媒体中拥有灵活的图像，在这些灵活的网格中工作。

最后，第三个要素是[媒体查询](https://developer.mozilla.org/en-US/docs/Web/CSS/Media_Queries/Using_media_queries)，它基本上就像 CSS3 中的一点点精灵粉，允许我们明确说明这些布局何时以及如何需要根据不同的条件进行更改和适应——更常见的是，这些是基于视口的大小。基本上，这些布局可以重新呈现信息，或者改变它们的层次结构，或者有效地响应浏览器窗口或设备显示器形状的变化。基本上，在一天结束时，它是关于假设你的设计没有完美的视图——我们可以创建这些没有理想宽度和高度的布局；无论你的屏幕有多小或多宽，我们都可以为网络创造出非常漂亮的、不受设备限制的布局。

Tim:

很好，很好的总结。说到这里，我想知道当你第一次想到这个概念时，这个想法是如何进入你的脑海的？要不要给你讲讲响应式设计的起源故事？

Ethan [3:46]:

是的，我可以试试。我的意思是，如果我们谈论的是事物的版本——而我一直处于测试阶段——我认为响应式设计的理念已经到来很久了。最后，当我在做这个演讲的时候，有一个顿悟的瞬间——这是我第一次创造这个术语。但是从一开始，我就非常热衷于将网络视为一种完全灵活的设计媒介。那是在 90 年代末，21 世纪初。关于网页设计的一个很大的灵感来源于这篇叫做 <q>[网页设计之道](http://alistapart.com/article/dao)</q> 的文章，它是由一个叫约翰·奥尔索普的人在 2000 年写的。

我在很多会议上发言，我和很多公司合作，当我问人们是否读过这篇文章时，没有多少人举手，因为它在网上已经超过三周了(这就像古代历史)！我仍然认为这实际上是更有先见之明的作品之一，我认为这是关于网络作为设计媒介的更相关的作品——即使在今天——因为约翰基本上是在说我们早期思考网络的方式基本上是把它当成一张纸。在 Photoshop 或 Illustrator 中，我们会创建一个画布，我们会创建一个具有固定宽度和高度的文档，然后我们会尝试将该宽度和高度转换到屏幕上，这是完全灵活的。

我的意思是，屏幕本身是有限制的，但是视窗——实际的浏览器窗口——我们无法预测它的大小或形状。所以他基本上是在说，我们真的应该试着把网络当作一个完全独特的媒介，并为它设计更灵活的设计。这要追溯到 2000 年，当时 4.0 浏览器是我们可以看到的新热点。我们没有很多工具来真正利用约翰所说的。所以我会为我自己的项目、我的博客(当我更新我的博客的时候)和其他项目做很多灵活的布局。但对于我的大部分专业工作来说，它真的是非常固定宽度的布局。

然后手机出现了，打破了一切。在移动的早期，至少在更发达的市场，我们正在重复同样的过程。这就像我们将拥有桌面网站，因为我们了解桌面网站，然后我们将拥有独立的 m.dot 网站，这是一个移动网站。我开始在客户合同中收到越来越多的建立 iPhone 网站的请求，这有点奇怪。这让我开始思考，我们下一个要建的网站是什么？在某个时候，其他一些新的很酷的东西将会出现在移动网络上。这不是特别好的扩展。

这就是我开始为几个客户项目使用流体网格的地方，媒体查询支持最终到达了一个点——无论是在桌面上还是在移动设备上——我们实际上可以拥有我们的蛋糕并吃掉它。我们可以设计这些灵活的布局，但仍然引入对媒体查询的轻微控制措施，以阐明如何使这些灵活的布局在小屏幕和大屏幕上看起来很漂亮。

Tim:

当你看这个的时候，你是从设计的角度，还是从工程的角度来看？你带来了什么背景？

Ethan:

哦，伙计，我还在想办法呢！*【笑】*我觉得都是，真的。我的意思是，我从来没有完全符合这个等式的任何一边。根据项目的角色，我要么是前端开发主管，要么负责设计。在我参与的这个项目中，我是这个完全灵活的基于网格的布局的设计负责人，过了一会儿，我作为一名前端开发人员在那个只有 iPhone 的网站上工作，并意识到这两件事不一定要完全对立。我认为我们可以再次考虑一种布局，从美学的角度更灵活地考虑它，但也要节省一点我们的努力，不一定每次有一个很酷的新设备上市时都要分叉我们的经验。

David:

很有意思。不是每个人都能从设计和工程的角度来看待一个项目。我认为，在早期，人们对你称自己为网页设计师，还是称自己为网页开发人员，有什么区别吗？当然，现在，这可以更清楚地分为两种不同的职业，但我认为能够弥合这两者之间的差距是非常重要的。

Ethan:

我认为这对于那些主要是前端开发人员或前端设计师的人来说是正确的。我认为，特别是在任何团队环境中，我们现在正在达到这样一个点，即我们为之设计的设备和屏幕的数量——我认为这两个学科确实需要更密切地合作。当我担任前端领导时，我合作过的最好的设计师可能会抛出一些 HTML 和 CSS 的粗略原型想法。这可能不一定是生产就绪，但他们至少可以真正有效地更清楚地向我传达他们的想法。

你知道，在前端设计领先的光谱的另一端，我正在与开发人员合作。如果他们知道如何推荐不同断点处的版式可能需要改变，或者可能推荐版式可能需要改变，我认为这是非常强大的东西。

Tim [8:23]:

所以，Ethan，我认为响应式设计运动成功的真正原因之一是行业有能力让商业领袖相信响应式设计是值得的。你想谈谈这个过程是如何结束的吗？如果有的话，你在整个过程中学到了什么？

Ethan:

是的，这是一个很好的问题，蒂姆。我想对我来说，这种转变不是我的功劳。坦率地说，我认为有很多人对那次讨论作出了贡献，现在仍然如此。我认为，当我第一次在*上发表[那篇文章时，人们对响应式设计非常兴奋，坦率地说，我仍然被那篇文章所淹没。我赶上了出版的最后期限，我以为那会是它的结束。人们真的接受了这个想法。他们开始重新设计他们的网站，使它们更具响应性。我开始看到代理机构积极响应，然后将解决方案带给他们的客户，最终我认为这只是一点一滴地往上爬。](http://alistapart.com/article/responsive-web-design)*

 *同样的事情发生在 2000 年代中期，当时我们真正开始为 CSS 而不是表格驱动的布局争论。你会看到这种讨论发生在社区的边缘，来自独立设计师和开发者，然后它开始得到一些主流的牵引。我要说的是，在 2010 年和 2011 年，我和一个更大的团队一起参与了《波士顿环球报》的重新设计。我认为在那个时候，我们有一个客户——他不是一个小出版商——对这种快速响应的想法非常兴奋，他们认为这是他们的业务需要。

当我们开始直播时，他们开始谈论他们从中看到的所有业务优势，并发布这些指标和数据。这让媒体和出版界的许多人刮目相看，这真的很令人兴奋。我认为这是整个行业都在重复的一种模式:当一家公司开始响应，并开始谈论他们在移动设备上看到的所有提升，而不是他们的旧桌面网站，或其他什么，这使得该领域的其他人注意到并对响应变得更加兴奋。* 

 *Tim:

Responsive 也给设计者和开发内容的人带来了更多的责任。我知道人们对 responsive 感到有点困惑的一点是，你向不同的受众呈现完全相同的信息。你能谈谈这个吗？

Ethan:

当然可以。至少对我来说，我的意思是，响应式设计就像——说到底，这是一种为尽可能多的设备和浏览器设计的廉价方式。这是一个灵活的布局。它没有理想的形状，所以默认情况下，这种布局会在各种不同的屏幕尺寸上扩展和收缩。这是基础。我认为，除此之外，你还可以寻找机会，进一步提升体验。我之前提到过我的同事兼朋友 Karen McGrane，她一直在谈论自适应内容的想法，根据业务或战略需求寻找机会，根据特定的上下文线索定制信息的呈现。

当她说上下文时，她不一定是在说他们正在使用的那种*设备*。更重要的是，我们如何根据我们对这个人的实际了解来调整这些信息——不依赖于设备。我认为这就是事情变得真正强大的地方。如果你能拥有这种灵活的响应基础，然后以对用户有意义的方式定制信息，不管他们手边碰巧有什么设备，我认为这就是事情变得真正强大的地方。当我们开始认为不同的设备需要不同的信息时，事情就变得有点模糊了。

Tim:

我想很快地谈谈这个问题，因为在我的工作中，我发现最难向利益相关者和设计师解释的一件事是，你真的不应该试图根据屏幕的宽度来呈现不同的内容。你想谈谈你是如何找到这些上下文线索的吗，或者你真正在考虑的向用户传递定制信息的方式是什么？

Ethan [12:08]:

是啊，当然。我的意思是，我可以以我自己的经验来说，当我参与讨论我们需要如何根据屏幕的大小来重新思考我们的演示时，这通常是一个非常注重桌面的设计过程的产物，或者至少当我遇到它时。进行这些讨论，好吧，这个信息图或这个表格不一定适合小屏幕，所以也许我们需要考虑如何调整它，使之适合移动设备。我认为一个更有效的方法是实际上——在某些情况下更难——但你知道，采用一种更注重移动的思维方式，并构建真正将东西放到小屏幕上的东西，首先考虑小屏幕的需求，然后从那里增强。

凯伦和我有一个[响应式设计播客](https://responsivewebdesign.com/podcast/)，这是我们从许多不同的公司听到的重复内容。维珍美国可能是我最喜欢的例子，他们基本上说他们从来没有在维珍美国展示过他们的客户——设计合作伙伴从来没有在维珍美国展示过他们的客户静态比较。基本上，从他们第一次见面开始，他们就有了一个响应式的原型——在非常早期的阶段，但他们有了一个每个人都可以在不同的浏览器、不同的设备上加载的原型。这个想法真的很有变革性。

如果你真的能把一个设计交到某人手中，我认为这就消除了很多关于我们如何开始考虑移动的模糊性，因为它真的把它直接交到了他们手中。只要有可能，我认为尽早进入原型是一种非常有用的方式，但这是一个很大的挑战——特别是因为我们的许多设计工具都在桌面上。

David:

这是非常重要的一点，当谈到移动平台上缺少的东西时——我正在和 YouTube Live 上的 *you* 交谈。那是今天，但我要告诉你，我注意到的一件事是，它最终成为一个非常政治化的讨论。人们为<q>的政治斗争，我需要在这个网页上有这个房地产，</q>，这是他们从他们的网页角度思考的原因之一。<q>我的部门必须出现在这个网页的这个区域。</q>

老实说，当我的公司在做这些网站转换时，我们能够为平板电脑平台和移动平台提供的最有价值的东西之一是，我们绕过了那场讨论，我们能够说，<q>好吧，移动受众不需要所有这些东西。他们只是需要这个。</q>

我们能够只放入用户真正想在页面上看到的东西。绕过这种讨论的政治实际上有助于提供更优化的体验——不仅仅是因为这些体验是为设备量身定制的，还因为这些体验是为用户实际需要和想要与之交互的东西量身定制的，而不是谁必须在哪个部分以及屏幕上有多少像素的政治。

这些讨论时有发生，当我们开始关注响应式设计时，意识到这些讨论，它将更多的讨论带到了表面。这让人们更容易想到它。这让人们意识到，就我们的部门和政策而言，我们这样做不是出于政治动机。这是我们为用户和用户需要体验的东西而做的事情。
