# 丰富的标点符号:如何做到这一点，为什么你应该麻烦

> 原文:[https://www.sitepoint.com/typographic-punctuation-web/](https://www.sitepoint.com/typographic-punctuation-web/)

今天的网络出版技术已经从它的美国 ASCII 起源走了很长的路——如今，它古老的印刷出版表亲享受了四个多世纪的印刷标点符号完全支持在我们的网络文档中使用。

漂亮的排版确实需要服务器端的一些额外配置和文档作者的更多关注——但是在本文中，我希望让你相信这是值得花时间的。丰富的标点符号是一种工具，如果使用得当，它会让你的网站文本变得生动起来！

##### 考虑“适当的”标点符号

环顾网络，很明显，使用和传递正确的标点符号是许多其他圆滑的专业网站作者的基本问题，包括直到最近才出现的 SitePoint(咳咳，迟到总比不到好)。尽管 QWERTY 键盘和以前一样有很多限制，但是 Unicode 的出现和提供广泛支持的浏览器意味着现在没有理由让网上内容充斥着不好看的直引号和连字符作为文本破折号。大多数网络浏览器和许多出版工具已经支持丰富的标点符号。只是入门的问题！

有无数的风格指南和标点符号的地方变化。一个高质量的风格指南，甚至是一个在网上找到的，将为你扩充标点符号库提供一个良好的起点。阅读印刷杂志或书籍也可以提供很多灵感，并举例说明丰富的标点符号的用法。

##### 产生非键盘字符

许多标点符号的共同问题是它们在普通的 QWERTY 键盘上根本不存在。这些键盘是由计算机程序员设计的，甚至当印刷工开始与计算机打交道时，键盘也没有得到真正的革新。

然而，最常见的误解是围绕着键盘上的几个常用字符*出现的。这些是直引号或哑引号，称为字形(")、直撇号(')和连字符或减号(-)。虽然这些字符看起来像 QWERTY 键盘上的标点符号，但它们最初只用于计算机编程，而不是文本文档。虽然一些文本编辑软件，如 WordPress Rich Text 和 OpenOffice.Org，试图将计算机编程符号转换为标点符号，但这通常会造成更多的错误。*

为了产生正确的“卷曲”引号、撇号、省略号、连字符以及键盘上没有的 en 和 em 破折号，作者需要做一些工作。我在下面列出了几个选项:

***利用键盘快捷键***

在许多操作系统中，用于产生许多 Unicode 字符的键盘快捷键是内置的。

例如，在运行 OS X 的 Mac 上，这些快捷键采取的形式是 **Option + <键>** 和 **Shift + Option + <键>** 。68kMLA wiki 包含了所有这些键盘快捷键的[列表；这里有一个例子:](http://68kmla.net/wiki/List_of_keyboard_shortcuts)

| 性格；角色；字母 | 键盘快捷键(OS X) |
| --- | --- |
| … | option+； |
| â€“ | 选项+– |
| – | option+Shift+– |
| ' | 选项+ ] |
| ' | Option + Shift + ] |
| " | 选项+ [ |
| " | Option + Shift + [ |

在 Linux 的许多发行版中，上面列出的键盘快捷键也是有效的；把 **ALT** 换成**选项**键即可。

Windows 用户也可以使用键盘快捷键输入 Unicode，但是有一些注意事项。首先，它要求你知道你想要输入的字符的 Unicode 编码(不像在 Mac 上那样容易记忆，而且更耗时)。第二，这取决于你打开的应用程序和你的语言设置。

在 Windows 中输入 Unicode 字符可以通过以下按键序列来完成:

1.  按住 **ALT** 键。

3.  在数字键盘上键入 0。

5.  再次在数字键盘上键入字符的 Unicode 编号。

7.  松开 **ALT** 键。

这里是输入常用字符的关键序列，我们在本文中重点讨论了这些字符。我在圆括号中标记了 **ALT** 按钮，表示在输入它后面的字符时应该按住它:

这种方法有一些替代方案——例如，许多微软应用程序也支持将数字转换成 Unicode 字符的**ALT**–**X**键盘快捷键。然而，由于这种方法要求作者知道字符的十六进制数*，并且是特定于微软应用程序的，我认为它不是普遍有用的，但是它可能适合您的工作流程。(事实上，这种技术在 OS X 也有效，通过按住**选项**键，但是除非你能说流利的十六进制，否则我建议你坚持使用常规快捷键)。*

***创建自己的键盘快捷键***

另一个很好的方法是手动给键盘上的键分配字符。例如，您可能想要创建一个键盘快捷键，当您按住右侧的 **Alt/Option** 并按下**时，该快捷键会产生一个省略号。**键。

根据操作系统的不同，修改键盘布局来赋予按键不同的含义通常很容易。请参考您的特定操作系统的文档或在网上搜索教程，了解如何实现这一点的完整详细信息。

起初，当你不得不停下来思考必要的组合键时，你可能会发现使用专用键很慢，但随着时间的推移，快速完成这些按键将成为你的第二天性。

***使用第三方应用***

在浏览器之外，还存在许多共享软件和免费软件应用程序来帮助输入 Unicode 字符。我在 Windows 上尝试过的一个轻量级(免费)实用程序叫做 [UnicodeInput](http://www.fileformat.info/tool/unicodeinput/) 。它允许使用可定制的按键输入 Unicode 字符。

***使用备忘单***

如果您的数据输入受限于 web 浏览器，并且不愿意(或无法)安装诸如 UnicdeInput 之类的应用程序，您可以考虑使用*备忘单*——包含所需字符的插件参考，提供简单的复制粘贴工作流程。

与拥有专用密钥相比，这显然是一种更耗时的方法。一个更方便的小抄方法是让网站作者利用浏览器的侧面板。Opera、Firefox 和 Internet Explorer 已经有许多浏览器面板: [Edicode](http://www.guntherkrauss.de/computer/xml/daten/edicode.html) 通过拉丁块和通用标点块提供了对丰富标点的快速访问。

在 Mac 上，这种类型的备忘单内置于操作系统中——但前提是你的应用程序是原生的 Cocoa 应用程序。您可以通过单击“编辑”菜单来确定是否是这种情况。如果你有一个特殊的字符菜单选项(键盘快捷键:**选项** + **命令** + **T** )，那么你就成功了！

当然，对于一次性查找，有很多在线参考资料，包括[这个列出了前 65，536 个 Unicode 字符](http://unicode.coeurlumiere.com/)和代表它们的数字。

**访问更多字符**

无论您采用哪种方法来访问非键盘字符，请记住，除了我们讨论过的最常用的字符和标点符号之外，还有更多字符和标点符号可供使用。有专门的符号，如笑脸、摄氏度、分数、罗马数字、数学运算符、质数和负号。要全面了解可用符号，请浏览 Unicode 标准中的[通用标点符号块](http://www.unicode.org/charts/symbols.html)和其他[拉丁语言块](http://www.unicode.org/charts/index.html)。

也就是说，某些印刷标点符号和字符的可用性仅限于广泛支持 Unicode 标准的字体。每当使用非典型字符时，设计者必须总是检查所使用的字体是否支持期望的字符。级联样式表(CSS)可以指定字体堆栈，从而确保跨平台和设备的广泛支持。字体堆栈应包含通用后备类型以及来自不同操作系统的字体。

但是等等——这还不是全部！

##### 用 Unicode 传送 Web 文档

不幸的是，以闪电般的速度访问非键盘字符和使用组合键并不是故事的结尾。您的 web 文档不仅要被编码，还要被标识为 Unicode 文档。

默认情况下，大多数 web 浏览器仍然将文档编码读取为 ASCII，这是旧的文档编码标准。必须通过在服务器对浏览器的文档请求的响应中指定 HTTP 头使用的编码来覆盖此默认设置。有关如何实现这一点的完整详细信息，请参考特定服务器的文档。还有一篇由 Tommy Olsson 撰写的关于网络字符编码的优秀文章——事实上，把那篇文章作为本文的配套阅读会很有帮助。

UTF-8 编码将满足所有基于拉丁语的语言，因为它将创建小文件并支持整个 Unicode 标准。HTML 文件和所有种类的 XML 文件，包括 Atom web 提要、站点地图索引和 XHTML，都可以使用 UTF-8 编码。

Apache 服务器只需要多一点说服就可以像 UTF-8 一样提供 web 文档。要实现服务器范围的更改，可以将以下内容添加到 web 根目录下名为`.htaccess`的配置文件中:

```
DefaultEncoding: UTF-8

AddEncoding .atom .htm .html .xht .xhtml .xml UTF-8
```

这个在 web 服务器上常见的以空格分隔的标准格式扩展列表将使用 UTF-8 编码。当使用服务器端编码时，也可以在每个文档的基础上实现，使用下面的 PHP 代码片段:

```
<?php header("Content-Type: text/html;charset=UTF-8"); ?>
```

要特别注意确保这些代码的正确性，因为 HTTP 头是区分大小写的！

其他服务器端语言有非常相似的修改 HTTP 头的方法。XML 文件需要 XML 声明中的编码信息(`<?xml [...] encoding="UTF-8"?>`)。虽然这可能不会影响实际的文档解析，但是在 HTML 文件中包含编码信息也是一个好习惯。

下面的 HTML 代码片段必须是`<head/>`元素的第一个子元素才能生效。同样，HTTP 头是区分大小写的！

```
<meta http-equiv="Content-Type" content="text/html;charset=UTF-8"/>
```

出于向后兼容的原因，XHTML 文件同时使用 XML 声明指定的编码和 HTML 方法。如上所述，如果服务器未能传递适当的 HTTP 头，这只是一种后备机制。

***逃避解析问题***

如上所述，转义字符是一种将字符作为标记中的代码序列写出的方法，在 HTML 和 XML 中执行相同的功能。当显示时，转义字符将显示为实际字符。

在 web 文档中包含实际的字符字形本身，而不是对其进行转义，对于非典型字符来说是一种更明智的方法——只是为了安全起见。此外，通过适当的文档编码和交付，转义所有内容的做法只会导致文档变得比实际大得多。

也就是说，*有*四个特殊的 Unicode 标准字符，它们必须被转义。下表中显示的这些字符在 HTML 和 XML 标记语言中具有特殊的含义，对它们进行转义可以避免潜在的解析和呈现问题。

| 性格；角色；字母 | 标志 | 换码顺序 |
| --- | --- | --- |
| 不到 | < | < |
| 大于 | > | > |
| 直接引用 | " | " |
| &的记号名称 | & | & |

***协商搜索引擎和不兼容设备***

在处理任何 web 文档时，作者几乎在开发的每个阶段都要做出一个至关重要的决定:倾向于读者还是搜索引擎？

正如我们所看到的，更丰富的标点符号无疑会给你网站的访问者带来更好的阅读体验。但有一个不幸的缺点——在某种程度上，它会让搜索引擎更难理解文档中的非典型标点符号。我们可以满怀希望地生活，在不太遥远的将来，丰富标点符号的广泛使用将增加搜索引擎对文档的理解，而不是承担减少理解的风险。

Unicode 不兼容的设备和搜索引擎可能需要您的页面的替代版本，其中所有内容都自动映射到 ASCII/ANSI。然而，随着手持设备和搜索引擎的智能化，这种做法变得越来越多余。通过 Atom web feeds 提供联合的网站可以很容易地解决这个问题。

例如，在构建提要时，发布工具应该用几乎等价的键盘字符对替换大海捞针中的 Unicode 字符。移动或手持版本的文档也是如此，因为这些设备往往不完全支持 Unicode 标准。因此，连字符(U+2010)将被替换为连字符-减号(U+2D)，短划线(U+2013)将被替换为两个连字符-减号(U+2D·2D)，卷曲撇号(U+2019)将被替换为撇号(U+27)，以此类推。一旦处理了这些字符，提要将只包含 ASCII/ANSI 字母和符号。

与网络版同时提供网络提要，但只使用基本的标点。然后，搜索引擎会找到两种已发布格式之间的`<link/>`,并将它们视为一个文档。作为一个副作用，当读者点击进入文档的网络版本以获得更好的阅读体验时，来自网络订阅源的点击率也可能增加。

##### 结论

全世界的网页设计者都遇到了在线内容中的字符问题——通常让直引号占据主导地位，有损于优秀的网页设计。正如我们所看到的，在今天这个时代，实际上没有什么可以阻止我们确保我们的排版以最好的方式增强我们圆滑的内容呈现。现在是设计师呈现我们文本的方式出现转折点的时候了，因为我们在这个问题上有了选择；事实上，你会注意到我们在 SitePoint 终于将这种说教付诸实践。三周前，我们冒险接受了当今技术为我们的文章提供的内容——我们再也不会回头了！

设计师在使用丰富的标点符号时遇到的第一个问题是输入法的局限性:键盘。但是，正如我们已经看到的，有几个体面的和非常琐碎的解决方案来绕过这个限制。第二个困扰我们的问题是文件递送。有必要通过微小的服务器端更改来仔细声明文档中使用的编码。

有了这些障碍，就没有理由保留标点符号了——所以拥抱 Unicode，给你的读者更丰富的排版体验吧！
