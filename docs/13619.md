# 微软会包容开源吗？

> 原文：<https://www.sitepoint.com/will-microsoft-subsume-open-source/>

所以我打算写一篇漂亮的、简短的、微软粉丝风格的帖子，告诉大家《ASP.NET 地图集》的生产版将于 2006 年正式发布。

但是，不幸的是，今天成为了非常悲伤的一天。网络世界。凯文·唐斯，这个非常有用的 NDoc 项目背后的智囊和打手，正在放弃这个项目并从社区退休。我可以列举他的理由，但是[凯文自己的声明](http://www.charliedigital.com/PermaLink,guid,95b2ab68-ba92-413a-b758-2783cde5df9c.aspx)比我能提供的更有说服力。

这引发了一些非常有趣的问题。首先，我像许多开发人员一样，使用 NDoc 和许多其他开源项目，而不花费我一分一毫的时间。这种做法真的需要停止了，因为我们将会看到越来越多令人悲伤的公告，因为开发人员和维护人员厌倦了毫无回报的付出。关于这个问题，[潜台词](http://subtextproject.com/)团队的仁慈独裁者[保罗·哈克](http://haacked.com/)有一个特别尖锐的想法:[让 2006 年 7 月 26 日成为第一个开源日](http://haacked.com/archive/2006/07/26/TheDemiseOfNDocAndAChallengeForUsersOfOpenSourceSoftware.aspx)。世界各地的开发者选择一个项目，然后捐赠代码、文档或者至少是现金。

第二个问题是新的、面向社区的微软明显产生的寒蝉效应。NET 开源项目。凯文退休的主要原因是宣布了沙堡项目——一个尚未发布的项目。仅支持. NET 2.0 的文档编译器与 NDoc 惊人地相似。好吧，最大的不同是沙堡的作者收到了微软的薪水，而且这个项目得到了微软机器的全力支持。这是一场相当不公平的竞争。

这个问题的另一个很好的例子是 [MSBuild](http://msdn2.microsoft.com/en-us/library/wea2sca5.aspx) ，一个在某些方面直接针对 [NAnt](http://www.google.com/url?sa=U&start=1&q=http://nant.sourceforge.net/&e=9797) 的工具。两者都是 Xml 驱动的构建工具，允许像受控构建环境、多种构建配置和持续集成这样的巧妙技巧。但是有一个得到了框架控制器的官方支持，并且集成了 Visual Studio。为了项目的长期使用，你会赌哪一个？接下来，你会为你的 ORM 赌上 [LINQ](http://msdn.microsoft.com/data/ref/linq/) 还是[伊比利亚](http://www.google.com/url?sa=U&start=1&q=http://www.nhibernate.org/&e=9797)吗？

不幸的是，我怀疑这一声明只是冰山一角。期待微软提供更多的免费工具和扩展。可能具有无可争议的高质量的工具。但是这些工具很可能会刺进你当地的心脏。NET 开源项目。Atlas 就是一个很好的例子——它在任何 ASP.NET 2.0 Ajax 框架被创建之前就有效地扼杀了它们。剩余选项的主要卖点是 ASP.NET 1.1 的兼容性。虽然微软这些年来发生了很大的变化，但似乎仍然存在很多拥抱、扩张和消灭的心态。
