## 采访记录

凯文:2009 年 12 月 4 日。Twitter 的 API Lead 分享了他对我们用来构建网络的技术的看法。这是 SitePoint 播客#39:亚历克斯·佩恩的网络未来。

**Kevin:** 我是 Kevin Yank，在珀斯参加 SitePoint 播客，参加[网络会议](http://www.edgeoftheweb.org.au/)今天和我在一起的是 Alex Payne，他是开发者之一，实际上是 Twitter API 的首席开发者。是这样吗？

亚历克斯:对，就是我。

凯文:我想很多人，当他们见到你时，他们会马上想到 [Twitter API](http://apiwiki.twitter.com/) 和所有可能的事情；从你的角度来看，Twitter 的 API 对 Twitter 意味着什么？

**Alex:** 对我们来说，这意味着专注于我们最擅长或至少是我们最想做的事情的灵活性。这意味着，在过去的几年里，当我们一直专注于扩展服务并试图添加真正高价值的功能时，比如我们刚刚推出的列表，人们多年来一直要求我们可以深入了解用户体验和类似功能的性能，以及稍微狭窄或稍微利基的东西，开发人员可以利用 API，为他们自己和他们的用户构建这些工具。

凯文:因此，从某种意义上来说，用户可以通过实现一些东西来告诉你他们想如何使用这项服务。

亚历克斯:是的，当然。

凯文:所以[你的演讲](http://www.slideshare.net/al3x/mind-the-tools)更笼统。这实际上是关于我们在网络技术上的发展和未来。你提到的一件真正引起我兴趣的事情是，越来越多的像 Twitter 这样的 web 服务可以并且应该探索 API 优先的方法，他们可以构建自己的 API，然后“吃自己的狗粮”，你说，在构建他们的 web 用户界面时。你认为这是 Twitter 采用的方法吗？你能想到这种方法奏效的其他例子吗？

Alex: 嗯，我听 Flickr 团队的人说，这就是他们的架构。这不是我们在 Twitter 采取的方法。我们的 API 有点脱离面向用户的应用程序，如果我们可以从头再来，我想我宁愿先从 API 开始，部分原因是这是我们的主要流量来源，部分原因是处理 API 问题、设计问题、性能问题真的会支持您对系统的思考。

凯文:我是一名技术人员，你认为 Twitter 的技术组合是怎样的？我想的不仅仅是你们使用的网络语言，我的意思是，你们都是 Twitter 的 Mac 用户吗？

亚历克斯:大部分情况下，是的。我认为现在每个人都在苹果硬件上运行，我们所有的工程师。我们有几个人每天都开机进入 Linux，但大多数情况下，我们都是 Mac 用户。

凯文:酷。你在你的演讲中提到，你曾经在 PowerPC 之前的 Mac 上工作，所以你一定经历过作为 Mac 用户的糟糕日子。

亚历克斯:是的。我认为那是 575 年前的一台 Performa，此后不久，我得到了我的第一台 Wintel 硬件，进入了 Linux 领域，直到 OS X 改进到人们可以实际使用它。

凯文:好的，所以你确实来回移动了？

亚历克斯:是的，我很高兴我这么做了。我的意思是，我认为像许多 Mac 人一样，我们都有几年的信仰危机，然后去学习 Linux，这最终成为一种职业资产，你知道吗？

Kevin: 你今天使用 Mac 的原因与你当时使用 Mac 的原因有所不同吗？

亚历克斯:我的意思是，那时候它碰巧就在房子周围。如今，更重要的是，没有其他人拥有那种开箱即用、一切都可以正常工作的体验，也没有其他人生产那种质量的硬件，这令人困惑。就像有些日子，我实际上更喜欢在 Linux 上运行，这样我就可以做一些奇怪的事情，比如运行一个平铺窗口管理器，但环顾非苹果硬件，没有人能做出好东西，即使你想花 2500 美元，也很难买到一台好机器。

**Kevin:** 是的，具有讽刺意味的是，我们的设计师是 SitePoint 办公室最后一个拒绝购买 Mac 的人之一，最近，他刚刚出去寻找一台笔记本电脑，根本没有打算购买 Mac，但他找不到质量好的 PC 笔记本电脑，因此最终购买了 Mac。

亚历克斯:是的，这真的非常非常令人惊讶。我不知道世界上的戴尔和索尼在想什么，但这很奇怪。

**Kevin:** 所以，当您谈到当今混合使用的各种 web 编程语言时，您指出的一个区别是，您区分了在 Web 上使用的通用语言与专为 Web 设计的语言，如 PHP 和 JavaScript，您提到这些专为 Web 设计的语言有寿命较短的趋势。你认为这是为什么？

亚历克斯:嗯，不一定会缩短寿命，但至少他们会受到严厉的批评。多年来，人们一直抱怨 PHP 鼓励不良的开发习惯，是一种非常随意设计的语言，人们也一直抱怨 JavaScript 的对象模型与大多数其他编程语言非常不同，有一些奇怪的东西等等。

凯文:说到奇怪的事，我们这里有一些噪音，但请继续。

Alex: 所以我认为部分原因是开发更多通用语言的人不得不满足各种不同领域的用户。他们有学术用户，科学用户，构建桌面软件的人，构建服务器站点软件的人，我认为这是一种完善的语言，它迫使像吉多·范·罗苏姆或 Matz 这样的人超越他们最初的语言概念，并找出我们如何改进 API，如何改进核心库，诸如此类的事情。如果一种语言的目标是 Web，并且它的受众更窄，我认为你永远不会看到程序员和不同学科之间的那种交叉交流。当你第一次开始在客户端使用 PHP，或者，抱歉，在命令行脚本之类的东西上，PHP 变得更好了。突然，这种语言变得更加强大了，现在我认为 JavaScript 也正在经历这个过程，因为人们正在使用 Rhino JavaScript 虚拟机之类的东西来进行服务器端编码。看起来他们正在重新思考语言，并对其进行改进。

凯文:所以用例越多，语言就越强大。

亚历克斯:似乎是这样，是的。

**凯文:**关于这一点，这也是你在演讲中提到的，在网络上，似乎最好的技术并不总是赢家，这也是马克·皮尔格林本周在查看关于`<img>`标签的旧列表帖子以及它如何最终成为 HTML 的选择时提出的问题。这是网络上的一个问题吗，仅仅在网络环境中发展的东西并不总是技术上更好的解决方案，这是变得更糟还是更好？

亚历克斯:我不知道。我反复思考这个问题。我的意思是，这是技术领域的一个古老的争论，可以追溯到盖伊·斯蒂尔的“越差越好”之类的东西。我认为它在网络上有用的地方是，我们正在经历这样一个不断进化的时期，有点像蜕皮，如果我们今天使用的技术有一点点缺陷，就更容易迁移到下一套技术。如果我们有完美的解决方案，或者甚至覆盖 80%或 90%情况的解决方案，那么每三年或四年重新发明网络，重新发明我们如何做这一切将会更加困难。

**Kevin:** 你在演讲中提到的另一个问题是，我们正在进入这样一个时代，越来越多的人不再学习语言，而是学习建立在语言之上的框架，以及随之而来的成本。作为一名程序员，你个人如何防范这些成本？你如何对冲你的赌注？你对此有什么办法？

Alex: 在过去的几年里，我尝试做的事情是，我的意思是，我花了很多时间在 Ruby on Rails 上工作，这在很大程度上是非常高效的，我们刚刚在 Twitter 的框架中完成了大量的工作，但在外面，我花了很多时间学习其他编程语言，试图看看 web 空间以外的问题，特别是一些低级别的问题，以便取得平衡。你有非常高层次的面向框架的思维，你只关注“我如何在尽可能短的时间内完成这些 Web 业务目标？”这是一种思考方式，但接下来要说的是，我将如何实现一种完整的编程语言，甚至是在我曾经工作过的安全领域中寻找问题。它只是给你一个非常不同的视角。

作为一个 API 开发者，一个 API 的创造者，你必须有一个独特的视角，因为使用这么多不同语言的这么多不同工具的人都在使用这个 API。所以你必须构建一些能自然工作的东西，不管消费者是 Rails 开发人员还是 PHP 开发人员？

亚历克斯:嗯，我的意思是，谢天谢地，因为 Twitter 或多或少是一个 RESTful API，所以大多数语言都有处理 RESTful web 服务的约定，所以我们可以从所有这些想法中受益。

Kevin: 你刚刚为 O'Reilly 完成了一本关于 Scala 语言的书，向我推销 Scala 语言。

Alex: 好的，简单来说 Scala:它的优势是你有面向对象的编程，每个人都已经知道并习惯了，你有函数式编程，很多人都接触过，但一般来说，他们发现你学的函数式语言中的语法或库是 Lisp 或 Occam 或 Haskell 或其他什么有点缺乏。Scala 的好处在于，因为它构建在 JVM 之上，并且可以与所有这些库完全互操作，所以您可以立即利用 10、12 年的 Java 历史。所以我们可能认为 Java 可能不再是最性感的语言，但多年来已经编写了大量的库，十年来人们一直在用 Java 解决非常困难的问题，在 Scala 中，你可以用一种漂亮的语言工作，看起来有点像 Ruby 或 Python 或更多开发人员熟悉的东西，但你仍然可以受益于 JVM 的所有优点。

凯文:这听起来就像是让 PHP 如此成功的一些因素，当 PHP 变得流行起来的时候，他们加入了面向对象的特性，但这些特性是可选的。同样的，Scala 提供了函数式编程的东西，但是你不必放弃你所有的技能和历史。

亚历克斯:对，没错。我的意思是，我知道很多人对 Erlang 感兴趣，认为它是一种第一函数式语言，Erlang 很棒，尤其是对于一种网络服务问题，但它有更多的限制。一切都是不可改变的。所以你给一个变量赋值一次，就不能再赋值了。这是一种与我们在许多面向对象语言中习惯的思维方式非常不同的思维方式，它非常特别地具有这种奇怪的源自 Prolog 的语法以及所有其他的内容。对于 Scala 来说，它更宽容一些。这种语言给程序员留下了更多的空间，让他们决定是想使用对象范式还是函数范式，是想用更 Erlang 风格的 actor 模型解决并发问题，还是想回到传统的 Java 线程等等。

凯文:因为它在 JVM 上，你可以自由地用 Java 构建一个特定的模块，然后使用 Scala 之类的东西来访问它。

亚历克斯:对，没错。就像如果你在像 Hadoop 这样的 Java 堆栈中工作，你已经得到了 Hadoop 的所有能力。你可以从 Scala 无缝调用它。诸如此类的事情可以节省大量时间。

Kevin: 正如你所说，Scala 是在 JVM 上运行的，你对 JVM 作为一个平台有什么看法？很明显，这背后有许多年的历史，它是一个坚实、强大的基础，但我不知道，我对 Sun 没有以前那么有信心了。你觉得 JVM 已经释放了足够多的 Sun，如果最坏的情况发生，Sun 崩溃了，JVM 会以令人满意的方式继续吗？

亚历克斯:我想是的。我的意思是，几个月前我参加了 JVM 语言峰会，无论如何我都不是 JVM 专家，但似乎在 Sun 和 IBM 以及所有依赖 Java 生态系统的人之间，有人将接过 JVM 的衣钵。这里有太多优秀的工程设计，现在还有 OpenJDK 项目以及其他各种 JVM 发行版。所以我认为有很多选择。

**Kevin:** 除了函数式语言之外，你对未来的另一个“黑暗中的刺”预测是，面向服务的架构将会有自己的一席之地，我们将会变得足够舒适地为 Web 构建可重用的软件组件，我们将开始能够以一致和标准的方式将它们连接在一起。这似乎就像我们经常听到的函数式语言一样，“哦，明年它们真的会流行起来。”这些干净的软件组件的想法，特别是在网络中，似乎总是在一年后才出现，然后当事情稳定下来时，手工编码者带来了一些新的和令人兴奋的东西。有一段时间，他们的 WYSIWYG 看起来几乎是一种合理的工作方式，然后突然一种 CSS 布局流行起来，不，不，回到手工编码。

亚历克斯:是的。

Kevin: 是否会出现另一波手工编码的疯狂浪潮，将 SOA 冲走？

亚历克斯:这完全有可能，是的。事实上，我认为在某些方面，JVM 语言的东西可能会刺激这一点。我已经看到越来越多我在 Twitter 上关注的程序员开始使用 Clojure，人们正在 Clojure 中构建这些微型 web 框架，这是人们在过去几年中在 Ruby 社区中所做的事情的一种密切反映，整个方法就像每个项目都是非常轻量级的，非常定制的， 你只需要把这个薄薄的单板放在其他库的上面，然后把一些东西拼凑起来，Lisp 一直鼓励这种非常敏捷的开发方法，而不是这种大型的企业级 SOA 之类的东西。 因此，如果越来越多的程序员加入这种潮流，那么我认为你是绝对正确的。我们可能会看到另一波“好吧，让我们一起破解它，直到它工作，然后继续下一个有趣的问题。”这一直是黑客文化的基本部分，对吗？

凯文:我还有最后一个问题，这个问题与网络服务的概念有关，这些服务似乎正在被构建，特别是在像 Twitter 这样的非网络用户界面的情况下——正如你所说，你发现向你妈妈解释 Twitter 是如何工作的更容易，因为它是一个桌面客户端，看起来像即时消息而不是网页。我们是否会到达这样一个阶段:web 服务不需要 web UI，它是一个纯粹的 web 服务，主要用户界面不在 web 浏览器中？我知道 Cameron Adams，他是 Google Wave 背后的设计师之一，大约一个月前他在 Web Directions South 上发表了演讲，他说作为 Wave web 客户端的前端设计师，他面临的最有趣的挑战是，他需要做出足够好的东西来吸引人们使用 Wave 服务，但又不能太好，以至于压制了第三方非 Web 客户端的市场。

我们能不能说一下网络浏览器对网络服务来说不是最重要的东西？

亚历克斯:是的，我当然希望如此。看起来网上越来越多的目的地实际上与网络无关。网络只是附带的传递工具，但我们不再传递超文本。比如，如果你想到在美国非常受欢迎的 Hulu 视频服务，我的意思是，它正在取代整个人口统计中的电视。这真的和上网没有任何关系。我的意思是，你可以从一个节目点击到另一个节目，但我相信他们很快就推出了基于 Adobe AIR 的应用程序，可能是在他们推出后的几个月或一年，对我的大多数朋友来说，这就是他们的电视。它运行这个 Adobe AIR 应用程序，可以与 Hulu 对话并传输视频。网站基本上就是你在上面注册，然后他们把你指向 Adobe AIR 应用，这是与服务互动的更好方式。

凯文:我不得不说，在澳大利亚，我们没有 Hulu，我们都非常嫉妒。

亚历克斯:你们有 Spotify 吗？

凯文:不，不，我们这里连那个都没有。

亚历克斯:你没有 Spotify？是的，所以在美国，这就像是一种交换，我们有 Hulu，但我们没有 Spotify，但你们这些家伙正在被骗。

凯文:我们什么也没得到。

亚历克斯:但是，我认为这绝对会成为一种趋势，老实说，展望未来解决 Twitter 之外的问题，我职业生涯的大部分时间都在浏览器中工作，稍微摆脱这种模式会非常非常好。我喜欢网络服务，我喜欢开发 API 之类的东西，但它看起来像桌面上的移动富客户端，所有这些东西都令人兴奋。那里有很多东西要学。

**凯文:**亚历克斯在推特上是[@ al3x](http://twitter.com/al3x)感谢你抽出时间，亚历克斯。

亚历克斯:非常感谢。
