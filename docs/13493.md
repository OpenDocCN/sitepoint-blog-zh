# 管理客户期望:用户接受文档

> 原文：<https://www.sitepoint.com/managing-client-expectations-user-acceptance-documentation/>

Sitepoint 的商业和法律论坛上的一个常见问题是，“*我已经兑现了我的承诺，但我的客户一直要求一些小的调整、支持和建议。我如何确定一个项目何时结束，我如何向我的客户传达这一点？*

大多数 web 开发人员都经历过“永无止境的项目”,能够同情处于这种困境的开发人员。在出色的服务和需要为你愿意为客户做的事情设定一些界限和限制之间取得平衡并不容易，尤其是当你刚刚起步并且需要 100%留住客户的时候，这一点尤其困难！

好消息是有一个简单易行的方法来防止这个问题:用户接受文档。

用户接受就像它听起来的那样——来自你的客户的确认，你已经完成了交易中你的部分(如原始合同中所述),项目可以结束了。用户验收在软件行业中很常见，通常包括用户验收(UA)文档，一旦工作完成
令客户满意，客户就签署并返还该文档。除了围绕项目的完成设置一些法律结构和责任外，UA 文档还向客户传达了一个清晰的信息，即您已经完成了交易。这也使客户明确地接受工作，这将使他们在要求正在进行的免费更改时三思。UA 文档也是保修期或下一阶段开始的明确起点。

有效的普遍获得文档可以只有一页长，并且有一个单独的段落，包含如下语言:

> 签署人同意 ABC WebDev，Inc .已满足 2006 年 1 月 1 日签署的协议中规定的要求
> ,项目已经完成。"

您可以扩展 UA 文档，以解释正在进行的工作的收费率(或其他条款)或保修期的长度，保修期最好从签署 UA 文档时开始。您想要阐明的任何其他要点也应包含在 UA 文档中。

为了使普遍获得文件有效，必须考虑以下因素:

*   客户需要理解 UA 文档的含义。UA 的概念应该在你的原始合同中引入，客户应该知道它如何适应整个项目流程。明确表示当你完成工作后，你将发送 UA 文档给他们签字。
*   There has to be some motivation for the client to sign the document, and some clarity about what happens if they don’t. A good way to address this is to add language to the original contract and the UA document which dictates fair terms for the end of the project, such as:

    1.从客户收到 UA 文档的那一天开始，他们有 30 天的时间来报告交付产品中的任何功能差距或其他缺陷(您承诺要做但没有做的事情)。明确表示你将尽快免费修复任何此类缺陷。

    2.xxx 天的保修期将在 UA 文件签署后开始。bug(但不是偏离原协议的实质性改动！)这段时间会免费修复。

    3.如果未在 30 天内签署 UA 文档，合同将自动终止，保修也将终止。在项目终止后或保修期外发现的任何增强、错误或更改都将产生新的账单。

使用上述简单的条款，普通客户会希望签署文件，以确保他们不会使他们的保修期无效。拥有一份明确签署的 UA 文档会让您受益匪浅，因为您不必担心永无止境的变更请求——如果客户在 UA 之后要求变更，他们会明白他们将被收费。

显然，每个开发人员都需要定制 UA 概念，以满足他们各自项目的需求。在某些情况下，开发人员可能不会交付一个完成的网站，直到 UA 文档被签署。在其他情况下，签署的 UA 文件必须附有全额付款。无论您如何应用 UA 概念，通过在项目结束时而不是仅仅在开始时拥有正式的、签署的文档，您将为您自己和您的客户都帮了一个大忙！
