# 第 4 集的答案(“什么是‘正常’，真的？”)

> 原文：<https://www.sitepoint.com/answers-to-episode-4-whats-normal-really/>

嗯,非常成功……**不！**

显然，web 开发人员对数据库规范化并不感兴趣。(但是感谢 <cite>malikyte</cite> 和 <cite>xhtmlcoder</cite> 让这个问题没有变成一个完全的鬼城！)

然而，这是一个遗憾——好的数据设计背后有各种各样的实际原因。仅举几个例子:设计合理的表通常比非规范化的表表现得更好，规范化的数据更容易成功聚合，而且(最重要的是)设计合理的表更容易理解。

最后一个是规范化表背后的症结所在。记住——计算机不在乎我们是否写出了好的代码；当我们写出好的代码时，是为了让未来的开发者不会诅咒我们的名字。数据规范化也属于面向未来的范畴。

不管怎样:继续寻找答案。我会很简短的，我保证。

1.  令我十分惊讶的是，FDA 的营养成分数据库实际上是 3NF(每样东西都有一个主键，每条数据看起来都是原子的)。我对几个设计选择有一些异议，但它们实际上很小。当你发现开箱即用的数据如此干净时，这是非常了不起的。
2.  尽管人口统计数据设计得相当好(并且容易转换成更好的形式)，但它甚至没有达到 1NF:记录缺少主键。公开数据通常都是这样，很臭；这使得跟踪版本之间的变化变得极其困难。
3.  SEC 的文件是一个棘手的问题。它们使用 XML 方言，所以普通形式不适用。

    我认为重要的是要注意不同的格式如何改变我们生产和消费数据的方式；SEC 数据是一个很好的 XML 例子，但是试图以任何结构化的方式将这些数据推入数据库是非常困难的。

    如果我想围绕这些东西建立一个网站，我可能会使用类似 Berkeley DB for XML 这样的东西，而不是关系数据库。

4.  虽然天然气价格数据被塞进了一个 Excel 表中，更多地用于表示而不是数据管理，但实际上是在 3NF 中(如果只是因为它是预聚合数据的话)。测量数据是主键，所有列都依赖于主键(即价格是日期的函数，仅此而已)。
5.  青少年逮捕率数据就像汽油价格一样，名义上是塞进 Excel 的 3NF 数据。
