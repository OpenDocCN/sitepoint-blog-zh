# 标准——只是合理设计的一部分

> 原文：<https://www.sitepoint.com/standards-sensible-design/>

这是一段漫长的旅程，但是，嘿，我们终于到了！

慢慢地，但稳步地，人们似乎已经认识到，Web 开发不是以一种我们不确定的方式把我们不真正了解的东西放在一起，然后在事情败露之前给客户开发票。

令一些人恼火的是，付钱给我们的人也开始要求可访问性和验证以及其他我们在过去疯狂的网络开发中不必处理的事情。

没问题，我们可以处理。毕竟，我们[希望标准得到认可](http://www.webstandards.org/act/campaign/buc/)，我们已经在邮件列表、论坛和聊天频道上向人们宣传了足够长的时间。现在，让我们沉浸在我们的荣耀中，对我们的荣誉感到舒适，并且看不起那些仍然使用`<font>`并且用大写字母写他们的标签的未洗过的乌合之众…或者我们应该吗？

##### 网络标准包括无聊的旧标记

使用 Web 标准是正确的做法，但是，和任何推荐一样，没有必要跟随它们走下坡路。新开发人员往往反应过度，对事情有点过于认真。例如，对表格数据使用表格并没有什么不好。很多时候，我看到新的 Web 标准拥护者害羞地求助于邮件列表，询问他们如何将一些东西转化为 CSS，因为“表格已经过时了”作为回应，[乐于助人的人会举出例子](http://www.meyerweb.com/eric/css/discuss/examples/notable-calendar.html),这些例子旨在展示 CSS 能做什么，而不是确定什么有逻辑意义。

一个好的经验法则是:如果你有一堆与一个或多个类别相关的数据，使用表格！

但是，不要建立在你的“老学校”经验之上，添加大量的间隔 gif 和不赞成使用的属性。拥抱那些当你开始筛选 HTML 规范时不敢回头看的增强可访问性的属性和元素。有些你认为没有意义的点，因为它们不会出现在屏幕上，现在可能会派上用场。

这里一个标题，那里一个范围，一些`TH`、`TFOOT`和`axis`……你已经将单元格数据连接到它们的标题上，对于使用任何种类浏览器的每个人来说——这是 CSS，尽管它很有用，也不能为你做的。

摘下我们的 CSS 帽子，看看 HTML 可能是一个非常好的主意。例如，为什么要用一个名为 highlight 的类来突出显示一个导航项目，而一个强大的元素会做同样的事情——甚至不需要 CSS？

##### 需求与做正确的事情

我们现在的问题是，我们必须生活在一种遗产中:客户和用户被洗脑，认为一个好的网站在每个用户代理上看起来完全一样，不管它可能有多过时或多糟糕。

当我们处理一个由不懂 Web 的设计师设计的设计时，混合表格布局仍然可以节省我们大量的时间和精力。如果我们转而使用 Web 标准并遵循分离表示和结构的神圣戒律，我们就有麻烦了。当然，我们可以[修改](http://css-discuss.incutio.com/?page=CssHack)设计，使其在所有浏览器上看起来都一样(至少，它通过了客户端过时且无用的测试方法)，但是当我们看到我们已经制作的东西时，事情看起来很糟糕。

CSS 充满了不可读的注释和选择器，它们只对这个和那个起作用——而且只对那些知道如何应对黑客攻击的人有意义；语义正确的标记可能有三个 div，其中一个就足够了；还有…

不要误解我的意思，我们确实需要那些“好利来”和“大约翰”,我对他们在研究和寻找解决方案上所付出的努力感到惊讶。所有的黑客攻击和修复都很有趣，在技术上也很有挑战性，但这确实有点像[浏览器嗅探](http://www.mozilla.org/docs/web-developer/sniffer/browser_type.html)。当 IE5 和 ie6 不复存在时会发生什么？我们会重新访问所有页面并破解 CSS 吗？几乎没有。如果我们真的深入调查，问题并不在于浏览器或它们对 CSS 的错误实现。问题是我们正在处理的设计。这是我们现在需要集中努力的。

让我们少一些链接到验证器的按钮，少一些关于 XHTML 的说教，尽管事实上[今天的用户代理还没有为此做好准备](http://hixie.ch/advocacy/xhtml)。让我们少抱怨缺乏 CSS3 支持，少指着我们最喜欢的浏览器，它做什么都好得多。客户和用户不在乎。他们选择了适合他们的浏览器，你需要一个非常好的理由来说服他们改变。

##### 击中痛处

然而，有一件事是每个人都担心的，当你提起这件事时，人们可能会听你的。钱。这就是我们开始让客户意识到好的符合标准的网页设计是有意义的。指出设计中的缺陷和问题，并如实告诉客户你还需要多长时间来根除这些浏览器特有的错误。展示独立于文本、语言或导航系统中元素数量的布局的好处。

为内容管理系统做广告——毕竟，你不希望每次客户想要更改页面文本中的一个单词时，你都收到电子邮件或打电话。如果客户想要“酷的下拉导航”，问他们为什么，以及网站将如何从中受益。询问他们是否希望访问者记住他们的内容和产品，或者他们的导航有多酷。你不是代码猴子:你是一个交互架构师或设计师，你的工作是交付一个有效且易于维护的网站。

让我们都少分析一些网络的技术细节，在我们开始开发产品之前，看看如何提高产品的可用性和逻辑性。一旦采取这一步骤，遵守标准的问题将不再值得讨论。
