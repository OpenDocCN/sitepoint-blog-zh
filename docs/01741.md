## 谈话要点

> 成为整个创业社区的一部分是我们的梦想。这是一种代际运动。

* * *

> 我们给父母打电话，我说，<q>爸爸，我要辞职了，</q>他说，<q>别这样。</q>我当时想，<q>我要去做，</q>剩下的就是历史了。

* * *

> 对我来说，每当你想启动一个应用程序时，你都必须去亚马逊，启动 web 服务器和数据库，解决安全性问题，解决可扩展性问题，以及所有这些真正可以用沙箱保护的事情，这是没有意义的。

* * *

> 他们推动我们发布，推动我们找出增长，让我们开始考虑指标——你会发现，一切都是由指标驱动的。这是他们首先宣扬的事情之一，如果你不与你的客户交谈，如果你不知道什么是有效的，什么是无效的，那么你就不会成功。

* * *

> 许多创始人落入陷阱，他们最终认为他们需要做所有这些不同的事情，无论是公关，无论是这个，无论是去这个会议，去那个会议，和这个人交谈。说到底，这一切都是为了打造一个用户会喜欢的产品——这意味着与你的客户交谈，也意味着围绕你的平台发展社区。

* * *

> 你不想做出伟大的产品。你想成为一个优秀的用户。我喜欢这个主意。产品的价值取决于用户的能力，这是我们花了一点时间才弄明白的。

* * *

> 我认为我们的早期采用者是对 GraphQL 非常感兴趣的人，我们早期的一些事情是它是及时的。我认为我们抓住了这项技术的危险，这是我们可以帮助开发它的有利条件。

Tim:

嗯，听起来确实很激烈。你们两个是第一批通过 Y Combinator 和我们交谈的人，所以我肯定我们会有很多关于这个过程的问题。但是首先，你提到了 Scaphold.io。你介意告诉我们一点这是什么吗，你试图解决的问题，等等？

Michael:

当然可以。创始故事是这样的，它是我们最初为自己创造的产品。我提到我们过去曾合作过几个项目。我们做了社交媒体的事情。我基本上为弗吉尼亚大学的学生建立了 craigslist 来交换书籍和所有这些东西，我们发现你经常一遍又一遍地重复同样的事情。大多数应用程序背后都有许多相同的基础设施。对我来说，每当你想启动一个应用程序时，你都必须去亚马逊，启动 web 服务器和数据库，解决安全性问题，解决可扩展性问题，以及所有这些真正可以用沙箱保护的事情，这是没有意义的。

这个想法实际上是从我第一次看到的一篇黑客新闻开始的，他们在谈论一种叫做 GraphQL 的新技术。我深入研究了它，并让 Vince 开始研究它，我们真的发现这项技术确实改进了构建客户端应用程序的方式，但在背面仍然有许多令人头痛的问题。你仍然需要做大量的服务架构，包括大量的数据库和备份等等。所以我们决定，<q>让我们建立一个系统或服务，让人们获得 GraphQL 的所有好处，这主要是从客户端来看，然后我们将做所有底层的事情，使这些系统变得非常容易。</q>我们基本上消除了摩擦，允许人们用 GraphQL 非常快速地构建数据系统。

斯卡霍尔德就是这么做的。例如，在几分钟内，假设你想建立 Instagram。在高层次上，你可以想，<q>好吧，我有用户，我有帖子，我有评论。他们必须以某种方式相互联系。当你是一名工程师时，情况就不一样了。你在担心我说的那些事。我们允许你在那个高层次上思考，然后你来到我们的系统，你说，<q>好的，我需要用户、帖子和评论，他们有这些字段，他们以这种方式相互关联，</q>我们的系统把所有混乱的东西旋转起来，然后向你展示这个 API，让你立即开始用 GraphQL 构建应用程序。</q>

Tim:

听起来很酷。只是更深一层…我很抱歉。对于那些不太理解 GraphQL 如何工作或者它到底是做什么的听众，你想深入了解一下吗？

Vince:

当然可以。GraphQL 的历史是有一个项目。我想这是脸书在 2012 年发明的一项技术，他们在内部大量使用这项技术，在生产中部署以支持他们所有的移动应用程序、web 应用程序以及几乎整个基础架构。正如您所看到的，这是一个非常可扩展的生产就绪型解决方案。大约在去年或 2015 年初，也就是两年前，他们向公众开源了这项技术，让每个人——网络上的各种开发者——开始使用这项令人敬畏的新技术，因为他们说，<q>嘿，它对我们很有用，我们是世界上拥有最多数据的最大的科技公司之一。这可能对其他所有人都适用。</q>

所以他们开源了它，这时候 Michael 带来了 GraphQL 的想法并告诉了我。我们开始挑选它，并认为这是一个惊人的想法，你只需要一个 API 来与你所有的数据进行交互。这就是这种服务理念的来源，在这里您不必担心创建 GraphQL 服务器所带来的冗长，并且能够从客户端利用它的好处。

Michael [8:34]:

没错。然而，常见的误解是 GraphQL 是一个数据库。我想澄清的一件事是 GraphQL 是数据库。如果您的听众熟悉 REST API，GraphQL 本质上就是 REST 2.0。它位于服务器的最前端，是客户端用来与服务器通信的语言，但实际上它并不真正了解数据。这就是我们所提供的。我们采用这项非常先进的技术，然后我们的客户与我们交流 GraphQL，然后我们与不同的数据库和 API 交流，让我们能够通过 GraphQL 以非常友好的方式向客户呈现数据。

David:

这是一个很好的解释方式。我正要问。我听到人们谈论 GraphQL，并将其与 REST 进行比较，似乎它也是在人们开始谈论 React 的同时开始进入时代精神的。

Michael:

是啊。这是另一件有趣的事情。我不认为这两个项目都出自脸书是巧合。我们越来越多地发现了这一点。我们使用 GraphQL 的次数越多，它几乎会激发你以更实用的方式思考。你开始考虑 React 宣扬这种单向的、单一的数据流——这种单向的、单向的数据流。GraphQL 遵守了这一点。它本质上允许您遍历这个数据树，如果您将您的应用程序视为一个组件树(React 启发您这样做)，它实际上是 GraphQL API 和 React 应用程序之间的一对一映射。

你会发现，当你用 GraphQL 构建 React 应用时，你通常会比你试图用 REST 或其他任何东西来改造 React 应用时写出更好的代码。

Tim:

你提到了 Y Combinator。我想问一点这个问题。是什么样的？*【笑声】*你们的过程怎么样？

这是我们第二次排练。我们在去年夏天做了这项研究，我们的基地在西雅图——因为我们在微软工作，我们有一栋房子。奖学金是一个远程项目，他们最近改变了它。我认为我们是前一轮伙伴计划的最后一批。但那基本上是处于早期发展阶段的年轻公司。我们带着一个功能原型去面试。我们还没有发射。他们推动我们发布，推动我们找出增长，让我们开始考虑指标——你会发现，一切都是由指标驱动的。这是他们首先宣扬的事情之一，如果你不与你的客户交谈，如果你不知道什么是有效的，什么是无效的，那么你就不会成功。

Vince:

是啊。

Michael:

然后我们在去年夏天结束了伙伴计划，取得了一些成功，在平台上获得了许多早期采用者，然后我们实际上在获得了伙伴计划后又重新申请了 Y Combinator，因为我们还没有得到足够的支持。这是一个为期两个月的项目；我们想要更多。我们被现在的冬季班录取了。我们现在实际上在一栋房子里。三家公司都住在这所房子里。唯一的问题是工作量很大。你无时无刻不在工作，这也是他们所宣扬的——你必须以度量为导向。你必须与你的客户交流，但这实际上是一个 3 个月的冲刺和增长。每周 10%。如果你能得到更多…他们希望你在 3 个月内翻 10 倍。他们只是在推动你真正找出什么在起作用，然后继续前进。

Vince:

是啊。这个程序没有多少魔力。他们告诉你，只要你走进去，你就需要非常努力地朝着一个目标努力。许多创始人陷入了一个陷阱，他们最终认为他们需要做所有这些不同的事情，无论是公关，无论是这个，无论是去这个会议，去那个会议，与这个人交谈。说到底，这一切都是为了打造一个用户会喜欢的产品——这意味着与你的客户交谈，也意味着围绕你的平台发展社区。

David [12:14]:

所以你们两个是创始人，你们创办了这家公司。你住在这个房子里，你在积极地建造这个东西。公司现在仅限于你们两个吗？

Michael:

我们还有几个人在帮助我们。我们联系了华盛顿大学的一名学生—

Vince:

圣路易斯华盛顿大学。

David:

我很好奇这样的物流是如何运作的。因为财政，让人们参与到这样的事情中的花费，而你们本质上还是学生，在建立这个东西的过程中。

是啊。我觉得我们很幸运。我认为我们找到的帮助我们的人真的相信这个产品，他们几乎来了，他们愿意为更少的工作，因为我们是一个年轻的公司。我们仍在努力提升我们的财务状况和一切，但他们非常愿意为未来的可感知收益而努力，而无需我们支付微软将支付的 10 万美元。

David:

确实如此。让人们为你工作是很昂贵的。

Vince:

绝对的。

Tim:

从工程背景到现在经营自己的公司，当你们刚刚开始人生的这个新阶段时，有没有什么教训对你们打击最大？

Michael:

对我来说，最重要的事情是我一直想打造产品，我认为我们发现自己处于这样一种情况，我们有一个伟大的产品，但它比我想象的更涉及社区。因此，现在我们正在学习如何让社区更多地参与进来，以及如何让人们在这个平台上进行建设……我听到了这句名言。我不想骂人，但就是这本书叫 <q>Badass</q> 。他们说，<q>你不想做一个伟大的产品。你想成为一个优秀的用户。我喜欢这个主意。产品的价值取决于用户的能力，这是我们花了一点时间才弄明白的。</q>

Vince:

是啊。在那一点上，我同意迈克尔的意见。我们都是产品驱动型的人，我们都想做出最好的产品，所以我们花了很多时间做这件事，但最后不要告诉别人我们的新功能。因此，最终发生的是，我们拥有所有这些无人知晓的功能，而在一天结束时，它只是被收藏在一个壁橱里。这就是我们现在试图更加关注的——努力建立这个社区，告诉人们我们在做什么，更加透明，更加以用户为导向来介绍我们的功能。

David:

这是一个挑战，因为有时你会觉得自己本末倒置，试图让人们参与到你作为创始人并不认为已经准备好展示的事情中。

Vince:

没错。

David:
