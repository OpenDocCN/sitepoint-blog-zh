# 简单性与特性:一个错误的二分法？

> 原文：<https://www.sitepoint.com/simplicity-vs-features-a-false-dichotomy/>

在其他条件相同的情况下，最简单的解决方案可能是最好的。这是对奥卡姆剃刀 T2 告诉我们的基本理解，这句格言是以 14 世纪英国逻辑学家奥卡姆的威廉命名的。换句话说:保持简单，笨蛋。

奥卡姆剃刀理论为软件和网络应用开发的设计决策提供了信息，这一理论也许可以通过 37 signals get Real 方法得到最好的体现，它建议开发者在竞争中处于劣势。“做得比你的竞争对手少，以击败他们，”该公司在他们的书*中说。“不要一上一下，试试一下一下。与其超越，不如尝试失败。”*

37signals 解释说，更少意味着“更少的功能”和“更少的选择/偏好”这个想法是简单将产生更容易使用和更优雅的技术。但是西北大学教授和前苹果副总裁唐·诺曼不同意。诺曼为计算机机械协会的*互动*杂志撰稿，他认为简单不是答案，也不一定会带来更好的设计。

“简单是 <u>*而不是*</u> 的目标。“我们不想放弃我们技术的力量和灵活性，”[写道。“车库门开启器可能很简单，但它几乎没有任何功能。如果我的手机只有一个按键，那肯定会很简单，但是，嗯，我能做的就是打开或关闭它:我就不能打电话了。”](http://www.jnd.org/dn.mss/simplicity_is_not_th.html)

人们实际上想要更多的特性和功能。他说，他们也想要易用性，但有一种流行的错误二分法，即简单等同于易用，功能等同于功能。诺曼列出了以下内容，他说这是一个隐含的假设:

*   功能== >能力
*   简单== >易于使用

“这两个陈述转化成简单的逻辑，”诺曼说。“每个人都想要更多的功能，因此他们想要更多的特性。每个人都希望易于使用，因此他们希望简单。”但是诺曼认为这是错误的逻辑。箭头从左到右:这与从右到左的方向无关。所以额外的能力不需要更多的特性。同样，易用性也不需要简单。”

人们很可能确实想要更多的功能和更容易使用的产品，但要实现这些东西并不需要设计师屈服于功能蠕变或使简单成为设计规则。诺曼列出了一些设计规则，以创造更好的产品，更容易使用，但不要以简单的名义牺牲功能。

*   模块化——将大而复杂的任务分解成更小、更易管理的任务。
*   **映射**–行动和它们带来的结果之间的关系应该是清晰的。
*   内聚的概念模型(Cohesive conceptual models)——人们应该理解他们应该做什么，做什么会发生，以及对他们的期望是什么。“看看任何苹果产品，”诺曼说。

“争论不在于增加功能和简单性，也不在于增加功能和可用性，”诺曼写道。“真正的问题在于设计:设计出具备工作所需功能的东西，同时保持可理解性、控制感和成就感。”
