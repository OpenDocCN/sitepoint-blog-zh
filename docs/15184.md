##### 基本托管功能

如果主机世界中有一个词误导了天真的冲浪者，那就是“无限制”。把事情放在一个角度来看，没有什么网站主机可以提供给你的是无限的。它可能是未计量的，也可能是无限制的(仅受服务器规格的限制)，但绝不是无限制的。

警惕大量使用这个词的主持人:这是一种营销策略，很多人都上当了，尤其是当他们没有做很多背景调查的时候。通常，提供无限带宽和磁盘空间的服务器往往会被过度订阅，其提供的服务可能会因此受到负面影响。

***1。磁盘空间***

所有托管帐户都提供一定数量的磁盘空间，您可以用来存储您的文件。不同主机的磁盘空间使用量会有所不同，因此在决定所需的磁盘配额时，请确保检查需要考虑哪些文件。

对各种任务所需的空间进行某种估计是个好主意。您需要多少空间来存储您的电子邮件、网络文件、数据库和日志文件？像这样分解你的使用情况将会使计算你需要多少空间的任务变得简单得多(一旦你确切地知道哪些文件被计入你的磁盘配额)。

在计算磁盘使用量时，所有计划肯定会包括所有可通过网络访问的文件。一些主机还会选择将电子邮件和/或日志包括在配额中，这使得估计您的磁盘空间需求更加困难。虽然您可能对 web 文件所需的磁盘空间有很好的了解，但您的电子邮件和日志文件的需求可能会不断变化。

如果主机在配额中包括所有类型的文件存储，请检查是否可以关闭日志或排除特定信息。如果该选项可用，它可能会为您节省大量的磁盘空间，尤其是如果您有一个繁忙的网站。但是，如果您想要使用统计包，您将需要为一些日志记录留出空间。

如果您启用了日志，那么值得记住的是，在服务器停止写入日志文件之前，您可能无法从服务器上删除日志文件。如果你记得登录并定期下载和删除日志，日常日志是相当容易管理的。如果您低估了您的日志空间需求，然后发现直到月底才能删除文件，那么每月日志的问题就会出现。如果你不小心，这很容易导致额外的费用。

如果您的电子邮件设置和收件箱包括在您的磁盘配额中，如果可能的话，最好为每个邮箱设置一个最大大小。这将避免你在服务器上存储太多的邮件，无意中超过你的限制。

***2。邮箱账号***

电子邮件帐户是主机服务的一个常见特征，尤其是当你托管一个域名的时候。有些主机会让您控制邮件设置，在服务器端限制邮件活动(例如，帐户数量或邮箱的最大大小)。其他主机会为你做所有的设置，即使设置新的邮件帐户可能比你想象的要容易，只要有合适的软件支持。

你如何配置你的电子邮件是个人喜好的问题，但本质上有四种主要类型的邮件帐户；POP3，转发，别名和自动回复。

*   POP3 账户是传统的“收件箱”。您在服务器上有存储邮件的空间，并且您使用电子邮件程序登录并下载邮件。每个登录名和密码组合通常相当于一个帐户。这有点像办公室的收件箱，邮件会一直留在那里，直到你对它做了些什么。如果收件箱已满，您的邮件将无法存储，并会被退回。
*   例如，如果您想在收到邮件之前将邮件发送到 SpamCop 之类的电子邮件过滤服务，转发邮件帐户非常有用。而不是存储在您的邮件服务器上，它会重定向所有邮件到另一个单一的电子邮件地址，这将处理它适当。这种账户有助于将你的电子邮件重定向到一个普通的 POP3 邮箱。
*   别名是可用于识别不同类型的电子邮件帐户的名称，将它们重定向到服务器上的 POP3 邮箱或其他地址，如果需要，将在那里再次处理它们。电子邮件的处理将取决于它们被发送给谁。一个包罗万象的别名经常被用来收集和处理发送给你的邮件服务器不能识别的人或部门的电子邮件。
*   自动回复邮件本身并不是一个电子邮件帐户，但是他们有自己的电子邮件地址，只是简单地回复给他们发邮件询问信息的人。如果您希望将预先准备好的信息发送给请求它的人，而不是您必须手动回复所有请求，那么它们非常有用。

***3。FTP 接入***

另一件在付费托管账户中常见的事情是 FTP 访问。FTP 程序允许你上传文件，编辑和删除服务器上的内容，比使用基于网络的界面要快得多。如果您在*nix 系统上托管，您还可以使用 FTP 更改您的文件权限设置。

我见过的主机提供的一个更好的功能是你可以创建自己的 FTP 账户。当有人在网站上帮忙时，或者如果您想共享您的网络空间，同时将您的用户文件与您自己的文件分开时，这非常有用。主持人如何做到这一点可以有所不同。

有些主机会让你像一个迷你托管公司，在那里你创建的 FTP 帐户将用户带到一个专门存放他们文件的特殊文件夹，这使他们与你的主文件分开。其他主机将允许您创建 FTP 帐户，您可以在其中准确定义用户可以访问哪些文件夹，以及他们可以(和不可以)对这些文件夹做什么。

如果您不打算在自己的网络空间上托管其他网站，那么拥有创建多个 FTP 帐户的能力可能看起来微不足道，但它对于允许任何帮助您的网站的人临时或永久访问您的网站非常有用，无需您提供自己的 FTP 帐户详细信息。这是一个值得拥有的特性！

***4。**数据传输*

计算您的数据传输需求、相关的带宽以及它对托管您的网站的影响绝非易事，但如果您想避免意外成本，这是您需要进行的计算之一。

在计算数据传输和带宽需求时会遇到各种各样的问题。你如何解释缓存的页面和图片？主机使用什么尺寸单位进行计算，你的带宽限制如何影响你的主机质量？

小心点！并非所有的数据传输限制都是相同的！不同主机之间的测量单位可能有所不同，如果他们不清楚，请务必询问他们在制定比较表时使用了多少字节/千字节。

**缓存及其对数据传输的影响**

许多 ISP 使用代理服务器来帮助他们的客户加快互联网接入速度。这个想法是，这些服务器在代理服务器上保存您的页面的副本(一段时间)，并在您更改页面时更新它们的副本。这样做的结果是，虽然您的页面可能只从您的服务器上下载过一次，但它可能已经被许多人看到了。

如果你有很少改变的静态页面，那么缓存可以帮助减少你的托管费用。巧妙使用 meta 标签可以告诉浏览器，如果用户的计算机或代理服务器上有页面，就不要麻烦从服务器重新加载页面。由于这些重新访问的页面不需要从您的服务器上重新加载，您的日志中的一个页面请求可能相当于该页面的数百次页面访问(特别是如果您从一些较大的 ISP 使用的大型代理服务器上进行访问)。

动态页面不太适合缓存，但是如果页面的动态元素对访问者或时间不敏感，那么在某种程度上可以使用缓存来降低数据传输费用。例如，使用日期的页面可能只需要由服务器每天构建一次。在构建动态页面的情况下，缓存还可以通过将数据库查询的结果存储在服务器缓存中来减少服务器负载。

***5。**带宽和托管质量*

当你寻找一个合适的主机时，一定要检查出你的站点可用的带宽量。如果你托管在一个共享服务器上，即使一个网站每月不需要大量的数据传输也可能运行缓慢；特别是如果你和一个更受欢迎的或者带宽密集型的站点在同一个服务器上。

服务器上托管的站点越多，它们就越有可能相互竞争可用资源，包括带宽。这就是为什么您可能会注意到在繁忙时期，或者在您共享的服务器上一个更繁忙站点的流量突然达到峰值时，服务器响应时间会减少。

**冗余问题**

有多少带宽未被使用以应对任何时候流量的突然高峰，这就是所谓的冗余带宽。特定服务器的冗余带宽越多，它就越有可能很好地应对突然出现的流量高峰，这就决定了广告活动是成功还是失败。

如果你的网站总是很慢很忙(尤其是在一天中的固定时间)，那么它肯定会从拥有更多带宽中受益。有可能您的主机没有提供足够的冗余带宽来处理正常需求，他们可能需要升级带宽容量，或者只是审查其带宽管理策略。

理想情况下，您考虑的任何主机都应该连接到主干网，至少有两个服务提供商，并且所有连接中至少有 25%的冗余。连接、服务提供商和冗余百分比的数量越多，系统在繁忙时就能更好地应对。

**突破带宽障碍**

通过购买一个允许突发带宽的计划，你将有助于防止你的网站在流量高峰期变慢。突发带宽意味着如果你需要的话，你可以使用比你的主机计划通常允许的更多的带宽，这在你的服务器流量突然达到峰值时非常方便。请注意，可用的额外带宽量将取决于您的站点可以访问的主干连接中的冗余带宽水平。

为您的包提供突发带宽的主机将允许您使用部分(或全部)冗余带宽来帮助您的站点处理流量。这只能作为一个短期的解决方案——如果你一直使用超过你的主机包允许的带宽，你真的应该升级。

**提供大型文件和多媒体服务**

如果你还没有猜到，带宽可用性对网站下载的数量和质量都有很大的影响，无论是大型档案还是多媒体演示。随着访问服务器变得越来越困难，丢失的数据包数量也在增加。对于小文件，如网页和图像，这不是一个问题；(统计上)小文件不太可能发生丢包。另一方面，较大的文件更有可能在传输过程中丢失数据，因为数据包丢失的机会更多，这反过来会对下载质量产生不利影响。文件可能会损坏，随着更多数据丢失，流媒体会变得更加不稳定。

如果你打算在你的站点上托管下载或多媒体文件，那就值得去找一个能提供足够带宽、数据传输以及在主干连接中有良好冗余度的主机。拥有一个支持 resume 的服务器是非常有用的，因为它可以节省不必要的数据传输。

你的带宽和数据传输费用将占你主机账单的大部分。虽然突发带宽可能是一种奢侈品，但在预算允许的情况下获得尽可能多的带宽和冗余总是一个好主意。

***6。文件支持***

你会期望所有的 Web 主机都支持运行最基本的网站所必需的基本文件类型——毕竟，如果一个服务器不知道如何处理一个 HTML 文件和与之相关的图像，那么它有多老了呢？然而，你不能把一个文件上传到服务器，然后简单地期望它第一次就能工作。不是所有的服务器都知道如何处理不同的文件，如果设置不正确，服务器甚至会混淆一些更常见的文件类型。

当你想使用除了典型的 HTML、JPEG 和 gif 之外的任何文件类型时，问题可能会变得很明显。如果与级联样式表相关联的扩展(.css)与 Corel 幻灯片放映中不太常见的 mime 类型相混淆。

**文件扩展名**

您的服务器如何处理各种扩展将取决于它运行的软件。在 Windows 系统上，扩展被映射到它们要用于的应用程序。因此，用 Perl 解释器打开一个扩展名为. pl 的文件，然后根据文件的内容输出页面。在*nix 系统上，同样如此。pl 扩展名并不重要:您必须在脚本的第一行明确给出 Perl 的路径，以便服务器能够知道应该使用什么程序来运行该文件。

当服务器不知道如何处理文件时，它会做以下两件事之一:

*   尝试以纯文本方式打开文件，或者
*   试图下载它。

这些默认行为通常是不可取的，特别是如果您希望文件“按原样”发送，以便您的浏览器及其插件可以自己处理它。如果任何人在下载文件的过程中(例如，您的一个脚本)看到存储在其中的路径和(有时是未加密的)密码，这也可能造成安全威胁。

在上传任何包含潜在敏感内容或源代码的内容之前，最好测试一下您的服务器如何处理各种文件类型。如果您使用的文件类型不被您所希望的方式支持，可能只是文件扩展名与正确的 mime 类型不相关。根据您的主机和服务器设置，您可以添加自己的 mime 类型。这甚至允许你创建自己的文件扩展名，让人们想知道你使用的是哪种脚本语言！

**Mime 类型**

必须强调的是，并不是所有的主机都会为您提供添加自己的 mime 类型的机会，所以最好确保服务器支持您计划使用的所有文件类型。

如果您的主机运行的是 Apache，那么您可以编辑 Apache 处理程序，让文件表现得完全不同。这当然是可能的。html 文件解析服务器端包括通过这种方式编辑处理程序。

但是，除非您需要解析所有文件中的 includes，否则这可能不值得，因为这会导致 all。html 文件在被发送到浏览器之前被解析，这会给服务器带来不必要的压力。

***7。脚本魔法***

虽然几乎可以将任何类型的 Web 技术安装到任何服务器设置中，但是您会发现服务器和脚本语言的某些组合比其他组合更好。因此，让我们看看更常见的 Web 技术，并讨论它们如何受到各种服务器环境的影响。

**Perl**

Perl 已经存在了足够长的时间，在许多服务器平台上都得到了广泛的支持，有些模块是特定于特定的 Perl 版本的，这取决于服务器操作系统。如果您打算使用它们，那么查看一下特定模块的操作肯定是值得的。为了最大限度地减少跨平台不一致的问题，请考虑在与您最终将上传 Perl 脚本的实时服务器相同的环境中开发您的 Perl 脚本，特别是当您使用特定于系统的函数或模块时。

在目前所有可能的脚本语言中，Perl 是提供脚本功能的主机最常支持的。除非您使用特定于系统或服务器的模块，否则您对 Perl 的使用不应该限制您对主机的选择。

Perl 特别适合在 Apache 服务器上运行，Apache 模块(如 Mod_Perl)被设计用来加速 Perl 脚本的执行。Perl 的 Windows 版本被称为 ActivePerl，虽然 Perl 可以在 Windows 机器上愉快地运行，但在某些功能上还是有一些细微的差别。

**PHP**

PHP 有两种风格:你可以把它作为 Apache 模块安装(如果你有 Apache 的话这很好),或者你可以把它作为一个独立的 CGI 应用程序运行。CGI 应用程序通常比 Apache 模块慢，所以如果你想提高 PHP 脚本的速度，最好是安装在 Apache 服务器上的 PHP。

请注意，PHP 不像 Perl 那样得到跨操作系统的支持，尽管它得到了主机领域所有主要参与者的支持。PHP 还有一些特定于 Apache 和特定数据库的函数。文件系统功能可能会导致 Windows 系统出现问题，因为许多问题都是*nix 特有的。幸运的是，这些函数相对较少，并且在文档中有记录。

虽然 PHP 在 Web 上的使用还不像 Perl 那样广泛，但最近它的使用已经增加了。这种增长可以归因于它的开源背景和接触到更广泛的受众。PHP 比 Perl 更易于使用，并且比 ASP 更好地支持与远程服务器的交互，因此它将继续流行。

目前，大多数 PHP 托管设施都是在运行 Apache 的*nix 系统上提供的，因此 PHP 开发人员更有可能找到满足他们需求的基于 Unix 的主机。除非您大量使用 ASP，否则如果您需要使用 PHP，使用运行 Apache 的*nix 系统可能是个好主意；毕竟，Apache 模块是专门为加速 PHP 的执行而设计的。

**动态服务器页面(ASP)**

ASP 本质上是微软的发明，直到 Chili 等项目的出现！软 ASP、即时 ASP 和 Apache ASP 模块，它仅限于 Microsoft IIS 服务器系列。ASP 支持被完全集成到 IIS 服务器中，所以如果你知道你被托管在 IIS 上(这对于 Windows 机器是典型的)，那么你可能能够运行你的 ASP 脚本。

如果一个主机在一台*nix 机器上为你提供 ASP 支持，他们很可能会使用 Chili！Soft ASP(我还没有找到使用其他两种解决方案的主机)。但是请注意，这不是真正的 ASP 支持；如果您打算将您的 ASP 脚本从一个平台迁移到另一个平台，有几个重要的差异应该考虑在内。

ASP 3.0 引入了特性(即服务器。Transfer method)，它允许基于服务器变量动态地包含文件(这是 ASP 开发人员的常见请求)。不幸的是，ASP 3 只受 IIS 5(Windows 2000 机器的一个特性)的支持，还没有提供给早期产品的用户。使用 Windows 98 或 Windows NT 机器来测试 ASP 页面的 ASP 开发人员，如果他们希望继续“在内部”测试，就必须限制他们使用新的 ASP 特性。

**FrontPage 扩展**

FrontPage Extensions 本身并不是一种脚本语言，但它确实在网站上提供了一些基本的 CGI 交互。虽然它们与 Microsoft FrontPage 系列相关联，但它们并不局限于 Windows 服务器。

FrontPage 扩展仅仅是 FrontPage 程序的一个扩展。当安装在服务器上时，这些文件允许程序与服务器交互；它可以在你对 FTP 一无所知的情况下上传你的整个网站。扩展的其他特性是它们允许你在你的站点上添加一些有限的功能。例如，您可以允许您的访问者在评论页面上留言，或者使您的整个网站可搜索，而无需了解任何脚本语言或安装任何脚本。

话虽如此，如果您需要主机提供 FrontPage 支持，请确保为您使用的产品安装了正确的扩展。事实上，当扩展比你的程序旧的时候，试图使用扩展的最新特性会导致问题。

***8。数据库***

由于对交互式 Web 内容的需求日益增长，数据库在动态内容的创建和存储中扮演着越来越重要的角色。像许多其他与虚拟主机相关的因素一样，你选择使用的数据库技术会影响你对主机的选择。

毫不奇怪，使用特定的数据库软件有利也有弊。在决定哪种数据库解决方案适合您的需求之前，您应该对希望在数据库中存储什么有所了解。

**平面文件数据库**

最简单的数据库是平面文件数据库，由单个文件或一组文件组成，其格式为希望使用它们的人所知。由于其简单且可预测的格式，它们可以与任何能够读取和解析文件的脚本技术一起工作(在一些编程之后)。每行要么是一个字段条目，要么被一个公共分隔符分割成几个字段条目。

*优势*

平面文件数据库:

*   提供对数据存储和访问方式的完全控制。
*   都是独立于平台和脚本语言的。
*   对于简单的小型数据集，可以快速实现和使用。
*   当在标准文本编辑器中查看时，他们的数据是可读的。

*缺点*

*   它们的伸缩性不好，因为执行搜索需要读取所有数据。
*   它们不太适合并发数据访问，这使得它们容易受到损坏。
*   在使用特定的平面文件数据库之前，您需要了解它的结构。
*   它们很难用作关系数据库；可能会有更多的数据重复。
*   没有可用的函数，所以所有的操作都必须由正在使用的语言来完成。

**MS Access 和 MS SQL**

这两种微软数据库产品通常限于在 Windows 服务器上使用 ASP 页面，但在正确的开放式数据库连接支持下，也可以在 Unix、Apple Macintosh 和 OS/2 系统上运行。

Access 只是 Microsoft SQL 的缩小版，尽管它在同时连接或功能支持方面不太容易扩展。如果您打算为大型或繁忙的数据库使用 Microsoft 解决方案，那么您应该选择 Microsoft SQL。Microsoft Access 更适合于小型业余爱好网站或低流量的商业网站，在这些网站上不太可能出现大量使用的情况，因为大量使用会降低网站的速度。

*优势*

*   关系数据库使数据管理更加容易。
*   两者都支持图像数据类型。
*   MS SQL 允许跨托管相同数据库的服务器进行访问。

*缺点*

*   编辑和使用这些数据库需要软件许可证。
*   如果使用不当，Access 可能会导致整个服务器停机。
*   难以在非 Windows 环境中使用(需要支持 ODBC)。

**MySQL** 的实现

虽然当前版本的 MySQL 特别针对*nix 平台进行了优化，但通过编译源代码，MySQL 可以构建为与大多数现代系统一起工作。MySQL 的性能很大程度上受系统性能的影响:操作系统和文件系统的稳定性，以及对多线程的支持影响最大。

*优势*

*   软件是开源的，所以可以免费使用，并且可以广泛获得。
*   这是一个被广泛使用的数据库解决方案，所以它的稳定性经过了很好的测试和记录。
*   良好的全面支持许多功能，尽管它的使用成本很低。
*   不同大小的文本和二进制数据类型为数据库设计提供了更好的优化。
*   关系数据库使数据管理更容易。

*缺点*

*   对于数据库新手来说更难使用。
*   使用 Codd 的规则可能不是真正合理的。

**甲骨文**

Oracle 可以被配置为在多种平台和操作系统上运行，并且已经被证明是非常可扩展和可靠的——所以它变得如此流行也就不足为奇了。

*优势*

*   具有可以直接写入文件系统的数据类型。
*   有能力处理极其大量的数据。
*   高配置的数据库服务器。
*   关系数据库使数据管理更容易。

*缺点*

*   高成本限制了它在低成本互联网应用中的使用。
*   大量的配置选项可能需要训练有素的数据库管理员。

    ***9。多媒体支持***

    如果多媒体支持对你的网站很重要，那么你真的应该花时间(和金钱)来获得满足你需求的主机服务。公平地说，你不能指望从一个预算价格的主机账户得到大量的支持。为你的访问者提供多媒体文件需要大量的资源，尤其是带宽。

    **闪光和冲击波动画**

    虽然 Flash 和 Shockwave 动画看起来相当动态(在某种意义上，可以根据您的系统功能改变它们的工作方式)，但实际上它们本质上是相当静态的。它们可以在任何支持 MIME 类型的服务器上工作，不需要任何专门的服务器软件。

    Flash 和 Shockwave 演示文稿可以被描述为交互式图像，因为演示文稿本身通常只是一个文件，可以在开始之前下载到您的计算机上。如果您不希望您的访问者能够在他们的计算机上存储演示文稿的副本，这可能会成为一个问题，因为完整的演示文稿可能会在达到其目的之前被下载。

    **同步多媒体集成语言(SMIL)**

    SMIL 是一种语言，允许在音频、视频、文本、动画和图形的演示中有更大的灵活性:您可以重复使用和混合这些多媒体元素，从而减少为每个不同版本的演示创建单独剪辑的需要。SMIL 允许用户在可用时选择观看选项(例如，考虑到他们缓慢的下载速度或语言选择)。

    虽然还不像 Flash 或 Shockwave 那样通用，但 SMIL 可以提供一种替代方案，因为它允许用户和网站所有者分别围绕服务器上可用的多媒体元素的使用和组合具有一定程度的灵活性。将来，SMIL 可以提供一种无需脚本语言就能将定时元素(例如，旋转横幅)整合到网站中的方法。

    **流媒体**

    互联网改变了人们对购物和交流的期望:我们现在期望事情比以前快得多。如果可以选择在缓冲了足够的信息后立即激活大型下载(并允许在后台下载其余数据)，很少有人会选择等待下载完成后再激活。

    如果您打算主持类似网络直播的活动，流媒体支持是必不可少的，如果您正在考虑这样的活动，您可能需要专门设置一个专用服务器来处理请求并提供所需的资源。即使您不直播活动，您可能仍然需要专业的商业流媒体软件来提供您需要的功能。

    这些协议与其他协议(如 HTTP 或 FTP)的主要区别在于，无论数据包是否在传输中丢失，它们都会继续传输数据:如果数据丢失，则不会重新传输。

    最令人愉快的多媒体演示是那些你可以控制的演示。RTSP 是一种双向流协议，允许访问者通过将消息发送回服务器来控制下一步发送演示的哪一部分，从而与媒体进行交互。当然，相对于 RTP，你可以期望为支持 RTSP 的服务器支付更多的费用。

    无论你决定向全世界提供你的多媒体文件，有一件事是可以肯定的:它们将使用大量的带宽和数据传输。音频和视频剪辑文件超过一兆字节并不罕见——即使乘以一个适中的数字，你也会开始发现在你的网站上提供几个大文件所带来的问题，即使它们并不经常被访问。
