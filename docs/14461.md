# /.艾德…又来了

雷击[两次](http://developers.slashdot.org/developers/04/08/04/1516258.shtml?tid=169&tid=192&tid=218)。为了省去实际阅读评论的麻烦，直接去看[尼尔 Blender 的](http://slashdot.org/~Neil%20Blender) [摘要](http://developers.slashdot.org/comments.pl?sid=116764&cid=9923312)——只有 6 个 fuecks 笑话——我很惊讶；)你应该看看 JFK 机场地勤人员对我的热情(实际上是歇斯底里的)欢迎，我去过几次纽约。感谢[马修](http://abel.math.harvard.edu/~leingang/)的好评。

一个真正切题的帖子是这里的——PHP 选集与乔治的[高级 PHP 编程](https://www.sitepoint.com/blog/)相比如何？

总的来说，我会说选集是“中级 PHP 编程”而不是高级。这些书的风格各不相同，选集采用了 Q &A;方法，旨在解决高级 PHP 编程中的常见问题，更具叙事性。在 PHP Anth II 的结尾有一些交叉，它进入了 API 文档、单元测试和设计模式等内容，但旨在为以前没有接触过这些概念的读者提供更多的介绍。当然，像性能问题这样的东西在高级 PHP 编程中得到更彻底的处理。我个人认为你可以两者兼得而不浪费你的钱。也就是说，如果你是一个 PHP 老手(例如，你完全理解 PHP4 中的引用是如何工作的)，PHP 选集可能不适合你。

当我谈到这个话题时，认为有一整部分的 PHP 书籍还没有人真正写过，主要集中在 PHP 应用程序设计(“策略”)和“战术”上，这在今天已经被很好地涵盖了。回到 2003 年 6 月，另一个/。帖子求[优雅的 PHP 架构？](http://developers.slashdot.org/article.pl?sid=03/06/03/193223&tid=169&tid=8)；

> 我的感觉是，我创建的所有网站一半是优雅的，一半是肮脏的。

在讨论 PHP 的“策略”时，很容易给出“绝对”的答案，我想这是人们对书的期望。谈论设计“策略”是一个完全不同的问题，我基本上要求作者提前承认“我可能是错的”或“我不确定，但这是对我有用的”。在编写 PHP Anth 时，我非常清楚这一点，并且避开了框架(甚至模板)之类的主题。

问题还在于，对于什么才是真正有效的方法，人们的看法大相径庭。让拉斯莫斯和马库斯写关于这个主题的书(如果他们好就好)，你会得到两个*非常不同的结果，两个结果可能都一样有效。*

在任何人将此作为反 PHP 火焰的燃料之前，请想一想这实际上是一个困扰所有 web 开发的问题，无论您是使用 PHP、Perl、Python、 [Ruby on Rails](http://www.rubyonrails.org/) 。NET、J2EE 或其他——见[网络应用:仍然没有理想的架构](http://www.phppatterns.com/index.php/article/articleview/95/1/11/)。
