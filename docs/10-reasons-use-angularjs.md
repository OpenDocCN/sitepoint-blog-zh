# 你应该使用 AngularJS 的 10 个理由

> 原文：<https://www.sitepoint.com/10-reasons-use-angularjs/>

如果你还没有尝试过 Angular，你就错过了为什么人们说 JavaScript 是世界上最灵活的语言。

Angular 是唯一一个不会让 MVC 看起来像给猪涂口红的框架。

如今，大多数框架只是现有工具的简单捆绑。它们是一个集成的工具集，但不是很优雅。Angular 是下一代框架，其中每个工具都被设计成以一种互连的方式与其他工具协同工作。

以下是你今天应该使用 Angular 的 10 个理由。

## 1.MVC 做得对

大多数框架实现 MVC 的方式是让你把你的应用程序分割成 MVC 组件，然后要求你写代码再把它们串起来。那是许多工作。Angular 实现 MVC 的方式是让你把你的应用分成 MVC 组件，然后让 Angular 去做剩下的事情。Angular 为您管理组件，并充当连接它们的管道。

因为 Angular 充当了中介者的角色，开发人员也不会为了使组件更容易适应而在组件之间编写打破抽象的快捷方式。

## 2.声明式用户界面。

Angular 使用 HTML 来定义应用程序的用户界面。HTML 是一种声明性语言，它比用 JavaScript 程序性地定义界面更直观，更简单。HTML 也不像用 JavaScript 编写的界面那样容易重组，这意味着不太可能出现问题。另外，当视图是用 HTML 编写的时候，你可以引入更多的 UI 开发人员。

HTML 也用于确定应用程序的执行。HTML 中的特殊属性决定了每个元素使用哪个控制器。这些属性决定了“加载什么”,而不是“如何加载”。这种声明式方法以一种所见即所得的方式极大地简化了应用程序开发。与其花时间在程序如何运行和什么应该先加载上，你只需要定义你想要的，Angular 会处理依赖关系。

## 3.数据模型是 POJO

Angular 中的数据模型是普通的旧 JavaScript 对象(POJO ),不需要额外的 getter 和 setter 函数。您可以直接在其上添加和更改属性，并随意遍历对象和数组。您的代码将看起来更干净、更直观，这是大自然母亲的意愿。

传统的数据模型是数据的看门人，负责数据持久性和服务器同步。这些数据模型的行为就像智能数据提供者。但是由于 Angular 的数据模型是简单的对象，它们的行为更像一个软木板。软木板只不过是人们存放和检索数据的临时存储区。然而，Angular 的软木板与控制器和视图密切相关。为了区别于传统意义上的数据模型，Angular 称之为“作用域”。

范围对象上的所有属性都通过 Angular 自动绑定到视图。也就是说，Angular 静静地观察这些属性的变化，并自动更新视图。

该范围没有数据开始，并依赖控制器根据业务逻辑需求向其提供数据。

## 4.指令行为

指令是 Angular 为 HTML 带来额外功能的方式。想象一个 HTML 拥有如此多丰富元素的世界(例如`<accordion></accordion>`、`<grid></grid>`、`<lightbox></lightbox>`等)。)我们永远不需要操纵 DOM 来模拟它们。我们的应用程序需要做的就是为元素分配属性，以获得开箱即用的功能。

指令通过让我们发明自己的 HTML 元素来实现这一点。通过将所有的 DOM 操作代码放入指令中，我们可以将它们从 MVC 应用程序中分离出来。这使得我们的 MVC 应用程序只关心用新数据更新视图。视图随后的行为取决于指令。

指令以自定义 HTML 元素的形式出现

```
<myticker></myticker>
```

自定义属性

```
<div data-myticker></div>
```

和自定义类名

```
<div class="myticker"></div>
```

允许它们像常规 HTML 元素一样使用。

指令被设计成独立的可重用元素，与你的应用程序分开。事实上，如果一个特定的元素被 HTML5 标准所采用，它应该像删除您的自定义指令一样简单，并且您的应用程序应该完全一样，而不需要更改您的应用程序。

记住，根据经验，控制器不应该直接操纵 DOM。所有的 DOM 操作都应该通过指令来执行。

## 5.过滤器的灵活性

筛选器在数据到达视图之前对其进行筛选，可能涉及格式化数字上的小数位、反转数组顺序、基于参数筛选数组或实现分页等简单的事情。过滤器被设计成独立的功能，与你的应用程序分开，类似于指令，但只与数据转换有关。

过滤器是如此的足智多谋，以至于不用写任何 JavaScript，只用过滤器就可以创建一个可排序的 HTML 表。

## 6.少写代码

到目前为止，所有这些都意味着您可以编写更少的代码。你不用自己写 MVC 管道。视图是用 HTML 定义的，更简洁。没有 getter/setter，数据模型更容易编写。数据绑定意味着您不必手动将数据放入视图。由于指令与应用程序代码是分开的，它们可以由另一个团队并行编写，集成问题很少。过滤器允许您在视图级别操作数据，而无需更改控制器。是的，这是一个总结要点，但是编写更少的代码是一件大事！

## 7.DOM 操作属于它们的地方

传统上，视图修改 DOM 来呈现数据，并操纵 DOM(或调用 jQuery)来添加行为。使用 Angular，DOM 操作代码应该在指令内部，而不是在视图中。Angular 将视图视为另一个带有数据占位符的 HTML 页面。这种查看视图的方式非常适合用户界面设计者。

通过抽象出 DOM 操作和 jQuery 调用，用户界面设计者能够专注于视图而不会分心。

通过使您的 MVC 应用程序纯粹是将业务数据呈现到视图中，而不必担心操纵 DOM，web 应用程序开发突然变得更有趣了。

## 8.他们所属的服务提供商

Angular 中的控制器是简单的函数，只有一个任务，就是操纵示波器。例如，您可以使用它将数据从服务器预填充到范围中，或者实现业务逻辑验证。与其他框架不同，控制器不是对象，不继承任何东西。

如果控制器如此简单，那么所有繁重的工作应该在哪里进行呢？Angular 引入服务就是为了做到这一点。

服务就像它们听起来的那样。它们不参与你的应用程序的 MVC，而只是提供一个向外的 API 来暴露你想让它暴露的任何东西。大多数情况下，它与服务器同步以维护离线数据存储，并公开向服务器推送数据和从服务器获取数据的方法。或者它可以用于创建资源共享服务，允许多个控制器共享相同的资源。

服务被设计为独立于应用程序的独立对象，允许您的控制器保持精简，并专用于它被分配到的视图和范围。当然，实现服务不是必需的，在控制器内部做一些简单的工作以避免过于复杂是完全可以接受的。

## 9.上下文感知通信

PubSub 系统是一个非常常见的工具，它允许解耦通信。web 上的大多数 PubSub 实现都不支持上下文。有时，您希望 PubSub 消息只对特定节点的子节点可读，或者只对特定子节点的祖先可读。换句话说，有时你不希望无关的 MVC 组件读取你的消息。

Angular 中的 PubSub 系统正是如此。`broadcast()`将向所有子控制器发送一条消息，而`emit()`将向所有祖先发送一条消息。

但是 PubSub 并不是控制器之间通信的唯一方式。事实上，如果您所做的只是告诉其他控制器在属性改变时更新它们的视图，那么您应该依赖数据绑定。我们已经知道视图可以绑定到当前范围的属性。但是我没有告诉你的是作用域继承了它们的父作用域的属性。这意味着如果一个属性存在于父作用域中，并且子作用域修改了它，那么从同一父作用域继承的所有其他作用域也将看到相同的修改，并且它们的视图将由 Angular 自动更新！这种自动化的方式总比使用 PubSub 要好。

## 10.单元测试就绪

如果不谈论 Angular 的单元测试就绪性，对它的什么描述是完整的？Angular 的整体是通过依赖注入(DI)联系在一起的。这是它用来管理您的控制器和范围。因为所有的控制器都依赖于 DI 来传递信息，Angular 的单元测试能够通过将模拟数据注入控制器并测量输出和行为来篡夺 DI 来执行单元测试。事实上，Angular 已经有了一个模拟的 HTTP 提供者来将假的服务器响应注入控制器。

这击败了更传统的测试 web 应用程序的方法，即创建单独的测试页面来调用一个组件，然后与它交互以查看它是否工作。

这 10 个理由应该能让你明白为什么 Angular 如此强大。不是所有的 web 应用都应该使用 Angular。例如，如果你正在编写一个游戏或者一个计算密集型的数学程序，没有理由 Angular 适合你的特定问题领域。但是对于一般的 web 应用程序，它应该作为一个可行的框架来构建。
