# 在你的 iPhone 应用中避免网络习惯

> 原文：<https://www.sitepoint.com/avoiding-web-habits-in-your-iphone-apps/>

如果你有幸为 iPhone 开发过 iOS 应用程序，你可能有过两次经历:

1.  甜蜜的自由！无需担心跨浏览器兼容性，您可以在应用中构建几乎无限的体验。
2.  瘫痪的自由！就像在麦片货架上挑选正确的盒子一样，太多的选择会让你感到沮丧，并求助于你最熟悉的东西。

在网站上创建优雅而复杂的交互并不困难。以 web 表单为例；一个页面上的许多字段和方向可以通过设计提示、易于浏览的布局和反应灵敏的控件顺利导航。但在 iOS 中却不是这样，在 iOS 中，一系列独立屏幕上的交互必须协同工作，才能创造出一种无缝的体验。

当我在 iOS 平台工作时，我注意到我的网络偏见是如何影响我的 iOS 线框和任务流的。以下是我学会的一些意识到这些偏见并避免它们的方法:

*   重新设定对打字的期望。如今，打字是如此平常，我们几乎不去想它。但是，无论一个人在移动键盘上的速度有多快，在触摸设备上选择文本总是比键入文本更容易。在添加文本字段之前，问问自己:这是绝对必要的吗？  If:这对你的应用程序来说是有意义的，一旦用户开始键入一两个字母，就在输入字段下动态填充一个可能的答案列表( [Google](http://itunes.apple.com/us/app/google-mobile-app/id284815942?mt=8 "Google app as an example for autocomplete") 和 [Path](http://itunes.apple.com/us/app/path/id403639508?mt=8 "Path app as an example for autocomplete") 是许多具有这种自动完成功能的应用程序中的两个)。尽可能地减少互动的时间，因为这些时间会累积起来！
*   **压缩表单和表格的标签。简洁在网络上总是可取的，但在 iOS 上是不可协商的。每当表格元素的标签文本被换行或截断时，都会中断可视流程。在不牺牲意义的前提下，尽量保持标签的简短。**
*   **反思形式**。不要在一个长屏幕上显示一系列连续的问题和输入字段，而是把 iOS 上的表单想象成几个快速交互串在一起。在顶部导航栏中保留取消/后退和提交/下一步按钮有助于保持与 iOS 标准控件的一致性。对用户来说，最好的表单不是苦差事，而是完成一些有用的事情必须采取的步骤。
*   利用手势，但是……手势很棘手，因为它们很容易被误解和滥用。我认为有三种方式最适合做手势:
    1.  像在桌面应用程序中提供键盘快捷键一样提供它们——它是为了改善体验和效率，但不是让应用程序正常工作的必要条件。
    2.  如果您已经创建了模拟真实世界交互的逼真控件，请使用它们。
    3.  当你需要要求故意的意图(比如接电话或删除账户)时，在你的应用程序中呈现它们。
*   用敲击的方式思考，而不是点击。指尖比鼠标指针的尖端大得多，这极大地影响了你的界面。一旦你有了一个应用程序屏幕的线框或模型，退后一步看看它。看起来像是*亲爱的，我把网页*缩小了，很多可触摸的区域挤在一起了吗？手指尖很粗的人会因为沮丧而忍不住把手机扔到房间的另一边吗？这似乎是显而易见的，但停止点击思考可能是一个令人生畏的挑战。
*   不要走极端。很容易反其道而行之，使用一组不同的本地元素，因为它们就在那里。开关是一个例子，它是一个因为新奇而被广泛过度使用的元素。放置得当的界面元素可以让用户控制你的应用，但是多余的交互会让他们感觉像是你的木偶。一双批判的眼睛和一些诚实的同事将有助于防止界面膨胀。

这些指导方针会随着平台的变化而变化，与其严格遵循这些指导方针，不如专注于批判性地思考你的产品所接触的媒介和人群。数以亿计的移动触摸设备的出现正在彻底改变我们对网络甚至桌面应用的看法。现在就挑战你的范式，保持领先！
