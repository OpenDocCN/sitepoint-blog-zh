# 终于有共享虚拟机了？

> 原文:[https://www.sitepoint.com/a-shared-virtual-machine-at-last/](https://www.sitepoint.com/a-shared-virtual-machine-at-last/)

经由，[一篇关于](http://java.sun.com/developer/technicalArticles/Programming/mvm/)的文章讨论了[项目巴塞罗纳](http://research.sun.com/projects/barcelona/index.html)，**孙实现多任务虚拟机(MVM)的研究成果。**如果成功，Barcelona 将使多个用 Java 编写的桌面程序或 Web 应用程序共享同一个虚拟机(VM)。对每个程序来说，看起来它拥有自己的虚拟机。

直到并包括 Java 2 Standard Editon(J2SE)1.4 版，每个启动的 Java 程序都会将 Java 运行时环境(VM)和核心 Java 类的新副本加载到内存中。J2SE 5.0 在某种程度上改善了这种情况，它在所有加载的虚拟机之间共享核心 Java 类的单个副本，但是这些单独的虚拟机使用的资源仍然会增加。

许多开发人员希望看到 Mustang(Java 6.0 的代号，目前正在开发中的)完成剩下的工作，并在所有活动的 Java 应用程序之间共享一个 VM，尽管在 Java 7.0 中这可能更现实。为了在不破坏大量现有 Java 应用程序的情况下实现这一点，Sun 将需要**共享虚拟机，而不让应用程序互相影响。**这就是巴塞罗那。

下面是这篇文章对多任务虚拟机给 J2EE 的 Java Web 开发带来的好处的评论:

> Java 2 平台企业版(J2EE)类似于操作系统，因为一台 J2EE 服务器可以托管多个应用程序。实际上，由于可伸缩性的限制、薄弱的应用程序间隔离以及底层 Java 平台中不充分的资源管理设施，很少这样做。这导致了服务器实例的激增，每个实例通常托管一个应用程序，随之而来的是总内存占用的急剧增加和更复杂的系统管理。MVM 通过隔离的高效实现支持 J2EE 1.3.1 参考实现的多个实例，因此，与 Java HotSpot 虚拟机相比，显著提高了可伸缩性，减少了内存占用和服务器启动时间。这是在不改变服务器代码和不损失性能的情况下实现的。
