# Page_Load 是邪恶的

> 原文:[https://www.sitepoint.com/page-load-is-evil/](https://www.sitepoint.com/page-load-is-evil/)

我们都见过，当我们第一次在代码视图中打开一个页面时，坐在那里等待代码。方便的 Page_Load(对象发送者，EventArgs e)方法。

好吧，朋友们，我今天在这里告诉你们，这种方法是邪恶的。没错，这个友好的代表实际上是邪恶的。原因如下:

1.  它鼓励在页面加载时为了方便而做任何事情。在一个方法中完成所有这些。我经常看到 300 多行 Page_Load 方法。我见过将整个网站编程为 Page_Load 方法。可能 95%的 ASP.NET 代码都存在于 Page_Load 方法中。
2.  它模糊了没有经验的开发者的页面生命周期。太多的人只是使用 Page_Load，因为这是他们知道的在请求页面时执行代码的唯一方法，除了处理回发事件。
3.  Page_Load 有些事情做的太晚了，比如添加控件，迷惑了一些开发者。
4.  Page_Load 做一些事情还为时过早，比如处理按钮点击，迷惑了一些开发者。
5.  从可维护性的角度来看，这使得在必要时将动作移动到页面生命周期的不同位置变得很困难。

**OMG！那是邪恶的！我该怎么办！？！？！？！**

所以，你不是用 Page_Load，是用什么呢？人们应该在构造函数中手工连接页面事件。

为什么这是一种优越的方法？首先，您的操作可以逻辑地分组到它们自己的方法中，而不是像一个庞大的 Page_Load()方法那样有 20 行。第二点，也是最重要的一点，您可以通过更改方法附加到的事件的名称，轻松地在页面生命周期中移动方法。最后，留给你的是更干净、更易懂的代码。
