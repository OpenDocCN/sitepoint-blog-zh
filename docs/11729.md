## 采访记录

凯文:2009 年 8 月 28 日星期五的 SitePoint 播客第 25 集:马特·莫楞威格的 WordPress。大家好，欢迎回到 SitePoint 播客:新闻，观点，以及网页开发者和设计者的新思维。我是主持人凯文·杨克，来自澳大利亚墨尔本的 SitePoint 总部，和我一起的还有我的主持人小组。

布莱德:网络开发工作室的布拉德·威廉姆斯。

iFroggy 网络的帕特里克·奥基夫。

斯蒂芬:来自德克萨斯州休斯顿的斯蒂芬·塞格拉夫斯。

布莱德:大家好，欢迎来到我们的节目。实际上，今天我在运行 master control，我的搭档是 WordPress 名人马特·莫楞威格。欢迎来到节目，马特。

马特:你好。很高兴来到这里。

布拉德:为什么你不花一点时间告诉我们你是谁，为了那 1%不知道的人。

当然可以。我的名字是马特·莫楞威格。大约 6 年前，我开始从事一个名为 [WordPress](http://wordpress.org/) 的项目，这是一个开源博客软件，从那以后，它已经发展到可以做任何事情。你可以用它来写博客，或者用它来驱动你的整个网站。大约几年前，我成立了一家名为[automatic](http://automattic.com/)的公司，其想法是将 WordPress 推向世界，我们最大的项目是[WordPress.com](http://wordpress.com/)，每月有超过 2 亿的访问者。

布莱德:太好了。WordPress.com，那实际上是关机的[对吗？](http://mu.wordpress.org/)

马特:是的。很明显。

Brad: 它是否与我可以下载或设置的 MU 版本相同，还是更像是为 WordPress.com 定制的？

**马特:**基本一样。因此，我们需要定制的东西，显然是因为它运行在一千多台服务器和多个数据中心等所有类似的地方，但我们实际上已经开源了我们几乎所有的系统。对于数据库，我们使用一个名为 [HyperDB](http://codex.wordpress.org/HyperDB) 的类，任何人都可以下载。我们已经发布了一个[工作系统](http://code.trac.wordpress.org/wiki/JobsDocs)，尽管它还不是很出名。我们使用一个 [memcached](http://www.danga.com/memcached/) 插件和一个叫做 [Batcache](http://wordpress.org/extend/plugins/batcache/) 的东西进行缓存，这些都是公开的。这个想法是，我们开发的任何软件，我们都希望它能被尽可能多的人使用。

**Brad:** 回到你所说的，六年前你第一次创建 WordPress 的时候——这实际上是一个来自 SitePoint 论坛成员的问题，[终极版](https://www.sitepoint.com/forums/member.php?u=32952)。ULTiMATE 想知道，当你第一次开始构建 WordPress 时，你是否认为它仅仅是一个博客平台，或者你认为它会发展成现在的样子，很多人把它作为内容管理系统或内容管理系统来使用？

Matt: 它仅仅是作为博客平台而创建的。差不多就是这样。我有一个博客，我想为博客提供动力。软件中的一个大问题是我如何将它嵌入到现有的网站中，并且在某个时候从我如何嵌入它切换到我如何用它来驱动我的整个网站。这实际上更有意义。

Brad: 的确如此，我认为至少在过去几年里，WordPress 在公众眼中已经发生了很大的变化，从一个博客平台变成了一个真正的内容管理系统——它是如此的可扩展，它可以为任何事情提供动力。

Matt: 在网络行家中，这是众所周知的，但更广泛的世界一般不知道这一点，事实上，我们的许多竞争对手说，“哦，如果你只是想写博客，WordPress 很好，但任何更严重的事情，你应该去 X，Y 或 Z…”我显然不同意这一点。

布拉德:哦，是的，当然，和它一起工作，我也会说同样的话。在开发者聊天中，或者在核心开发者之间，或者只是和你一起，有没有讨论过实际上从 WordPress 的不同设置页面和文档中移除博客引用之类的事情？

未来的网站，不管博客是否是它的主要部分，大多数网站都将有一个博客。我非常反对不必要的抽象。

Brad: 在我的工作中，我更多地把它作为一个内容管理系统，而不是博客，但正如你所说，我发布的几乎所有东西都有一个博客功能，所以我可以明确地看到你的想法。下一个问题来自以[为中心的效应](https://www.sitepoint.com/forums/member.php?u=101685)，他们想知道的是，你认为 WordPress 在什么程度上会变得过于臃肿，或者说有没有一个临界点会变得过于臃肿？

是的，绝对是。这些年来，WordPress 已经变得越来越大，但它已经非常一致了——我试图保持它与宽带速度和服务器容量以及一天中的一切相关。我试着让 WordPress 保持大概相同的膨胀水平——可以这么说。一般来说，特别是对于新特性，我们说这可能不适合核心，这是属于插件的东西，这是我们每天都要做出的最困难的决定之一。绝大多数的东西都是插入式的。当我认为某样东西会进入内核时，要么是因为——因为这是两件事之一；80/20 法则——超过 80%的观众会用这个，或者我认为 80%的观众应该用这个。有时候会有一些。偶尔我们会走在曲线的前面，但通常情况下，世界会迎头赶上。

布莱德:当然。有没有讨论过 WordPress 的一个更轻便的版本，它实际上是运行一个网站的必需品？

Matt: WordPress 已经相当轻便了。我担心代码开销的地方主要是在可伸缩性方面。我们已经表明，在一台服务器上，将 WordPress 扩展到每天数百万个页面实际上并不困难，这远远超过了 99.9%的网站所能获得的页面。所以我觉得这种担忧已经得到了很好的解决。

Brad: 我知道曾经有一个分支，我想它叫 LightPress，我不太了解它，但我记得读过一点，他们分支了 WordPress，并试图制作一个更轻的版本。我想它现在已经死在水里了。

马特:是的。这些年来，WordPress 大概有十几个分支。只是它有什么不同，对用户真的重要吗？很难决定，但是我一点都不介意叉子。这是一种开源的方式，如果有一个 fork 真的非常非常酷，我们可能会考虑把它的功能放回核心。

Brad: 从技术上来说，WordPress 是从一个分叉开始的，不是吗？

马特:的确如此。

Brad: 没有 forks，就没有 WordPress，这听起来很棒。

Matt: 这也是开源的基础部分。

**Brad:** 下一个问题实际上来自 [AlexDawson](https://www.sitepoint.com/forums/member.php?u=316859) ，他是 SitePoint 的导师之一，他想知道为什么 WordPress 选择使用 XHTML 1.0 作为 [DOCTYPE](https://reference.sitepoint.com/html/doctypes) 。

那大概是五六年前的事了。

Brad: 嗯，我想一个更相关的问题应该是，既然 XHTML 2.0 已经奄奄一息，有没有计划在 XHTML 之外改变它？

所以，还没有 XHTML 2.0 的计划。HTML 5 中有一些非常酷的功能，整个过程，web 应用程序组，我对此非常非常兴奋，我们将继续接受这些功能，因为它们在浏览器中得到广泛支持。至于代码的实际语法，我们是否要停止自结束 BR 标签和结束列表项？大概不会。即使我们不是 XHTML，它仍然有点像——我喜欢 XHTML 的规则，所以我不认为继续这样有什么错。实际的 DOCTYPE 不像以前那么重要了。

Brad: 我同意这一点，很多人都在谈论 XHTML，现在看来它真的不会有任何发展，HTML 5 会是答案吗？他们会完成 HTML 5 吗？谁来收养它？仍然有很多问题存在，我认为随着时间的推移，这些问题可能会根据 HTML 5 的采用情况得到解答。

**Matt:** 是语法上的事情；没什么大不了的。我不太关心 XHTML 是否是未来的答案，或者在不设置应用程序/xml、MIME 类型和所有这些规则的情况下提供 XHTML 是否正确，而不仅仅是因为它是一个语法清洁器，它是否会迫使你对代码进行一点整理。

Brad:WordPress 中实际嵌入的 WYSIWYG 编辑器实际上并不清理代码，所以它特定于任何文档类型，不是吗？

马特:它竭尽全力。它显然不能清理一切，不能修复一切，但如果你只是使用编辑器，它会产生有效的代码。如果你从一个不同的网页或微软 Word 中粘贴内容，它会再次尝试清理，但有时我们什么也做不了。

Brad: 我知道我认为在 WordPress 的愿望清单上，WYSIWYG editor 总是排在最前面，但我认为这是你工作的任何平台的问题。

**Matt:** 所见即所得一般来说，是超硬的。我们只是试图在每一个版本中做得更好——每一个版本都变得更好，如果你将我们现在所做的与我们两三年前第一次引入它时所做的相比，它已经发生了很大的变化。

布莱德:我觉得稳定多了。我是典型的视图源类型的人，坚持使用 HTML，但最近几个版本，我发现自己使用 WYSIWYG 多一点。

马特:那真是太棒了。对我来说，我会在写作时使用所见即所得；也就是说，如果我想写一篇文章或一些有意义的东西，HTML 会在精神上妨碍我。这并不是说我对 HTML 了如指掌，只是杂乱分散了我对文字的注意力，让我更难编辑自己的文字，而编辑你自己的文字已经很难了。

布拉德:我认为每个人都有自己的做事方式。我仍然习惯于写 hrefs 和图片标签，这对我来说很自然。我不像你那样经常写博客，也不像外面的许多观众那样，但是当我写博客时，我喜欢确保它非常干净和精确，我认为为我查看代码有助于我有这种感觉。但是 WYSIWYG 编辑器是一个很好的选择。

我会经常在所见即所得编辑器中编辑，然后在我发布之前，我会在 HTML 中检查它。

布拉德:查看代码。

这就是为什么我喜欢 WordPress 的新版本，因为它可以让你非常容易地在它们之间切换，它最酷的一点是无论你选择什么，它都会很粘，所以当你下次回到这个页面时，它会默认你最后一次查看的内容，无论是 HTML 还是 WYSIWYG。

布莱德:这是一个很棒的功能。我想谈谈 WordPress 和 WordPress MU，我知道你们几个月前宣布他们将会合并。从那以后我就没有听到太多关于它的消息，所以我想知道你是否可以给我们一个关于它的进展的想法，或者它的版本。

Matt: 这将是一个 3.0 的时间线，就像我在 WordCamp 上说的那样，我们还没有开始工作，所以随着我们开始工作，它可能会发生变化。基本上，这个想法是我们有这两个代码库；它们 99%是相同的，现在让我们把它们结合起来，把开发资源集中起来，这样我们就不会有一个错误追踪器在那边，一个错误追踪器在这边，一些开发人员在那边工作，一些开发人员在这边工作。让我们把 WordPress 的所有资源、意图和想法都用在 MU 上，我认为这将会大大改进产品。

布拉德:哦，是的。我完全同意。我在 WordPress MU 上工作了很长时间，看起来 MU 总是有更多的错误。还是那句话，这只是我基于看 trac 和社区的个人看法。感觉就像 MU，它不像 WordPress 那样有一大群开发人员或者其他人在帮助它。

MU 支持的博客和 WordPress 支持的博客一样多。事实上，有更多的 MU 博客是有争议的。问题是运行它们的人，我猜他们真的很忙，所以他们不像 WordPress 的人那样参与。

Brad: 是的，但是将它们结合起来将基本上消除这种情况，所以你基本上让所有的开发人员都在一套代码上工作，我认为这将使它在现在和未来之间日夜变化。所以我对这次更新非常非常兴奋。我很期待。所以这就是为什么我想从你身上套出一些信息。

**马特:**爽。你应该参与进来。

布莱德:是的，事实上我已经开始涉足 WordPress 了。我会说我想我是我正在谈论的人之一。我在 WordPress 方面的工作比 MU 多，但你是对的，我可能应该更多地参与。我想稍微谈一下 WordPress.org 的[，实际的网站。我知道最近你发布或推出了商业主题页面。它得到了很多媒体的关注，很多人的谈论，我认为这是一个伟大的功能，我认为这是一种普遍的共识。有没有计划对插件做同样的事情，并有一个商业插件部分，本质上只是促进那些符合 GPL 的商业插件？](http://wordpress.org/)

短期内可能不会。说实话，没那么多人要。这真的就像一两个人，他们问了很多，但就人数而言并没有那么多，而主题页面上有很多人…我认为主题和插件有一点不同，主题更像是设计网站的基础，有点像是构建模块，插件通常只是其中的一小部分。所以说实话，我觉得主题比插件有更好的商业价值。

是的，我同意。这实际上是我的后续工作，感觉这是每个人的心态，商业主题被社区接受和理解；而商业插件几乎是不被接受的。如果我发布一个插件并收费，更多的人可能会告诉我不应该这样做，即使它仍然符合 GPL。

Matt: 更有可能的是，有人会创造出一个好的免费替代品，但这种情况不太可能发生。

布拉德:完全正确。也许我也是这样，我遇到了一个需要花钱的插件，很可能你是对的，有一个免费的替代品可以做一些类似或非常接近该功能的事情。

WordPress 的大部分功能首先以插件的形式开始。所以如果所有的插件都是商业化的，那将会严重抑制，因为即使它们是 GPL，这些家伙显然会希望我们在 WordPress 中加入功能，我认为这将会严重抑制 WordPress 的发展。

Brad: 一些插件保证有价格标签，一些更复杂的电子商务表单，诸如此类的东西……如果你看看这些插件背后的源代码，你会发现很多工作都被放入了插件中。

马特:这不是工作量的函数，因为显然 WordPress 投入了大量的工作。

布拉德:没错。这是一个非常好的观点，最终，这是每个人的决定，他们想如何释放它。我是一个大…

我喜欢说生活中最好的东西都是免费的。

嘿，WordPress 就是一个很好的例子。我是 GPL 的忠实粉丝，我喜欢看到发布 GPL 的东西。如果人们为某些东西付钱或要钱，我真的没有问题。

此外，如果你看看商业主题页面的方向，他们并没有真正对下载收费，越来越少；他们对支持和定制以及相关工作收取更多费用。我认为插件也完全可以向那个方向发展。

Brad: 是的，这实际上是一个很好的商业模式，我知道有几个这样的插件。我认为电子商务插件就是其中之一。下载是完全免费的，如果你需要支持，你需要付费。我认为对于插件开发者来说，这可能是一个很好的商业模式。WordPress 2.8.4 最近发布了，我这里有一些关于版本发布的统计数据。所以在这里请原谅我。因此，WordPress 2.7 于 2008 年 12 月 10 日问世，整整两个月后，WordPress 2.7.1 问世了。WordPress 2.8 于 6 月 11 日发布，整整两个月后，WordPress 2.8.4 也发布了。基本上，在 2.7 有一个次要版本的同时，2.8 有四个次要版本；所以我很好奇，2.8 怎么了？是不是有很多功能还没有准备好推出？也许你可以解释一下这里的不同。

**马特:**其实正好相反。如果我们发布一个 bug 修复版本，就像 2.7.1 一样，可能会花同样多的时间，因为发布版本在质量和通过的 bug 方面是相似的，但是 WordPress 2.8 到目前为止的四个版本都是安全修复。因此，这些事情，而不是如果它是一个错误，我们把它们都拉起来，我们得到了一大堆的修复，这更多的是一个时间问题，就像我们的目标是在主要版本发布后一到两个月。如果是安全问题，我们会尽可能快地发布出去，对吗？因为否则的话，博客就危险了，你的网站、你的内容或任何东西都可能发生问题。所以这是我们的责任，我们的义务去尽快获得释放。当我们发现一个严重的安全问题时，比如 2.8.4 中的问题，有人可以远程重置你的密码；尽管我讨厌发布太多的内容，但这是正确的做法。

布莱德:对，绝对是。这是一种有趣的错误。我想每个人——显然，我可能不应该笑它——但我想每个人都有一点乐趣，重置他们所有朋友的密码。我认为你们认真对待安全问题非常好。我知道每当我和不熟悉开源的人谈论开源时，这是他们口中的第一个问题。如果有人能看到我的源代码，那怎么保证安全呢？我如何知道我的网站是安全的？我认为，当发现安全漏洞时，你发布消息的方式证明了你对你的软件或 WordPress 和开源软件的安全性有多重视。为此脱帽致敬。它实际上回避了一个好问题。当这个漏洞出现的时候，我正在跟踪这个故事，或者说这个漏洞，它被广泛报道，它被报告给 WordPress 的方式实际上是一个博客帖子，基本上说，“这是你如何利用这个漏洞。”你认为提交安全漏洞的适当方式是什么，而不是仅仅写博客让全世界都看到？有没有更好的方法是人们应该使用的？

Matt: 那么，在过去几年中，我们发布的 2.8 的其他版本以及所有安全版本都是在漏洞发布之前发布的。大多数负责任的安全研究人员所做的是他们会联系供应商——如果是我们——我们有一个开放的电子邮件地址，任何人都可以通过电子邮件向我们发送任何信息，我们会非常认真地跟进每一份报告。我们研究它，我们验证它，我们修复它，我们发布它，然后他们在几天后宣布它。这让每个人都有机会升级和一切，所以每个人都受到最大的保护。因为他们的利益实际上是——发现问题的恶名，他们并没有真正从零日漏洞或一堆被搞乱的博客中受益。所以大部分时间都是这样的。在这种特殊情况下，我们和其他人同时发现，这并不理想，这也是我们不得不开夜车解决问题的原因之一。
