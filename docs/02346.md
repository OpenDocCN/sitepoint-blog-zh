David:

在这一集中，我们将与 Alex Walker 交谈，他是 SitePoint 的设计师和网站经理——根据他的简历，他自 2001 年以来一直在对 CSS 做残忍和不寻常的事情。

Tim:

紧接着，我们有一个问题想问所有的客人，那就是，<q>在你目前的职业生涯中，你是什么样的，为什么？</q>

Alex:

我想我大概是 3.0 吧。嗯，实际上，如果你谈论我的整个职业生涯，我会回到印刷业。我从事印刷业。所以，如果你想把这些内置进去，我可能会更喜欢 4.0。但就互联网和技术之类的东西而言，我认为第一部分是，我是[香肠](https://en.wikipedia.org/wiki/Sausage_Software)的网站管理员。那是人们曾经拥有的——实际上我仍然怀念那个电子邮件地址，站长@香肠……我的第一个严肃的电子邮件地址。所以如果可以的话，我想拿回那个，但是我觉得这是不可能的。

但那是一次真正的学习经历。我第一次花了很长时间处理代码，之前我更多的是设计背景。

所以，这是一个飞跃。然后，SitePoint 的开始是与客户打交道，参加客户会议，以及建立网站本身。我认为第三版是——我现在更像一个编辑和作家。

我每周写一篇新文章，我为网站处理其他人的工作，在那里我们编辑、设置和协商内容。所以，我估计我现在是 3.0。

David [2:13]:

我看得出你也是这个网站的一名多产作家。你从 SitePoint 的最开始就一直在写作，是吗？

Alex:

我想我的第一篇文章是 2002 年或者 2003 年。不完全是从一开始。凯文·杨克是我们真正的早期作家之一；我们在网站上仍然有他的一些书。在早期，我大概每六个月写一篇文章，然后从 2004 年到 2010 年，我做了一个月度时事通讯。

我记得每个月我都会有点恐慌地想，哦，不，我得想出今年要写的 12 个想法。我该怎么做呢？在过去的两年里，我每周都会写一篇文章，这种文章会出现在设计时事通讯的顶部。

很奇怪。我以前听人说过——经常写作比偶尔写作更容易，事实确实如此。我记得我在想，<q>我会在六个月后完全没有想法，</q>但我仍然有很多积压的东西可以写。

令人惊奇的是:即使我已经把事情安排好了，一周内还是会发生一些事情，或者突然出现一些我们正在做的事情。或者有人写了一些刺激的东西…甚至我可能只是在看历史频道或古董巡回展览或类似的东西，那里会有一些我会联系到我们所做的事情的东西。

它会驱使我朝着一个方向前进，你走吧。这比我想象的要容易，但希望不会停止。我仍然总觉得总有一个月我会碰壁，就像是，我无话可说了。

David:

Alex [4:03]:

我想对于我五年或十年前做的时事通讯，我肯定对 CSS 更感兴趣，我想我做了一些非常有趣的东西。但是这需要大量的研究，大量的测试和阅读说明书。一个月做一次并不太难，但是后面的工作——我不认为你可以每周都做，特别是像 CSS 这样的东西。你可以重申你以前做过的事情。但是要想出新的想法，从不同的角度谈论一些有趣的事情，我发现只是坐在那里阅读材料要容易得多。

我有一块特雷罗板，可以在手机上调出。我会看电视，有人会说些什么，我会说，哦，我的上帝！

我的伴侣有时会转过身来对我点点头，然后说？

我会去，是的，时事通讯。因为那时我们只会在电视上看到一些东西。她知道——我不知道是否有一个光球出现在我的头上。但也许我会坐得更靠前或站得更直些。

正如我之前提到的，有一个节目叫做古董路演，这是一个谈论古董和这类事情的英语节目。但是我已经写了一些文章，这些文章是我在那里完成的，比如你敲响的会说话的钟。

1780 年有个人发明了复印机。这就像一个有滚轴的胡桃木桌子，他(一个商人)会把他所有的东西写出来，他有湿润剂和其他纸张，他可以把信滚过去，它会在另一面复印出来。

所以，复制是我们这个行业的关键要素之一。如此多的问题都围绕着数字版权管理和知识产权，你会想，从哪里开始呢？我们谈论的是 18 世纪 80 年代，这是我们第一次有了能够如此精确地复制东西的想法。

David [6:13]:

你写的东西中有哪些对你来说很突出？

Alex:

我认为文章[蝉的原理和它为什么对网页设计者重要](https://www.sitepoint.com/the-cicada-principle-and-why-it-matters-to-web-designers/)是那些在黑客新闻和一些大的来源——Slashdot 上爆炸并被选中的文章之一。这可能是我的终极文章，因为我总是喜欢将科学和历史与设计融合在一起——这是我的风格。我一直是我的艺术朋友中的极客，也是我的极客朋友中的极客，所以这是一种融合——我认为这是我特别擅长的。我不认为我很擅长这些事情，但是混合是我能做的事情。

大约四年前，我写了一篇关于蝉的原理的文章，这篇文章一直在我的脑海里打转。我想我看过一个关于蝉的历史频道节目——它们在地下挖隧道很长时间，然后出来繁殖，用同时出来的蝉的数量淹没环境，然后再次消失。

奇怪的是，对于其中一些物种来说，这是 13 年和 17 年。它们是质数，没有人能找出这个奇怪数字的原因。它们会在 13 年后出现——彼此相隔不到一个小时——从地下冒出来。他们最终发现，这是因为以蝉为食的动物——如蜥蜴和蛇——这些动物往往处于四年或六年的盛衰周期。如果你试图将 6 年的繁荣和萧条周期排列起来——比如说一条蛇或一只鸟或类似的东西——这种模式需要很长时间才能排列成 17 年的交叉。

所以，这些数字对打破模式真的有好处，我记得我认为这意味着什么。我不知道它是什么，但它意味着什么。最终，我在一个阶段使用了带有平铺背景的多层图案。而这种事情不排队的想法，正是你在不想重复的背景下想要的。

所以，使用重叠质数在瓷砖实际宽度上的想法，你开始得到这些真正疯狂的事情发生。所以你可以用这些不重复的多层瓷砖让成千上万的像素横向排列，或者它们确实重复了，但是当你开始让这些数字相互作用时，你会发现惊人的数学现象。

所以，当我写它的时候，就有一些东西在那里，我开始通过它工作，做数学。当它开始出现在我面前的时候，我在和人们核对，然后说，<q>哇，是我错了吗，还是这很神奇？</q>人们从椅子上站起来说，<q>不，不，这不可能。</q>

但是，是的，进行得非常顺利。它在网上火了起来。很长一段时间以来，这一直是可能的，没有人真正注意到这是你可以做的事情。所以，是的，它肯定还在我脑海中浮现。

Tim [9:36]:

这是一个非常酷的突破，尤其是在一个人的职业生涯中——当你知道那里有什么东西，而你只是看着并说，<q>嘿，等一下，如果我们只是把这个东西和这个东西放在一起…</q>

Alex:

有时像 SVG 这样的东西出现了，这些都是新事物，所以有很多新的方向。但是，是的，在我们已经用了很长时间的东西里找到有用的东西是一个美好的时刻。

Tim:

是啊，肯定的。你之前提到你从印刷行业转型到了网络行业。你想谈谈你是如何完成这一转变的，以及你为了能在不同的领域工作而获得的技能吗？

Alex:

是啊。实际上，我 17 岁的时候在一个商业领域——或者说学徒领域——做学徒，这个领域叫做<q>图形复制</q>。这是杂志制作过程的一部分。这也是报纸业务的一部分。

当我开始的时候，这都是基于电影的。我们使用画笔和 Rubylith，以及我们使用的这种可剥离的蒙版物质——这很像你在 Photoshop 中使用的快速蒙版——这种红色的可剥离的东西甚至有点类似于他们今天在 Photoshop 中所做的。我们通过在其他胶片上重叠和曝光胶片来制作杂志封面。

这些年来(那是在 80 年代)，麦金塔电脑开始进入整个行业，所以我开始接受这方面的培训。我们在那个阶段使用的扫描仪有汽车那么大。这些东西旋转着，它们有一个鼓，你必须给这些透明的鼓上油。我们现在仅仅用桌子上的一个愚蠢的小东西就能做这么多，这太令人惊讶了……我们花了几个小时做这种事情。

我在那工作了大约十年，开始时是一份相当有趣的工作，但它逐渐变得越来越自动化，它不像我开始时那样有用和有动力。

所以我得到了一点奖金，因为他们在减少公司的人数。这实际上给了我两年的时间重返校园。我基本上是在学习所谓的电子设计，在那个阶段(我们在谈论 97 年，98 年)，几乎我班上的每个人都在学习为 Macromedia Director 设计。

每个人都只是，<q>我们不想学—</q> …那是什么代码？这是某种代码——<q>ExtendScript</q>或者某种疯狂的东西。每个人都想制作光盘，我真的不认为这是未来。我记得有一次我和另一个女孩对做互联网的东西很感兴趣，其他大多数人都很喜欢，<q>呃，代码，不要给我看代码！</q>

*【咯咯地笑】*

所以我开始涉足 HTML，我认为我很早就做了一个好决定。我开始使用一个名为 Arachnophilia 的小文本编辑器，这是一个非常简单的文本编辑器，用于编排代码。班上的每个人都被告知使用 Adobe PageMill，而且——唉！那东西太可怕了。它写的代码！它把代码放在屏幕后面，就像绿野仙踪一样。你真的不知道它在做什么。这就像使用 Word，你真的没有学到很多使用它。

所以我从那时起就开始摆弄这些代码，我认为这为 CSS 开始成为一种东西做好了准备。我马上就开始玩它了，在 SitePoint 早期，这当然是一件大事。我们正在推动一些事情，比如远离基于表格的设计。我认为我们是最早这样做的网站之一——当时技术并没有很好地支持它。

很难记得在那些日子里，仅仅用浮动来实现非常简单的事情有多难。

David [13:47]:

这些都是你自己手工制作的吗？

Alex:

是啊。我们有一个从零开始构建的 PHP 后端，是我和 Julian 设计的，它不能以我们现在谈论的方式响应。但是它可以在更小的设备上运行，我认为我们当时已经走在了这条曲线的前面。

David:

有趣的是，你已经从实际动手，在网站上做所有的基本编码，到现在处于一个不干涉的位置——我假设还有其他人在为网站动手编码。

Alex:

是的，我们有一个团队，我倾向于贡献更多我想出的组件。目前，我们可能在网站上看不到它，但在业余时间，我正在研究一个关于 SVG 和黄金比例的想法。我基本上有一个可重复使用的图标网格，我想为每篇文章生成一个新的艺术作品，在 SVG 中，我使用文章的标题，作者的名字和日期来创建一个随机数，它被输入到这个小图中，它基于一组规则创建一个新的布局，这将生成一个调色板。

我正看着这些单词，如果有一个单词像 <q>make</q> 或 <q>PHP</q> 或其他实际上意味着我可以将它附加到我得到的 40 个图标中的一个上，它就会使用它。但除此之外，我一直在观察包豪斯的艺术家，并从中获得灵感。他们是我最喜欢的设计师和艺术家。

所以我试图把这一点带过去，并使用许多人——康定斯基和那些人——使用的那种真正的几何概念；试图梳理出他们使用的布局规则，看看我是否可以将它们重新输入到一个不断自我再生的单一图形中——我猜是为每篇文章。

所以，如果你给它同样的信息，你每次都会得到同样的图。

David [16:08]:

我很期待看到你肯定会写的那篇文章。

Alex

是的，进展很慢。你不想把它弄得太复杂，否则没人能理解你在做什么，这真的很难解释。

所以，我认为实际上当你写作时，这是一件非常有用的事情——当你设计你将要写的东西时，因为你知道你必须解释它——当你建造它时，你必须努力使它简单，这样你就可以解释它！

这几乎就像是一个因素，将事情拉回到一个合理的水平，这实际上对设计是有用的，我认为。

David:

这种方法——你必须考虑如何解释这些事情的事实——改变了你的发展方式吗？

Alex:

是的，当然，这也是我在编辑和与作者一起工作时努力追求的东西。

很多时候，人们甚至会以一种有很多细节并且正确的方式解释事情，但有时在那个特定的句子中可能有五件事情，它们都是正确的，但只有两件是真正有用的——其他的只是碍事，弄乱了句子。

所以这整个想法实际上是写作完成的。我认为海明威谈论健壮的写作和充满活力的写作和简化。我想他说过这样的话，在你开始写作之前，你必须了解一个故事的所有部分。你要知道所有的细节。你不能略读并假装那些东西不重要，但你也不能把所有这些都放进故事里。你必须在开始之前了解它，但你必须选择故事的部分，并让其他东西几乎在水下，这样才能工作。这两种方法——要么遗漏内容，要么试图全部塞进去——实际上都不适合好的写作。所以我认为这实际上适用于编码和其他各种事情。

所以，这绝对是我认为写作教给我的一个原则，我把它带进了设计，甚至是我做的编码工作。除了写作，我认为海明威还可以教你很多东西。

David [18:15]:

我记得几年前有一本 JavaScript 书出版，作者是 Angus Croll，书名是[如果海明威写了 JavaScript](https://www.nostarch.com/hemingway)——关于不同作者编写代码的方式。

Alex:

我看过那本书，是的。亚当把它放在这里，所以我浏览了一下。是的，我相信里面还有莎士比亚和其他一些作家的作品。

David:

我喜欢你的文章的一点是，你没有将自己局限于经典的技术设计问题，甚至是代码设计问题。你有关于如何写作，如何处理版权问题的文章。

Alex:

是啊，绝对的。正如我所说的，我认为当我把自己限制在与 CSS 直接相关的主题上时，我对找东西写感到恐慌。

所以，如果这只是我会读的东西，我会想，<q>哇，这真的很有趣，</q>或<q>这是对我有用的东西，</q>或<q>这是我这个星期学到的东西，我不知道它，它让我的生活变得更好了一点，</q>那么我通常肯定有人会发现它很有用。

所以，如果我可以把标尺放在它上面，说<q>这是值得的，</q>我会写一篇关于它的文章。如果有人写信给我说，<q>这与设计无关，我愿意被责骂。</q> *【笑】*那好吧。

Tim:

所以，我想你已经看到网络不断接受新的挑战，并能够完成新的任务。你认为网络所能处理的范围会达到顶峰吗，或者你认为我们会继续发现网络所能做的我们以前从未想象过的新事物吗？

Alex [19:55]:

我认为，就我们能把事情推进到什么程度而言，我们还有很长的路要走。

我的一部分渴望 21 世纪初，在那里，作为一名设计师/程序员，你几乎可以自己设计和做任何事情。你是编码员，你是后端人员。我曾经在其中写过 PHP 和一些 JavaScript。跨越一切是一件很酷的事情，现在事情变得如此复杂和强大，以至于我们让 <abbr title="user experience">UX</abbr> 的人处理小的部分，而设计师则交给前端人员。
