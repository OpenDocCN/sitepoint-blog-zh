# 当心自动化可访问性工具陷阱

> 原文：<https://www.sitepoint.com/automated-accessibility-trap/>

自动化辅助工具是一种可以测试网页甚至整个网站的辅助功能的软件。自动化辅助工具非常有用，因为它们可以节省你大量的时间。不想在网站的每一页上检查图片中的替代文字吗？通过一个自动测试器运行这个网站，它会为你做所有的检查！

自动化易访问性测试工具已经存在很长时间了，并且一直是检查网站易访问性的有用方法。Bobby 是第一个也是最著名的自动化可访问性测试工具之一，现在已经快 10 岁了，虽然它不再免费提供，但现在已经有很多其他免费工具存在，如 [WebXact](http://webxact.watchfire.com) 和 [Wave](http://wave.webaim.org) 。

但是这些工具是不是好得有点不真实？我也这么认为你真的能这么容易地测试一个网站的可访问性吗？不幸的是，答案是响亮的“不”。大量的问题与单独使用自动化工具来测试网站的可访问性有关。

##### 自动化工具逐字解释可访问性指南

作为一个软件，自动化的易访问性测试工具并没有太多的常识。它将逐字解释每一个可访问性指南，而不考虑页面上发生的其他事情。

Dictionary.com 认为,“准则”一词的定义是“为适当行为提供指导的规则或原则”。指南只是提供了最佳实践的指导:它不应该在不考虑其他因素的情况下照原样应用。

例如，W3C 可访问性指南中的[规定](https://www.w3.org/TR/WCAG10/full-checklist.html)[应该为每个表](https://www.w3.org/TR/WAI-WEBCONTENT-TECHS/#tech-table-summaries)提供一个摘要。该摘要不会出现在屏幕上，但会在阅读表格内容之前大声朗读给屏幕阅读器用户。表格摘要很有用，因为它们告诉屏幕阅读器用户在表格中可以看到什么。太好了！…但是假设您在表格的正前面放置了一个标题，它也描述了表格的内容。在这种情况下，该摘要基本上是多余的，因为它可能会重复标题。然而，无论如何，自动化的可访问性工具会将这识别为一个问题。

##### 自动化工具无法检查内容的可访问性

内容的组织方式——无论是在页面上还是在整个网站上——是网站可访问性的重要组成部分。一个网站可能是完美的编码，并符合最高的编码标准，但如果其内容结构不良，该网站将被证明很难或不可能为一些有特殊需求的访问者所用。

自动化可访问性测试工具无法检查许多重要的内容可访问性考虑因素，包括:

*   内容的“前置”，这样每一段都以结论([检查点 13.8](https://www.w3.org/TR/WAI-WEBCONTENT-TECHS/#tech-front-loading) )开始
*   确保内容被分成可管理的块，前面有描述性的副标题([检查点 12.3](https://www.w3.org/TR/WAI-WEBCONTENT-TECHS/#tech-group-information) )
*   在适当的地方使用列表([检查点 3.6](https://www.w3.org/TR/WAI-WEBCONTENT-TECHS/#tech-list-structure) )
*   确保使用简单明了的语言([检查点 14.1](https://www.w3.org/TR/WAI-WEBCONTENT-TECHS/#tech-simple-and-straightforward) )

##### 自动化工具无法检查许多编码问题

大量的可访问性指南往往与站点的编码方式有关。然而，自动化的易访问性测试工具不能测试对这些指南的遵守情况。自动化工具无法检查:

*   文本是真实的文本，并不嵌入图像中([检查点 3.1](https://www.w3.org/TR/WAI-WEBCONTENT-TECHS/#tech-use-markup) )
*   该站点无需使用 JavaScript 或 Flash 即可运行([检查点 6.3](https://www.w3.org/TR/WAI-WEBCONTENT-TECHS/#tech-scripts) )
*   在使用服务器端图像映射的地方提供了等效的文本链接([检查点 1.2](https://www.w3.org/TR/WAI-WEBCONTENT-TECHS/#tech-redundant-server-links) )
*   HTML 中的结构反映了页面的视觉外观(例如，标题被标记为 HTML 代码中的标题)([检查点 5.1、3.5、3.6、3.7、5.4](https://www.w3.org/TR/WCAG10/full-checklist.html) )

如果我们仅仅依赖自动化工具，我们将无法检查这些重要的指导方针是否得到了满足。

##### 自动化工具在许多情况下使用过时的指南

自动化易访问性测试工具通常使用 W3C 易访问性指南，该指南已经有五年多的历史了。其中一些准则已经过时，不再适用。事实上，其中一些现在被认为是阻碍可访问性而不是帮助，所以最好忽略这些指导方针。

例如，一个自动化的易访问性测试工具可能会坚持表单项目包含默认的占位文本( [checkpoint 10.4](https://www.w3.org/TR/WAI-WEBCONTENT-TECHS/#tech-place-holders) )。它也可能坚持链接由非链接文本分隔([检查点 10.5](https://www.w3.org/TR/WAI-WEBCONTENT-TECHS/#tech-divide-links) )。这些准则都不再相关；事实上，它们的实现可能会使可访问性变得更差而不是更好。(必须注意的是，W3C 声明，只有当用户代理继续需要它们时，这两个指南才是相关的；尽管如此，大多数自动化工具仍然会因为网页违反这些准则而失败。)

##### 自动化工具不能正确检查大多数指南

自动化可访问性工具可以检查大量的指导方针，并在没有达到指导方针时告诉您。然而，在某些情况下，该工具声称正在满足某个准则，这实际上可能是不正确的。

例如，如果所有图像都包含 alt 文本，软件将报告该准则通过。但是如果替代文本不能描述它的图像呢？如果 alt 文本被搜索引擎塞满了无意义的关键词会怎样？自动化辅助工具怎么可能知道这些？

##### 自动化工具生成的警告可能会被误解

自动化可访问性工具生成的报告提供警告和错误。这些警告基本上是自动化工具无法检查的指导原则，但可能是错误。通常他们不是；事实上，通常它们甚至不相关。但是，一些阅读包含此类通知的报告的人可能会尝试通过对其网站进行适当的更改来消除警告消息。这样做的话，他们可能会执行不必要的指导方针，并且在这个过程中，无意中降低了网站的可访问性。

##### 不要落入自动化工具陷阱！

自动化的易访问性测试工具可能是有用的——它们可以在执行一些非常基本的易访问性检查时为您节省大量时间。但是，必须谨慎使用它们，并且不能将它们用作可访问性检查的独立指南。事实上，在评估一个网站的可访问性时，专家的可访问性知识应该总是被应用，也许与 Internet Explorer 的奇妙的 Web accessibility [工具栏或 Firefox](http://www.nils.org.au/ais/web/resources/toolbar) 的同样优秀的 [Web Developer 工具栏一起使用，这两者都可以帮助大大加快人工检查的速度。](http://chrispederick.com/work/firefox/webdeveloper/)
