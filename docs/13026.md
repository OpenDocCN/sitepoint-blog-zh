# 虚拟主机的变化

> 原文:[https://www.sitepoint.com/the-changing-face-of-web-hosting/](https://www.sitepoint.com/the-changing-face-of-web-hosting/)

本周早些时候，谷歌[在](https://www.sitepoint.com/google-reveals-app-engine)宣布了[谷歌应用引擎](http://code.google.com/appengine/)的预览版，这是一项可以让你在谷歌自己的服务器基础设施上构建和运行网络应用的服务(目前只有 Python 版本)。这一最新举措延续了脱离自管理托管的趋势，并通过提供完全托管的应用程序环境来扩展这一趋势。

![](../Images/b05f208672031465a0b1634139a0f775.png)

首先，让我们礼貌一点，抛开谷歌搞糟应用引擎发布的事实——首先是[被自己的配额](http://valleywag.com/377304/google-app-engine-cant-handle-demand-for-google-app-engine)绊倒，然后屈服于[基本的跨站脚本攻击](http://valleywag.com/377603/google-app-engine-page-already-owned-by-hackers)。我们就假设预览版在完全准备好之前就被赶了出去。我们也会忽略 App Engine 与竞争对手 [AppJet](http://appjet.com/) 惊人的相似。

谷歌应用引擎加入了亚马逊弹性计算云(EC2)的行列，提供可立即扩展的托管服务(T2)。在这个不确定的时期，在托管服务上建立你的网站会令人难以置信地安心，如果你的业务永远不会起飞，它会保持便宜，但它也可以毫不费力地跟上失控的打击。

App Engine 击败 EC2 的地方在于对初学者的友好性。您可以使用一个命令将新应用程序部署到 App Engine。谷歌处理所有的基础设施细节，所以你甚至不必考虑负载平衡和数据库备份之类的事情。

App Engine 的衰落是锁定的。为 App Engine 构建的应用程序将只能在 App Engine 上运行，而不是为 App Engine 构建的应用程序*将需要进行重大修改才能在谷歌的服务中工作。*

相反，EC2 的优势在于灵活性。EC2 允许您建立一个虚拟服务器集群，其行为就像专用的物理计算机一样。您可以选择在这些服务器上运行的操作系统和软件，并且可以控制在任何给定时间托管您网站的虚拟服务器的数量。但是所有这些设置都需要时间，而且需要大量的技巧。

很明显，EC2 提供的灵活性使其非常适合托管具有大量基础设施需求的已有应用程序，但也给刚刚起步的初学者带来了障碍。但 EC2 平台的开放性意味着这一障碍只是人们建造(并可能出售)梯子的机会:

*   Scalr 是一个最近推出的开源项目，它自动化了 EC2 集群的管理。虽然仍处于早期阶段，但它已经做了我们必须为 SitePoint 的 EC2 托管应用程序编写的定制系统所做的一切。

*   Heroku 是一个基于浏览器的开发环境，用于构建 Ruby on Rails 应用程序。在您的浏览器中编辑您的应用程序代码，然后单击一个按钮将应用程序自动部署到 EC2！所有细节都是为你管理的。

不可避免的下一步是有人在 Amazon EC2 平台上实现 Google App Engine 提供的服务。将这两个竞争对手的优势结合在一起，事情将会变得非常有趣！
