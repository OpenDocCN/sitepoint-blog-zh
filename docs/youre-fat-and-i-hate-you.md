# 你很胖，我讨厌你

> 原文：<https://www.sitepoint.com/youre-fat-and-i-hate-you/>

所以这种情况最近在我身上发生过几次(没有提到名字)——我读了一些某人在 JavaScript 中发现的巧妙技巧，我就像<q>哦，太酷了，我想知道它是如何工作的</q>。所以我继续追踪下去，结果发现作者*不知道*是如何工作的，阅读他们的代码也没弄清楚，因为**的大部分工作是由外部框架**完成的。

让我恼火的是，这么多现代 JavaScript 开发都依赖于框架。这并不是因为从实用的角度来看这有什么不对，而是因为**我对事物的机制**感兴趣，而**用框架编程掩盖了机制**。要处理错综复杂的依赖关系链并了解脚本实际上在做什么实在是太费力了。框架本身的代码通常被优化到难以辨认的程度——这在实践中有助于提高速度和效率，但很难阅读和理解。

当然，从使用框架的开发者的角度来看，这正是重点。机制应该被隐藏起来，这样应用程序可以更容易更快地创建。当然，实际的最终用户根本不应该关心，就像我不关心我的车如何工作一样，我只想让它跑起来。

但是如果我是一个汽车修理工，我可能会有不同的观点…我们在这里！我们发现，整整一代开发人员现在都在开发复杂的应用程序，却没有深刻理解它们是如何工作的。我想我可能听起来有点精英主义，也许我是，但它仍然困扰着我，因为它使许多开发对我来说毫无用处。我根本不在乎 X 用 jCloth 或者别的什么做了一个更好的图像预览脚本——我并不印象深刻，因为就我而言，他们并没有写出来；如果有人使用卡西欧键盘上的预设来制作音乐，我会印象深刻(虽然他用的是雅马哈键盘！).

我可能仍然喜欢音乐，但我不会认为他们是音乐家，也不能和他们谈论音乐理论的细节。

如果我想使用一个巧妙的技巧，或者一个有意义的特殊方法，但是如果不在代码中引入依赖关系，我就无法使用它——依赖关系会使代码库膨胀，而 T2 会使应用程序变慢。JavaScript 已经是一种解释语言，框架是解释环境，所以使用框架编写的应用程序本质上是使用元代码——由解释器解释的代码！增加解释者的工作和代码库的大小怎么会是一个好主意呢？

这让我想起了一部名为《最大输家》的电视剧。这个节目就像有慢性体重问题的人的老大哥——每周他们试图通过速成节食和过度锻炼来减肥，每周他们中的一个被投票淘汰，最后剩下的人被冠上<q>最大减肥者</q>(很棒的双关语哈！).但该节目与健康或健身无关；节目制作人根本不关心参赛者的健康，也不关心他们在鼓励如此密集和不可持续的减肥方面树立的坏榜样。这个节目不是关于这些，它只是关于<q>让我们一起嘲笑胖子</q>。

在我看来，框架驱动的开发与此类似。它可能会产生快速而简单的结果，但它不是真正的编程，我对它一点学术兴趣也没有。我不在乎它是否能成为好的电视节目，我在乎的是其背后的实质。

我对这种语言本身很感兴趣，我发现很少有人真正用它来写作，这非常令人沮丧。
