# 添加离线处理的应用程序引擎

> 原文：<https://www.sitepoint.com/app-engine-to-add-offline-processing-xmpp/>

![](img/5677ebf13b4f25cb6efa7a12d1824594.png "google-appengine")上周末，[谷歌更新了他们的应用引擎路线图](http://googleappengine.blogspot.com/2009/02/roadmap-update.html)，纳入了一些他们计划在未来六个月内发布的新 API。这些路线图更新可能不像去年 10 月 [Q1 2009 路线图承诺](https://www.sitepoint.com/google-reveals-app-engine-product-map/)那样吸引眼球，但仍然很重要。

在接下来的六个月里，或者大约在 5 月份的谷歌 I/O 活动期间，当时只有一个月大的应用引擎是去年的明星，谷歌计划向应用引擎添加以下新的 API:

*   支持运行计划任务
*   用于执行后台处理的任务队列
*   能够接收和处理收到的电子邮件
*   支持发送和接收 XMPP (Jabber)消息

根据一个叫做“App Engine 上的离线处理”的 Google I/O 会话的描述，App Engine 是为请求驱动的 web 应用程序设计的。通过添加对离线处理的支持，开发人员将能够在 App Engine 上创建全新类别的通信应用，如游戏、微博和即时消息应用。

几周前，当谷歌宣布将 Jaiku 移植到平台上时，开发者 Gareth Rushgrove 实际上已经预测到了 App Engine 的这一变化。“我认为，没有 XMPP 和离线处理 API，你不可能真正做到这一点，”他说。

预计 App Engine 还将在未来几个月内增加一项存储大文件的服务，一种让开发者为更多资源付费的方式，以及一种新的运行时语言——可能是 Java。去年 12 月，谷歌暗示将为 App Engine 推出类似 AdWords 的定价系统。

App Engine 实际上是谷歌网络操作系统战略的第四个方面，包括 Gears(离线数据存储)、Native Client(网络应用的本地 CPU 资源)和 Chrome(客户端网络应用交付)。谷歌正在推动一个计算的未来，在这个未来中，应用程序从云端交付，计算机操作系统并不重要。客户端运行的是 Windows、Mac、Linux 还是其他系统，这与谷歌的 Web OS 愿景无关。

App Engine 为应用生态系统提供了开发、托管和部署框架，Google 希望为他们的 Web 操作系统创建应用生态系统。自从 App Engine 去年 4 月推出以来，他们一直在快速迭代和改进它。如果他们能够实现过去几个月宣布的路线图目标，这个已经引人注目的平台将对开发者更具吸引力。
