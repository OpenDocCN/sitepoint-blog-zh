# 不应发布支持的浏览器列表的 5 个原因

> 原文:[https://www . site point . com/5-reasons-to-avoid-supported-browser-lists/](https://www.sitepoint.com/5-reasons-to-avoid-supported-browser-lists/)

桌面开发者在转向网络时遇到的最棘手的问题之一是跨浏览器的兼容性。我们现在有五种主流浏览器的多个版本，新应用以惊人的频率发布。开发新手很容易依赖一个很酷的 CSS 技巧，却发现它在浏览器 X *(我没有提到名字……)*中惨败

因此，公司求助于发布官方支持的浏览器列表，即 WebFoozle 2.0 在 IE8 上运行，Firefox 3.5 在 Windows 上运行。**不要这样做！**支持浏览器列表不是一个好主意有几个原因:

## 1.你的应用程序应该支持每一种浏览器

Web 应用程序应该是为 web 而不是浏览器编写的。您应该努力实现设备独立性，而不是针对特定的操作系统和浏览器版本。很少会遇到无法在一个浏览器中实现的功能。

我承认现实世界的开发并不那么简单，但是 web 标准和渐进的增强技术使您能够创建几乎在任何地方都能工作的应用程序。如果商业利益证明额外的开发成本是合理的，那么即使没有 CSS 或 JavaScript 的浏览器也可以得到支持。

## 2.浏览器支持列表需要维护

不可能在操作系统和浏览器的所有组合上测试你的应用程序，所以你的列表将被限制在你测试过的平台上。它会随着系统的发展和问题的解决而膨胀和收缩。

这份名单一公布就会过时。你可能会提供对 Firefox 3.5 的支持，但这是否意味着 3.5.1 和 3.5.2 版本也能工作？

## 3.它会将焦点切换到您不支持的浏览器

你可能支持许多浏览器，但是你的列表也突出了那些你不支持的。当粉丝们质疑他们最喜欢的浏览器被遗漏时，这可能会导致负面宣传(即使应用程序确实工作)。

较大的公司有更多令人担忧的问题。如果微软开发了一个只支持 IE 的网络应用，阴谋论者会很快指出他们反垄断的指控。

## 4.它暴露了糟糕的测试实践

在开发阶段更容易修复问题，因此早期和频繁的跨浏览器测试是绝对必要的。如果你只支持两三种主流浏览器，用户会质疑你的应用的可靠性。

## 5.它减少了客户的数量

假设你已经自豪地宣布兼容 Windows 上的 Firefox 3.5。它可能在 Mac 和 Linux 的 Firefox 3.5 中运行良好，但这些用户会费心去尝试吗？

公司发布受支持浏览器列表的主要原因是为了减轻支持负担。然而，由于兼容性问题，一个经过良好测试的应用程序应该不会收到大量的支持电话。此外，您的应用程序需要用户——从商业角度来看，并找到您无法识别的边缘问题。

你发布了你的应用程序支持的浏览器列表吗？它节省了时间和成本还是导致了更多的问题？
