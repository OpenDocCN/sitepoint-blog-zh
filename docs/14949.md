# 单独销售的逻辑–开发人员的法律问题

> 原文:[https://www.sitepoint.com/legal-issues-developers/](https://www.sitepoint.com/legal-issues-developers/)

网站是由什么材料制成的？或者更确切地说，它不是由什么材料制成的？对我来说，这才是真正的法律问题，因为一位同事向我咨询了一件涉及他的客户的纠纷，这位客户是一名网络主机开发商。

该客户最近扩大并升级了其内部技术资源。然而，它忽略了重新调整其法律思维，以反映其从建立简单网站到开发强大的、软件驱动的网站和基于网络的软件的技术转变。这种疏忽导致了不合适的合同，并最终危及该公司对一个有价值的源代码库的版权，该库在为许多不同客户的项目中发挥了作用。

幸运的是，如果你让你的技术洞察力来指导你起草合同，你应该能够避免这家公司的错误。

##### 这是谁的网站？到底什么是网站？

通常情况下，客户会要求拥有其订购的网站的知识产权。一个客户可以很好地争辩说，如果它不拥有网站，它将被束缚在开发主机上。公平地说，这是一个看似强大的论点，许多开发者主机很难反驳。但是这里面有一个陷阱。

客户在订购网站时会讨价还价什么？开发者主机提供了什么？随着 Web 开发技术的变化，这些问题的答案可能不像以前那样清晰了。

***之前:网站=观感=客户的品牌***

过去，网站只包含纯超文本标记语言(HTML)。Web 页面是视觉元素的静态显示:徽标、图形、布局、颜色主题，当然还有文本。HTML 独自实现了网站的设计和布局，即“外观和感觉”。

定制设计的网站视觉元素是客户独特品牌的核心。当然，客户为网站这些元素的所有权提出了令人信服的商业论据。很明显，客户对网站这些元素的兴趣可能受到法律保护(尽管具有讽刺意味的是，这种“外观和感觉”的保护可能更多地来自商业外观法，而不是版权法)。

因此，网站开发者通常会签订合同，授予客户对其“网站”的所有权。但是这在今天到底意味着什么呢？

***现在:网站=逻辑=开发者工具包***

纯静态的 HTML 网站越来越少了。今天，我们有客户端脚本交互。我们也有许多平台用于服务器端数据库处理和其他基于服务器的逻辑操作，或者换句话说，完全成熟的编程。

无论是在服务器端还是在客户端，驱动网站的逻辑都不会在视觉上给网站打上烙印。

##### 抱歉，逻辑单独出售…也许

那么，客户对于支持其网站的后端逻辑的所有权的论点有多大说服力呢？在我看来，客户没有公平的权利拥有为网站提供动力的逻辑——除非客户明确地讨价还价，并支付了额外费用，特别是对代码的所有权(当然，客户有权许可，非排他性地，使用逻辑)。

关于网站背后的软件，法律是怎么说的？在我看来，答案往往是不明确的。如果没有合同，非雇员开发者通常拥有他们编写的软件的版权和他们创造的其他知识产权。

然而，请注意，这条规则有一些重要的例外，法院有时对“雇员”的定义不同于我们对普通人的定义。

##### 让合同表达你的意思

实际上，开发者和客户之间通常有一个合同。不幸的是，该合同通常指的是“客户的网站”，而没有定义该术语。然后，合同声称将该未定义的“网站”的所有权转移给客户。对我来说，这回避了一个问题，即网站支持软件是否包含在“网站”中。

软件驱动网站所有权的转移应该会引起开发者的一些担忧。如果你没有得到你的合同，你可能会危及你的有价值的源代码的所有权。

好消息是，通过适当的关注，你应该能够给你的客户所期望的和应得的网站品牌特征的所有权，并且仍然保留你的软件的权利。这是一个双赢的局面。

要做到这一点，你所要做的就是清楚地区分驱动网站的软件和“网站”本身。通常情况下，如果在合同中正面处理，这是一个可管理的任务。

##### 永远不会太晚…

如果你已经在一份合同中签字放弃了一个“网站”,而这份合同并没有恰当地划分这种区别，那么你的客户对你编写的在网站后面运行的软件提出了权利要求，那该怎么办？在这种情况下，我相信你面临着更昂贵和更具挑战性的任务来证明(向非技术法官或陪审团)授权软件和网站其余部分之间的区别:

1.  cuts to the heart of whether there really was an agreement at all (the proverbial “meeting of the minds”), and
2.  在你的案件的具体事实下，在决定结果时有很大的价值(因为没有合同定义)。

***为法官和陪审团编程 101***

你如何说服一个非专业法官或陪审团，这种区别是如何支撑整个交易的？我认为视觉元素基本上是一次性交易。当一个开发者为一个客户设计一个标志时，他必须从那个客户身上榨取全部的价值或价格——因为他永远不能为另一个客户利用(即重复使用)那个标志。类似地，HTML 代码也不能真正被利用。

对于真正的逻辑元素，无论是脚本还是编程语言，都不是这样。通常，高效和老练的开发人员会更加努力和聪明地开发模块化代码，以便他们可以广泛地重用，并反复从中提取价值。

因此，放弃代码的版权实际上会让开发者一次又一次地“付出代价”。精通技术的常识、公平和商业动态都迫使开发者只在知情的情况下，为了一个合适的价格，放弃代码所有权。

***服务器机架和审判室的网站架构***

对于法官或陪审团来说，如何有说服力地区分软件和网站？你的软件应该尽可能的与网站的其他部分不同和分离。理想情况下，你的软件应该存在于完全独立的文件中(dll，exe 或类似的文件),并驻留在一个独立于网站的服务器上。该架构的美妙之处在于，从法律和技术的角度来看，它都是同样可取的。

##### 结论

基于广为流传的网络相关合同，网站和网站驱动软件之间的区别是模糊的。但是，在正确的情况下——如果巧妙地画出来，无论是在法庭上还是在合同中——这种区别可能会造成利润和损失之间的差异。

我同事的客户发生了什么事，本文开头讨论的？它决定谈判，以避免预期诉讼结果的不确定性。你看，那个开发者主机有几个反对它的罢工。

首先，它的即兴合同没有试图在网站和供电软件之间做出任何区分。相反，根据开发商自己不计后果的条款，他们将“所有开发的财产”的知识产权转让给了客户。

第二，开发者主机没有试图从技术上把 Web 用户界面从支持网站的大量逻辑中分离出来。相反，它将所有逻辑嵌入到完全驻留在呈现网页中的服务器端脚本中。当然，你我都知道它仍然是软件，但是试着向非技术法官和陪审团解释一下。

无论如何，将你的 Web 用户界面与你的逻辑分离是非常有意义的——无论是在法律上，最好是在一个清晰透彻的契约中，还是在技术上，在实现架构中。
