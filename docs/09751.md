# 移动设备上的互联网代理案例

> 原文:[https://www . site point . com/the-case-for-internet-agents-on-mobile-devices/](https://www.sitepoint.com/the-case-for-internet-agents-on-mobile-devices/)

在这篇文章中，我想提出一种编写服务器端代码的方法，移动电话可以执行这些代码来执行长时间运行、周期性和电池密集型任务，这样就不需要单片集中式服务器来托管它们，并且执行它们所需的数据(例如用户名/密码或信用卡信息)的安全性比当前的系统更难获得或读取。

我将从一点哲学和历史开始，这对许多人来说可能是旧闻，但我认为它设置了场景。新的东西从第二部分开始出现。

### 现有的信息所有权

互联网最初的前提是自由和去中心化。任何人都可以拥有自己的服务器，运行邮件系统，或 HTTP 服务器，或其他任何东西。这仍然是许多网民珍视的理想，然而互联网的货币化意味着它以一种非常不同的方式演变。99%的人不关心他们自己的服务器，即使他们关心，也不知道如何设置。人们希望向他们提供易于使用的服务。互联网公司利用了这一点，在他们的用户存在的生态系统中建立起完整的围墙。

脸书、谷歌和苹果是这一领域最大的玩家，事实上也是互联网上最大的创新者，因为他们的财务模式证明了开发工作的合理性。然而，为此已经做出了一些妥协。安全性、稳定性，甚至您的互联网角色和数据的可用性都由单个实体控制。总的来说，这些公司在保护你(他们)的数据方面做得很好，可靠性也很好。

但是当事情确实出错时(例如最近的索尼数据泄露事件)，这是一个大事件，影响的不仅仅是几十人或几百人。更重要的是，集中信息为控制它们的公司创造了一座金矿(几乎是字面上的意思)，而[政府希望获得访问权限](http://en.wikipedia.org/wiki/USA_PATRIOT_Act)。

虽然像 [Diaspora](https://joindiaspora.com/) 这样的项目在努力将控制权还给人民方面值得称赞，但仍有许多挑战。最重要的，也是我认为我们不应该争论的，是创新金融模式。Diaspora 正在生产开源软件，这不会直接创造收入(至少不会达到大公司想要的规模)，他们必须生产人们想要使用的有竞争力的产品，同时与拥有数量级以上资源的公司竞争。简单的逻辑表明，除非人们有令人信服的理由选择开放路线，否则大公司将会获胜。数据隐私是显而易见的选择，但目前几乎每个人都对信息安全漠不关心。

最后一个挑战，也是我想在这篇文章中谈论的话题，围绕着易用性。人们不运行自己的服务器，因为他们不知道如何运行，或者即使他们知道如何运行，他们也要费力设置和花钱运行。即使是对安全性(有点)认真的人也会犹豫。我运行了一段时间我家人的电子邮件和网站，最终放弃了，把它们转移到了谷歌应用程序，对此我仍然感到有些尴尬。

### 移动代理的案例

如果部署服务并不难，并且可以通过只有您想要的设备/人才能访问的方式来完成，那会怎么样？云计算提供了机会。目前，大多数人使用云计算服务的方式类似于传统的网络服务器。它是一个整体服务器，为来自相同数据库和计算机的所有用户提供服务，因为这就是我们被教导如何使用它们。

相反，我可以看到一种机制，通过这种机制，您的手机或笔记本电脑只需向云计算服务注册代码，您的需求就可以通过这种方式得到满足。这可以由设备本身管理，无需用户交互，因为现在已经有了创建和消费云计算服务的 API。此外，交互可以(在大多数情况下)被加密，这样只有创建代理的设备才能检索信息，甚至获得处理请求所需的数据，从而使代理更少成为黑客攻击的目标。

我发现这对于手机来说特别有意思。手机的功能越来越强大，但却受到电池的限制。在服务器上，开发人员可以浪费或扩大资源，但在编写移动应用程序时就不会这样了。解决这些问题的传统方法是在服务器上执行大量处理，并使用移动设备作为 I/O 平台来访问 web 数据。这对于拥有完整体验的平台来说很好，但对于 mashup 风格的应用程序来说可能很困难，因为它们可能没有运行企业服务器的资源。此外，向第三方服务器发布信息还会带来安全隐患。

### 在实践中

我将从我自己的应用程序 [NodeDroid](http://www.8bitcloud.com/nodedroid.html) 中举一个例子，这个应用程序的屏幕抓取电信网站来报告电话使用情况。NodeDroid 将用户名和密码存储在手机上，访问电信公司的网站以下载 HTML 页面，然后从中解析出使用信息(这有点占用处理器)。

如果电信公司能提供一个 API 来做到这一点而不需要屏幕抓取，那就太棒了，但总的来说他们没有。目前，所有的处理都在手机上进行。我曾经想过建立一个服务来获取电话，但是由于两个原因我没有这样做。首先，我没有从中赚到任何钱，所以我不明白为什么我要承担运行服务器的费用。其次，也是更重要的一点，我不想看到别人的密码。

如果我有一台服务器，电话必须发送用户名/密码对，这样我的服务器就可以将它们传递给电信公司并获取数据。叫我理想主义者吧，但可能有更好的方法。我之前已经对 T2 的这个 T3 咆哮过，总的来说，每当我想要一个新的服务时，我总是对创建另一个网络应用感到不舒服。

谷歌提供名为[谷歌应用引擎](http://code.google.com/appengine/)的云计算服务。每一个拥有谷歌账户的人(几乎是每一个拥有安卓手机的人)都可以创建他们自己的云应用，并且有非常慷慨的免费限额。这款手机可以创建一个名为“代理”的特殊网络应用，专门用于获取其使用数据并将其部署到 GAE。为了安全起见，部署的代理将包括访问电信公司的用户名/密码，但它将使用[公钥加密技术](http://en.wikipedia.org/wiki/Public-key_cryptography)对其进行加密，并将私钥保存在手机上，这样即使有人入侵了 GAE 的 web 应用程序，他们也无法检索到信息。当手机想要获取信息时，它会在交换期间将密钥发送给代理，代理将解密执行该功能所需的参数，将数据返回给用户，然后忘记密钥。通过这样做，我们获得了许多优势:

1.  获取信息的工作负载被卸载到服务器，节省了电池电量和带宽使用，因为只有所需的信息被发送回手机，而不是整个屏幕抓取会话
2.  用户信息安全地保存在服务器上，因为它只在设备的单独调用期间可用
3.  用户的手机不再存储敏感信息，如果有人偷了手机，他们将能够获取您的使用情况(通过调用代理)，但无法检索用户/通行证来访问电信公司网站上的其他信息。他们需要侵入手机和服务器来获取，即使这样也只能获得一个用户的详细信息。
4.  如果用户希望他们的信息更加安全，代理可以分布在任何云计算提供商或他们自己的服务器上运行
5.  如果一个云提供商出现故障，it 部门可以在其他地方重新创建代理来实现可靠性
6.  服务器代理可以运行后台或重复的任务，使手机不必这样做。通知可以由 [C2DM](http://code.google.com/android/c2dm/index.html) 执行，即使源服务不支持它

我喜欢它作为一种方法，目前我正在试验一个系统在 Android 上做同样的事情。它并不完美，仍然存在一些安全问题，但它比今天使用的大多数方法安全得多，也更加个人化和隐私化。为了实现这一点，编写代理代码必须比编写服务器和创建托管它的基础设施更容易。代理将是电话应用程序的扩展，捆绑在应用程序中，不需要任何特殊的服务器。我想我能做到。一旦我有了运行系统，我会发更多的帖子。
