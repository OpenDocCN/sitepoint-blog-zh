# 在此配置中，谁是我的 Sql Server 用户？

> 原文:[https://www . site point . com/who-be-me-SQL-server-user-in-this-configuration/](https://www.sitepoint.com/who-be-me-sql-server-user-in-this-configuration/)

一个很少被记载，但却非常重要的事实。NET 应用程序的一个缺点是经常使用 Windows 身份验证与 Sql Server 数据库进行通信。不幸的是，到底默认使用哪一个身份最多只能说是很少被记录。所以我做了这张方便的桌子给你参考:

| Windows 版本 | 网络服务器 | 默认 Windows 上下文 |
| 任何的 | 卡西尼(VS.NET 2005 内置网络服务器) | 交互式用户环境* |
| 2000(专业版或服务器版) | （同 ImmigrationInspectors 移民检查） | ASPNET |
| 经验值 | （同 ImmigrationInspectors 移民检查） | ASPNET |
| 2003 服务器 | （同 ImmigrationInspectors 移民检查） | NT 权威网络服务** |

备注:
*:这是指你，开发者的上下文。这通常转化为管理员组的成员，意味着您的应用程序以完全数据库权限运行。这意味着这是一个糟糕的测试环境，因为 web 应用程序永远不应该拥有这些特权。
**:这是 IIS6 中默认应用程序池的默认标识。在任何情况下，web 应用程序都承担宿主应用程序池的上下文。

至于成功地将安全性从开发转移到生产，我能给出的最大提示是在您的数据库中创建数据库角色，并为这些角色分配权限。这里的主要问题是转移 Sql Server 用户可能很困难，而数据库角色完全包含在数据库中，更容易转移。而且，当您进入生产环境时，您总是可以创建一个具有适当角色的新用户。这比手动为新用户分配适当的权限要容易得多。并且比仅仅将生产用户的 db_owner 权限授予 web 用户要安全得多。
