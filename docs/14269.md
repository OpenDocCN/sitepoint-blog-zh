# 已编译和解释的垃圾评论

> 原文：<https://www.sitepoint.com/comment-spam-compiled-and-interpreted/>

继[之后的自动化博客评论垃圾邮件？](https://www.sitepoint.com/blog/)和反馈(非常感谢)，我想我应该把其中一些编辑(和解释)成更有序的东西。

**侏儒还是机器人？**

谁(或什么)在发布垃圾评论的答案似乎是:既有没有生命的可悲的侏儒，也有自动化的脚本/程序。鉴于这种情况，我的结论仍然是，如果我们想要*防止*人类提交的垃圾邮件与脚本提交的垃圾邮件，需要不同的方法(重点是防止-参见下面的“消除激励”)。

我还没有找到任何确切的数字，但我也认为更严重的问题是垃圾邮件自动化，这是基于与攻击一些知名博客应用程序以及人们采用的对减少垃圾邮件有显著效果的解决方案有关的轶事证据。显然，任何自动化过程都能够产生比手工数据输入大得多的数量。

**禁止合法使用酒吧**

…或者说是“无障碍诅咒”。人们似乎普遍认为，合法用户在博客上发表评论必须很容易。事实上，理想的情况是合法用户不应该受到任何垃圾邮件保护机制的影响。

有些人愿意要求用户注册/认证，并发现这已经足以阻止垃圾邮件发送者。然而，风险在于不鼓励合法使用。此外，像 Hotmail 这样的网站已经发现，用脚本自动注册和登录是完全可能的，尽管这需要更多的工作。我真的认为这表明让你的评论发布 API 变得更复杂足以阻止当今的垃圾邮件制造者(稍后会有更多)。

有一些关于使用验证码从脚本中筛选出人类的讨论。反对的主要论点集中在合法用户的可访问性上；这些图像真的可读吗？听力受损的人怎么办？这里有几个答案——看看 Wez 在他的博客上使用的基于 ASCII 的验证码——可读性很强，但仍然需要计算机科学博士学位才能进行编程分析。还可以通过音频查看科林对图灵的看法。

另一个关于验证码和巧妙绕过验证码的方法的问题是由克里斯蒂安和 T2 提出的。人们对此的反应似乎是“验证码的终结！”。事实上，我认为这种情况很少发生，而且也不难阻止——要么研究[防止热链接](http://www.alistapart.com/articles/hotlinking/),要么使用 Wez 的 ASCII 验证码，这些验证码本质上是不可热链接的。

虽然以一种安全和可访问的方式来设置验证码是可能的，但是对于合法用户来说，这仍然是一个额外的步骤，而且我相信对于这个问题来说，这是多余的。真正需要的不是筛选出人类用户，而是从脚本中筛选出合法的*用户代理*(网络浏览器)

**防止自动化**

对我来说，现在有足够的轶事表明，使你的发布 API 稍微复杂一点就足以阻止脚本自动发布垃圾邮件。

一条评论提到了皮特·鲍耶的简单而有效的解决方案，这需要用户通过网络浏览器多做一步，但不仅仅需要 LWP::简单编写脚本。

在其他地方，一位 WordPress 用户描述了简单地将帖子 url 重命名对垃圾邮件的直接影响。随后的[评论](http://wordpress.org/support/3/13443#post-88056)中有一条特别有趣；

> 重命名技巧适用于大多数垃圾邮件机器人——只要你记得把 wp-comments-post.php 也从你的服务器上删除，正如有人提到的:p 然而，有一些机器人似乎解析整个 index.php 文件来找到评论文件名，我也改变了评论表单变量，但仍然有一些通过，可能是因为机器人解析评论表单并获得变量名。因此，正如有人提到的，这就像冷战，你必须适应不断保持领先的垃圾邮件发送者。
