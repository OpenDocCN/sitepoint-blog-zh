罗斯:是的，我们对增长和未来的挑战感到兴奋。

路易:我真的很期待看到它成长。我猜你更出名的可能是卡布奇诺的共同创造者。现在，我想卡布奇诺第一次发布的时候可能会在 SitePoint 播客上讨论，虽然我当时没有做这个节目，但对于不熟悉卡布奇诺的人来说，你想谈谈它是什么以及它是如何产生的吗？

当然可以。首先，快速概述一下，正如你在介绍中所说，Cappuccino 是一个构建基于浏览器的应用程序的框架，我们的意思是，我认为现在被称为单页应用程序，或者有人称之为胖客户端 web 应用程序，诸如此类。这个想法是，像谷歌文档或我们自己的 280 张幻灯片，甚至在某种程度上谷歌地图这样的应用程序，实际上是你只需加载一次，运行应用程序的大部分逻辑都发生在客户端，它进入服务器只是为了获取新数据并对这些数据进行操作。所以，我认为在过去两年左右的时间里，我们看到的变得非常流行的应用程序的整个模型就是卡布奇诺真正要构建的。卡布奇诺本身是从 2005 年在我们大学宿舍开始的，但那只是一段时间的爱好，然后在 2008 年我们开始全职研究它，并成立了一家名为 280 North 的公司来做这件事，然后我们向公众发布了它，我相信是在 2008 年 8 月。所以它已经推出三年多了。

**路易:**对。值得一提的是 Cappuccino 是开源软件。

是的，它是开源的，在 GitHub 上，我相信它目前的版本是 0.9.5，并且是 LGPL 授权的。

**路易:**对。所以卡布奇诺背后的想法是，根据我的理解，在某种程度上重现了用 Cocoa 和 Objective-C 为桌面 Mac 环境开发原生应用程序的体验

罗斯:是的。

**路易:** —浏览器内。

**罗斯:**对。我想我应该提到这一点。当我们考虑写卡布奇诺时，我们实际上——我们想要解决的问题是在 web 浏览器中构建一个应用程序，我们已经为此工作了一段时间，随着每个项目的深入，我们会遇到这些障碍，并意识到这真的很难，在浏览器中构建这种应用程序真的很难。但是我们三个当时正在开发它的人也有编写 Mac 应用程序的经验，所以我们想为什么我们不试着开发一个看起来像 Cocoa 的框架；Cocoa 似乎在构建这种复杂的应用程序方面工作得很好。所以我们开始尝试模仿可可的创意，最终，我的合伙人之一弗朗西斯科有了一个想法，为什么我们不试着写可可呢。为了好玩，他写了 Objective-J，这是 Cappuccino 所用的语言，它是一组 JavaScript 的扩展，可以在浏览器中动态编译，所以我们添加了这组语言扩展来添加某些功能，如代码导入、经典继承和动态调度。

**路易:**对。

Objective-C 添加到 C 中的相同的特性，所以在他完成之后，我们开始着手移植大量实际的 Cocoa 框架，这就是今天的 Cappuccino。

**路易:**对。你说编译发生在浏览器中，是动态的，就像 CoffeeScript 一样，你把它放在一个预处理器中。

罗斯:这其实和 CoffeeScript 极其相似。我们有能力在浏览器中或者在命令行上提前完成，我相信 CoffeeScript 也有这样的功能。因此，卡布奇诺应用程序的开发周期通常是对您的。j 文件，并像平常一样刷新浏览器，这些更改将在浏览器中动态编译。但是，当你部署时，我们建议提前编译并运行我们所有的各种优化脚本，以尽可能快地完成工作，这样你就不会在每次用户加载你的应用时都做重复的工作。

**路易:**对。这是——因为它运行在 JavaScript 中，并且编译成或多或少直接向上的 JavaScript，您从中获得的浏览器支持水平如何？

罗斯:所以，官方上我相信我们支持几乎所有的浏览器，IE7 或更新的版本。我认为卡布奇诺在 IE6 中也能工作，但我们并不真的支持它，因为我们使用了很多透明的 PNG，对那个问题并不感兴趣。

**路易:**太奇怪了；已经很久了，直到你现在提出来，我都忘了 IE6 不做幻灯片。

罗斯:是啊。

路易斯:那就像哦，哇，那东西真的很古老。

罗斯:是啊。是的，当我们真正发布，或者当我们编写第一个 Objective-J 时，IE6 是最新的 Internet Explorer，所以我们花了很多时间来弄清楚如何让在 IE6 中编译这种语言足够快以达到现实。

**路易:**对。我的意思是，这听起来——所以，对我来说，告诉我你是如何想出你如何开始工作的想法的，这听起来完全像是这样的事情，哦，哇，用 JavaScript 编写复杂的应用程序很难，用 Cocoa 编写复杂的应用程序真的很容易，我们应该用 JavaScript 重写所有的 Cocoa，然后你会花一天的时间尝试这样做，然后意识到这真的很难，然后停下来

罗斯:(笑)

路易斯:你们没有做我想我会在你们的位置上做的事情，真是太棒了。

罗斯:是啊。这绝对是有趣的，有挑战性的，可能比我们任何人想象的都难，但这也是一次非常好的学习经历，我对 JavaScript 如何工作以及 Cocoa 如何工作有了更多的了解，我肯定很喜欢这方面。

Louis: 你现在还在日常工作中或正在进行的项目中使用卡布奇诺吗？

我一直活跃在 Cappuccino 邮件列表上，处理一些 bug 和 pool 请求，但在过去的一年左右，我自己并没有太多时间来开发 Cappuccino。在 Stripe，我一直在做许多不同的事情，包括许多后端代码，但我们最近也重写了我们的前端，我们选择了 CoffeeScript 和 Backbone，然后我们在此基础上构建了一个定制框架。

**路易:**对。

罗斯:所以，实际上我并没有在 Stripe 开发卡布奇诺。

**路易:**对。我实际上是想问你，你觉得卡布奇诺和诸如 Backbone 之类的东西竞争或相对，感觉如何，因为看起来它们似乎都在某种程度上试图解决一个类似的问题，尽管我猜卡布奇诺有很多 Backbone 并不真正关心的 UI 组件。

**罗斯:**对。

Louis: 但是处理在浏览器中运行的更复杂的应用程序的想法。

从某些方面来说，他们试图达成一个相似的目标，从某些方面来说，他们实际上是一种恭维。Backbone 并没有真正尝试在视图层上做任何事情，你知道，这里有一些方便的小函数来帮助定义事件，但是这里并没有真正做很多事情。我想，Backbone 的真正优势在于，它让将模型和集合绑定在一起变得容易，就像一个 restful 后端，而且代码越少越好；我认为这可能是 Backbone 的最大优势。卡布奇诺实际上并没有试图做太多，它试图成为后端不可知论者，我们的目标是，你知道，我们想写这个前端，前端集中的东西，然后你想如何来回获取数据将取决于你。另一方面，Backbone 真的很小——我认为它是 10k 的 JavaScript，或者是框架，而没有图像的 Cappuccino 可能接近 300k 或者更多。因此，这里肯定有大小差异，卡布奇诺内置了许多功能，许多不同的视图组件，许多图像资源，因为正如你所说，你知道，我们有自己的完整风格，你可以使用。

**路易:**对。Cappuccino 的工作方式是否类似于 jQuery UI，其中有一组接口组件，但它们可以用您可能想到的任何其他自定义组件进行切换？

罗斯:从某些方面来说，是的，是的，但是从其他方面来说，实际上很不一样。因此，jQuery UI 所关注的事情之一，以及 Backbone 和所有其他框架正在做的事情，是他们仍然希望使用 HTML 和 CSS 来定义事物的个人风格和元素的组织。Cappuccino 采用了我想有些人可能会称之为根本不使用 CSS 或 HTML 的激进方法，所以当你与 Cappuccino 交互时，你永远不用编写 CSS 或 HTML 代码，一切都是 JavaScript 对象；在 Cappuccino 中，它被称为 CP 视图，这显然映射到 NS 视图层次结构。所以，你知道，在某些方面，我真的认为 Cappuccino 使与其他人共享组件变得更加容易，因为传统 web 应用程序和共享视图组件的一个问题是，CSS 和 HTML 中的所有内容都是紧密联系在一起的，如果你改变 HTML 的布局，你的 CSS 结构很容易被破坏；你必须有很多关于 HTML 如何排序和嵌套以及 CSS 如何应用于此的知识，并且你必须在不同的地方随身携带许多这种依赖。卡布奇诺没有这个问题，因为所有东西都是用代码写的。

**路易:**对。

而且 Cappuccino 确实失去了一些灵活性，因为你失去了 CSS 的表现力，像静态视觉风格那样声明，这可能很好，但 Cappuccino 确实有一个我们添加的主题系统，让你可以这样做——它添加了一些功能，所以你可以使用标准视图组件，但通过我们的主题系统修改它们的外观。

路易斯:对，是的。你提到你在 Stripe 的工作中也做了一些后端的工作，你在 Stripe 做的是什么样的服务器端代码？

大多数代码都是用 Ruby 写的，通常是在 Sinatra 之上，尽管并不总是这样。我在我们的 API 上做了很多工作，设计了 API 的结构和我们向用户公开的实际接口，而不是具体功能的实现，我已经做了一些，但在我开始之前还有很多。

**路易:**对。

罗斯:我最近一直在做的事情是，我们本周刚刚发布了一个新的 Webhook 功能，它的工作方式与 GitHub 的 post-receive 挂钩非常相似，如果你熟悉的话。因此，您只需在您的 Stripe 帐户中注册任意数量的 URL，我们将在任何有趣的事情发生时向这些 URL 发送事件，这样您就可以跟踪您所有 Stripe 对象的更改历史，而无需拉我们。

**Louis:** 你能举一个例子，说明什么样的东西会被使用，或者如何在常规用例中发挥作用吗？

**罗斯:**当然，所以人们最常使用的一件事是我们有一个循环计费系统，很多人会想在月底增加使用费，所以你向你的用户收取某个月的费用，假设你收取手机分钟数，然后他们使用他们所有的手机分钟数再加上 60 分钟， 因此，您希望在这额外的 60 分钟的基础上增加额外的费用，所以我们将向您发送一个网络挂钩，告诉您他们的计费期已经结束，我们已经为他们创建了一张发票，以及关于该发票的所有详细信息和它受影响的日期，我们给你一个机会向我们发送一条消息，说，好的，很好，在您尝试支付之前，将这笔钱添加到发票中。

**路易:**对。是啊，听起来真不错。所以在你与 Ruby 的合作中，你说你正在与 Sinatra 合作，我猜是因为你不必像传统的 web 应用程序那样处理那么多的完整视图堆栈，因为你更多地使用 API，或者因为你的主要产品是 API，正如你之前所说的，使用像 Sinatra 这样的轻量级产品而不是像 Rails 这样的更全面的框架是有意义的？

是的，我认为这是一个准确的描述。我认为 Rails 肯定有一些不错的特性，即使你正在编写 API，但对我们来说，无论是从我们正在做的事情还是我们自己的需求来看，这都不是我们想要的方向。因此，我们使用 Sinatra 作为我们的 API，我们也将其用于我们的网站，我们的管理网站，我们实际上将我们的 API 用于该管理网站，所以，正如我所说，它是用 Backbone 编写的，Backbone 实际上只是调用我们的常规 API，即我们的客户使用的 API。因此，我们在网站上展示的大多数功能都是以某种方式展示的，如果用户愿意，他们可以构建自己的竞争性仪表板，这实际上非常有助于我们了解在 API 中构建的正确功能，以及最大的性能问题在哪里。

**路易:**对。实际上，我认为这是一个非常有趣的方法，而且这是我以前从未听说过有人做过的事情。我想大多数人构建包含 API 的 Web 应用程序的方法是构建带有一些视图的应用程序，然后使用 API 来呈现 XML 或 JSON 等，但我想如果您构建的应用程序或仪表板完全类似于这些单页 JavaScript 应用程序， 那么简单地使用你自己的 API 首先是一个很好的方法来确保设计或者你的 API 是合理的，这对于其他开发者来说是很容易使用的，并且正如你所说的，可以指出其中可能存在的任何问题。

罗斯:是的，这是我们的目标，网站的第一个版本不是这样写的，它是一个更传统的网站，当我们作为一个单页应用程序进行重写时，我们认为我们可以真正做到的事情之一是，在标准的基础上，你知道，让它响应更快，加载更快，等等。 这种直接与 API 通信并使用相同的 JSON 数据交换的能力能够真正确保我们的用户在使用 API 时有良好的体验。

路易斯:对，非常好，下次我做类似的事情时，我一定会记住这一点。那么，在这种情况下，您的整个前端都是基于主干网构建的，并且您正在使用 API，这是否意味着您已经放弃了对仪表板的非 JavaScript 支持？
