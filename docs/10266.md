# 保护可用性测试不受你的影响

> 原文：<https://www.sitepoint.com/protect-usability-tests-from-yourself/>

无论你是在测试一个现有的产品，参与一个网站的重新设计，还是在一个全新的网站上工作，你最大的挑战往往是你自己。以下是一些你可能无意中引起问题的方式:

*   你对一个公司、产品或应用的个人经历会扭曲你的理解。想一想您日常使用的应用程序。你知道它最让你沮丧的地方，甚至知道你会如何解决它。现在假设你被雇佣来改进这个应用程序。你会想直接找到你沮丧的根源。但是如果你把你的测试仅仅局限在那个领域，你可能会因为你狭隘的视野而错过一个更大的问题。
*   多年的网络使用和工作经验有助于你成为一名专家，但也可能产生盲点。基于过去成功和失败的经验来假设网络应该如何运行是不可或缺的，但也是危险的。做写实插画的时候，艺术家说“画你看到的，不画你知道的”，这也是一个道理。
*   来自公司高层的大量肯定他们理论的压力会潜移默化地影响你进行测试和提问的方式，尤其是如果你是新人的话。人们对制药公司资助的测试持怀疑态度是有原因的。你可能永远不会故意歪曲结果，但当风险足够高时，很容易无意中这样做！

无论你是在进行正式的可用性测试，还是仅仅和你的团队开会讨论线框，你根深蒂固的信念可以在任何地方影响你！以下是一些积极抵制偏见的方法:

**避免问诱导性问题**

这样的问题暗示了你期望得到的答案。例如，分析以下陈述可能导致的结果:

> 我希望您使用此表格来完成您的订单。当你点击提交按钮时，你的卡不会被扣款，所以请完成整个过程。

这种引导是微妙的，但是这种说法假设使用网站的人知道你应该点击表单末尾的一个按钮，这个按钮叫做提交按钮。更微妙的是，仅仅通过说出“提交按钮”这几个字，你就可能让你的参与者对这个元素给予了比平常更多的关注。

**对参与者感同身受**

如果你认识到许多人都有强烈的愿望去正确地完成一项任务，你的测试将会非常成功。不断的道歉可能会让测试进行得太久，而且会让参与者感到紧张(让你觉得自己很卑鄙)。

通过仔细注意你的语气，写一个保证和信息丰富的脚本来引导会议——以及在会议期间温和地提醒参与者，你正在测试网站或应用程序，而不是个人——你更有可能创造一个产生公正结果的环境。如果参与者只是努力想弄清楚你所做的界面，并且认为它非常清楚，那么你就非常有可能变得具有防御性，或者问一些引导性的问题来达到预期的结果。

当心你的假设和糟糕的记忆

测试一结束，你的大脑就开始将新信息融入你对事物如何运作的现有假设，忘记或改变那些新的讨厌的细节。你的大脑保持高效的步骤并不总是导致最准确的信息合成——特别是如果项目阶段之间已经过了很长时间。要准确记住考试中发生的每一件事是不可能的，即使你是一个好的笔记者(尽管，笔记有帮助！).考虑至少在会话期间记录屏幕和音频，以便您可以在分析期间引用。

没有必要在这些事情上强调太多，但是随着时间的推移，要学会警惕你的倾向。当你学会消除偏见时，你最好的测试和内部讨论就会发生。
