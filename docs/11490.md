# 7 大骗局——以及如何避免它们！

> 原文:[https://www.sitepoint.com/7-great-rip-offs/](https://www.sitepoint.com/7-great-rip-offs/)

几乎所有的网络开发人员都发现自己在任何一年都在向某些客户讨债。有些情况我比其他人更能忍受。在这篇文章中，我确定了网络专业人士陷入付费陷阱的七种最常见的方式，并提出了克服这些陷阱的方法。

## 有问题的支付等式(或一半+一半=一半)

常见的——也是最令人沮丧的——建筑商的付款条件通常由以下等式组成:一半的总额预付，另一半在项目完成时支付。这些条款的问题是，通常是客户，而不是开发商，决定项目何时被批准和结束。因此，开发商可能已经完成了整个项目，并将其提交给客户进行审查，但自几个月前的初始存款以来，没有看到任何资金。除非客户同意网站是完整的，否则开发商无法发送最终付款发票。那么如何避免这种情况呢？

补救措施是不同的支付结构。我鼓励我的 web 开发客户要求一笔可观的预付款，金额在总工作量的三分之一到一半之间。之后，与某些里程碑相关的分期付款具有良好的商业意义，例如设计批准、关键站点方面的交付或整个测试站点的展示。在项目进行之前收到进度付款将有助于防止开发商偏离客户太远。

## 了解核心服务和附加服务的区别

项目需要在一开始就明确定义。否则，客户在整个过程中要求的额外服务可能很难被识别和收费。对你的 web 开发业务来说，时间和金钱的最佳投资之一就是标准服务合同。有一个涵盖所有基础，但允许根据每个客户的要求进行修改。它将为项目的范围提供一个参考点，并防止任何以你为代价的[范围蔓延](http://articles.techrepublic.com.com/5100-10878_11-1045555.html)。

例如，如果一个网站的 CMS 包括一个博客模块，它是否也包括与 Twitter 和脸书的社交媒体整合？一个电子商务网站包括与 QuickBooks 或其他簿记软件的集成吗？如果合同没有涵盖这类问题，客户可能会认为项目价格中包含的功能比开发商打算提供的要多。

## 即付即付但书

开发人员通常在项目的某些方面使用分包商。除非客户已经预先支付了足够的项目费用，否则开发商可能会拖欠其分包商尚未从客户处收到的款项。为了保护现金流，考虑为分包商提供“即付即付”的条款，类似于“当客户付款给我时，我会付款给你。”虽然它不会加快客户的付款速度，但它会让你不必自掏腰包。

## 从你所在的位置，你还能看到客户吗？

开发人员保持在客户的范围内，避免在没有客户同意的情况下在项目上走得太远，这一点很重要。例如，如果开发者准备交付一个完整的网站，而客户只支付了最初的线框图，那么开发者已经向客户提供了大量的信用。除非付款有担保，否则投资可能永远无法收回。管理您的发票、工作进度和付款时间，使提供给客户的信用额度保持在可控范围内。从你的客户那里获得更大的预付款，以及更频繁的分期付款或定期账单可能是有效的。然后，如果你的发票没有被支付，可能是时候考虑中止项目工作了。

## 无免费发布

保留对项目的控制权，直到你得到全额报酬。这可能是让客户有动力向你付款的最好方法。您应该在您的服务合同中加入条款，规定使用图形设计和底层代码的版权许可证的转让必须付费。您可能还想考虑在开发过程中控制的服务器上托管站点；然后，您只需在支付约定的金额后，将网站激活并传输到客户的服务器上。

## WIP 上诉

如果客户——不管什么原因——失去了对项目的关注，在制品(WIP)可能是开发人员最大的敌人。如果分期付款在开账单前需要客户的反馈，开发人员可能要等很长时间了。一个好的服务协议允许开发人员发送当前工作进度的发票，即使客户在一段时间内保持沉默，也能让开发人员有机会结清账目。

没有人能强迫客户关注并提供反馈，但当面对沉默之墙时，能够寻求付款可以帮助网络公司做一件大的、没有报酬的工作，并使他们不必等到客户回来。

## 像竹子一样，准备好弯曲一点

如果客户拖欠付款，考虑一下你是否愿意在一段时间内接受部分付款。这可能有助于让现金短缺的客户将你的成本视为一系列更小、更合理的可能性，而不是一个大的不可能。例如，如果你将一笔未偿还的金额分成四部分，而只有前两部分最终得到了偿还，这一点点重组已经让你离家近了一半——在你不得不进行一些严重的债务催收之前。

就像项目一样，客户有不同的形状和大小。当然，困难之处在于，问题客户从一开始就不那么明显。通过在开始工作之前制定一份全面的开发协议，并在开发过程中注意提供最低限度的信贷，开发人员可以有效地管理这些风险。
