# PHP 项目管理

> 原文:[https://www.sitepoint.com/php-project-management/](https://www.sitepoint.com/php-project-management/)

不管我们喜不喜欢，除非我们做一个业余爱好项目只是为了自娱自乐，否则即使我们中最有技术含量的人也只是会编码的项目经理。并且，作为项目经理，我们可以体验项目失败的心碎。

项目失败的方式有很多。它可能会失败，因为没有满足截止日期，超出预算，结果不是所要求的，等等。但是，它也可能在情感上失败；项目可能成功了，但是人们的期望没有达到，没有满足感或“工作做得很好”

那么，作为一名技术项目经理，你能做些什么来最大限度地减少你在简历中添加“失败项目的领导者”的机会呢？答案是:差不多就是非技术项目领导做的事情。

## 管理预期

认为现实很重要没问题，但是商业界的人都知道，人们认为发生的事情永远比真正发生的事情更重要，项目也是如此。即使一些最成功的项目也可能遭受期望的失败，这将永远掩盖已经完成的良好工作和获得的利益。如果期望的失败足够严重，它甚至会破坏和干扰其他做得很好的项目。那么你如何管理期望呢？

首先，确保每个人都非常清楚项目要做什么，有什么好处，用户需要什么，需要什么资源来支持它，它可能依赖于什么其他项目，以及它要做什么。我知道我把最后一点列了两遍，但它真的很重要！很多时候，用户听到的是他们想听到的，这可能需要比一份备忘录或一个简短的对话更多的东西来澄清这一点。

特别要注意上层管理人员对项目的评价，无论是一开始还是项目进行的后期，并且在它们成为公认的智慧和“从一开始就是项目范围的一部分”之前，迅速纠正任何错误的说法

与此同时，迅速而礼貌地应对你听到的关于项目的任何负面言论。这个想法不是要压制反叛的言论(尽管获得一份名单对以后来说并不是一个坏主意)，而是要找出这是否是一个误解，对事实的错误陈述，或者这是否是一个可以在以后变成问题并应该立即解决的问题，要么通过修改项目规范，要么通过重新审视基本假设。

## 不要传统，要迭代

在一个传统的瀑布型项目中，你开发一个项目计划，从开始到结束，并以一个总的时间和费用预算结束。不要那样做。相反，使用一个迭代的、敏捷的项目模型，其中每一步都以一个易于测试的、高度可见的产品结束，人们可以根据这个产品来衡量他们的期望。

尽管有相似之处，技术项目并不像造桥。桥梁建设的主要未知因素是地基的土壤特性。一旦进行了评估，就消除了许多不确定性。技术项目可能没有更多的变量，但它们分布在项目的整个生命周期中，你永远不会真正到达关键点，直到你完成(也许)，所有这些变量都是已知的。瀑布是基于这样一个事实:未知的东西将会被评估并得到补偿。出于各种原因，技术项目似乎并不是这样工作的，所以瀑布模型是一个糟糕的模型。

然而，瀑布的主要问题是它把人们的注意力集中在一些接近最后的“测试”阶段。它不是为支持几乎从第一天就开始测试的情况而设计的。因此，直到项目基本完成，问题，甚至是主要问题，才会被识别出来。

另一方面，敏捷完全是关于迭代的:工作两周，生产一些真实的东西，让用户测试，然后重复。没有 Power Point 演示或马拉松式的团队会议，只是一些人们可以登录并玩的东西。这才是吸引他们注意力的地方，你不知道吗。

当你在做敏捷的时候，我也建议你做 Scrum。它们不必一起完成，但我认为这是一个好主意。你知道人们喜欢什么吗？他们喜欢短的东西；开始，发生，结束。砰。这就是 Scrum 的本质——频繁，非常短，没有时间放松或请求，在项目团队会议上，我们直话直说，语速很快。它可以防止任何人忘记这个项目和它正在做的事情，并且不会占用他们一天中太多的时间。

这一切对你有什么帮助？敏捷和 Scrum 让你更清楚地看到你在整个项目评估中是如何做的。随着严肃的用户测试有规律地进行，你会很早就注意到可能必须添加到应用程序中的东西和可以删除的东西。如果存在重大问题，您可以在流程的早期开始重新设计。这让你变得积极主动，这有点像拥有漂亮的头发；它总是会带来回报，并使项目处于积极的状态。

## 范围蔓延的弊端

更多的项目注定会受到范围蔓延的影响。范围蔓延导致项目延期、超出预算，并且无法满足来宾的主观期望。所以，不要让它发生。就像享受与公婆在一起的时光一样，我知道说起来容易做起来难。

最常见的形式是，当有人提出一个观点，并说“但这是我们从一开始就想要的方式”时，范围蔓延就会发生。他们可能是诚实的；他们可能真的认为特性 X 将执行功能 Y，事实上这是项目存在的主要原因。因此，因为它从一开始就应该在应用程序中，所以你把它添加进去。有时这种影响无关紧要，但大多数时候并非如此。这在一个项目的生命周期中不会只发生一次，而是会一次又一次地发生。那你是做什么的？

首先，无论其他什么，你必须有外交手腕。你可能想跳过桌子，抓住某人的喉咙，你必须保持冷静，保持力量的平衡。回到你最初的项目陈述，表明这个请求从未被讨论过，更不用说被认为是申请的一个组成部分了(这就是拥有一个非常清晰和详细的项目陈述的关键所在)。

其次，为了限制范围蔓延出现的次数，您应该确保您的用户在他们的迭代测试中做得很好。当范围蔓延发生在开发的后期时，它是最具破坏性的，在这种情况下，必须进行许多用户或多或少看不见的改进。迭代测试，而不仅仅是谈论项目如何进行，是一种更好的方式，让用户了解系统到底在做什么。

最终，问题是双重的。首先，你可能没有足够详细或具体地说明应用程序会做什么和不会做什么；两边都需要拼出来。第二，你需要记住，当你说话或者当你的用户阅读你写的东西时，他们可能没有 100%的注意(即使他们正在开会讨论应用程序应该做什么)。不要期待，因为你已经明确地把事情说清楚了，人们会知道。一个好的技巧是展示你的计划，然后让他们告诉你他们认为这个系统会做什么，以及他们可以从中期待什么。记录会议-您可以在审判中使用它。

遗憾的是，没有防止范围蔓延的方法。你能做的最好的事情就是尽力去阻止它，当它抬起它丑陋的头时，尽你所能去阻止它。最后，很有可能你必须做你的用户想要的，但是不要只是做；把它作为你项目的附录。详细说明做这件事需要的步骤，需要的时间，对其他事情的影响，然后从你的用户群那里得到他们真的想做这件事的认可。

## 小心奇怪的东西

小心奇怪的东西。我不是说奇怪的需求，而是对你的团队或环境来说可能是新的东西。例如，如果您正在构建一个接受信用卡数据的应用程序，您真的[了解 PCI](https://www.sitepoint.com/pci-compliance-and-the-php-developer/ "PCI Compliance and the PHP Developer") 吗？或者，您可能会从 CodeIgniter 转向 Lavarel。你将需要一些额外的时间(甚至超出正常曲线)。留意可能会绊倒你的东西。

## 摘要

底线是:项目管理是一场你赢不了的游戏。如果你在预算和时间之内到达，就好像每个人都会认为你是个骗子，不会再相信你的估计。关注重要的事情。具体点，友善点，敏捷点，抱最好的希望。
