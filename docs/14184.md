# 导航开源许可

> 原文:[https://www.sitepoint.com/open-source-licensing-2/](https://www.sitepoint.com/open-source-licensing-2/)

选择开源许可的决定对不同的人有不同的意义。那些从最终用户的角度考虑这个问题的人可能会觉得这个决定是不相关的:相当于商业点击包装许可证，通常也不会被阅读。对于这种类型的用户来说，软件仅仅代表了一种更经济或者更有效率的途径来完成任务或者获得功能。

许可对于开发者来说更为关键。开放源码许可的美妙之处在于它将版权(和专利，如果由作者持有的话)无偿转让给最终用户和再经销商。因此，例如，Web 专业人员可以免费利用应用程序，在商业活动中使用它，并在与客户的互动中从中获利。

通常，在他们工作的过程中，开发人员发现软件并不完全满足他们的需求:它缺乏给定的能力。为了解决这个问题，开发人员可能会决定构建新的功能。这就是开源许可的缩影:没有任何附加条件！根据许可证的选择，修改后的新解决方案可以在原始许可证下重新分发(或者从原始许可证中分离出来，我们很快就会看到)。

这项工作的结果是数百个新的开放源码软件包可以自由使用。坏消息是，其中一些软件包可能会有侵犯专利的风险，而软件作者可能并不知道。

这种可能性，加上对不断发展的知识产权(IP)问题和责任的关注，已经成为开发人员困惑的一个因素，特别是因为存在大量不同的开源许可证。随着开源环境的扩大，选择许可证、审查代码和发布产品而不担心责任的过程变得更加令人烦恼。

##### 许可难题

当开发者考虑他或她的开源许可选择时，第一站应该是[开源计划(OSI)](http://www.opensource.org/) 。该组织维护开源软件的定义，并对遵守该定义的许可证进行认证。顺便访问该网站，你会发现超过 50 个许可证的描述。

最近，一些许可证是否应该合并的问题出现了，尽管一些，如学术免费许可证，服务于利基市场。OSI 的共同创始人和名誉主席 Eric Raymond 指出，OSI 列出的许多许可证实际上是个人或公司的虚荣项目。

“只有大约六个被广泛使用，”雷蒙德告诉 LinuxInsider。“我们正在考虑阻止进一步扩散的方法，但到目前为止，我们的政策是不拒绝符合 OSD 的许可证，即使它们是重复的。这种情况可能很快就会改变。”

在我们继续深入之前，让我们后退一步，看看许可证是如何变成许可证的。

##### 执照是怎么诞生的？

Raymond 的组织通过一系列审查和讨论来审查新的开源许可证。

“我们寻求符合[开源定义](http://opensource.org/docs/definition.php)的十点。我们有律师，也有精通法律的非律师，一个接一个地讨论许可证。董事会考虑他们的建议和投票，”他说。

最近，Sun Microsystems 贡献了通用开发和分发许可证(CDDL ),通过它该公司发布了其开放的 Solaris。Sun 官员表示，他们选择退出现有的许可证，以便为新 Solaris 平台的用户提供专利保护。

就雷蒙德而言，他并不总是认为有充分的理由发放新的许可证，即使公司很大。

“大多数新执照都是公司法律部门建造纪念碑的活动，他们有太多的时间，”他说。“偶尔你会得到一个许可证，以一种真正新的或有趣的方式解决潜在的法律问题。但这种情况以前并不常见，现在变得极为罕见。”

##### 许可内幕

虽然没有正式的公开统计，但最近独立完成的测量开源许可证使用的研究表明，绝大多数开源项目使用 GNU 通用公共许可证(GPL)。遵循 GPL 的是有限 GPL (LGPL)，Berkeley 软件分发(BSD)和 Mozilla 许可。

在提炼众多许可证的过程中，法律专家 Lawrence Rosen 最近出版了一本名为 [Open Source Licensing](http://www.phptr.com/title/0131487876#) 的书，对许可证进行了分类。这也有助于开发人员减少为他们的软件研究不同许可类型所花费的时间。

目前由 OSI 维护的 50 多种许可证分为四种不同的类型:

***1。学历证书***

学术许可证代表了最“自由”的开源许可证，对许可证用户没有任何要求——甚至不要求用户共享修改或重新分发它们。此类别的许可证包括 BSD、MIT 和 Apache 许可证。

学术许可证旨在提供绝对的自由。唯一明显的限制是，这些许可证禁止在营销活动中利用原始许可方的名称作为背书。除此之外，这些许可证真正是为那些寻求完全控制软件、其使用、修改以及随后独立或与另一个软件包一起重新发布的人准备的。

BSD，开源许可的鼻祖，起源于加利福尼亚大学，允许自由使用、修改和分发该机构内部开发的软件。从那以后，它就成为了一个开放源码开发者可以使用的公共许可证。

MIT 由麻省理工学院创建，是对 BSD 许可证的重写。Apache 许可证与 BSD 和 MIT 的不同之处仅在于它要求在修改作品的文档或源代码中包含一个通知，以标识新发行版包含由 Apache 软件基金会创建的软件。

***2。互惠许可证*T3**

像其他许可证一样，互惠许可证授予开发者和最终用户使用软件的完全权利。唯一的区别在于要求软件的任何衍生产品都必须在相同的许可下发布，并且源代码必须发布。由此产生的新软件也必须是免费的。

互惠的目的是确保一个不断增长的自由软件世界出现，并且原创作品——以及修改过的和新的作品——仍然对用户免费。由于使用了 GPL，现在一些最流行的软件仍然是免费的，包括 Linux、MySQL、Bash shell、Mailman、gzip 和 grep。

这一类别的核心是 GPL，它最后一次更新是在 1991 年，尽管预计今年将由原作者理查德·斯托尔曼和 Eben Moglen 根据开源社区的意见进行更新。Mozilla 公共许可证也属于这一类。

***3。标准许可证*T3**

标准许可证旨在创建软件和文档的基本标准。修改和重新分发的源代码通常必须作为补丁分发，以免修改核心。

例如，设想这样一种情况:创建一个 Web 应用程序，允许在各种流行的博客应用程序之间导入和导出。Web 开发人员获取了这种新软件的源代码，并构建了一个额外的功能来迁移和转换特定的设计元素以及数据。在标准许可下，核心应用程序将与插件一起分发，以支持后者的新功能。

标准许可证的目标是保存现有的代码库，以便原始作者可以毫无困难地重新开发它。在某些情况下，插件不会受到影响。在其他情况下，原作者将更新文档，以允许第三方更新他们的插件(通常也称为补丁)。

***4。内容许可***

最后，内容许可证涵盖代码之外的元素，如艺术、拷贝和音频/视频。那些熟悉 [Creative Commons](http://www.creativecommons.org) 的人会认可这个许可，尽管有一些在 OSI 列出，包括学术自由许可。

关于知识共享(CC)许可的一个警告是，如果一个类似共享的属性被包含在 CC 许可中，它使得许可是互惠的，类似于 GPL。

##### 知识产权问题

在理解许可过程时，区分版权、商标和专利是很重要的。所有这些元素在我们每天使用的软件中都发挥着作用。在某些情况下，免于专利风险已经包括在许可证中(如太阳微系统公司的 CDDL，以及在某种程度上，其他)。

然而，正如 Rosen 在他的书中明确指出的，许多许可证保护用户免受原始软件的专利要求，但是一旦你修改和重新分发源代码或二进制文件，就不一定能扩大这种保护。

也就是说，简单性仍然是开源的核心吸引力——找到一个满足需求的应用程序，下载，安装，然后开始使用或开发。

像 Sourceforge 这样的组织加速了这个过程。Sourceforge 可以被认为是开源软件项目的知识共享，拥有超过 90，000 个开源项目，注册用户超过 100 万。它是众多开源开发者和用户寻找软件的热门目的地。Sourceforge 将只托管一个使用 OSI 许可的项目。

仅此一点并不构成威胁。但是，如果源代码被修改、打包到另一个解决方案中并进行分发，那么 IP 和许可证的考虑就变得至关重要。

一个家庭手工业正为此而酝酿，像 Black Duck Software 这样的公司(见[这篇文章在开源](https://www.sitepoint.com/blog/)中)希望通过提出代码审查解决方案来利用开发者对开源和混合源代码的创建、使用和分发的关注。据 Raymond 称，这对于某些人来说可能是一个昂贵的提议，并且可能使完全遵守变得不可能。

他说:“世界上有那么多受版权保护和专利保护的代码，通过审查获得肯定的保证实际上是不可能的。”“你能做的最好的事情就是确保你的代码中没有其他人的明确版权，这还远远不够。”

##### 尽职调查

定期进行审查，主要目的是显示尽职调查。

Raymond 认为，在现代知识产权法(尤其是专利)创造的疯狂和有害的环境中，唯一有意义的策略是完成足够的形式审查，以记录审查的进行，然后基本上忽略你的风险，直到你被起诉。

“这正是专利律师会给你的建议。你不想事先‘知道’你可能侵犯了哪些专利——这就成了‘故意的’,并使损失增加了两倍，”他说。

“是的，这很疯狂，”他承认道。"这反映了现代知识产权法的根本疯狂."

考虑到当今不断发展的知识产权问题，GNU 通用公共许可证——最广泛的变体之一——将在十三年内首次更新。预计将于 2005 年修订。然而，Raymond 建议开发者不必为了等待新的 GPL 而推迟选择许可，因为目前它本质上是一个软件。

##### 降低风险

政策制定者、律师和法官将最终引导一套古老的知识产权法律进入 21 世纪。这个过程的方向可能取决于个人和组织对开源的理解。

虽然 OSI 为政策制定者进行了一些关于开放源码和知识产权状况的宣传，但据 Raymond 说，“我们的主要重点是向企业推销这个想法，相信他们会向政府推销这个想法。我们合作的政治焦点团体更多，比如 OSIA(开源产业联盟)。”

与此同时，开发者可以仔细研究许可证，自己选择一个。虽然昂贵的法律建议会有所帮助，但其他人可能会冒险选择一个许可证来抵押他们的代码。

了解选择许可的影响非常重要。正如您所看到的，每一类许可证都有其特定的用途，无论是确保最终用户自由、防止商业用途还是保留标准代码库。

用户在做出选择和分发软件后，可以切换许可方案；然而，当涉及到做出这样的改变时，现有的代码和许可协议的问题是模糊的。这些未经证实的水域意味着开发者需要非常小心地选择一个他们可以长期使用的许可证。

例如，如果一个开发者预计长期只销售支持和定制服务，选择一个可能阻止软件本身销售的互惠许可证就足够了。然而，如果未来的应用程序有可能成为部分专有的，同时包含原始或修改后的开源软件，学术许可可能是一个更好的途径。

开发者能期待一个工具来简化 OSI 中的许可证选择吗？

雷蒙德只会说，“还没有。我们正在努力。”
