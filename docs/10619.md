# 对软件自由的威胁

> 原文:[https://www.sitepoint.com/the-threat-to-software-freedom/](https://www.sitepoint.com/the-threat-to-software-freedom/)

对许多人来说，尤其是非开发者，自由软件可能指的是不需要支付许可费就可以获得的软件——免费软件。不幸的是，这是一个错误，可能会让这些消费者失去一些自由。

自由软件(有时被称为 *libre 软件*或*自由开源软件*)可以在最小限制的情况下被使用、研究和修改——如果有的话——并且可以被复制和重新分发，不管修改与否。因此，标准之一是授予用户访问源代码的权限。

一些用户认为开源软件和自由软件是一样的，但是这些人冒着失去重要自由的风险，尽管风险很小。虽然开源倡议(它提供了该术语的官方[定义](http://opensource.org/docs/osd))现在似乎是模仿自由软件的标准(如自由软件基金会[定义](http://www.gnu.org/philosophy/free-sw.html))，OSI 的焦点传统上是软件开发模型，而不是用户自由。这个名字反映了这一点；“开放”从来不是指自由。

开源也没有考虑设备是否会让你运行你修改过的程序版本，这意味着解决[电视化](http://en.wikipedia.org/wiki/Tivoization)的问题似乎没有优先权。

## 为什么它很重要

也许你是少数几个从未接触过自由软件代码的开发者之一。也许你总体上不同意自由软件的理念。然而，我保证这些年来您已经使用了很多次了——这是不可避免的！有一些我们都知道的亮点:GNU、火狐、MySQL、Apache HTTP 服务器(httpd)、Python、Ruby、PHP、WordPress、WebKit、OpenSSH 和 Android，仅举几例。还有不那么明显的；例如，Linux 内核嵌入在各种路由器、无线接入点、玩具和家用电器中。

即使你能想到可能不直接使用自由软件的个人，每个能上网的人肯定至少间接地使用它；自由软件对于网络(以及一般的网络)技术至关重要，无论是后端代码、数据库和操作系统，还是运行网络设备。

## 问题是

虽然自由软件自动成为解决供应商锁定、垄断行为、高昂的许可费用、数字版权管理和各种其他技术问题的解决方案，但它所代表的哲学和理念与许多公司销售软件许可证的传统商业模式直接冲突。不幸的是，一小部分人没有试图通过改变现有的商业模式或寻找公平竞争的方式来适应，而是以某种方式进行报复。

## 自由软件的潜在弱点

公司对抗自由软件威胁的一种方法是简单地忽略它。奇怪的是，苹果继续使用这种方法来对抗自由软件的竞争，尽管他们当前的操作系统很大程度上基于各种自由软件。我不认为公司直接从这种实践中受益，但我怀疑他们这样做是为了最小化开发、测试和支持成本。举例来说，没人会想到苹果会支持从 GNU/Linux 桌面流媒体播放 AirTunes，因为他们拒绝承认这种替代方案。与微软不同(微软最终在公众面前承认了自由软件的竞争)，苹果的大部分收入来自硬件。

第二种攻击是谎称被盗代码的所有权是作为自由软件发布的——一种 [FUD](http://en.wikipedia.org/wiki/Fear,_uncertainty_and_doubt) (恐惧、不确定和怀疑)攻击。是的，我指的是[上海合作组织争议](http://en.wikipedia.org/wiki/SCO-Linux_controversies)，这基本上变成了商业自杀！许多自由软件项目比它们的专有项目更可信(想想 Firefox 和 IE)，而且太多的人和公司都希望看到自由软件成功。

另一个想法是获得自由软件的版权所有权。这实质上允许一些自由软件项目的专有分支，并阻止原始版权所有者通过双重许可模式开发软件赚钱。这是最近甲骨文收购 MySQL 的主要问题之一。尽管购买版权永远不会扼杀自由软件，但它可以通过一些策略来帮助减缓其发展速度。

这种攻击也只能在特定情况下起作用。例如，Linus Torvalds 并不拥有所有 Linux 内核代码的版权；许多贡献者拥有部分项目的版权，这是故意的，因此没有人或实体可以获得对它的控制。下次您发现自己在查看内核源代码时，从根 Makefile 目录运行下面的命令，了解一下拥有 mainline 版权的贡献者的数量。请记住，输出仅代表 USB 驱动程序的版权所有(仅占完整内核代码的一小部分):

## 软件专利

为什么软件专利比其他领域的专利更成问题？关于这一点，还有很多可以在这篇文章中讨论的。软件专利维基是必不可少的读物——或者至少浏览你感兴趣的部分；肯定会有你感兴趣的部分，欢迎投稿。还有一个关于这个主题的优秀的可下载视频纪录片，名为[专利荒谬](http://patentabsurdity.com/)。

### 意外侵权

软件专利一般来说是一个坏主意，但是对于自由软件来说尤其糟糕。虽然所有软件都有可能无意中侵犯专利，但在专有应用程序中要检测出来要困难得多。也就是说，轻松检测某些专利侵权行为需要访问源代码。

即使你后来被发现侵犯了一项专利，专利所有人可能会放弃立即采取行动，而是等到侵权行为尽可能广泛地蔓延，以最大限度地提高潜在的法院裁决。例如， [GIF](http://en.wikipedia.org/wiki/GIF#Unisys_and_LZW_patent_enforcement) 格式在与之相关的专利成为一个问题之前已经被广泛使用，不管诉讼的延迟是否是故意的。

开源风险管理在 2004 年委托进行了一项专利研究，确定 283 项已发布的(但尚未得到法院确认的)软件专利可能被用于支持针对 Linux 内核的专利主张，如果法院确认其有效的话。几乎可以肯定的是，随着核心专利和专利数量的增长，这个数字也在增加。2007 年，[微软声称](http://www.linux-watch.com/news/NS6670466370.html):

> Linux 内核侵犯了它的 42 项专利，而 Linux 图形用户界面侵犯了另外 65 项专利。此外，OpenOffice 程序套件侵犯了另外 45 项专利，各种电子邮件程序侵犯了另外 15 项专利，各种自由和开源程序据称侵犯了另外 68 项专利。

任何自由软件开发者都希望他们的代码避免侵犯任何专利；然而，对于软件专利持有者来说，分析这种情况而不简单地重写潜在的侵权代码部分似乎是一种规范。在这种情况下，看起来微软对其专利细节保密，只是将这些声明作为其 FUD 战役的一部分。

很可能，微软认为真正的专利战风险太大；正如我们在上海合作组织的案例中所看到的，自由软件有许多捍卫者准备拯救它。但是如果微软和其他公司真的决定采取行动呢？如果微软因为你使用运行 Linux 内核的网络服务器而起诉你，却不告诉你你是如何侵犯他们的专利的，直到你上了法庭(假设你已经走了那么远)，这公平吗？

很有可能，如果一个专有软件供应商声称一项专利被一个独立开发的自由软件项目侵犯了，那么这项专利就太明显了，不应该被授予。毕竟，由于代码不可用，自由软件项目通常不能容易地从专有软件中借用编码技术思想，尽管专有软件开发人员肯定更容易摆脱它。

我们之前已经多次看到明显的专利被授予。如果亚马逊可以为它的[一键式结账过程申请专利，或者微软可以为它的](http://en.wikipedia.org/wiki/1-Click)[操作系统关机](http://yro.slashdot.org/story/10/09/01/1456235/Microsoft-Patents-OS-Shutdown)申请专利，那么在软件世界里，几乎任何东西都可以申请专利。在这种情况下，专利有时可以通过证明现有技术的存在而无效，假设一个人有资源来对抗这样的索赔。

由于通常很难知道专有软件是否侵犯了专利，你会认为这部分解释了为什么 NVIDIA 和 ATI 最快的 Linux 内核驱动程序直到今天仍然是专有的。打开它们的风险太大了，尽管将代码包含在主线内核中有明显的好处。这些厂商可能也想保住自己的软件专利，所以如果需要的话，他们可能会在专利战中反击。我毫不怀疑，如果软件专利不是这样一个问题，更多的软件开发人员会选择自由他们的项目。

### 专利“标准”

另一种专利问题是 MPEG LA 类的。当公司试图基于他们拥有的专利创建标准时，就会发生这种情况，这样他们就可以以高昂的价格获得与这些格式互操作的许可软件的好处。我们在过去已经看到了 MP3 的这个问题，但当前的热门话题是 H.264。

由于领先优势，H.264 肯定(不幸地)比 WebM 更常见。如果 H.264 赢得了 HTML5 视频格式之战，这意味着像 Firefox、Chromium 和 Konqueror 这样的自由软件浏览器(它们不应该允许自由受到专利的限制)不可能有原生视频支持。鉴于其中一些浏览器的流行，从长远来看，WebM 或 Ogg Theora 似乎是唯一可能获胜的格式，但苹果和微软仍将试图无限期地拖延仅支持 H.264 与仅支持 WebM/Ogg Theora 的战争。通过能够代表他们的用户群支付专利许可证，他们对自由软件竞争有不公平的杠杆作用，他们打算榨取它的全部价值。web 开发者又一次失败了。

## 介入

关于软件专利我们能做什么？当然他们只是生活的一部分？不一定。通过为[终端软件专利维基](http://en.swpat.org/wiki/Software_patents_wiki:_home_page)做贡献，你可以帮助像自由软件基金会这样的组织鼓励政府废除它们。

目前的一个例子是澳大利亚，该国政府预计将通过关于专利主题的新立法。目前正在起草一份提交给澳大利亚政府众议院的正式请愿书(有原始签名的书面请愿书)。我们正在寻找创新的澳大利亚软件的例子来证明软件专利对于生产创新软件是不必要的。你可以[在这里](http://en.swpat.org/wiki/Australia)阅读细节，而相关的讨论可能会在 australia-public-discuss@endsoftwarepatents.org 的邮件列表上进行。签名预计将在软件自由日活动期间以及理查德·斯托尔曼的一些澳大利亚巡回活动中收集。
