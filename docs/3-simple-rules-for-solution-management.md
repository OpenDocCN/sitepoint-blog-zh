# 解决方案管理的 3 个简单规则

> 原文：<https://www.sitepoint.com/3-simple-rules-for-solution-management/>

拥有一个结构合理的项目——或者更恰当地说，解决方案——对于确保开发计划的成功是绝对重要的。必要时，开发人员需要能够从源代码控制中提取解决方案，强制恢复数据库和任何其他外部依赖项，对配置进行微小和适当的更改，并拥有一个构建的功能性解决方案。在实践中要做到这一点有很多细微的差别，但是保持事情可管理的本质可以用三个相当简单但有实质内容的规则来表达。

1.  **Always use some sort of source control.**

    有许多源代码控制解决方案可以覆盖几乎所有的开发情况，从个人爱好者到地理上分散的大型团队。具体使用哪一个应该由您的需求和基础设施决定。但是在没有源代码控制系统的情况下进行开发是对你的客户、雇主或者你自己的不负责任。你甚至不必亲自参与管理系统——每月只需支付几美元就可以获得托管的 SVN 解决方案。几个 [站点](http://www.codeplex.com/) [为开源项目提供](http://www.sourceforge.net/)免费的公共源代码控制系统。

    为什么这个源代码控制系统是如此关键的组件？因为每个软件解决方案都需要一个，而且只有一个真正的来源。否则，当由出价最低者提供的硬盘发生可怕的碰撞死亡时，您将面临恢复原状的挑战。

2.  **If it is an integral part of the project, it is in source control.**

    二进制文件不属于源代码控制，这曾经是常识。反对这一点的论点围绕着这样一个事实，即二进制文件相对较大，不能被区分，所以以受控的方式存储它们是很昂贵的，也没有什么意义。这两个事实都不是假的，但是如果使用源代码控制的操作目标是为您需要的一切提供一个真正的源代码，那么这些二进制文件应该在源代码控制中。这包括第三方库(包括单元测试框架)、数据库、艺术文件，尤其是技术文档。当我坐在迪拜的海滩边时，如果有人需要做紧急维护，我真的不想被打扰。

    这里一个非常有趣的附带问题是“这个系统中应该包括数据库 drop 吗？”我的一般回答是响亮的是，与理解，这可能是后勤上不可能的，由于这些文件的大小或敏感性。如果不是这样，应该尽一切努力将一个示例数据库附加到受源代码控制的解决方案中，以确保开发人员是在与一站式商店打交道。

3.  **Redeployment to a virgin environment must be a minimum fuss operation.**

    有多少次你收到一个项目，即使它建立在第一次尝试，需要几个小时的调整和徒劳的配置文件，系统设置和数据库细节，使其实际工作？这不应该发生。

    在一个完美的世界里，你应该能够从源代码控制中得到解决方案，运行一个脚本，一切都应该正常工作。对于许多复杂的应用程序来说，这种场景通常有点希望。但是在最坏的情况下，重新部署应该是一个简单的、可重复的操作。这意味着应该有一个数据库创建和/或恢复脚本，包括适当的代码来重新连接用户和角色。这也意味着任何反言系统设置依赖关系都应该被脚本化。

我有意避免在这里给出具体的建议，因为具体做什么取决于您的项目和环境的性质。这并不是说人们不能从具体的例子中学习。为此，我将从[潜台词](http://www.subtextproject.com/)开始。NET 博客包，因为它总体上组织得非常非常好。此外，该项目在许多方面类似于一个定制的长期的、中等规模的 web 应用程序。你们大多数人可能正在做的事情。享受并妥善管理这些解决方案。
