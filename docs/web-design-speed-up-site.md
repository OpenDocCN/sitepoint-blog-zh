# 实用的网页设计——加速你的网站

> 原文：<https://www.sitepoint.com/web-design-speed-up-site/>

我想我一直喜欢简单的东西:功能重于形式，性能重于美观，诸如此类。当我还是个小男孩的时候，我和爸爸参加了“松木比赛”——一个由童子军发起的比赛，看谁能从一块前臂大小的松木上削出最快的赛车。在比赛当天，你和你的家人(通常包括一个非常争强好胜的父亲，他为你做了大部分工作)会把你雕刻的作品带到比赛现场。你会把你的赛车放在一个滑梯的顶部，旁边是一些其他孩子的赛车艺术作品，当哨声响起时，你会让他们离开。他们会在倾斜的跑道上翻滚，或滑行，或蹒跚，满怀希望地走向胜利，这时你的父亲会用拳头捶失败者父亲的汗水脸，而你会尽量不显得尴尬。

我很早就决定，我的赛车将是简单的，符合空气动力学的，功能性的。没有剧透、小摆设或花哨的油漆工作挡道——只有几盎司光滑的雕刻木头，可以像风一样滑行。所以我们不停地雕刻(“给我那把刀，孩子，你会伤到自己的”)我们修剪，打磨，直到我们有了一个如丝般光滑的子弹形状的炸弹，它肯定会让其他孩子哭着跌跌撞撞地回家。这很简单，没有什么花哨的东西阻碍它沿着轨道前进。要不是我们用胶水把轮子粘得太紧，以至于它们几乎不能转动，它可能会赢。我们没有获得最后一名，但那是因为另一名参赛者的赛车顽固地坐在起跑滑道上，根本拒绝移动。

…这让我想到了今天专栏的主题。如果安迪·金制造松木赛车，他会把它们做得简单、时尚、符合空气动力学、实用——而且他的车轮会正常转动。相反，他制作网站。他还创建了 [Webreference](http://www.webrefrence.com/) 和[JavaScript.com](http://www.javascript.com/)，并为其撰写文章，这两个网站是任何称职的网页设计师都应该经常参考的。自从万维网诞生后不久，安迪就一直致力于网站优化技术的研究。换句话说，他知道自己在说什么。

最近安迪出版了一本书，[加速你的网站:网站优化](http://www.websiteoptimization.com/) ( [新骑手出版社](http://www.newriders.com/)，2003 年)，这本书用简单明了的英语阐述了如何让你的网站从头到尾都更快。我很幸运地向《新骑手》讨要了一份评论副本，读完这本书后，我知道我必须与你分享安迪的一些见解。我加倍幸运地得到安迪本人的合作；他允许我与 SitePoint 的读者分享他的书的一些内容(当然是在版权限制范围内)，甚至回答了我问他的关于书的内容和范围的几个问题。你不能要求更多的合作。

我甚至不会试图总结这本书，也不会试图涵盖书中的所有内容。Andy 的大部分工作是为更复杂的 Web 创建者做的——即优化 DHTML、JavaScript 和 CSS 代码的技术，优化图形和多媒体演示，搜索引擎优化，等等。这个专栏更多的是为业余或非专业的网页设计师写的，所以我将在我的评论中坚持这样做。但是请记住，我正在讨论不那么深奥的东西。安迪的书去了本专栏不敢涉足的地方！

在你的副本到达你的办公桌之前，你可以看看补充网站，[WebSiteOptimization.com](http://www.websiteoptimization.com)。书中介绍的许多技术在这里都有解释和扩展。

##### 概述和基本原理:加载速度

第一部分给我们一个基本原理，为什么我们应该关心优化我们的网站。原因是强有力的和令人信服的:基本上，我们越优化我们的网站以快速加载和易于使用，我们就越能吸引和留住访问者。很简单。

一个统计数据很好地总结了这一点——如果你的页面在 8 秒或更短的时间内没有加载(至少足够使用),一般访问者会退出并去别的地方。这对于一般的“爱好”或个人页面，或者非盈利网站来说已经够糟糕了。但是对于一个商业网站来说，这是一个关键的信息，应该驱动整个网站的设计理念。

有没有想过为什么雅虎！是网络上最著名和最常用的搜索引擎/目录？它不是最全面的，甚至不是最容易使用的(尽管它肯定是用户友好的)；然而，就加载时间而言，它是绝对可靠的。你点击进入雅虎！而且每次都会出现，一定会出现，通常在 2-3 秒内。Andy 详细解释了为什么 Yahoo 加载如此之快，但我们要记住的是，它每次加载都快速可靠。我们的网站应该加载得很快。

<q>*“用户将缓慢加载的页面与劣质产品和服务、不安全和低可信度联系在一起。用户满意度低会导致网站和购物车被抛弃。”* —金，第 25 页</q>

即使你不依靠你的网站来支付租金，你也应该认真考虑一下你的页面的加载时间。对绝大多数人来说，我们把网站放在网上的全部原因是为了让人们使用它们。你希望，或者你应该希望，你的访问者在你的网站上有一个轻松愉快的体验。良好的网络体验的首要标准是快速和简单的页面加载。

当然，也有例外:“艺术”页面有许多大而复杂的图形；包含大量音频流和其他多媒体功能的页面；展示“前沿”网页设计效果的页面；以及并非严格为公共消费而设计的网站，如用于容纳的“存储”网站。PDF、PowerPoint 和其他文件。但是，如果设计者花时间和精力去做的话，即使是大量图形的网站也能很快加载。

我最喜欢的两个图形网站，瑞安·布利斯的[数字亵渎网站](http://www.digitalblasphemy.com/)和我的朋友罗比的[低预算设计网站](http://www.lowbudgetdesigns.com/)，都有在 5-6 秒内加载(对我来说)的主页。一旦你到了那里，你知道点击大的图形会花一些时间，这没关系。你及时赶到了那里。

如果你的网站需要花费更多的时间来加载，你可以做一些事情来让那些不耐烦的访问者保持兴趣。事实证明，如果你在几秒钟内给访问者一些可行的内容，你的页面可以用 20-30 秒的加载时间留住访问者。一些需要阅读的文本，一两张介绍性的图片，或者至少，一些线性的或者完成百分比的进度指示器将有助于保持他们的注意力，并减少他们对你的页面不能立即出现的沮丧。

让您的页面在加载时间上保持一致也是一个好主意。如果你的主页能在几秒钟内加载，那么其余的页面也应该加载，除非有一个明确的压倒一切的理由。加载你的主页是让访问者停留在你的页面上的第一步，但是如果其他页面加载缓慢，快速加载主页的乐趣会很快消失。不要调整你的主页，让你的内容下滑；如果你要优化，在你的网站上做。

<q>*“关键属性(对于一个具有良好“流量”的优化网站)是，它首先应该非常用户友好和透明，但人们应该能够立即发现其中的复杂性，以便快速找到与自己的技能相匹配的“行动”机会的正确水平。这些‘挑战’既包括内容，也包括视觉。”*——米哈里·契克森米哈，引自金，第 34 页</q>

有几个因素有助于你的网站开放、容易的“流动”。加载速度是第一要素，但不是唯一要素。易于导航是让你的网站运行良好的关键；这包括提供“路标——如网站地图、面包屑路径和‘你在这里’的地标”(第 36 页)，以帮助你的访问者在你的网站上方便直观地移动。快速清晰的反馈是好的网站流的另一个要素:清晰的内部和外部链接，导航工具，以及性能变量的显示，如进度指示器，可下载文件大小显示等等。

您的页面应该针对用户的技能和知识水平进行优化:提供一个“适应性强/可调整的界面，让用户能够根据自己的技能水平控制环境的复杂性”(第 36 页)是一个值得包含的内容。布局和功能的简单性比炫耀你尖端的编码技能更有价值。请记住，这个想法是让你的访问者运行你的网站，尽可能轻松愉快地浏览，然后带着希望很快回来的愿望离开。

安迪推荐 [SpaceAgent](http://www.insidersoftware.com/) ，这是一个 99 美元的程序，可以优化网站中的图形和文本，以获得最佳效率，这是实现最大优化的良好第一步。当然，像这样的程序不能完成全部工作，但是这是一个很好的开始方式。

##### 卷起袖子:优化您的代码

在这里，我们进入优化您的网站的最大效率的本质。Andy 给出了很多关于调整和修剪 HTML 的具体信息；我将介绍一些亮点。

***最小化 HTTP 请求***

我问了安迪以下问题:“你会特别向业余和/或非商业网页设计师推荐哪些技术？那些不怎么使用 CSS 的网页设计师怎么办(要么是出于选择，要么是对如何使用 CSS 缺乏了解)？”他回答说，“通过减少你的页面中的对象数量来最小化 HTTP 请求，并组合相邻的图形。这可能是最重要的技术。”

每当您包含一个对象时——一个图形、一个 Java applet、一个外部样式表、一段 JavaScript——页面必须“调用”服务器来传递对象。理想情况下，优化后的网页会有尽可能少的服务器调用。网络大师雅各布·尼尔森是安迪影响较大的人之一，他估计每个“呼叫”需要 0.5 到 2 秒才能完成。所以你需要考虑你的页面需要做的每一个“调用”:有必要吗？你能让你的页面在没有特定对象的情况下实现它的功能吗？

安迪给了我们一个现实生活中的例子，网络广告制作人[Elivad.com](http://www.elivad.com/)的主页的原型。最初的主页有 7 个带有翻转的图形，总共有 14 个服务器调用，以及 6 个针对样式表、JavaScripts 和页面本身的其他调用。该页面在 56k 的速度下加载大约需要 15 秒。

在优化之后——使用 CSS 翻转将图像和 JavaScript 翻转转换为文本，将按钮转换为带彩色背景的文本链接，合并和优化图像，以及优化 JavaScript 和样式表——页面能够将其服务器请求从 20 个减少到 6 个，将 56k 调制解调器的加载时间加快到 6 秒。通过消除 JavaScript 翻转代码，HTML 代码本身减少了 41%，即使考虑到稍微大一点的 CSS 代码(由于在导航栏中添加了 CSS 翻转)。该网页的功能一样，因为它的加载速度几乎快了一倍(在 56k 调制解调器上平均 6 秒)，它将吸引和留住更多的访问者(参见 pgs。50-53).

但是，你会说，我的页面有几十个服务器调用，加载不需要 15 秒！也许不是，但你确定吗？您是在 56k 拨号连接上测试您的页面，还是假设每个人都在使用您正在使用的宽带或 T1 连接？我们知道，现代的浏览器和服务器能够同时发送多个请求，以加快每次调用的下载时间。但是请记住，“即使使用 HTTP keep-alive，每次往返都要增加更多的时间，因为消息必须在互联网上从客户端到服务器再返回”(第 53 页)。

当考虑文档的`<HEAD>`中的外部文件时，这一概念尤为重要。页面的`<HEAD>`中的任何样式表和 JavaScript 都必须在可视化显示之前进行处理；包含多个 CSS 和 JavaScript 文件的页面可能需要非常长的时间来处理，而且整个过程中，您的站点访问者都在坐着等待某件事情发生。

***清理代码***

我假设你们中没有人使用膨胀代码来制作页面，比如微软的 Front Page。如果你是，那么你就太离谱了，这篇文章帮不了你多少。

我能告诉你的是，有一段时间，我受雇于一家大公司的网页设计团队，帮助优化和标准化他们内部的技术“帮助”页面。我一半的时间都花在把弯刀拿到头版上。我通常能够去掉 50-80%的代码，而丝毫不影响页面的显示(尽管我被告知 FP2000 和更高版本不太容易膨胀)。换句话说，如果你使用 MSFP 的旧版本(或者另一个膨胀码特辑——那里有很多)来生成你的页面，找到另一个替代品！

大多数访问者不能阅读 HTML，也不知道代码是好是坏，他们只是对好的内容和导航感兴趣。 —安德鲁·史达琳

对 Andy 提供的一些优化技术的最大批评之一是，它使得那些细读源代码的人很难阅读和修改它。确实如此，但是这是不简化和优化您代码的理由吗？几乎所有网站的访问者都不会盯着你的代码看——他们一点也不在乎。至于你的设计师同事，他们将不得不全力以赴地处理它——为了更低的带宽和更低的救助率，这是值得的。

***删除线返回***

这是一个为了效率而放弃美观的代码的完美例子。浏览器不关心你是否在每个命令后有一个回车；删除换行符可以显著减小文件大小。这是第 48 页的一个例子:

```
<TABLE>   

<TR>   

<TD>speed</TD>   

</TR>   

<TR>   

<TD>me</TD>   

</TR>   

<TD>up</TD>   

</TR>   

</TABLE>
```

变成了:

```
<TABLE><TR><TD>speed</TD></TR><TR><TD>me</TD></TR>   

<TD>up</TD></TR></TABLE>
```

文件大小减少了 11%。它没有那么漂亮，但展示时看起来一模一样。在代码美学上值得牺牲吗？你决定吧。

***选择正确的文档类型:严格与古怪***

使用“严格”标准的 HTML 页面将比使用“过渡”标准的页面解析得更快，并且比使用旧标准或根本没有 DOCTYPE 的页面快得多。关于文档类型的进一步解释，请参见我在 SitePoint 的上一篇文章[Web 设计基础](https://www.sitepoint.com/blog/)，并访问 [Mozilla 的文档类型嗅探页面](http://www.mozilla.org/docs/web-developer/quirks/doctypes.html)了解各种标准的详细信息以及 Mozilla 等浏览器如何决定使用哪种文档类型。

当然，使用 Strict 标准的页面必须用 strict HTML 4.01 编写——没有不赞成使用的元素。使用所有结束标记的严格 HTML 4.01 编写的页面将比使用不推荐的元素和“古怪”的 HTML 的类似页面呈现得更快，因为浏览器必须做更少的工作——即使由于使用了所有结束标记，它会稍微大一些(稍后将详细介绍)。这件事的好处是，它也让我们离符合 XHTML 标准更近了一步…这是一个完全不同的主题。

***空白***

HTML 代码中的“空白”——缩进、跳过的行等等——完全是为了方便设计者和查看代码的人使用。它对代码的显示方式没有任何影响，实际上还会降低加载速度。53-4 页有一个简单的例子。准备好惊讶于代码大小降低了多少吧。

```
<TABLE   ID='whitespace-repository'   WIDTH="800">    

  <TR>    

    <TD>speed</TD>    

    <TD>me</TD>    

    <TD>up</TD>    

  </TR>    

<TABLE>
```

成为

```
<TABLE ID='whitespace-repository' WIDTH="800">    

<TR>    

<TD>speed</TD>    

<TD>me</TD>    

<TD>up</TD>    

</TR>    

<TABLE>
```

或者更好的是:

```
<TABLE ID='whitespace-repository' WIDTH="800"><TR>    

<TD>speed</TD><TD>me</TD><TD>up</TD></TR><TABLE>
```

最后一个示例比原始示例节省了 50%以上的文件大小。

例外:某些 JavaScript 在某些情况下需要空格。除非你知道自己在做什么，否则不要试图优化 JavaScript。

***逗号分隔属性***

一些标签，最常见的是`<KEYWORDS>` 元标签、`<MAP>`坐标属性和`<STYLE>`标签，允许逗号分隔的列表。浏览器和搜索引擎都会忽略逗号后面的空格。通过消除空格，您可以节省宝贵的文件大小。同样，对于阅读代码的人来说，这并不容易，但其他人都会从中受益。

这里有一个例子，抄袭自 pgs。54-55:

```
<STYLE TYPE="text/css">    

<!--    

body {    

font-family: arial, helvetica, sans-serif;    

font-size: 1.1em;    

}    

-->    

</STYLE>
```

变成了:

```
<STYLE TYPE="text/css">    

<!--    

body{    

font-family:arial,helvetica,sans-serif;    

font-size:1.1 em;    

}    

-->    

</STYLE>
```

或者甚至:

```
<STYLE TYPE="text/css"><!-- body {font-family:     

arial, helvetica, sans-serif;font-size:1.1 em;}--></STYLE>
```

另一个很好的减少文件大小的小损失美观。

*部门冗余**部门冗余**部门冗余*

您会惊讶地发现有多少标记和属性是多余的，可以在不影响代码运行的情况下删除。像`<FONT>`和`<BASEFONT>`、`<CENTER>`、`<STRIKE>`和`<U>`这样的废弃标签可以用样式表更有效地处理。许多种类的`<BODY>`属性也是如此—`BGCOLOR,``BACKGROUND,``LINK``ALINK``VLINK`等等。越依赖样式表，就越符合 XHTML。

表格是查找冗余元素的好地方。一个简单的例子，抄袭自第 56 页，告诉我们如何收紧我们的表代码。

你可以用这个:

```
<TABLE>    

  <TR>    

    <TD ALIGN="right">tastes great</TD>    

    <TD ALIGN="right">less filling</TD>    

    <TD ALIGN="right">burma shave</TD>    

  </TR>    

</TABLE>
```

但是这个效果更好:

```
<TABLE>    

<TR ALIGN="right">    

<TD>tastes great</TD>    

<TD>less filling</TD>    

<TD>burma shave</TD>    

</TR>    

</TABLE>
```

这样可以节省更多的空间:

```
<style type="text/css">    

<!--    

table.tr.right{text-align:right;}    

-->    

</style>    

...    

<TABLE><TR CLASS="right"><TD>tastes great</TD>    

<TD>less filling</TD><TD>burma shave</TD>    

</TR></TABLE>
```

您可以使用相同的样式表在该页面上编写更多的表格。

***剔除可选引号***

HTML 4.01 中的一些属性可以不加引号。如果属性仅包含字母数字字符、连字符、句点、下划线和冒号，则可以不加引号。如果属性包含其他字符，则必须用引号括起来。

这是可行的:

```
<IMG SRC="t.gif" WIDTH=1 HEIGHT=1>
```

但这并不:

```
<TABLE WIDTH=100%>
```

雅虎！最繁忙的网站省略了链接标签中的引号。其他网站完全省略了引号，违反了标准的 HTML 建议，但仍然相信个人浏览器能够处理代码。通常这是有效的……通常(第 57 页)。

***省略可选的结束标记***

好吧，这里有一个节省了一些字节，但违反了 XHTML 标准，该标准要求所有标签必须是封闭的，属性必须是完全限定的。如果这与你无关，那么这些标签就不必关闭:`<P>`、`<LI>`、`<OPTION>`，甚至`<BODY>`、`<HTML>`。您也可以省略`<TR>`和`<TD>`标签的结束标签，但是如果没有这些结束标签，Netscape 3 将无法正确呈现表格。

这是页面大小和呈现速度之间的一个权衡:带有所有结束标记的 HTML 呈现速度稍快，但是省略结束标记会减小页面大小，并且仍然允许代码进行验证。这是你必须考虑的选择。

***评论***

代码正常显示很少需要注释。它们通常用于标记文档的特定部分，或者为设计者提供他们自己使用的信息。你越是能够在没有注释的情况下运行，或者你越是能够缩写和压缩注释，你的代码运行起来就越紧凑和高效。

***最小化`<HEAD>`节***

在你的`<HEAD>`部分，元标签、样式表和 JavaScript 占据了大量的空间。你能相处的越少，你的网站加载就越快。记住，`<HEAD>`部分中的所有内容都必须在页面加载之前解析。

元标签经常被过度使用。虽然它们在某些情况下是必要的——某些搜索引擎的关键字规范、定义默认脚本、样式表语言和字符集、增加从服务器发送的 HTTP 头等等，但是它们中的许多都是多余的或不需要的。从最严格的意义上说，您绝对需要的唯一元标签是关键字和描述标签。其余的都是多余的，或者可以通过服务器设置更好地处理(关于使用 META 标签的更多信息，请参见我的实用网页设计专栏，[15 大 META 标签技巧](https://www.sitepoint.com/blog/))。

***优化表格***

Andy 花了整整一章的时间来讨论表优化，我不会试图在这里总结所有的内容。我要指出的是，浏览器需要时间来整理和解析表格，页面设计越依赖表格，访问者访问网站内容的时间就越长。在理清各种表格结构、在表格单元格中放置图像、来回运行几十个 HTTP 调用以使一切正常运行之间，加载一个页面可能要花很长很长的时间。

尽可能多地告诉浏览器表的结构和复杂性，尽可能降低表的复杂性，这是效率最大化的方法。尽可能简化。使用 CSS 样式化表格。使用背景色而不是背景图片。如果你真的想加快速度，完全去掉表格，使用各种`<DIV>`元素和 CSS2 来布局你的页面(Andy 给出了另一章关于这个特定主题的建议)。

我们都可以做的一件事是给浏览器足够的表格信息，以便能够增量地呈现表格，而不是必须等待一切都提前加载。您可以使用`COL` 和`COLGROUP`元素告诉浏览器您使用了多少列以及它们的宽度(以像素、百分比或相对宽度为单位)。这使得浏览器在呈现之前只对表进行一次传递，而不是两次传递(第一次传递查找列数)。如果一个表格没有固定的宽度，浏览器必须接收每个字节的数据，然后才能调整表格的大小。

您还可以使用`THEAD`、`TFOOT`和`TBODY`元素将表格行分组为表格头、表格脚和表格体部分。Andy 说得好:“当您希望在每页上打印页眉和/或页脚信息时，您可以单独设置这些组的格式，并使用它们来打印更长的多页表格”(第 69 页)。

许多网站(包括我的网站)使用一个包罗万象的表格来展示页面的大部分或者全部。这通常意味着浏览器必须下载该表中的所有内容，然后才能显示它。一个页面坐了又坐，看起来什么也没做，然后突然一下子显示出来，大概就是这样布局的。为什么要强迫你的观众这样等待？相反，将你的表格分层。以一个小的、快速加载的表格开始你的页面，包含一些有用的东西，比如搜索框、导航条，或者至少是一个标题图，来奖励你的访问者的努力，并向他们保证你的页面确实是成功的。一个“折叠上方”的表格允许额外的内容在第一个表格之后加载，让你的访问者在加载页面的其余部分时有所应对。即使你的整个页面加载速度比以前慢，给你的访问者一些即时的东西也是值得的(不，首先弹出的广告不算数。访问者来到你的网站不是为了看广告)。

如果您严格使用符合 CSS2 的浏览器，您可以使用`TABLE-LAYOUT` 属性来加速表格的呈现。这个属性让我们控制浏览器用来布局表格单元格、行和列的算法。

***URL 中的尾随斜线***

有些人认为这两个网址没有任何区别:

http://www.mysite.com 和 http://www.mysite.com/

但是有。尾部的斜杠告诉浏览器 URL 指向一个目录，因此浏览器不必决定 URL 是指向一个目录还是一个文件。您可以通过在目录列表(不是文件，不要这样做:http://www . my site . com/index . html/)后面加上斜杠来加快 URL 的响应时间。

***让 CSS 开始工作***

我会让你买这本书来了解这方面的优化细节，但简而言之，使用 CSS 来取代图像翻转和基于表格的设计是非常值得你花时间和精力去转换的。随着越来越多的用户转向 IE 5+、Netscape 6+和 Mozilla 等现代浏览器，那些浏览器拒绝支持 CSS 的用户就不会有什么问题了。
