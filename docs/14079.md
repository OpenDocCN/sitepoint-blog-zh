# PHP 的业务

> 原文：<https://www.sitepoint.com/the-business-of-php/>

也许 PHP 领域的一个异常现象是，尽管大量采用 PHP 作为软件开发平台，但其中存在的商业软件产品的生态系统相对较小。人们可能很容易把这归因于 PHP 是一个开源的——而且很大程度上是免费的——产品。使用“零成本”平台的人通常不太愿意为在其上运行的软件付费。

但是，在开源领域，这种争论经常发生，我们都需要吃饭，如果做不到这一点，我们还需要购买 iPods 和固定齿轮自行车。许多伟大的软件工程头脑[在众所周知的项目](http://www.kernel.org/)上修修补补，而[却因此获得报酬](http://www.osdl.org/about_osdl)。然而，对于我们其他人来说，商业动机需要存在于我们日常活动的某个地方。

我敢说大多数专业 PHP 开发人员(我不严格地使用这个术语；指以编写 PHP 代码为生的人)在某种客户服务实体中工作。也就是说，代表客户建立 PHP 网站和应用程序并收取费用。

(顺便提一下，我认为这可能是 PHP 社区不可抑制的重新发明轮子的欲望的一个原因——但是我将把它留到另一篇博文中。)

紧随该组之后的是那些为组织“内部”构建解决方案的开发人员，这些解决方案用于该公司的运营，并因此用于商业目标。我自己就属于这一类。

尽管如此，PHP 社区中有一小部分人——而且还在不断增加——专注于为该平台开发专有软件。我发现他们的商业模式和他们面临的挑战很吸引人。

**[【vbu tin】](http://www.vbulletin.com/)**

第一个伟大的专有 PHP 成功故事？我敢打赌，很少有 PHP 开发人员没有听说过(或者确实使用过)Jelsoft 的 vBulletin 论坛软件。

他们已经存在五年了，而且看起来很高兴地在他们的领域里击退了“免费”的挑战者。

从表面上看，这确实令人惊讶，因为公告牌软件已经高度商品化，并且存在大量非常成熟的应用程序。

**[地狱犬求助台](http://www.cerberusweb.com/)**

另一个成熟的专有 PHP web 应用程序，它处理客户支持/服务台场景。这个特殊的类别似乎有更少的免费竞争对手，并旨在直接减少大量的业务费用:客户支持。很容易理解，人们乐于为这样的应用付费。

**应用与图书馆**

有太多其他“成熟的”商业 PHP 应用程序可用；安装人员自带的交钥匙解决方案，通常需要客户很少干预。相比之下，我想看看软件业的另一部分:库和组件。

来自专有平台背景的开发人员——显然 Windows 就是一个例子——习惯于在他们的应用程序中使用丰富多样的预构建组件生态系统。

其中许多都相当昂贵，低端开发人员和业余爱好者无法企及。长期以来，PHP 社区的一个优势是，在微软 ASP 平台中，大量可能需要付费的功能可以在各种 PHP 中免费开源获得。

然而，这导致专有 PHP 组件的生产非常有限，这意味着如果您需要解决一个模糊的中间件问题，您可能只能靠自己。开源社区倾向于只解决常见的问题。

然而，我设法找到了两个专有的 PHP 库:

**[JpGraph](http://www.aditus.nu/jpgraph/)**

JpGraph——很大程度上是 PHP 的权威图形库——以开源形式提供，用于非商业用途，但任何商业用途都需要支付许可费。

**[PHP AdWords API 库](http://www.phpadwordsapi.com/)**

这是一个有趣的问题。一个有事业心的家伙，Georgi Mirchev，预见到了 Google 的 AdWords API 发布将会产生的需求，并率先向市场推出了一个用 PHP 编写的、文档完整的 web 服务 API 实现。

然而，人们不得不期望，在如此需求旺盛的应用中，开源克隆不会太遥远。

**关于源代码保护**

有趣的是，专有 PHP 应用程序中的主要参与者并不编码或混淆他们的源代码。

无可否认，这样做的选择最多是有限的，但它证明了许可和保护知识产权不一定会成为一个可行的业务的障碍。

我敢打赌，大多数以微软为中心的开发人员会因为害怕向客户提供他们的全部源代码而逃之夭夭！

那么谁在使用专有的 PHP 软件呢？你们中有谁开发并销售这种软件？
