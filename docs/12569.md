# 谷歌取消 OpenID 白名单要求

> 原文:[https://www . site point . com/Google-removes-OpenID-whitelist-requirement/](https://www.sitepoint.com/google-removes-openid-whitelist-requirement/)

在[宣布支持 OpenID](https://www.sitepoint.com/the-single-sign-on-war-will-ruin-openid/) 的一天后，谷歌决定放弃依赖方必须首先获得谷歌批准才能接受来自 Gmail 的 OpenID 的要求。谷歌表示，他们取消要求的原因是申请 API 访问的网站数量超过了他们的处理能力。

谷歌的埃里克·萨克斯[在博客文章](http://google-code-updates.blogspot.com/2008/10/moving-another-step-closer-to-single.html)中解释了这对消费者意味着什么:

> 这一注册要求也导致了一些混乱，因为用户希望能够使用接受 OpenID 2.0 兼容登录的现有网站，只需在这些网站的登录框中输入“gmail.com”(或在某些情况下输入完整的电子邮件地址)。通常情况下，用户输入 gmail.com 后，依赖方网站会在 gmail.com 服务器上寻找一种特殊类型的文件(XRDS ),检查 Gmail 是否运行 OpenID 身份提供商。对于昨天的发布，我们特意选择不在 gmail.com 上发布那个特殊的 XRDS 文件，因为如果我们发布了这个文件，如果用户试图登录的网站没有在我们这里注册，他们就会在谷歌收到一个错误。既然我们已经删除了注册要求，我们将尽快推送 XRDS 文件。一旦 XRDS 文件上线，最终用户应该可以通过在任何支持 OpenID 2.0 的登录框的 OpenID 字段中键入 gmail.com 来使用该服务，就像雅虎用户可以键入 yahoo.com 或他们的雅虎电子邮件地址一样。(与此同时，如果你觉得真的很怪，你可以在一个符合 OpenID 2.0 的登录框中键入“https://www . Google . com/accounts/o8/id”，并查看定向身份工作流的运行情况。)

谷歌还解决了他们何时以及是否会成为依赖方的问题。萨克斯说，他们没有的原因是一个技术问题。他说，问题是，如果谷歌支持消费者用户的联合登录，富客户端应用程序将会崩溃，因为用户名和密码的概念被硬编码到这些桌面和移动应用程序中。他说，对于使用自己的身份提供商的企业电子邮件客户来说，这正是今天已经发生的事情，谷歌是他们的依赖方。

萨克斯说，谷歌正在解决这个问题，但没有承诺谷歌将成为一个依赖方(或谈论发生这种情况的时间表)。

我们的一位读者 Deron Meranda 提供了一些有趣的想法，解释了为什么像 Google 或 Yahoo！可能不想成为昨天评论的依赖方。以下是摘录版本:

> 如果他们的一个电子邮件账户被黑了；他们可能有一些法律责任，或至少坏的公关内容。这已经够糟了，当雅虎！当[美国州长莎拉]佩林的账户被破解时，受到了很多负面报道；想象一下，如果第三方操作也参与进来，会发生什么。雅虎！仍然会得到所有不好的关注，但违规甚至不是他们的错，也不在他们的控制之下。
> 
> 此外，希望大公司比小网站更安全。他们有能力正确和安全地管理登录，加密密码，处理密码恢复，防止僵尸帐户，等等。他们也倾向于对用户的隐私有更多的保护(或者至少有更多的钱和层次)；当然它并不完美，但是当一些公司说它需要用户的名字来注册一个账户时，谷歌将会极力抵制；没有合法的授权。我不确定所有较小的运营商是否都“安全”或值得信赖，因此大型运营商应该担心，当他们将身份认证外包给另一方时，这可能会危及其用户的隐私。
> 
> 这并不是说我们不应该强迫他们也成为注册会计师，但我们应该意识到他们有一些特殊的情况需要仔细考虑。我认为这只是时间问题，会让 OpenID 更加成熟。
> 
> 还有，除非你是少数几个大玩家之一(谷歌，雅虎！)，那你应该是个 RP。仅仅作为一名特工的理由是站不住脚的。

换句话说:这是政治。

不管是什么原因，我仍然坚持我的观点，OpenID 不会起作用，对消费者来说不会是一个容易的销售，直到他们真正相信他们的 ID 可以在任何地方登录，不管他们的提供商是谁。雅虎的尝试。谷歌通过鼓励开发者添加“登录雅虎！/Google”按钮也没有帮助。不过，希望三巨头真的想成为依赖方，而不是控制一种打上*烙印的*通用身份体验。这是一件我很想弄错的事情。
