# 增强用户登录过程的 11 个专家提示

> 原文:[https://www . site point . com/11-ways-to-enhance-your-web-application/](https://www.sitepoint.com/11-ways-to-enhance-your-web-application/)

你和你的团队为你的 web 应用程序付出了辛勤的劳动——这是你的骄傲和快乐。另外，你测试过的用户都很高兴。他们只是喜欢你添加的所有社区建设功能。太好了！然而，在推出六个月后，该网站并没有在社区中获得任何吸引力。人们似乎渐行渐远。为什么？一切看起来都很完美，那么还会有什么问题呢？

这种情况太普遍了，根源往往在于基础。即使是网站的一个非常小的组件也会对用户体验产生巨大的影响！

例如，如果登录过程本身提供了一个糟糕的体验，那么人们会不愿意使用它，你添加的所有杀手级功能都将是徒劳的。在最糟糕的情况下，它可能会阻止人们再次登录网站，这意味着没有社区，没有重复销售…所有这些加起来就是一个失败的网站。

## 那么是哪里出了问题呢？

大多数时候，网站或 web 应用程序中的问题都很简单。尽管如此，即使是小问题也可能等同于负面体验。而且如果可以的话你真的不希望有什么负面的体验。用户是敏感的，你懂的！

的确，有些人会全力以赴，试图解决他们遇到的任何可用性问题——我们都喜欢解决问题，对吗？这是我们的天性。但是不要忘记，随着网络变得越来越容易使用，人们越来越不能容忍糟糕的交互设计，如果能提供更好的体验，他们通常会寻找替代服务，这取决于他们在你的网站上投入了多少。

## 这很简单…然而事实并非如此

登录交互设计表面上很简单。然而，对于用户登录页面的最终设计考虑，有相当多的因素。当它们结合在一起时，事情会很快变得复杂。以下是一些需要考虑的因素:

*   安全性
*   以前的用户体验
*   现场遗留程序
*   内部业务流程
*   页面界面设计
*   受众平台考虑因素

你或许可以想出更多的因素来添加到这个列表中。不管怎样，我们仍然可以关注一些简单的事情，以确保体验良好。以下是我列出的确保你的用户不断回来登录的建议。

## 1.使用电子邮件地址作为用户名

研究表明，人们在记住自己的密码方面有很多困难，甚至不需要记住用户名。使用人们更容易记住的字符串，比如电子邮件地址，可以进一步降低用户忘记登录信息的可能性。

网站的用户名采用由用户指定的电子邮件地址的形式的惯例变得越来越普遍。当然，使用电子邮件地址作为用户名的方法可能存在一些问题，例如:

*   一些服务提供商回收电子邮件地址
*   用户更改他们的名字，以及他们的电子邮件地址
*   采用不同格式的电子邮件地址

然而，这些问题没有一个是不可克服的——只是要确保允许(并测试)上面列出的不同场景。

强迫用户使用会员号(或分配给他们的其他用户名)登录的常见方法没有任何帮助。如果你的网站用户名必须使用电子邮件地址以外的东西，至少让你的用户通过创建他们自己的用户名在你的网站上使用来个性化他们的帐户。

## 2.让你的用户使用长密码

在这个新时代，我们不断被提醒使用安全的密码。你知道规矩——时间越长越好；特殊字符越多越好…

随着自动填写用户名和密码的密码记忆插件和浏览器的出现，人们可能会认为今天的 web 应用程序中密码字段的平均长度会越来越长，并且能够接受 64、128 甚至 256 个字符长的密码。人们甚至希望八个字符密码的时代正在迅速消失！

事实上，这种接受长密码的转变正在发生。然而，我们还没有到那一步，似乎正处于过渡期。

请注意，当用于接受用户名或密码的文本字段与存储它的相应数据库字段的长度不同时，可能会出现问题。这可能导致用户输入的密码被截断，甚至更糟的是，整个记录完全无法访问。如果您提供长密码，请务必测试它们！

有趣的是，银行通常不允许密码包含特殊字符或长度超过 12 个字符。这种限制通常是由于他们与之交互的遗留系统的限制，而不是他们的 web 服务的限制。大多数 web 服务不受此类约束的约束，不要遵循旧的模型。

解决方案很简单—从一开始就计划允许长密码。通过在设计文档中获得正确的信息，确保您定义了密码的长度，并测试该上限，您应该能够避免与密码长度相关的任何障碍。

另外，至少在注册过程中，一定要告诉用户您的系统允许的最小和最大密码长度。这应该使用位于相关字段旁边的文本消息来实现。用户不会意识到这样的细节，除非你告诉他们！

## 3.在表单验证中添加一些 Ajax

我们并不都是完美的——有时我们会输错用户名和密码。我们的用户也是如此。

当然，出于安全原因，在我们授予用户访问权限之前，用户输入的密码需要完全匹配。

然而，让我们稍微放松一下用户名。例如，如果用户名是一个电子邮件地址，作为一个用户知道我是否不小心输入了就好了。反对“代替”。com”。如果在提交表单之前提供这个警告就更好了！

在这种情况下，一点 Ajax 风格的验证会大有帮助。检查用户名以确定它是否是唯一的或者格式是否正确会使事情变得简单一些。

进行验证时要考虑的另一个因素是，如果用户在用户名后面输入一两个空格，会发生什么情况。你看，这个空格在屏幕上对用户来说并不明显，但实际上他们输入了一个无效的用户名，因为它在末尾包含了额外的空格。

这个障碍很容易克服——只需修改用户名字段。您可以在客户端或服务器端执行这种修剪(两者都更好)。重要的是在表单验证中加入一些智能，并对用户想要的用户名进行一些简单的、有根据的猜测。

## 4.维护持久登录

曾经有一段时间，你可以登录到一个 web 应用程序，并保持登录状态，直到你注销！还记得那些日子吗？这不就是“记住我”或“保持我登录”复选框的作用吗？当然，那些提供这一功能的网络应用并不是像网上银行或股票交易网站那样的关键服务。但是男孩方便吗！

那个功能最近似乎已经过时了。如今，“记住我”复选框只意味着“记住我一小段时间”。最近有一个明显的趋势，web 应用程序要求您在一周、两周或其他任意时间段后再次登录。

![Persistent Login Example](../Images/d84a65f288e7e7696ce5b03815db57b1.png)

这些有时间限制的持久登录当然是一种安全措施，但它们只适用于正在使用的计算机。因此，如果计算机实际上位于一个安全的环境中，时间限制对用户没有好处，也不方便。不得不一次又一次地登录一个你宁愿无缝使用的 web 应用程序变得很烦人。

在这种情况下，通知用户他们的登录将持续多长时间当然有助于管理预期，但这并不会使每两周重新登录的过程变得不方便。

更好的方法是允许用户控制持续登录的有效时间。通过让用户控制这个设置，我们可以让每个人都满意。一个只从家用电脑访问网站的用户可能会将其登录设置为“永远”，而另一个从图书馆或网吧访问相同服务的用户可能会将其设置为“从不”。毕竟，登录信息属于用户(他们用它访问的数据也是)，所以我们的用户一定有足够的智慧做出这个决定吗？

## 5.让您的文本字段靠在一起

将用户名字段及其关联的密码字段放在单独的页面上是我鼓励的一种做法——拥有一个专用的“登录”页面比将登录过程集成到另一个页面中要容易得多，尤其是当该页面包含其他表单和文本字段时。

但是我们的用户名和密码字段应该如何对齐呢？有两种主要的思想流派——并列的和相互重叠的。

重要的是要确保这两个领域彼此非常接近——记住，它们毕竟在信息和功能方面是相关的。因此，它们也应该在空间上相关联。

![Login Layout Examples](../Images/8cee16e55a804ccd96d345c99396833e.png)

当用户名和密码在屏幕上不在一起时，登录页面可能会有问题(信不信由你，我见过多达三分之一的页面出现这种情况！)用户名栏和密码栏之间的区域不适合大横幅广告！请将这两个字段放在一起，以减少混淆。

这是一个小问题，但是让你的用户寻找他们需要用来登录的字段，并让他们怀疑他们是否在正确的字段中输入了他们的登录信息——特别是如果他们已经在页面的更下方——并不能真正让用户对 web 应用程序充满信心。

## 6.将您的登录链接放在顶部

正如用户期望点击一个网站的标志会把他们带回到网站的主页一样，现在许多用户期望在页面的顶部(通常在右手边)看到一个到登录页面的链接。

把你的登录链接放在别处会导致访问者玩“寻找登录”的游戏，这对你的目标没有帮助。当然，应用程序的用户会习惯它的位置。然而，当一个新用户最容易受到挫折的时候(通常会对你的网站留下持久的印象)是基于他们最初的一些使用你的网站的经验，在他们知道各种功能存在于哪里之前。你只有短暂的时间，所以你希望这些经历是积极的。

## 7.标记您的登录链接

您可能已经注意到，在登录过程中有许多约定，登录标签也不例外。许多访问者可能寻找的确切文本不是“登录”就是“登录”。[有多种变体](http://www.maxdesign.com.au/2008/12/12/sign-in/)，但是这两个词几乎被普遍理解，所以使用起来非常安全。

不幸的是，现在有一些网站(没有指名道姓！)认为适合使用唯一标签来表示“登录”。在最糟糕的情况下，链接会被贴上“机会”、“推荐”甚至“新功能”这样的标签——这些都与登录无关。当用户在他们期望登录的位置看到一个链接时，他们开始质疑这个链接是否是一个广告，以及这个链接后面的页面可能在卖什么。

![Login Link Example](../Images/94fe1cbb859dcada840012943c092a31.png)

当营销部门坚持要把登录链接标上新的或不同的东西时，争论这个问题可能会很困难。如果你不能赢得这场战斗，一个折中的办法可能是在一个更标准的登录链接之外添加营销链接。指向同一个页面的两个链接所造成的混乱要比弄乱一个访问者期望找到的更标准的链接所造成的混乱要少。

## 8.让用户找回忘记的用户名和密码

另一个已经成为标准的约定是为用户提供一个链接来重置或恢复忘记的密码，并在登录表单的同一页面上列出该功能。

然而，在用户输入任何信息之前，在登录页面上显示这个特性会让人有点分心。这几乎就像是嘲弄他们说“来吧——我就知道你会犯错！”
很容易修复——只在用户登录出错后显示您的密码恢复解决方案的链接..只需要一点 JavaScript 来改变包含这个链接的段落的`text-indent`属性。
另一个需要考虑的情况是，用户甚至记不起他们用作用户名的电子邮件地址。这种情况下你该怎么办？

这很容易处理——通过忘记密码功能，您已经可以获得这些信息。

密码恢复过程显示的错误消息通常足以让用户确定他是否输入了正确的用户名。例如，如果错误显示“没有用户使用该电子邮件地址注册”，那么您的用户将立即知道他输入的电子邮件不是正确的。

当然，你还可以提供其他的退路——秘密问题、用户的个人信息等等。然而，对于大多数网站，一个简单的密码恢复过程就足够了。

## 9.显示有用的错误消息

众所周知，错误消息在网络上是个问题——在一般的软件中也是如此。是的，在出现错误时通知用户是很重要的，但是没有必要用技术术语来迷惑他们(也不应该泄露不必要的信息，以防看到错误消息的人是试图危害您的系统的恶意黑客)。

当然，还有另一个极端——没有给用户足够的信息。假设你只想登录你最喜欢的社会化书签服务。这不是诺克斯堡——人们不会因为你的用户知道他们输入了错误的电子邮件地址而决定生死，所以让他们知道。比起一条什么都不告诉他们、让他们蒙在鼓里的简短信息，他们会更欣赏它。

编写有用的错误消息是一门艺术——听起来好像我在陈述显而易见的事情，但是要尽可能地清晰。根据内容交付方式和你对品牌的自由度，你应该尝试在个人层面上与用户互动。

例如，“无效授权”是机器人和令人困惑的。“你似乎输入了错误的密码”是一种更友好的说法。

对于非关键任务、低容量的应用程序，您甚至可以考虑通过多做一点来改善用户体验，让用户知道他们输入的用户名/密码对的哪一部分不正确。最后(这真的是可用性 101)，不要侮辱人。Matthew Magain 之前在 SitePoint 上写了一篇关于 Reddit 错误信息的文章。记住:用户并不愚蠢，他们只是普通人。就像你和我一样，他们也会犯错，而且他们经常很匆忙。考虑到不可避免会出现的人为错误。

## 10.谨慎使用额外的问题

对于需要更高安全级别的应用程序(如互联网银行站点),包含一个额外的问题来验证 web 用户已成为一种流行的方法。

但是，必须提出这样一个问题:这种额外级别的身份验证是否真的有必要——也许可以通过另一种方式(使用需要额外的服务器端开发的方法)来获得对用户真实性的这种程度的信任。

一些示例包括 SMS 代码确认、智能令牌和智能卡确认(用于外联网连接)、实施更严格的密码策略，或者仅当从不同的计算机到用户指定的机器访问帐户时才激活额外的问题。

如果您必须向用户询问其他问题，请考虑使用可通过键盘访问的问题/答案格式(即，不仅仅依靠鼠标)。

这让我想到了验证码。对验证码系统有个人仇恨的人不在少数。如果你必须使用验证码，尽你所能使它尽可能容易获取。验证码是规则的例外，因为它们最好与用户名和密码字段位于同一页面。

## 11.降低你的页面权重

您的 web 应用程序登录页面真的需要所有这些按钮和图形吗？想想看——你的登录页面的核心功能是作为到主站点的过渡。你的用户想要使用这个页面，然后尽快离开它。除了导航和品牌，添加任何东西都会降低用户浏览页面的速度。

记住，不是每个人都使用 T1 连接上网。如果一个页面加载时间太长，用户会开始问这样的问题“我应该以后再做吗？”或者“这到底值不值得做？”如果用户有过几次类似的经历，那么她会开始寻找其他网站来代替访问。

降低页面权重通常很容易做到(只是不要告诉营销人员！)查看页面，并确定哪些内容可以归类为网站上使用页面的核心，并仍然使页面看起来像它所属的那样。移除其余部分——有时这甚至可能包括扩展导航系统的一部分。当然，如果你的登录过程使用了 JavaScript 弹出窗口，这个建议就无关紧要了。

## 12.关于 OpenID 的一句话

等等，这个数字 12 是怎么回事？好吧，孩子们，就当是假期的圣诞奖金吧。不过，严肃地说，如果不讨论一下 OpenID，登录主题就不完整。毫无疑问，在阅读这个列表时，你们中的许多人都在想，使用 OpenID 可以解决很多这样的问题。

现在关于 OpenID 的辩论离题了，但基本上我的观点是 OpenID 对于非技术社区来说太难使用了。

我说出来了。它出来了。咻，感觉好多了！

使用 URL 登录的模式与大多数人所经历的预期行为相去甚远。如果有选择的话，大多数人会回到提供用户名/密码对的传统方法——即使这意味着他们会有更多的登录。

这将随着时间的推移而改变。我希望有一天 OpenID 真的会成为主流。但是在一些大玩家加入进来并允许可互换的 OpenID 使用之前，我建议不要让 OpenID 成为用户访问 web 应用程序的唯一方式。

## 保持简单

改善登录体验可以归结为简化用户的整个过程。当然，实现这一目标意味着在 web 应用程序的设计规范、实现和开发方面需要做更多的工作。但是建立网站到底是为了什么呢？设计/开发团队的汗水和泪水，还是客户的满意？

在一天结束的时候，买单的是顾客。
