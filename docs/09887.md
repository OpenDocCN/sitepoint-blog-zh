# 最好测量一下网络的温度——它正随着另一场“炎症”而下降

> 原文:[https://www . site point . com/better-take-the-web-temperature-it-down-with-other-itis/](https://www.sitepoint.com/better-take-the-webs-temperature-its-coming-down-with-another-itis/)

在标准革命的黎明，出现了 DIV-itis 现象。

这是善意但幼稚的开发人员，对他们来说，从表格到布局的转换仅仅意味着将每个`<table>`和`<td>`标签换成相应的`<div>`。

许多人甚至谈到将我的网站从表格转换成 divs(T1 ),好像这是唯一的目的；同时完全忽略了*真正的*目的，即将**从视觉转换为*语义*标记**。

然后是列表——一个接一个的列表——随之而来的是一个新的，但又密切相关的疫情，后一种现象是 <dfn>LIST-itis</dfn> 。最初的灵感来自于[给导航列表](http://www.alistapart.com/articles/taminglists/)添加含义的想法，对一些人来说，成为了伪语义标记的神奇万灵药，因为每一种可能的结构都被批准到列表中。老实说，人们几乎什么都没有就建立了整个网站！

## 一点点历史的重复

现在又发生了！

我喜欢浏览有趣网站的源代码(我一直是一个标记观察者)。但是最近我看到了对`<section>`元素的过度使用，就像这个真实世界的例子(为了简单起见，我去掉了各种属性和内联标记):

```
<body>

   <section>
      <section>
         ... 
      </section>
   </section>

   <header>
      <section>
         ... 
      </section>
   </header>

   <section>
      <section>
         <section>
            <section>
               ...
            </section>
         </section>
      </section>
   </section>
```

但是**section 元素不是一个容器**，不应该为了样式或布局的目的而用来包装任意的内容块。

引用[草案规范](http://www.whatwg.org/specs/web-apps/current-work/multipage/sections.html#the-section-element):

> section 元素不是一般的容器元素。当出于样式目的或者为了方便脚本而需要一个元素时，鼓励作者使用 div 元素。一般规则是，只有当元素的内容在文档的大纲中明确列出时，section 元素才是合适的。

所以大多数的`<section>`元素应该是`<div>`。

## 逃离过去

我不怪任何人迷茫。规范草案本身在许多地方含糊不清或自相矛盾；毕竟这是一份草案。

例如，仍然不清楚`<article>`元素的真正用途。它是否环绕链接到文章的部分？它围绕文章的内容吗？它是否环绕了包含文章的整个页面？还是只是为了辛迪加？到底什么是<q>文章</q>？(还有为什么没有`<content>`元素？)

最终，规范将不得不澄清它所有的模糊之处；要么这样，要么我们将最终在一堆普遍接受但未被批准的公约上解决事情的含义(这将是不幸的，特别是因为这正是 HTML5 试图避免的事情)。

与此同时，我担心我们会看到越来越多热情但不成熟的关于 <abbr title="HyperText Markup Language, Version 5">HTML5</abbr> 的实验，滥用其元素来传递混乱或矛盾的语义，或者作为显然是`<div>`工作的物理布局块。

围绕`<div>`在 <abbr title="HyperText Markup Language, Version 5">HTML5</abbr> 中的使用，似乎有一种“对学校来说太酷了”的态度，好像它们被视为过时、不时尚、粗糙，或者它们不再是必要的。毫无疑问，对于现在有特定语义元素的事物，我们将不再需要它们，比如`<header>`和`<footer>`；但是在很长一段时间内，我们仍然需要它们作为中性的、结构化的容器。这种容器在网页和界面构造中是非常必要的，仍然可以弥补 CSS<abbr title="Cascading Style Sheets">和</abbr>在为块和伪块定义可工作的结构化布局属性方面的不足！

## 面向未来的设计

我认为，在新元素的语义没有明确定义的情况下，根本不应该使用它。因为谁是的真正语义*？它们用于访问技术(如屏幕阅读器)、搜索引擎蜘蛛和其他种类的机器人。然而，当甚至那些语义的*作者*都不确定它们的真正含义时，这些机器读者怎么能从元素语义中获得可互操作的含义呢？*

哦好吧。我不指望改变任何人的想法。

所以，对于那些热衷于开拓这一未来标准的人，我只想请你们记住——不要把它视为视觉的或结构的标记，什么才是真正的 ***语义的*** 标记。

并且要小心<dfn>节-它是</dfn>！
