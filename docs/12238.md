# 避免 CSS 攻击和条件样式表的 5 个理由

> 原文:[https://www . site point . com/5-reasons-to-avoid-CSS-hacks-and-conditional-style sheets/](https://www.sitepoint.com/5-reasons-to-avoid-css-hacks-and-conditional-stylesheets/)

与其说 CSS 是一门科学，不如说它是一门艺术。你可以了解选择器的每种组合，并成为每种样式属性的专家，但仍然无法按照你的意愿来布局和设计你的网站。

传统编程背景的开发人员不喜欢 CSS 有几个原因:

1.  学习规则只是游戏的一半——你需要经验来真正理解 CSS。
2.  CSS 可能不合逻辑，需要太多的尝试和错误。
3.  即使浏览器 X 完美地呈现了您的页面，浏览器 Y 也可能会严重失败。

许多网站和应用程序被认为是完整的，但却发现它在某个浏览器中崩溃了。因此，Web 开发人员求助于黑客和有条件的 CSS 来尽快解决问题。

不幸的是，从长远来看，快速修复的开发会导致更多的问题…

**1。CSS 验证可能不行**
有很多工具，比如 Firefox 控制台和 [W3C 验证服务](http://jigsaw.w3.org/css-validator/)，可以帮助开发者发现 CSS 代码中的错误。不幸的是，黑客和条件样式表经常依赖无效的 CSS 来利用已知的浏览器漏洞或应用特定于供应商的属性。如果你的 CSS 已经抛出错误，你可能会错过真正的问题。

**2。你的 CSS 变得更加复杂**
快速修复一般会对你现有的规则集应用进一步的规则。在条件 CSS 的情况下，那些规则也将在一个单独的文件中，并且只在某些条件下适用。虽然像 [Firebug](http://www.getfirebug.com/) 这样的工具可以帮助定位和分析代码，但它们并不是在每个浏览器中都可用。增加的复杂性使得调试和维护更加困难。

**3。你的 CSS 可能不是面向未来的**
黑客依靠浏览器漏洞。不幸的是，该浏览器的下一个版本可能会修复您所依赖的错误。这无疑是 IE7 中 IE6 兼容站点中断的原因之一，例如 IE6 错误地应用了`* html`选择器，但 IE7 中已修复。

如果你的网站已经依赖于 IE6 和 IE7 条件样式表，那么你很可能需要一个 IE8 条件样式表。

**4。它是浏览器检测**
网页应该是设备无关的。虽然我们并不是生活在一个理想的世界，但浏览器检测总是让人感觉有点脏。在 JavaScript 和服务器端代码中肯定是避免的，那么为什么要在 CSS 中使用它呢？

**5。很少有需要它们的时候**
一个有经验的程序员使用标准的 CSS 不能创造出一个在所有流行的浏览器(是的，包括 IE6)中都能工作的设计是不寻常的。微小的渲染差异总是不可避免的，但是 CSS 代码可以保持整洁，对所有浏览器都有效。

最终，黑客和有条件的 CSS 会导致糟糕的开发实践和无法诊断或理解真正的问题。关键是尽早测试并经常测试:在项目开始时发现问题，这样你就不需要在项目结束时求助于快速修复。
