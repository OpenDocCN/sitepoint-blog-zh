路易斯:大家好，欢迎来到 Sitepoint 播客的另一集。今天在节目中，我很高兴和我一起的是杰里米·基思先生，他来自阳光明媚的英国。你好杰里米。

杰瑞米:优秀。听起来不错。

杰里米:是的，我想就是这么回事。现在你可以指着某人说，“就像这样。”我甚至注意到了客户，他们现在找到我们，指着《波士顿环球报》说，“他们是怎么做到的？”“我们能拥有它吗？”这正是千禧年开始时网络标准大变革的情况。

路易斯:你已经在野外见过这个了。客户是否立即意识到这种方式比传统的移动独立网站或应用程序更有吸引力？

Jeremy: 我认为它对客户有说服力的地方是那些试图为不同平台开发独立应用的客户。可能两三年前他们开发了一个 iOS 应用，然后在过去的 18 个月里，哦，我们需要开发一个 Android 应用。后来，他们意识到，哦，我们需要开发 Windows phone。那些客户现在说，“好吧，够了。这越来越不可收拾了，这种情况会持续到什么时候？”他们看着自己的数字，意识到这是无法扩展的。正是这些客户对响应式设计及其承诺非常感兴趣。尽管响应式设计在不制作网站的人中间也成为了一个时髦、时髦的术语，但这也有一点危险。当它被开发人员和设计人员使用时，这很好。我们都明白响应式设计的含义。

Ethan Marcotte 在创造这个术语时非常非常清楚，它是一套特定的技术。这不是一种空洞的人生观，也不是某种禅宗哲学。是特定的技术。现在，它在市场营销和咨询领域真正起飞了，它像 HTML5 或 CSS3 一样被当作术语来使用，却没有真正理解这些术语的含义。我必须判断客户什么时候来，他们在谈论响应式设计，我判断他们知道这是什么吗？或者他们真的需要多谈一点来理解它的意思吗？

我对 HTML5 和 CSS3 之类的术语肯定有过这样的感觉。他们到处乱说这些术语，但当我对他们进行更多的提问时，我意识到，哦，实际上他们只是在使用这些词。他们实际上不知道 HTML5 是什么，CSS3 是什么。他们只知道这是时髦的术语，他们把它等同于现代。这里有一个问题，响应式设计已经成为这些术语之一。这也是成功的标志，当你的术语被营销人员劫持，并开始在某些对话中失去意义，因为人们只是随便说说，这实际上是一个很好的迹象，表明你已经被主流采用。

路易斯:就在野外的实际使用而言，我想这同时也是你想去的地方和你不想去的地方，因为你必须教育客户。

杰里米:正是。这是一把双刃剑。一方面，你想让每个人都知道这个术语，你想把这个词传播出去，但是要小心你的愿望。因为当人们实际上开始用这个术语来指天底下的一切时，你就失去了最初对这个术语的清晰理解。

路易斯:我认为对于我们大多数在网页设计领域的人来说，我们真的很高兴看到这种趋势越来越强，尤其是考虑到设备尺寸和分辨率的激增，甚至在过去的一年里。事情变得有点疯狂，试图用独立设计的、特殊用途的网站来支持所有这些网站将很快变得不可维护。

杰里米:正是。规模问题是关键。这种设备的激增很好，我认为它实际上强调了已经存在的东西。甚至在几年前，当你制作一个网站时，你真的只考虑桌面网络浏览器，它仍然是浏览器的一个变种。你有网络浏览器、Opera、Chrome、Safari 和所有这些浏览器。web 开发人员的口头禅是，“它不必在每个浏览器中都看起来一样。”实际上很难推销这个想法，因为从表面上看，这些浏览器似乎没有什么不同。背景看起来没什么不同。我有一台电脑，上面有一个浏览器，为什么我的网站在这些浏览器中看起来不一样？

随着设备的爆炸式增长，这些设备上的浏览器具有不同的大小和功能，现在对人们说，“看。你不能指望所有这些不同的设备看起来或工作起来都一样。”因为有了这些设备，告诉人们“不必在每个浏览器上看起来都一样”是一个更容易理解的概念。一直都是这样，只是现在每个人都更容易理解，好吧，现在我明白了。该网站没有一个特定的外观，一个特定的感觉。一个网站在不同的浏览器和不同的设备上会有不同的外观和感觉。这个概念现在被更好的理解，更广泛的理解。

**路易:**对。我想这也是一把双刃剑，因为现在作为开发人员，我们的任务是试图在 100 或 200 种不同的屏幕尺寸和分辨率上维护一个网站，而以前可能是 12 种。

Jeremy: 当然，但是在不同尺寸和不同设备上进行测试和针对设备进行优化是有区别的。如果我们真的必须为 200、300 种不同的设备进行优化，那么是的，我们的生活将会非常悲惨。更重要的是确保这些设备上的东西不会损坏。只要每个人都有这样的想法，即所有这些设备看起来不一定都一样，那么它实际上并没有那么糟糕。这种设备的泛滥并不是世界末日。这是值得欢迎的事情。

是的，我在你的博客上注意到你和其他几个人建立了一个公共测试实验室？

杰里米:是的。所以在 ClearLeft，在过去一年左右的时间里，我们一直在一点一点地收集设备。这永远不够，你永远不会有足够的设备，但我们正在努力做到最好。我会去这些不可靠的二手商店，买一些从卡车后面掉下来的便宜的旧设备。我们一点一点地把一些设备放在一张桌子上，这样我们就可以测试我们在这些设备上构建的东西。真的，不够。我意识到这有点浪费。它们一天中的大部分时间都放在桌子上，当我不主动测试某个东西时，它们就不会被使用。

我刚刚在博客上提到了。我说，“看，我们在 ClearLeft 一直有一个开放的政策。如果你要来，如果你就在附近，就过来，用我们的 WiFi，坐在沙发上，喝杯茶。”诸如此类的事情。我只是延伸了一下说，“请随意使用这些设备。”我没想到的是，真的，真的很好，就在我说的同一天，布莱顿的所有其他人，像雷米·夏普，来自[ministryoftype.co.uk](http://ministryoftype.co.uk/)的埃吉尔，他们开始在推特上插话说，“哦，我有这个设备就放在我的桌子上，没有被使用”或者，“我有这个设备在抽屉里。你要我把它放在你的办公室吗？”

我说是的。“设备越多越好。”在大约 24、48 小时内，设备的数量翻了一番。这不是计划，这不是我宣布的想法，带着设备来。这只是一个非常非常好的副作用。现在我已经收集了相当多的设备。尽管如此，这永远不够，你永远不会有足够的设备，但这是一个相当不错的数量。我不得不出去买很多电源板和 USB 线来给所有设备供电，给它们充电。现在我觉得还好，这个测试实验室相当不错，但现在它不再是我们的设备，而是每个人的设备。任何想过来试试的人。

我之所以写了这么多博客，不仅仅是为了让布莱顿的每个人都知道你可以通过 Clear Left 办公室使用我们的设备，也是为了让其他地方的人知道，无论你碰巧在哪里。我敢打赌，有一个 web 开发人员社区，我敢打赌他们都和你有同样的感觉，那就是我没有足够的设备来测试。相信我，每个人都这么想。如果每个人都拿出他们的资源，你会有一个相当好的公共图书馆。我没有做的一件事是太担心法律方面或保险之类的东西。我只是决定如果发生了什么事，我会处理的。如果我们办公室里的某个设备被盗或与保修有关，但不属于我们，我会在问题出现时处理它，而不是试图预测所有可能出错的事情。到目前为止还没出什么问题，如果真的出了问题，我会处理的。我认为这也是一种重要的态度，因为这种打开你的大门，拥有一个公共测试实验室的想法，并不是一个新的想法，但对其他人来说很难起步。那是因为他们试图预测所有可能出错的情况。简单地忽略这个问题，它就消失了。

路易:是啊，要是我们所有的问题都是这样就好了。我认为这是一个奇妙的想法，我期待着看到同样的事情在其他城市发生。你在 E3 看到了吗？我想是的，微软宣布他们将在 Xbox 上推出 IE 的一个版本。你可能还需要为你的测试套件准备一个 Xbox。

杰里米:我们别无选择，真的。我们将不得不为办公室买一个 Xbox。

路易:这是现在的业务需求。

杰里米:正是。这是一笔开支。

路易:太棒了。我想和你谈的另一件事是，很明显，在过去的五六周里，有很多关于响应式设计的讨论，因为整个对话都是关于响应式图像的。这是一直在进行的事情，因为当 Ethan 第一次提出它的时候，它是响应式设计的基石之一，你可以有流动的图像。然后，当人们开始考虑带宽时，我们开始寻找其他更聪明的方法，使用 JavaScript 加载图像，并提出所有这些技术。我认为长远的观点是总要有符合规格的东西。过去几周，这方面有了一些进展，尽管并不令所有人满意。你对事件发生的时间顺序做了很好的分析。对于那些没有密切关注的听众，你能解释一下目前的情况以及我们是如何走到这一步的吗？

杰里米:好的。这个问题主要是带宽问题。因为你不想在小屏幕设备上显示大图像。太浪费了。首先，在响应式设计中，这种情况的发生并不是因为人们利用现有的桌面网站并使它们具有响应性。他们添加了取消布局样式、线性化布局和缩小 CSS 中图像的样式。实际上你还在加载所有这些图像。从那时起，我认为这种态度已经改变了，人们开始建立更多的移动第一，因为它被称为，响应设计，你从一个线性化的布局开始，你从小的图像开始。然后，您构建更大的桌面屏幕视图。

首先，这本身已经是一个进步。如果你从较小的图像开始，现在你的问题不是我如何检测某人是否在移动设备上，并给他们一个较小的图像。现在你的问题很简单，我如何检测某人是否在桌面设备上，并给他们一个更大的图像？直接来说，这是一个更小的问题空间。还有一个事实是，如果有任何事情出错，用户默认会得到一个较小的图像，而不是较大的图像。这很好，有一堆技术可以帮助你确定，这是一个大屏幕设备吗？如你所说，现在解决这个问题本质上涉及到黑客。使用 JavaScript，使用服务器端检测。使用 cookie，也许可以用 JavaScript 设置一个 cookie，然后在服务器上读取。然后，根据 cookie 的内容，您可以提供较小或较大的图像。

很多非常非常聪明的解决方案。现在就是这种情况。一般来说，当你看到开发人员用 JavaScript 做一些事情，绕过这个问题，这是一个好的迹象，这是需要在标准中的东西。它需要从编程式转变为声明式。要么它需要从围绕 JavaScript 的黑客行为转向 CSS，要么转向 HTML。你可以在过去看到这样的例子，比如在 HTML5 中，很多输入类型是我们过去不得不在 JavaScript 中处理的。有一个日期选择器，在输入字段中有占位符文本。所有这些事情我们过去都可以做。我们不得不使用 JavaScript。但是因为很多人都用 JavaScript 来做，这是一个信号，嘿，这需要在规范中声明，并且跨浏览器标准化。你看到它从 JavaScript 变成了 HTML。这里也是一样，我们现在用 JavaScript 解决这些问题，而且比最初出现时要困难得多。它真的需要进入声明性标记。它需要是标准的，并且需要在不同的浏览器中以相同的方式工作。这些非常非常聪明的黑客所遇到的一个问题是，浏览器开始优化他们下载网页的方式，通过提前查看源代码来查看是否有任何图像元素，并预先下载源代码属性中的内容。如果您正在做一些聪明的 JavaScript 来替换 source 属性中的内容，那就太晚了。浏览器已经开始下载初始图像。

**路易:**对。

你不能责怪浏览器这样做，因为就速度和性能而言，这实际上是一件非常聪明的事情，这是浏览器所关心的。光说“别这样，你在破坏我们聪明的剧本”是不够的我们一直在试图用 JavaScript 解决这个问题。我们需要把它转变成像 HTML 这样的声明性标准。我认为从长远来看，我认为这确实需要一个很大的视野。你会在黑客和 JavaScript 中发现一些东西。它变成了声明性的或者 HTML 或者 CSS。也许从长远来看，它实际上可以进入格式本身。因为这是一个需要响应的图像。实际的图像格式。有些人也一直在谈论这个问题。你有一个图片，它可以根据浏览器的功能，内容协商，以不同的文件大小提供服务。

现在，我们正打算将 JavaScript 中的流行模式移植到 HTML 的声明性标准中。很多非常聪明的人都在从事这项工作，很多人都想出了这些 JavaScript 技巧。像斯科特·吉尔，马特·威尔科克斯，所有这些人。他们去了 WHATWG，他们不习惯花太多时间在邮件列表上，开始谈论这些东西。WHATWG 的默认态度是，这不是问题。你必须证明这对我是个问题。好的。我很确定这确实是个问题，因为所有的开发者都在试图解决这个问题。很明显，这是需要解决的问题。最终 Hixie 确信了这一点。事情是这样的，邮件列表上的某个人，他实际上并不是 WHATWG 的长期贡献成员，他说了类似这样的话，“哦，你应该去 W3C 建立一个社区组织。”W3C 社区组织提出了这个新的想法，但是人们基本上可以围绕这些东西进行头脑风暴，但是这与任何官方规范都没有关系。它只是一个人们聚集和走到一起的地方。

问题是，来过 WHATWG 的人认为这个建议是 WHATWG 作为一个团体的建议，但事实并非如此。只有一个人这么说。WHATWG 没有太多的层级。你，我，我们都可以成为 WHATWG 的成员，只要给 WHATWG 列表写一封邮件。这被理解为，“哦，如果你去社区小组解决这个问题，然后回来找我们，这就是解决这个问题的正确方法。”这些人组成了一个社区团体，W3C 上的 Responsive Images 社区团体，在那里他们真正开始了大量的工作。他们做了很多伟大的工作，非常好的工作。首先，弄清楚用例，因为事实证明这并不像你第一次想的那样简单，只是简单地用较小的图像替换较大的图像。有时图像中显示的内容可能会有所不同。它可能是大图的一部分。
