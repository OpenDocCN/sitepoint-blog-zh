##### 5.选择一个伟大的域名

当你建立你的网站的移动版本时，你需要决定用户访问它的域，并想出如何宣传它的存在。有几个选择移动域的选项。让我们逐一探究。

***1。完全使用单独的域(例如`www.example-mobile.com`)。***

对于我工作过的任何网站来说，创建一个独立的域名从来都不是必须的。一个单独的域名会损害你网站的整体品牌，会让用户困惑——有多少人会真正记住你传统网站和手机版网站的网址？你应该在广告中宣传哪个网站？这种方法可能会导致麻烦，因此不推荐使用。

***2。使用子域(如`m.example.com`)。***

创建一个子域可能是最流行的选择。使用子域(比如 mobile，或者简称 m)可以让你的移动网站成为你品牌的一部分，而不会造成混淆。显然，如果适合您的客户，您可能希望将术语“移动”本地化。如果在你的域名前加上“m”意味着可怕的事情，你可以考虑把移动元素放在你的 URL 的末尾(例如`http://example.com/mobile/`)。

不管怎样，你应该假设人们会把你的移动网站地址搞错。通过设置几个不同的公共子域——m、mobile 和其他——并将它们全部重定向到一个移动网站来预测这种可能性，就像您应该将传统网站的 ww 和 wwww 之类的错别字重定向到 www 一样。

***3。使用. mobi 顶级域名(如`example.mobi`)。***

自从. mobi TLD 出现以来，注册它的问题一直是 web 开发人员激烈争论的原因。宣布了 mobi 域名。就我个人而言，我从未在我参与的任何项目中使用过. mobi 域名，但这一决定可能取决于个人偏好。如果你有几个小时可以消磨，一个[快速搜索将为你提供大量关于使用. mobi 域名利弊的信息](http://www.google.com/search?q=should+i+use+a+.mobi+domain)。

***4。什么都不做，让服务器根据用户代理发送最佳页面。*T3】**

从开发人员的角度来看，执行用户代理检测可能是最有趣的选择。从用户的角度来看，这也是最优雅的，但不幸的是，这是最容易出现问题的方法。

它是这样工作的:当一个用户访问 example.com 网站时，服务器会查看访问者的用户代理，并试图检测访问者使用的是移动设备还是桌面浏览器。给定该信息，服务器将为该用户代理发送最合适的页面。这种方法的美妙之处在于，它允许您使用单个 URL，任何人都可以在任何设备上使用该 URL。无论是在电脑上还是在移动设备上，所有用户都会神奇地看到对他们来说最佳的网站版本。如果做得好，这将是一个很好的解决方案。但如果做得不好，可能会是一场灾难。

我们讨论的第二个选项(使用子域)还有一个优点:使用高端设备浏览的用户可能首先访问移动网站，然后意识到这个网站不包含他或她想要的信息。然后，用户可以选择浏览传统网站来查找信息。用户体验可能没有得到优化，但是用户应该仍然能够找到想要的信息。

然而，使用用户代理检测，只有一个 URL，因此只有一个版本的网站可以返回到该设备。客户不能选择要查看的版本，即使他或她能够查看该站点。如果您计划执行用户代理检测，您将需要允许例外，以便设备可以覆盖检测过程，以便查看网站的特定版本。如你所见，如果你走这条路，事情会变得非常混乱，非常快。

当你决定一个移动网站的域名时，我共事过的同事和公司总是选择第二个选项。创建一个子域是最容易设置的选项(你已经拥有了这个域)，也是最便宜的选项(不需要注册。mobi)，这意味着您不必花费数小时来调整服务器(并可能扰乱正常的流量)。最终结果是一个专门为移动设备构建的沙箱子域。如果流量不是你所期望的(不管是好是坏)，子域可以让你在不影响传统网站的情况下灵活地扩大或缩小移动网站。

##### 6.验证您的标记

桌面浏览器可以相当宽容。这里或那里的一些放错位置的 HTML 标签，通常会被动态地修复，以便正确地呈现您的页面。然而，为了执行这种错误处理而内置到桌面浏览器中的“智能”相当于更多的代码，这意味着更大的安装和更强的处理能力。

另一方面，移动浏览器就不那么宽容了。在移动设备上运行的浏览器一般不会有 2 GHz 的处理器和 100MB 的磁盘空间。因此，您必须一次又一次地检查、验证、再检查您的标记。

移动网站的大部分检查和验证可以通过普通的桌面浏览器来完成。例如，如果您使用 XHTML 进行开发，您可以重用所有用于验证传统站点的工具:

*   W3C 验证器可以用来发现简单的标记错误——甚至有一个特定于移动设备的版本。
*   Ready.mobi 网站列出了一些额外的手机特定检查。

你会注意到，在这一点上，我们还没有讨论如何检查移动网站是否能在移动设备上正常工作。一旦你收到了几个验证者的绿灯，你就需要得到几个不同类型的手机来执行一些实际的测试。

##### 7.测试测试测试！

在台式电脑上用网络浏览器测试你的网站只能模拟移动体验。移动设备使用中有许多元素无法通过这种方式准确复制。例如，移动运营商可能会限制数据包的大小，使其小于您的预期，因此甚至不会发送您的网页或图像！此外，内容 mime 类型可能是浏览器之间的一个问题——您是否使用正确的文本/html 或应用程序/xml+xhtml 来提供页面？手机可以显示哪种图像格式？

由于内存和磁盘占用空间小，移动浏览器不如桌面浏览器健壮，所以最好的建议是测试，测试，测试！当然，并不是每个人都有能力在市场上的每部手机上进行测试，但还是有其他选择的:

***1。仿真器***

有大量的[在线仿真器](http://mtld.mobi/emulator.php)和[离线仿真器](http://developer.openwave.com/dvl/tools_and_sdk/phone_simulator/)可以让你快速查看图像的上下文和总体布局，但它们不是真正的设备，因此它们有自己的怪癖和不同之处。这些工具可以作为找到常见问题的第一步。此外，它们可以免费使用，所以没有理由不在模拟器中查看，但你不能称在模拟器上查看的站点为“测试过的”。

***2。*租房时间**

租用时间是另一种选择。有些服务可以让你在多部手机上实时上传或观看内容。您可以远程控制不同手机的功能。这项服务确实要花钱，但仍比购买大量不同的手机便宜。对于一个基本的移动网站，这项服务可能是不需要的。

***3。买一把手机***

购买一个小的子集或代表性的手机是可能的。如果你计划做更多的移动开发，预先花点钱真的会有帮助。你可能需要购买 5 或 6 部代表主要品牌和类型的手机。

你需要某种 Windows 智能手机——我们在 HTC Mogul PPS-6800 上使用 Windows CE 上的 Internet Explorer 进行了测试，它给出了所有 Windows Mobile 设备的代表性覆盖范围。你还需要一部诺基亚型号的手机。我们在运行 Symbian Series 6 操作系统、内置服务和可选 Opera 浏览器的诺基亚 6600 上进行测试。你需要一部索尼爱立信手机。我们用的型号是一款 T630，涵盖了索尼的互联网服务。最后，你应该测试在你的观众中最受欢迎的模型。你不需要这些确切的手机，但这些类型的设备将让你很快看到手机浏览器的细微差别，并涵盖大多数潜在客户。

***4。问问你的朋友***

最后，你可以简单地问问你的朋友。我们都知道，在 east 5 或 6 个同事或同事会很乐意把他们的电话借给我们几分钟。一旦你用在线模拟器和验证器测试了你的网站，在借来的手机上测试一下。预测试应该已经解决了除了每部手机特有的问题之外的所有问题。记笔记，修改，一遍又一遍地测试，直到你满意为止。

根据你的预算，其中一个选项会适合你。花一点时间和金钱真的可以走很长的路——特别是如果这是你的第一个手机网站。获得正确的测试套件将允许您发现常见问题，同时帮助您构建一个工作 HTML、CSS 和 JavaScript 代码库，这些代码已知是经过测试的，并且可以在未来的项目中重用。

##### 结论

在这篇文章中，我们探讨了构建移动网站时应该考虑的七个关键因素。一路上，我证明了移动 web 开发的入门并不像您曾经认为的那样可怕。通过一些背景阅读，你现有的网络开发知识和大量的测试，你有可能为自己建立一个非常有用的、成功的移动网站。

我期待着下次我遇到交通堵塞时看到你努力的成果！
