## <cite>标准快速设计原型</cite>，Eric Meyer

在 Eric 的第一天演讲中，我们错过了一个机会，可以看到他将一些现实生活中的 CSS 代码转化为提交代码，而本次会议弥补了这一点。他演示了如何在与客户的会议中做出页面布局和颜色选择等设计决策，方法是通过入侵 CSS 代码来生成设计想法的即时预览。为了做到这一点，他用旧的 Netscape DevEdge 网站的设计带领观众经历了这一过程，征求关于什么应该移动到哪里的想法，并通过一些快速的 CSS 编辑在飞行中模仿它。

如果你对自己的 CSS 技能有足够的信心，可以在真实的客户会议中做到这一点，Eric 从经验中学到了一些程序上的小技巧:

*   使用 CSS 定位——没有花哨的浮动方法，这些方法可能会非常临时——来布局你的原型。
*   选择一个原型平台——一个显示你的原型的浏览器——并坚持下去。
*   将任何跨浏览器测试或者跨浏览器问题的想法排除在原型开发过程之外。

## <cite>无障碍设计:超越基础</cite>，德里克·费瑟斯通

德里克向我们展示了更上一层楼的可访问性，解决了不明显的问题，这样网站就可以很好地为残疾人服务，而不会牺牲设计师们拼命追求的视觉吸引力。

在许多情况下，解决方案确实是“了解你的 CSS”。如果你知道你在做什么，一个必填字段图标或验证错误消息必须出现在输入字段之前以获得可访问性是没有关系的——你可以把它放在使用 CSS 的可视化浏览器的那个字段的右边。但是 CSS 不是万能的解决方案。

有时归结为深思熟虑的标记:为什么不把所有必填字段放在“必填”字段集中，把所有可选字段放在“可选”字段集中？有时候你真的必须使用屏幕阅读器来想出非常简单的想法，比如向#results 片段标识符提交一个表单，它直接指向结果页面上的搜索结果(#details)的链接，从而允许屏幕阅读器跳过页面顶部的“再次搜索”字段。

德里克展示了一个完全无障碍的 DHTML 填字游戏，这是他的致命一击。为了看看这个启用和未启用样式表的例子，我们有必要从本次会议的幻灯片和源代码中寻找答案。

## <cite>了解 Ajax:一窥幕后真相</cite>，蒂姆·卢卡斯

Tim 对 AJAX(尽管他为替代方案辩护，SitePoint 仍然全部大写)和异步 JavaScript 和 XML 做了相当详细和实用的介绍。Cameron Adams 不久前在 SitePoint 上用 <cite>[AJAX:与远程脚本的可用交互性](https://www.sitepoint.com/article/remote-scripting-ajax)</cite> 很好地解决了这个问题，所以我不会深入讨论这项技术的基础知识。然而，蒂姆确实提到了几个特别有用的金块:

*   避免 AJAX 应用程序中的跨站点脚本(XSS)漏洞通常很简单:通过要求 POST 请求来保护破坏性/不可逆的操作，如注销用户和删除记录，并通过要求将有效的会话 id 作为链接 URL 中的参数(而不仅仅是 cookie)来检查请求的凭证。).
*   所有现在受到广泛关注的 AJAX 应用程序都是为了要求 AJAX 工作而从头开始设计的(即打开 JavaScript、ActiveX)。但是最佳实践是构建一个“哑”站点，然后用 AJAX 以一种优雅的可降解方式增强它。

在展示了一些使用中的 AJAX 的经典例子(动态地从服务器更新结果，拖放列表项以记录它们在服务器上的新顺序，等等)之后。)，Tim 简单介绍了一下目前的一些工具包: [DOJO](http://dojotoolkit.org/) (处理大问题，4MB 的 JavaScript 预压缩，有点吓人) [Rico](http://openrico.org/) (没有 Safari 支持，所以不值得你花时间) [SAJAX](http://www.modernmethod.com/sajax/) (很好地解决了小问题，目前以 PHP 为中心，但正在增长)，以及[Prototype](http://prototype.conio.net/)/[script . aculo . us](http://script.aculo.us/)(与 Ruby on Rails 捆绑在一起，生成性感的 JavaScript 代码)

他强调说，这些工具包都没有解决 AJAX 的可访问性问题，但这是创新者需要解决的问题。如果你现在想使用 AJAX 可访问性问题并没有停止[谷歌地图](http://maps.google.com/)！—工具包可能是最好的方法。

## <cite>标题属性:有什么用途？</cite>史蒂夫·福克纳

我有幸介绍的两次会议中的第一次，Steve 谈到了围绕一个属性的令人惊讶的复杂的可访问性问题，具有讽刺意味的是，引入这个属性是为了提高可访问性:title 属性。

这个属性的一些限制:当显示为工具提示时，Mozilla 浏览器会截断长标题值，工具提示在大多数浏览器中只显示 5 秒钟，并且在大多数情况下需要鼠标来访问 title 属性中的内容。

特别是，现代的屏幕阅读器(例如 [JAWS](http://www.freedomscientific.com/fs_products/software_jaws.asp) )通常会默认忽略文本链接上的标题(通过调整让您听到标题属性值*而不是*链接文本)。然而，他们似乎非常可靠地读取文本字段上的 title 属性。以最近[的一篇文章](http://www.alistapart.com/articles/pdf_accessibility)为例，这篇文章大量使用了标题属性及其内联链接，他演示了如果关键信息放在标题属性中，链接会变得令人困惑或完全误导。

史蒂夫从测试中得出了一些实用的建议:

*   首字母缩略词和缩写词的标题:使用它们，但是在第一次提及时，要在正文中包含扩展形式，以方便那些无法访问标题内容的人
*   为链接添加注释:仅将标题用于非关键信息，并在附加内容之前包含链接文本的副本。

对于本应提高可访问性的东西来说，这并不太好，对吗？为了实现这个属性的一些承诺，Steve 建议使用服务器端 XSLT 转换，将标题属性转换为带注释元素后面的内联内容，以便用户能够以可访问的形式从这些信息中受益。

## 微格式:进化网络

简而言之，[微格式](http://microformats.org/)是简单的标准，使用 HTML 的可扩展位 meta 标签和 class、rel 和 rev 属性——为当今的 Web 添加有意义的功能，而不必等待新标准的发展来支持该功能。Tantek 在本次会议中做了出色的工作，他解释了微格式的动机以及关于微格式的基本设计决策，以使它们在很短的时间内变得有用和成功。

微格式的非显而易见的动机是用户应该控制他们自己的数据。例如，把你的电子邮件从一个电子邮件程序转移到另一个程序不应该有今天这样的障碍。简单、集中，最重要的是*开放*标准可以让用户在给定的应用/服务不符合标准的情况下，带着他们的数据跳槽，并把这些数据带到更好的服务中。

对用户来说很棒，但是为什么会有公司想要这样呢？到目前为止，这个游戏一直是通过以专有格式存储数据，并设置障碍将数据转移到其他地方，从而将用户锁定在一个软件上。支持开放标准的理由是，知道一家公司会让你更容易将业务转移到其他地方，会让你更容易相信这家公司会提供你的数据。此外，潜在地使用与相关服务相同的数据格式的好处意味着您的应用程序可以从其他人的工作中受益，这些工作将普及和增强该数据格式的潜力。

Tantek 然后带我们回顾了微格式的简史，按照它们出现的顺序展示了今天可用的微格式。特别吸引人的是 [hCard](http://microformats.org/wiki/hcard) 和 [hCalendar](http://microformats.org/wiki/hcalendar) 格式，这两种格式允许您将联系人和事件数据以标准化格式嵌入到您的网页中，vCard 和 iCalendar 兼容的应用程序可以通过一个简单的 bookmarklet 读取这些数据。

微格式在野外被采用方面似乎正一个接一个地成功(例如，现在有 40，000 多名雅芳代表在网上有 hCard 联系方式——这是历史上第一个被女性比男性更快采用的个人数据格式？)，Tantek 概述了一些为微格式做出的设计决策，他认为这些决策推动了微格式的采用:

*   微格式是可样式化的，在网络上是可见的，而不是隐藏在只有搜索引擎才能看到的地方。
*   微格式今天仍然有效。
*   微格式的实现成本很低，甚至没有成本。
*   微格式可以手工编码。
*   微格式可以嵌入到现有类型的内容中，而不会干扰它们。

最重要的是，微格式是 Web 2.0 文化不可或缺的一部分，Web 2.0 文化释放数据，供可互操作的服务访问和使用，而无需将数据锁定在一个集中的存储库中。

## <cite>设计下一个网络:为 Web 2.0 翻译的以用户为中心的设计原则</cite>，杰夫·维恩

这是会议的强制性 Web 2.0 会议，谢天谢地，Jeff 将所有的能量带到了这个新运动/技术/架构/无论什么的舞台上，同时驱散了所有的宣传。Jeff 提出了几个对 Web 2.0 至关重要的想法，然后指出，对于开发人员来说，Web 2.0 意味着使用体现这些想法的新平台来解决老问题。

所以，是的，播客实际上只是流媒体音频的一个新名字，但它是一个开放的标准平台，它产生了各种各样的工具、社区和可能性，这些都是以前从未有过的——就像维基和博客到内容管理系统，以及预期导航(AJAX *à la* Google Maps)到以前的网络地图技术。

Web 2.0 意味着将你的用户视为构建产品价值的参与者。Web 2.0 意味着开放和放弃对数据(您的和您的用户的)的控制。Web 2.0 意味着把简单的东西送给社区，对难的东西收费。Web 2.0 令人兴奋，我迫不及待地想在其中找到自己的位置。
