## 回到世界年轻的时候

回到开源的早期——那时恐龙还在地球上游荡，拉斯姆斯还是个年轻人——我们谈论过两种类型的开源项目:不需要任何成本的项目，以及给你重新发布和修改代码的自由的项目。我们使用的类比是，如果项目不需要任何成本，那么项目就是“免费的”,如果项目给了你与他人分享的自由，那么项目就是“免费的”。

![vector illustration of beer](../Images/11bde8b7f853167f264f15b56fb531f3.png)

那时候我们就是这样向麻瓜解释开源的，到处都是麻瓜。应该注意的是，这还是在学习 PHP 的第一步是在服务器上编译 Linux 内核的时候。

## 现代

几十年后的今天，开源已经成为一个生态系统，而不是你和五个朋友参加的用户组，或者你订阅的杂志。问题是，我们大多数人已经不再谈论不同类型的开源，我们只是假设两者都有。在我们这个世界角落的大多数项目——PHP——实际上都是。PHP 许可证 BSD 许可证的衍生物——非常开放，给你自由，几乎没有责任。其他项目使用 GPL、MIT、Apache 和其他许可证。每个开发人员或团队都有权为他们的代码选择他们认为最合适的许可证。如果你使用他们的代码，你有责任遵守他们的许可证的限制和责任。

大多数使用开源的开发者理解我在最后一段所说的一切。我们明白了。如果是 GPL，我们要把代码出货，如果是 MIT，我们要把版权和名字留在上面，等等。我们中的大多数人都理解许可证的基本分支，即使我们没有深入研究黑暗的细节。但是使用开源代码带来了另一种责任，另一种代价。

我的朋友 Elizabeth Smith 在 2017 年[ZendCon](http://www.zendcon.com/)上的演讲中说得好:“如果你使用开源软件，但你不为开源软件做贡献，开源软件将会消亡。”

![Open Source Logo, made dead](../Images/39edbd2309a37da9105bb33e9554b268.png)

这就是开源的代价:回馈。这是一些开发者和大多数依靠开源谋生或获利的公司忘记支付的代价。回馈并不是你回馈给开源的小费，回馈是一种隐含的责任。

## 如何给予

"好吧，卡尔叔叔，我们明白了，我们该付给谁？"

看，如果你这么想，那你就真的不明白。

是的，几乎所有的 PHP 核心开发人员都希望你能给他们几百美元的时间。如果我们中有足够多的人这样做，他们中的一些人可以将他们所有的工作时间奉献给使 PHP 变得更好。

![Fry from Futurama yelling shut up and take my money](../Images/0ee35c3cbc9346f3c27529b7ab07a598.png)

但这不是我们回报的方式。钱是可以赚的。当我说回馈的时候，我的意思是给予一件不能获得的东西，一旦给予，就无法取代。当我说你需要回馈开源时，我指的是付出你的时间。对于公司来说，这意味着给你员工中的每个开发人员时间来回馈他们选择的开源项目(他们的选择，不是你的)。对于个人来说，这意味着参与到自己喜欢的项目中。对我们中的一些人来说，这是写代码，对其他人来说，这是文档。对于一些像我一样的人来说，这意味着写教程，在用户群和会议上发表演讲，尽我们所能进行教育；帮助培养下一代开发人员。

我们都有一个最喜欢的项目。对我来说，它只是碰巧是 PHP，但我第二喜欢的是 Zend Framework 的一个小组件。我不能直接为 PHP 做出贡献——我不是 C 程序员。然而，我可以帮助传播 PHP 很棒的事实。我可以直接贡献我最喜欢的 Zend 框架组件。它是我经常使用的一个部分，所以当我发现它不能做一些事情时，我卷起袖子，编写代码让它做我需要的事情。然后，我将更改发回生产线，以便其他人也能受益。

“等等！等等，倒车！你只在对你有利的时候才捐款，不是吗？”

也许吧？那又怎样？我没说不能自私，我说你必须回馈社会，这样别人才能受益。开源就是“这是我写的，如果你想用，你就可以用。”我们在帮助自己，但也在帮助别人。

## 如果他们不想要我的礼物呢？

我喜欢 WordPress。尽管我给了马特·莫楞威格很多废话——在过去的几年里，我给了他很多——但我尊重他所建立的一切。我几乎所有的项目都使用 WordPress，因为它是一个易于使用的工具，我可以用它完成工作。即便如此，我几次向 WordPress 投稿，我的修改都被完全忽略了。

我在万神殿工作时，工作的一部分就是每天花时间回馈社会。WordPress 是一个非常庞大和复杂的系统，每天都有数百名开发人员在工作。信不信由你，他们并没有坐在那里等着卡尔叔叔来修复漏洞拯救他们。尽管如此，我还是去了 bug 追踪器，找到了几个我觉得我可以修复的，分叉了项目，写了代码，提交了 PRs。

其中一个被忽略了。我不确定为什么有时候 WP 修复非关键 bug 要花很长时间。我提交了 PR 的一个 bug，我也必须提交一张罚单，因为还没有人注意到它。那一个被简单地拒绝了，因为它太琐碎了，他们甚至不想把它通过审查过程。在这个过程中，我“修复”了我在代码中看到的几个问题。IF 语句中缺少花括号等。琐碎的事情但对我很重要。我在我的公关上得到一个礼貌的评论，实际上是说:“请停止”。

![Retro illustration of policeman showing the stop gesture](../Images/ba8cc32d064b52645b87c845fd47a5b3.png)

WordPress 有一个非常严格的变化过程。这么大的系统，被这么多网站使用，必须有一个。我没有阅读他们的指导方针，也没有遵守他们的社区惯例。

WordPress——以及其他你想命名的开源项目——没有义务接受我的 PRs。他们也没有义务接受你的。我不知道有哪一个开源许可证要求一个项目要么接受 PR，要么给出一个不接受的正当理由。只是不在那里。他们可以有理由地拒绝，也可以毫无理由地拒绝。项目不欠你，你欠项目。

也就是说，如果你写代码，如果你为文档做贡献，或者如果你以其他方式为一个项目付出时间，而这个项目决定，不管出于什么原因，你的改变是不必要的或不可取的，那很好，你已经完成了你的部分。你的债务已经部分偿还了。你付出了你的一些时间和才能，这就是对你的全部要求。

## 付出代价

无论你在做什么项目，都要花时间回报。不要让开源在我们这一代消亡。

保存这个伟大的概念；我们帮助建立的这个生态系统让我们可以建造如此多的东西。如果你是开发者，找到自己喜欢的项目，回馈。如果你经营一家公司或一个开发团队，用你的钱给他们时间来回报一个项目。帮助下一代开发者保持开源生态系统的繁荣。
