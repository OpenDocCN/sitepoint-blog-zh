# 反应与角度:一个深入的比较

> 原文：<https://www.sitepoint.com/react-vs-angular/>

**应该选择棱角分明还是反动？每种框架都有很多优点，在它们之间做出选择并不容易。无论你是一个想知道从哪里开始的新人，一个为你的下一个项目挑选框架的自由职业者，还是一个为你的公司规划战略远景的企业级架构师，你都可能从这个主题的教育观点中受益。**

为了节省您的时间，让我先告诉您一些事情:本文不会给出哪个框架更好的明确答案。但数百篇类似标题的文章也不会。我不能告诉你，因为答案取决于各种各样的因素，这些因素使特定的技术或多或少地适合你的环境和用例。

既然我们不能直接回答这个问题，我们就试试别的办法。我们将比较 Angular 和 React，展示如何以结构化的方式处理比较任意两个框架的问题，并根据您的环境进行调整。你知道，古老的“授人以鱼”的方法。这样，当一年后两者都被 BetterFramework.js 取代时，您将能够再次重新创建相同的思路。

## 从哪里开始？

在您选择任何工具之前，您需要回答两个简单的问题:“这本身是一个好工具吗？”以及“它是否适合我的用例？”它们本身都没有任何意义，所以你需要时刻把它们都记在心里。好吧，问题可能没那么简单，所以我们试着把它们分成小问题。

关于工具本身的问题:

*   有多成熟，背后是谁？
*   它有什么样的特点？
*   它采用了什么架构、开发范例和模式？
*   它周围的生态系统是怎样的？

自我反思的问题:

*   我和我的同事能够轻松地学习这个工具吗？
*   它和我的项目吻合吗？
*   开发者体验是什么样的？

使用这组问题，您可以开始评估任何工具，我们也将基于它们来比较 React 和 Angular。

我们还需要考虑另一件事。严格来说，将 Angular 与 React 进行比较是不公平的，因为 Angular 是一个成熟的、功能丰富的框架，而 React 只是一个 UI 组件库。为了公平起见，我们将结合一些经常使用的库来讨论 React。

## 成熟

作为一名熟练的开发人员，重要的一点是能够在成熟的、经过时间验证的方法和评估新的尖端技术之间保持平衡。一般来说，在采用由于某些风险而尚未成熟的工具时，您应该小心谨慎:

*   该工具可能有问题且不稳定。
*   它可能会被供应商意外放弃。
*   如果你需要帮助，可能没有一个大的知识库或社区。
*   React 和 Angular 都是家境不错的，看来在这方面可以自信一点。

### 反应

[React](https://reactjs.org/) 由脸书开发维护，用于他们的产品，包括 Instagram 和 WhatsApp。它从 2013 年开始就存在了，所以并不算新。它也是 GitHub 上最受欢迎的项目之一，在撰写本文时有超过 150，000 颗星星。其他一些使用 React 的著名公司包括 Airbnb、优步、网飞、Dropbox 和 Atlassian。我觉得不错。

### 有角的

[Angular](https://angular.io/) 自 2016 年以来一直存在，使其比 React 略显年轻，但它也不是新上市的产品。它由谷歌和[维护，正如 Igor Minar](https://www.youtube.com/watch?v=rbFLorQWlOQ&feature=youtu.be&t=358) 提到的，即使在 2018 年，它也在谷歌的 600 多个应用程序中使用，如 Firebase 控制台、谷歌分析、谷歌快递、谷歌云平台等。除了谷歌，Angular 还被 Forbes，Upwork，VMWare 和[其他](https://www.madewithangular.com/)使用。

## 特征

就像我之前提到的，Angular 比 React 有更多现成的特性。这可能是一件好事，也可能是一件坏事，取决于你如何看待它。

这两个框架有一些共同的关键特性:组件、数据绑定和平台无关的呈现。

### 有角的

Angular 提供了现代 web 应用程序开箱即用所需的许多特性。一些标准功能包括:

*   依赖注入
*   模板，基于 HTML 的扩展版本
*   带有生命周期挂钩的基于类的组件
*   路由，由`@angular/router`提供
*   使用`@angular/common/http`的 Ajax 请求
*   `@angular/forms`对于建筑形式
*   组件 CSS 封装
*   XSS 保护
*   代码分割和延迟加载
*   用于单元测试的测试运行程序、框架和实用程序。

这些特性中的一些已经内置到了框架的核心中，你不能选择不使用它们。这要求开发人员熟悉诸如依赖注入之类的特性，以便构建哪怕是一个小角度的应用程序。HTTP 客户端或表单等其他特性完全是可选的，可以根据需要添加。

### 反应

使用 React，您将从一个更简单的方法开始。如果我们只看反应，这是我们有的:

*   它没有使用传统的模板，而是使用了 JSX，一种基于 JavaScript 的类似 XML 的语言
*   具有生命周期挂钩的基于类的组件或更简单的功能组件
*   使用 setState 和钩子进行状态管理。
*   XSS 保护
*   代码分割和延迟加载
*   错误处理边界
*   单元测试组件的实用程序

开箱即用，React 没有为依赖注入、路由、HTTP 调用或高级表单处理提供任何东西。您应该根据自己的需要选择添加任何额外的库，这可能是一件好事，也可能是一件坏事，取决于您对这些技术的经验。经常与 React 一起使用的一些流行库有:

*   [React-router](https://reactrouter.com) 进行路由
*   为 HTTP 请求获取(或 [axios](https://github.com/axios/axios) )
*   多种多样的 CSS 封装技术
*   用于附加单元测试实用程序的[酶](https://enzymejs.github.io/enzyme/)或[反应测试库](https://testing-library.com/docs/react-testing-library/intro)

我工作过的团队发现选择图书馆的自由是一种解放。这使我们能够根据每个项目的特定需求来定制我们的堆栈，而且我们还没有发现学习新库的成本有那么高。

## 语言、范例和模式

从每个框架的特性退一步，我们来看看两个框架都流行什么样的高层概念。

### 反应

当考虑 React 时，有几件重要的事情浮现在脑海中:JSX、组件和钩子。

#### JSX

与大多数框架不同，React 没有独立的模板语言。React 没有遵循分离标记和逻辑的传统方法，而是决定使用 [JSX](https://reactjs.org/docs/introducing-jsx.html) 将它们组合在组件中，这是一种类似 XML 的语言，允许您直接在 JavaScript 代码中编写标记。

虽然将标记与 JavaScript 混合的优点可能有争议，但它有一个无可争议的好处:静态分析。如果您在 JSX 标记中出错，编译器将发出一个错误，而不是继续静默。这有助于立即捕捉错别字和其他愚蠢的错误。使用 JSX 在不同的项目中流行起来——比如允许在降价文件中使用 JSX 的 [MDX](https://github.com/mdx-js/mdx) 。

#### 成分

在 React 中，你可以使用函数和类来定义组件。

类组件允许您使用 ES 类编写代码，并将组件逻辑组织成方法。它们还允许您使用 React 的传统生命周期方法在组件被安装、更新、卸载等时运行定制逻辑。尽管这种符号对于熟悉 OOP 编程的人来说更容易理解，但是您需要了解 JS 的所有细微差别——例如，`this`是如何工作的，并且不要忘记绑定事件处理程序。

功能组件被定义为简单的功能。它们通常是纯的，并在输入属性和渲染输出之间提供清晰的映射。功能代码通常耦合度更低，更容易重用和测试。在引入钩子之前，功能组件不能是有状态的，也不能替代生命周期方法。

React 开发人员中有一种趋势，放弃类组件，转而支持更简单的功能组件，但是随着钩子成为一种新的特性，您通常会在更大的 React 项目中看到这两种方法的混合。

#### 钩住

[钩子](https://reactjs.org/docs/hooks-intro.html)是 React 版本引入的新特性。它们是允许您对功能组件中的组件状态和生命周期方法特性进行分类的功能。React 提供了两个钩子:`useState`用于管理状态，而`useEffect`用于创建副作用——比如加载数据或手动编辑 DOM。

钩子的引入是为了使功能组件更简单，更容易组合。现在，您可以将大型功能分割成更小的原子部分，允许您划分相关的功能块，将它们从渲染逻辑中分离出来，并在不同的组件中重用。与使用类组件和其他模式(如渲染函数和更高阶的组件)相比，钩子是一种更简洁的替代方式，因为它们会很快变得过于复杂。

React 提供了构建应用程序的方法，而不涉及大量复杂的抽象层。利用功能组件和钩子可以让你编写更简单、更原子化和可重用的代码。尽管将代码和模板结合起来的想法可能有争议，但是将表示和应用程序逻辑分离到不同的功能中可以让您获得类似的结果。

### 有角的

Angular 也有一些有趣的东西，从组件、服务和模块等基本抽象开始，到 TypeScript、RxJS 和 Angular 元素，以及它的状态管理方法。

#### 主要概念

Angular 比 React 具有更高的抽象层次，因此引入了更多的[基本概念](https://angular.io/guide/architecture)来熟悉。主要的有:

*   **组件**:定义为专门修饰的 ES 类，负责执行应用程序逻辑和呈现模板
*   **服务**:负责实现业务和应用逻辑的类，由组件使用
*   模块:本质上是 DI 容器，用于将相关的组件、服务、管道和其他实体连接在一起。

Angular 大量使用了类以及 DI 等概念，这些概念在前端开发领域不太流行，但任何有后端开发经验的人都应该很熟悉。

#### 以打字打的文件

TypeScript 是由微软开发的一种建立在 JavaScript 之上的新语言。它是 JavaScript ES2015 的超集，包括该语言新版本的功能。你可以用它代替 Babel 来编写最先进的 JavaScript。它还有一个非常强大的类型系统，可以通过使用注释和类型推断的组合来静态分析您的代码。

还有一个更微妙的好处。TypeScript 深受 Java 和。因此，如果您的开发人员具有其中一种语言的背景，他们可能会发现 TypeScript 比普通的 JavaScript 更容易学习(请注意我们是如何从工具切换到您的个人环境的)。尽管 Angular 是第一个积极采用 TypeScript 的主要框架，但它现在也在许多其他项目中受到关注，例如 [Deno](https://www.sitepoint.com/deno-introduction/) (一个 TypeScript 本地运行时)、Puppeteer 和 TypeORM。

将 TypeScript 与 React 一起使用也是可能的(也是明智的)。

#### RxJS

RxJS 是一个反应式编程库，允许更灵活地处理异步操作和事件。它是观察者和迭代器模式与函数式编程的结合。RxJS 允许您将任何东西视为连续的值流，并对其执行各种操作，如映射、过滤、拆分或合并。

Angular 在其 HTTP 模块中也采用了这个库，供内部使用。当你执行一个 HTTP 请求时，它返回一个可观察的而不是通常的承诺。这种方法为一些有趣的可能性打开了大门，比如取消一个请求、重试多次或者处理连续数据流(比如 WebSockets)的能力。但这只是表面。要掌握 RxJS，你需要了解不同类型的观察对象、主题以及大约一百种方法和操作符。

#### 状态管理

与 React 类似，角度组件也有一个组件状态的概念。组件可以将数据存储在它们的类属性中，并将值绑定到它们的模板。如果您想在应用程序之间共享状态，您可以将它移动到一个有状态的服务，该服务可以在以后注入到组件中。因为反应式编程和 RxJS 是 Angular 中的一等公民，所以基于一些输入利用可观察值来重新计算状态的一部分是很常见的。然而，在较大的应用程序中，这可能会变得棘手，因为更改一些变量可能会触发难以跟踪的多方向级联更新。Angular 有一些库，可以让你大规模地简化状态管理。我们稍后会更仔细地观察它们。

#### 角度元素

[角度元素](https://angular.io/guide/elements)提供了一种将角度组件打包成定制元素的方法。也称为 web 组件，自定义元素是一种与框架无关的标准化方法，用于创建由 JavaScript 代码控制的自定义 HTML 元素。一旦定义了这样一个元素并将其添加到浏览器注册表中，它就会自动呈现在 HTML 中引用它的所有地方。Angular 元素提供了一个 API，它创建了必要的包装器来实现定制组件 API，并使其与 Angular 的变化检测机制一起工作。这种机制可以用于将其他组件或整个 Angular 应用程序嵌入到您的主机应用程序中，这些应用程序可能是在不同的框架中以不同的开发周期编写的。

我们发现 TypeScript 是提高项目可维护性的一个很好的工具，尤其是那些具有大型代码库或复杂领域/业务逻辑的项目。用 TypeScript 编写的代码更具描述性，更容易理解和重构。即使您不打算使用 Angular，我们也建议您在下一个 JavaScript 项目中考虑使用它。RxJS 在你的项目中引入了管理数据流的新方法，但是需要你很好的掌握这个主题。否则，它会给项目带来不必要的复杂性。角度元素具有重用角度组件的潜力，看看这在未来如何发展是很有趣的。

## 生态系统

开源框架的伟大之处在于围绕它们创建了大量的工具。有时候，这些工具甚至比框架本身更有帮助。让我们看看与每个框架相关的一些最流行的工具和库。

### 有角的

#### 角度 CLI

现代框架的一个流行趋势是拥有一个 CLI 工具来帮助您引导您的项目，而不必自己配置构建。Angular 有 T2 这个名字。它允许你用几个命令就能生成并运行一个项目。在`node_modules`中，所有负责构建应用程序、启动开发服务器和运行测试的脚本都是隐藏的。您还可以使用它在开发和安装依赖项期间生成新代码。

Angular 引入了一种有趣的管理项目依赖关系的新方法。使用`ng add`时，您可以安装一个依赖项，它将自动配置使用。例如，当您运行`ng add @angular/material`时，Angular CLI 从 npm 注册表下载 Angular 材料，并运行其安装脚本，该脚本自动配置您的应用程序使用 Angular 材料。这是使用角度示意图完成的。Schematics 是一个工作流工具，允许库对您的代码库进行更改。这意味着库作者可以提供自动的方法来解决您在安装新版本时可能会遇到的向后不兼容的问题。

#### 组件库

在使用任何 JavaScript 框架时，重要的一点是能够将它们与您选择的组件库集成在一起，以避免不得不从头构建一切。Angular 提供了与大多数流行的组件库以及自己的本地库的集成。例如:

*   [ng-bootstrap](https://ng-bootstrap.github.io) 用于使用 bootstrap 小工具
*   [材质 UI](https://material.angular.io/) ，用于 Google 的材质设计组件
*   NG-ZORRO，一个实现 Ant 设计规范的组件库
*   【Angular 的 Onsen UI，一个移动应用的组件库
*   [PrimeNG](https://www.primefaces.org/primeng/) ，集合了丰富的角度分量

#### 国家管理图书馆

如果本机状态管理功能对您来说还不够，那么在这方面有几个流行的第三方库。

最受欢迎的是 NgRx。它的灵感来自 React 的 Redux，但也利用 RxJS 来观察和重新计算状态中的数据。使用 [NgRx](https://ngrx.io/) 可以帮助你执行一个可理解的单向数据流，同时减少代码中的耦合。

[NGXS](https://www.ngxs.io/) 是另一个受 Redux 启发的状态管理库。与 NgRx 相反，NGXS 通过使用现代的 TypeScript 特性和改进学习曲线和整体开发体验来努力减少样板代码。

Akita 是一个较新的产品，它允许我们在多个存储中保存状态，应用不可变的更新，并使用 RxJS 来查询和传输值。

#### 离子框架

Ionic 是开发混合移动应用的流行框架。它提供了一个 Cordova 容器，很好地集成了 Angular 和一个漂亮的材质组件库。使用它，您可以轻松地设置和构建移动应用程序。如果你更喜欢混合应用而不是原生应用，这是一个不错的选择。

#### 角度通用

Angular universal 是一个捆绑了不同工具的项目，用于实现 Angular 应用程序的服务器端渲染。它与 Angular CLI 集成，并支持多个 Node.js 框架，如 Express 和哈比神，以及。网芯。

#### 预兆

[Augury](https://augury.rangle.io/) 是 Chrome 和 Firefox 的浏览器扩展，有助于调试在开发模式下运行的 Angular 应用。您可以使用它来探索您的组件树，监控变更检测，并优化性能问题。

#### Compodoc

Compodoc 是 Angular 的静态文档生成器。与其他文档生成器类似，它可以基于代码中的 TSDoc 注释创建静态 HTML 文档。然而，Compodoc 附带了专门针对 Angular 的便利特性，比如浏览您的模块结构、路由，以及将类分类为组件、服务等等。

#### ngx-管理

Ngx-admin 是一个流行的框架，用于创建带有 Angular 的定制仪表板，并使用 Nebular 或 Angular 材料作为组件库。

在[牛逼的角度列表](https://github.com/PatrickJS/awesome-angular)中还有很多其他的库和工具。

### 反应

#### 创建 React 应用

Create React App 是一个 CLI 实用程序，用于 React 快速设置新项目。与 Angular CLI 类似，它允许您生成一个新项目，在开发模式下运行应用程序，或创建一个生产包。它使用 Jest 进行单元测试，支持使用环境变量的应用程序概要分析、本地开发的后端代理、TypeScript、Sass、PostCSS 和许多其他特性。

#### 组件库

与 Angular 类似，React 有各种各样的组件库可供选择:

*   [蚂蚁设计](https://ant.design/docs/react/introduce)
*   [材料界面](https://material-ui.com/)
*   [反应-引导](https://react-bootstrap.github.io/)
*   [语义界面](https://react.semantic-ui.com/)
*   [Onsen UI](https://onsen.io/) ，针对移动应用进行了优化
*   [蓝图](https://blueprintjs.com/)，用于创建桌面应用程序

#### 国家管理图书馆

钩子的引入无疑动摇了 React 的国家管理。关于是否需要第三方状态管理库的讨论还在进行中。即使钩子解决了处理状态的直接需求，其他库仍然可以通过允许我们使用经过时间考验的实现模式、大量附加库和开发工具来进一步推动这一需求。

Redux 是一个受 Flux 启发的状态管理库，但是做了一些简化。Redux 的关键思想是，应用程序的整个状态由一个对象表示，这个对象由称为 reducers 的函数变异。减速器本身是纯函数，与组件分开实现。这使得关注点和可测试性更好的分离。

MobX 是一个管理应用程序状态的备选库。它不像 Redux 那样将状态保存在一个不可变的存储中，而是鼓励您只存储最少的必需状态，并从中派生出其余的状态。它提供了一组装饰器来定义可观察对象和观察者，并将反应逻辑引入到您的状态中。

#### 样式库

与 Angular 不同，React 不提供原生 CSS 封装能力，所以需要寻找第三方解决方案。这个问题有许多解决方案，但没有明确的领导者。一些受欢迎的是:

*   [Styled Components](https://styled-components.com/) ，一个库，允许您创建应用了样式的 React 组件，以及样式化您的组件
*   [CSS 模块](https://github.com/css-modules/css-modules)，允许您导入 CSS 文件并生成唯一的独立类名来引用样式
*   情感(Emotion),它将样式化组件和 CSS 模块的方法合并到一个库中

#### 属性类型

[属性类型](https://reactjs.org/docs/typechecking-with-proptypes.html)是 React 的一个可选特性，它允许您引入组件运行时属性验证。与对 TypeScript 使用静态类型检查不同，PropTypes 将在应用程序实际运行时执行类型检查。当开发库时，如果您不能确定您的客户机是否在使用 TypeScript，即使您在使用，这也是非常方便的。从 React 15.5 开始，道具类型被移到一个单独的道具类型库中，现在完全是可选的。考虑到它的好处，我们建议您使用它来提高应用程序的可靠性。

#### 反应自然

React Native 是脸书开发的一个平台，用于使用 React 创建本地移动应用。与生产混合应用程序的 Ionic 不同，React Native 生产的是真正的原生 UI。它提供了一组标准的 React 组件，这些组件被绑定到它们的本地对应物。它还允许您创建您的组件，并将它们绑定到用 Objective-C、Java 或 Swift 编写的本机代码。

#### Next.js

[Next.js](https://nextjs.org/) 是 React 应用的服务器端渲染框架。它提供了一种灵活的方法，可以在服务器上完全或部分呈现您的应用程序，将结果返回给客户端，并在浏览器中继续。它试图使创建通用应用程序的复杂任务变得更容易，因此设置被设计得尽可能简单，使用最少量的新原语和对项目结构的要求。

#### 管理反应

React-admin 是一个框架，用于在现有的 REST 或 GraphQL APIs 之上构建 CRUD 风格的 SPA 应用程序。它附带了一些便利的特性，比如用材料设计、国际化、主题化、数据验证等构建的 UI。

#### UI 开发环境

在过去几年中，前端开发的一个主要趋势是开发工具的繁荣，这些工具允许您独立于应用程序，以交互方式开发、测试和记录您的组件。Storybook 已经成为该领域的领导者之一，同时支持 React 和 Angular。但是，React 还有其他替代方案。

与 Storybook 类似，React Styleguidist 允许您创建组件的交互式文档。与 Storybook 相比，生成的 UI 看起来更像一个交互式的自述文件，而不是一组独立的故事。虽然 Storybook 作为开发环境大放异彩，但 Styleguidist 更多的是一个文档工具。

我们在本文中还提到了 [MDX](https://github.com/mdx-js/mdx) 。它允许你通过添加交互式 JSX 组件来增加你的减价文件的趣味。

#### 测试助手

测试 UI 组件通常需要在沙盒环境中呈现它们，模拟用户交互，并验证输出结果。这些日常任务可以通过使用适当的测试助手来简化。对于 Angular，这是内置的[试验台](https://angular.io/api/core/testing/TestBed)。对于 React，有两个流行的候选:酶和测试库。

[酶](https://enzymejs.github.io/enzyme/)是事实上的标准选择。它允许您将组件呈现为完整的或浅层的 DOM，并与呈现的组件进行交互。它主要遵循白盒测试方法，在这种方法中，您的测试可以引用组件的一些内部信息，如其子组件、道具或状态。

测试库遵循一种不同的方法，推动你像用户一样与你的组件交互，而不需要知道技术实现。以这种方式创建的测试通常不太脆弱，也更容易维护。虽然它最受 React 的欢迎，但测试库也可用于 Angular。

#### 盖茨比（姓）

[Gatsby](https://www.gatsbyjs.org/) 是一个使用 React.js 的静态网站生成器，它允许你使用 GraphQL 查询你在 Markdown、YAML、JSON、外部 API 和流行内容管理系统中定义的网站的数据。

#### 反应 360 度

[React 360](https://facebook.github.io/react-360/) 是一个为浏览器创建虚拟现实应用的库。它提供了一个声明性的 React API，构建在 WebGL 和 WebVR 浏览器 API 之上，从而更容易创建 360 VR 体验。

#### React 开发人员工具

[React Dev Tools](https://chrome.google.com/webstore/detail/react-developer-tools/fmkadmapgofadopljbjfkapdkoienihi?hl=en) 是 Chrome 的一个浏览器扩展，用于调试 React 应用程序，允许你遍历 React 组件树并查看它们的属性和状态。

在 [Awesome React 列表](https://github.com/enaqx/awesome-react)中还有很多其他的库和工具。

## 采用、学习曲线和开发经验

选择新技术的一个重要标准是它有多容易学。当然，答案取决于多种因素，比如您以前的经验以及对相关概念和模式的大致了解。然而，我们仍然可以尝试评估在开始使用一个给定的框架时你需要学习的新东西的数量。现在，如果我们假设你已经知道 ES6+，构建工具，以及所有这些，让我们看看你还需要了解什么。

### 反应

有了 React，你首先会遇到的是 JSX。对于某些开发人员来说，这确实显得有些笨拙。然而，它没有增加太多的复杂性:只是表达式，即 JavaScript 和特殊的类似 HTML 的语法。您还需要学习如何编写组件，使用 props 进行配置，以及管理内部状态。您不需要学习新的模板语法，因为所有这些都是普通的 JavaScript。尽管 React 支持基于类的组件，但随着钩子的引入，功能开发越来越受欢迎。这需要你理解一些基本的功能开发模式。

官方教程是一个开始学习 React 的好地方。一旦你完成了，熟悉路由器。React 路由器可能有点复杂和不传统，但没什么可担心的。根据项目的规模、复杂性和需求，您需要找到并学习一些额外的库，这可能是棘手的部分。但之后，一切都应该是一帆风顺的。

我们真的很惊讶 React 的使用是如此简单。即使是有后端开发背景、前端开发经验有限的人也能很快赶上。在这个过程中，您可能会遇到的错误消息通常很清楚，并且提供了关于如何解决底层问题的解释。

缺点是您需要投入时间来选择支持您的开发活动的库。考虑到他们的数量，这可能是一个挑战。但是这可以在您熟悉框架本身之后，在您的开发活动中完成。

虽然 TypeScript 不是 React 所必需的，但是我们强烈建议您在项目中评估和采用它。

### 有角的

学习 Angular 会让你接触到比 React 更多的新概念。首先，您需要熟悉 TypeScript。面向具有 Java 或。这可能比 JavaScript 更容易理解，但是对于纯 JavaScript 开发人员来说，这可能需要一些努力。开始你的旅程，我们可以推荐[英雄之旅](https://angular.io/tutorial)教程。

框架本身有丰富的主题需要学习，从基本的主题如模块、依赖注入、装饰器、组件、服务、管道、模板和指令，到更高级的主题如变化检测、区域、AoT 编译和 Rx.js，这些都包含在文档中。Rx.js 本身就是一个沉重的话题，在官网上有非常详细的描述。虽然在基础层面上相对容易使用，但在进入高级主题时会变得更加复杂。

总而言之，我们注意到 Angular 的准入门槛比 React 高。新概念的数量之多可能会让新来者不知所措。甚至在您开始之后，体验可能会有点粗糙，因为您需要记住诸如 Rx.js 订阅管理、变更检测性能和盒子里的香蕉之类的事情(是的，这是文档中的一条实际建议)。我们经常会遇到难以理解的错误信息，所以我们不得不在谷歌上搜索它们，并祈求一个精确的匹配。

看起来我们倾向于在这里做出反应，我们确实是这样做的。我们有让新开发人员参与规模和复杂性相当的 Angular 和 React 项目的经验，不知何故，React 总是进行得更顺利。但是，正如我前面所说，这取决于一系列广泛的因素，可能对你有不同的作用。

## 做决定

您可能已经注意到，每个框架都有自己的一套功能，既有好的一面，也有不好的一面。但是这种分析是在任何特定的环境之外进行的，因此不能提供您应该选择哪种框架的答案。要决定这一点，你需要从你的项目的角度来审视它。这是你需要自己做的事情。

首先，试着回答这些关于你的项目的问题，当你这样做时，将答案与你对这两个框架的了解相匹配。这个列表可能不完整，但应该足以让你开始:

*   项目有多大？
*   要维持多久？
*   所有的功能都是预先明确定义的吗？还是希望你灵活一些？
*   如果已经定义了所有的特性，那么还需要什么功能呢？
*   领域模型和业务逻辑复杂吗？
*   你的目标平台是什么？网络、移动、桌面？
*   需要服务器端渲染吗？SEO 重要吗？
*   您将处理大量实时事件流吗？
*   你的团队有多大？
*   你的开发人员经验如何，他们的背景如何？
*   有没有你想用的现成的组件库？

如果你正在开始一个大项目，并且你想最小化做出错误选择的风险，考虑首先创建一个概念验证产品。选择项目的一些关键特性，并尝试使用其中一个框架以简单的方式实现它们。POC 通常不需要花费很多时间来构建，但是它们会给你一些关于使用框架的有价值的个人经验，并允许你验证关键的技术需求。如果您对结果满意，您可以继续进行全面的开发。如果没有，从长远来看，快速失败会让你省去很多麻烦。

## 一个框架来统治他们？

一旦你为一个项目选择了一个框架，你就会被诱惑为你即将到来的项目使用完全相同的技术栈。不要。尽管保持技术堆栈的一致性是个好主意，但不要每次都盲目地使用相同的方法。在开始每个项目之前，花点时间再次回答同样的问题。也许在下一个项目中，答案会有所不同，或者景观会有所改变。此外，如果你有机会用不熟悉的技术做一个小项目，那就去做吧。这样的实验会给你提供宝贵的经验。保持开放的心态，从错误中学习。在某种程度上，某种技术会让人感觉自然而正确。
