# 创建舒适的可用性测试

> 原文：<https://www.sitepoint.com/creating-comfortable-usability-tests/>

可用性测试就是简单地观察一个人使用你的网站或应用程序，并有目的地学习如何改进。

测试的内容本身很重要，但是如果参与者感到不舒服和不自在，你的测试将会大受影响。无论你是在会议室还是在咖啡店，带着笔记本电脑，在开始可用性测试之前，你都应该考虑一些事情:

## 做好准备！

如果你没有准备好，你的参与者将会知道，这将为接下来的测试定下基调。这对他们来说是一个新的情况，知道你在控制会让一切感觉正常一点。这三个步骤可以帮助你做好准备:

1.  创建你的草稿。写一份介绍，这样你就可以自信地开始会议。确定你想要回答的关于你的网站的问题。然后，写下具体的任务，让参与者完成，这些任务可能会揭示这些问题的答案。
2.  **预测试。运行一两个与真实任务完全一样的样本测试。同事、朋友或家庭成员都非常适合。记下测试流程在哪里出了问题，你在哪里最不舒服，以及是否有任何参与者误解你的地方。确保你将使用的所有技术都按预期工作。**
3.  **拧一下。**检查你的笔记，看看草稿中的漏洞在哪里。润色引言，练习多读几遍。如果有必要的话，重新组织任务顺序(特别是如果做一个任务给如何做另一个任务提供了线索)，调整措辞，并再三检查你没有以领导的方式呈现任务。

## 设定期望值

你将某人置于一个充满未知的情境中，所以预先告诉参与者会发生什么有助于缓解紧张情绪。有人看着你使用网站会感觉像是一种个人侵犯(尤其是如果内容是个人的)。

非常清楚，一切都是保密的，你不是在测试这个人；您正在测试产品。他们没有做错什么。在整个过程中，当他们在表达思想和提出修辞性问题方面做得特别好时，重申他们，因为这将带来更多同样的问题。

另一方面，更固执己见的人可能会抓住机会告诉你网站的所有问题。这可能很快演变成夸张和/或愤怒。使用你的介绍来帮助每个人理解什么类型的反馈是有用的，什么是无用的。你看着他们，听着他们说出自己的想法，问一些关于网站是什么的具体问题。这不同于询问他们的意见，你可以很好地解释测试是以帮助你的发现保持完整性的方式建立的。

## 听起来像人类

通过照本宣科的方式与人交流可能会令人不快，尤其是对你的现场参与者而言。如果你觉得读剧本很痛苦，那么试着根据要点清单即兴发言。这样，你可以涵盖一切，但又不会听起来像个机器人。

如果您确实使用了项目符号，在开始测试之前先练习一下，这样您就不会无意中引入新的概念或想法。从某些方面来说，剧本更容易，也更一致，但是如果它不真实，会破坏你们之间的融洽关系。

## 保持灵活性

你的参与者可能会不时偏离你的预期路径，最终完成你留到以后再做的任务。在这种情况下，我喜欢让他们完成它，然后在运行中切换测试。他们不会知道其中的区别，你也不会因为不得不重复做同样的事情而感到沮丧(没有人喜欢这样做！).

在几次测试之后，你可能还会意识到你创建的一些任务没有帮助或者不清楚。这给了你不好的信息，当你的志愿者试图解释你的请求时，会使整个测试嘎然而止。为将来的测试改进任务，不要把所有的测试都安排在后面。这样，你就有时间反思和改进。

最重要的是，让参与者知道你尊重他们的时间，感谢他们的参与。和他们握手，给他们提供茶点，给他们一个舒适的座位，询问他们的近况——像对待一个重要客户一样对待他们。这将为他们定下正确的基调，并帮助他们理解他们的工作有多重要！
