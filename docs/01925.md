以下是我们的书的摘录，由 Alexis Goldstein、Louis Lazaris 和 Estelle Weyl 撰写的《真实世界的 CSS3，第二版。世界各地的商店都有出售，或者你可以在这里买到电子书。

我们今天所理解的 HTML5 有一段相对动荡的历史。您可能已经知道 HTML 是用于描述万维网上的内容或数据的主要标记语言(另一种较少使用的标记语言是 XML)。HTML5 是 HTML5 语言的最新版本，包括新特性、对现有特性的改进和 JavaScript APIs。

也就是说，HTML5 并不是该语言以前版本的重构——它包含了 HTML4 和 XHTML 1.0 的所有有效元素。此外，它的设计考虑了一些原则，以确保它可以在几乎所有平台上工作，与旧浏览器兼容，并优雅地处理错误。在 W3C 的 HTML 设计原则页面上可以找到指导 HTML5 创建的设计原则概要。

首先也是最重要的，HTML5 除了允许 web 设计者在描述页面内容时更具表现力的新元素之外，还包括对现有标记元素的重新定义。当你可以使用`article`、`section`、`header`、`footer`等元素时，为什么还要在页面中加入`div`元素呢？

术语“HTML5”也被用来指代许多其他新技术和 API。其中包括使用`canvas`元素绘图、离线存储、新的`video`和`audio`元素、拖放功能、微数据和嵌入字体。在这本书里，我们将会涉及到很多这样的技术，甚至更多。

### 注意:应用程序编程接口

API 代表应用程序编程接口。用看待图形用户界面(GUI)的方式来看待 API 除了它不是人类的界面，而是代码的界面。API 为您的代码提供了一组“按钮”(预定义的方法)，您可以按下这些按钮从系统、软件库或浏览器中获得所需的行为。

基于 API 的命令是一种抽象更复杂工作的方式，这些工作在后台完成(有时由第三方软件完成)。一些与 HTML5 相关的 API 将在本书的后面部分介绍和讨论。

总的来说，如果你对 JavaScript 或其他 API 没有什么经验，你不应该害怕。虽然拥有一些 JavaScript 或其他语言的经验肯定是有益的，但这不是强制性的。无论是哪种情况，我们都将带您逐步浏览本书的脚本部分，确保您不会感到困惑。

在撰写本文时，HTML5 在语义元素和各种 API 方面得到广泛使用已经有 5 年多了。因此，将 HTML5 归类为“新”技术不再正确——但它仍在不断成熟，仍有问题需要解决(如浏览器中的错误，以及跨浏览器和平台的不一致支持)。

还应该注意的是，有些技术从来不是 HTML5 的一部分(比如 CSS3 和 WOFF)，但有时会被归入同一标签下。这促使了广泛的、无所不包的表达方式的使用，例如“HTML5 和相关技术”为了简洁起见——也冒着引起激烈争论的风险——我们将这些技术统称为“HTML5”

## 我们是怎么到这里的？

web 开发行业在相对较短的时间内有了显著的发展。在 20 世纪 90 年代末，一个包含图片和醒目设计的网站在网页内容和表现方面被认为是顶级的。

今天，情况大不相同。简单的、性能驱动的、基于 Ajax 的网站(通常被区分为“web 应用程序”)越来越普遍，它们依赖客户端脚本来实现关键功能。如今的网站经常类似于独立的软件应用程序，越来越多的开发者也这样看待它们。

一路走来，web 标记不断发展。HTML4 最终让位于 XHTML，后者实际上只是具有严格 XML 风格语法的 HTML4。HTML5 已经成为最常用的标记版本，我们现在很少看到用 HTML4 或 XHTML 构建的新项目。

HTML5 最初有两种不同的规范: [Web Forms 2.0](https://www.w3.org/TR/web-forms-2/) 和 [Web Apps 1.0。](https://whatwg.org/specs/web-apps/2005-09-01/)两者都是 web 环境变化和对更快、更有效的可维护 web 应用程序的需求的结果。表单和类似应用的功能是 web 应用的核心，所以这是 HTML5 规范的自然发展方向。最终，这两个规范合并成了我们现在所说的 HTML5。

在很短的时间内，曾有过关于 XHTML 2.0 的讨论，但是这个项目很久以前就被放弃了，让人们关注更实用的 HTML5。

## 真的 HTML5 请站起来好吗？

因为 HTML5 规范是由两个不同的团体(WHATWG 和 W3C)开发的，所以该规范有两个版本。你可能很熟悉 W3C(或万维网联盟):它是维护原始 HTML 和 CSS 规范的组织，也是一系列其他 Web 相关标准的组织，如 SVG(可缩放矢量图形)和 WCAG (Web 内容可访问性指南)。

另一方面，WHATWG(又名 Web 超文本应用技术工作组)是由一群来自苹果、Mozilla 和 Opera 的人在 2004 年 W3C 会议让他们心灰意冷后成立的。他们觉得 W3C 专注于 XHTML 2.0 而忽略了浏览器制造商和用户的需求，而不是致力于一个向后兼容的 HTML 标准。所以他们独立开发了我们讨论过的 Web 应用和 Web 表单规范，然后合并成一个规范，他们称之为 HTML5。看到这一点，W3C 最终让步，基于 WHATWG 的规范创建了自己的 HTML5 规范。

这似乎有点令人困惑。是的，在幕后有一些我们作为设计者和开发者无法控制的政治。但是我们应该担心这个规范有两个版本吗？简而言之，没有。

WHATWG 的规范版本可以在 http://www.whatwg.org/html/找到，并于 2011 年 1 月更名为“HTML”(去掉了“5”)。它现在被称为[“生活标准”，](http://blog.whatwg.org/html-is-the-new-html5)，这意味着它将不断发展，不再使用递增的版本号。

WHATWG 版本包含了 HTML 特有的特性，包括 HTML5 的新特性。此外，WHATWG 正在开发涵盖相关技术的单独规范。这些规范包括微数据、画布 2D 上下文、Web Workers、Web 存储等。

W3C 版本的规范可以在[https://www.w3.org/html/wg/drafts/html/master/](https://www.w3.org/html/wg/drafts/html/master/)找到，其他技术的单独规范可以通过[http://dev.w3.org/html5/](http://dev.w3.org/html5/)获得。[](#ftn.d5e236)

那么 W3C 规范和 WHATWG 的有什么区别呢？除了名字(“生活标准”相对于“HTML5.1”)，WHATWG 版本更非正式，更具实验性(有人可能会说，更具前瞻性)。但是在大多数地方它们是相同的，所以它们都可以作为研究 HTML5 新元素和相关技术的基础。[](#ftn.d5e241)【2】

## 我为什么要关心 HTML5？

如前所述，HTML5 的核心是许多新的语义元素，以及一些相关的技术和 API。这些对语言的添加和改变是为了让开发人员能够构建更容易编码、使用和访问的网页。

这些新的语义元素，以及其他标准，如 WAI-ARIA 和微数据(我们分别在附录 B 和附录 C 中介绍)，有助于使我们的文档更容易被人类和机器访问，从而有利于可访问性和搜索引擎优化。

特别是语义元素，在设计时考虑到了动态 Web，特别关注于使页面更易于访问和模块化。我们将在后面的章节中更详细地讨论这一点。

最后，与 HTML5 相关的 API 有助于改进 web 开发人员已经使用多年的许多技术。许多常见的任务现在都得到了简化，将更多的权力交给了开发人员。此外，HTML5 音频和视频的引入意味着在 Web 上发布富媒体内容时对第三方软件和插件的依赖将会减少。

总的来说，有很好的理由开始研究 HTML5 的新特性和 API，我们将在本书中讨论更多的理由。
