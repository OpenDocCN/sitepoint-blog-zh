# 没有黑仔离线网络应用的 5 个原因

> 原文:[https://www.sitepoint.com/killer-offline-web-applications/](https://www.sitepoint.com/killer-offline-web-applications/)

让网络应用在没有互联网连接的情况下也能工作的技术听起来是个好主意。一般来说，它允许开发人员:

1.  在电脑上运行流程以改善延迟和响应，以及
2.  创建脱机工作并在本地存储数据的应用程序。

这些技术模糊了网络和桌面的界限。这是一个几年前让业内每个人都兴奋不已的概念，现在已经被 HTML5 采用。参见我之前的帖子: [**如何创建离线 Web 应用**](https://www.sitepoint.com/offline-web-application-tutorial/) 。

那么，为什么只有少数平庸的 web 应用程序使用离线功能呢？

谷歌已经把它添加到 GMail 中，我怀疑 Chrome OS 是这么做的主要原因。但这并不是一个突破性的概念——离线电子邮件已经存在了几十年。其他产品倾向于随机收集密码管理器和 WordPress 等系统的速度提升工具。

以下是离线应用没有成为主流的 5 个原因:

## 1.糟糕的浏览器实现

虽然 HTML5 逐渐出现，但离线功能还不成熟，实现也不尽相同。在这项技术普及之前，开发者必须依赖浏览器插件。谷歌 Gears 是最知名的，但它并不是在所有平台上都可用，而且[开发已经停止](https://www.sitepoint.com/google-gears-gone/)。

## 2.额外的开发工作和复杂性

创建一个在线和离线都能工作的 web 应用程序并不容易。在某些情况下，一个应用程序可能需要两个不同的版本来处理差异。如果大多数用户保持连接，这种努力可能是不值得的。

## 3.埃阿斯

如果服务器可以将工作卸载给客户机，则处理开销是分散的。还可以更高效地批量发送数据，而不是整页加载。然而，许多优秀的 web 应用程序已经使用 Ajax 实现了这一点。不使用支持脱机的代码，响应时间会很快。

## 4.安全性

本地存储数据应该可以防止许多安全问题，这就是为什么它被一些密码管理器使用的原因。然而，如果你真的关心安全，为什么不使用一个从不连接网络的桌面应用程序呢？

## 5.用户受益很少

我能理解离线存储的吸引力，但是有多少用户在网络连接不良的地方做大量基于网络的工作呢？如果他们这样做了，那么使用电子邮件客户端或文字处理器而不是在线应用程序就很简单了。

web 应用程序的一大优势是数据存储在云中。离线存储可能会提供一个安全网，但这有必要吗？如果有一件事我们可以确定的话，那就是互联网连接将会改善。

也许现在还为时过早，但对我来说，离线功能就像是一项等待应用的技术。然而，我想到了一个杀手锏…明天再来了解更多。
