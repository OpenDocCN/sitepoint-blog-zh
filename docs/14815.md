# 使富 Web 应用程序架构变得可用

> 原文:[https://www.sitepoint.com/architecture-usable/](https://www.sitepoint.com/architecture-usable/)

“关注用户，一切都会水到渠成”是一种哲学，它可以成就一个产品，也可以毁掉一个产品。

软件设计师因专注于实现模式而忽视用户而臭名昭著。人们很容易迷失在抽象层次的宏大概念中，并对那些使开发人员的工作更容易的东西感到兴奋；因此,“真正”用户的需求和愿望有时可能会退居二线。识别可用性限制，并在其中进行设计，将焦点放在用户身上。

##### 可用性限制

用户对各种类型的交互都有一定的基本好恶。它们不仅适用于人机交互，也适用于现实生活。去掉用户不喜欢的，集中他们喜欢的，这是提高用户满意度的第一步。下面列出了用户讨厌的 10 件事:

***用户讨厌学习***

学习需要用户的认知投资。如果我们学习新的交互方式所做的投资没有回报，那么学习新的交互方式就没有意义。在界面中使用标准元素，并在交互中保持这些元素在视觉上的一致性，使得用户只需学习一次，就可以将知识应用到任何地方。真实世界的隐喻可以加速学习，帮助用户理解最细微的细节，因为它们允许设计师利用用户在现实生活中已经学到的东西。

例如，万维网上使用的旅行隐喻使每个人都可以轻松上网，按钮的斜面邀请用户“推动”它。

每个用户在接近界面时，脑子里都有一个不同的预先存在的模型。界面应该足够灵活，以适应所有这些用户模型。例如，用户可能输入一个电话号码 617-493-4984 或(617) 493 4984，所以界面应该准备好接受其中任何一个。

典型地，用户将尝试学习界面中各种元素的位置，并随后尝试与它们交互，而不是每次都看着元素。帮助用户记住重要的元素并确定这些元素的位置将提高学习的效率。界面应该简洁明了——这让高级用户容易记住东西，防止新用户迷失在界面中。将元素分成不同的组可以保持界面整洁。人物角色还可以帮助开发人员防止功能膨胀，并专注于用户。

用户会发现一点一点地学习新信息变得更容易，所以一次给用户几个新任务会比用信息淹没他们更有效。然而，学习材料不应该妨碍用户对真实内容的访问；因此，界面中的元素应该是可学习和可扫描的。这将允许新手用户学习新功能，而专家用户将能够快速执行任务。

例如，按钮应该有文本/工具提示和图标。文本/工具提示使按钮可学习，而图标使其可扫描。保持界面的缓慢发展也有助于用户学习，并在先前成功的基础上创造一个成熟的产品。在真实用户的一小部分上测试新特性为开发人员提供了修改的真实性能的指示，并允许缓慢的发展发生。

***用户讨厌重复***

没有人喜欢输入数据；我们更不喜欢重复数据。你有多少次跳过在网上输入个人信息？当您丢失花费时间和精力输入的数据(例如，Web 表单中的问题或评论)时，您有什么感受？自动数据收集和信息重用将为您的用户省去数据输入的麻烦。

用户希望一次获得或写入数据，并能够在任何地方使用它。可以自动收集的任何数据，例如关于位置、任务、状态等的上下文信息。应该是这样获得的。富 Internet 应用程序的主要优点是，即使在客户端，它们也能保持数据的丰富性，并允许使用客户端逻辑重用下载的数据。根据用户与网站的交互，信息必须能够流向其他连接的内容。使用标准格式存储数据将使这种互操作性成为可能，从而让用户满意。通过将数据锁定到特定的应用程序中，用户可以使用该应用程序，但是如果它不支持他们需要的特定任务，他们就会讨厌它。

界面中各种组件的默认值应该反映大多数人的偏好，但是它们也应该很容易被删除，这样用户就可以设置新的值。例如，文本字段应该填充默认值并预先选择，以便于删除。自动完成是重用数据的另一种方式，即使软件不能完全理解这些数据。将收集到的所有数据存储在服务器上可以让用户在不同的设备上获得持续的体验。因此，应该备份数据，而不是覆盖数据，以防止数据丢失。

***用户讨厌等待***

等待某人或某物出现是乏味的。有多少次你厌倦了等待朋友的出现？如果我们不知道要等多久，我们会更讨厌等待。因此，快速加载数据并显示进度非常重要——用户可以计划并适应其他任务。让用户访问前一个任务并显示进度将会吸引用户的注意力，并在数据加载时让他们得到通知。等待时在任务间切换需要用户努力，这也是速度如此重要的另一个原因。显示部分加载的数据也有助于最小化感知等待。

如果长时间的等待是不可避免的，用户应该被告知他们将在数据加载后收到一个警告。

当用户重新联机时，应在完成未决请求时发送适当的警报。需要不同类型的警报来防止在重要任务期间打扰用户。例如，不管用户在做什么，一种类型的警告会弹出声音。另一种提醒类型聚集在一个列表中，用户有空时可以看到。另一个会在侧面板的自动收报机上弹出，用户可以在有空的时候浏览一下。警报应该由用户配置。

***用户讨厌搜索***

找不到你想要的东西会令人沮丧。如果东西的位置经常变化，就不可能记住它们在哪里。保持一致的导航对于帮助用户找到他们想要的信息很重要。当我们在一个新的地方时，我们通过向别人寻求帮助或看标志来四处走动。同样，搜索和导航对于在网上寻找资料也非常重要。

在现实世界中，我们将时间分配给不同的任务，并按主题排列信息。我们也喜欢在一个地方满足尽可能多的需求——既简单又节省时间。

这就是超市和购物中心如此受欢迎的原因。在 Web 应用程序的情况下，用户寻找他们可以遵循的任务，以便实现特定的目标。一系列任务构成一个过程；该过程因不同用户而异。设计者可能无法在设计阶段设想用户可能希望在应用程序中遵循的所有过程。有些情况可能会在稍后阶段发生变化；有些场景在设计阶段甚至可能不存在。因此，询问用户想从一个任务中走到哪里比列出所有可能的过程更容易。如果界面是围绕一个僵硬的过程设计的，用户可能会觉得被束缚住了。html 的一个优点是，用户可以以设计者可能没有预料到的方式快速方便地浏览网站信息。应用程序需要允许用户根据需要构建跨越一组服务和内容的个人流程。

过程成功地将用户引向目标，施加最小的认知负荷，然而它们限制了创造性用户的有趣工作。新手用户喜欢遵循默认流程中的任务。通过严格的导航系统连接的任务对他们来说更容易使用，尽管这些相同的系统经常限制高级用户。为了不扼杀用户的创造力，基于任务的界面应该松散连接。从一个任务导航到另一个任务，通过这些任务所属的多个方面，非常适合高级用户。设计不良的界面更有可能是围绕特定的功能或内部结构来组织的，让用户去弄清楚哪个功能序列将完成他们特定的任务。面向任务的界面通过设置任务并通过上下文导航将它们联系起来，引导用户完成一系列操作。

通过有效的信息架构，用户也可以被导向理想的过程。收集到的导航数据还可以用于引导用户使用其他人已经使用过的流程。微软的任务窗格是在不同任务之间提供上下文导航的一个很好的例子。

将应用程序分成基于任务的界面，并给每个任务一个 URL，这样就可以从 Web 链接到它们。从 Web 链接到应用程序中的各种任务将使用户很容易找到应用程序。

***用户讨厌阅读***

一张图胜过千言万语。如果您能以图表的形式查看相同的信息，您还会费力地阅读一页页的表格数据吗？以文本形式阅读信息需要我们的大脑将数据视觉化。用图形表示这些信息可以减少这种工作。

人们擅长扫描，我们识别和记忆图像比文字更容易。图形元素更易于比较长度、形状、方向和纹理。动画可以显示时间的变化。颜色帮助我们区分。图像比文本更有效地帮助我们实现这些目标。

对于大多数人来说，图形元素比文本更容易理解，但是一些残疾用户在评估图形元素时有问题。计算机也不能很好的理解图形数据。

文本是一种可以通用解释的格式。因此，从服务器传到接口的数据如果是文本的，会更有用。界面应该以一种所有用户都可以访问的方式呈现这些数据。

***用户痛恨安全漏洞***

虽然我们很乐意与一些人分享一些信息，但我们更喜欢将一些信息保密。无意中让其他程序访问私有数据可能会带来灾难性的后果。例如，允许任何程序访问用户的 Microsoft Outlook 地址簿将会打开一个蠕虫的罐子。

用户不喜欢输入他们的信息，他们更喜欢信息被自动收集。开发人员必须能够控制这些数据的共享，以防止安全漏洞。在共享用户的个人信息或其他收集的数据之前，应征求用户的同意。列出用户允许网站与其他人分享信息可能带来的问题和好处，将有助于用户做出决定。如果用户允许共享数据，那么其他各方应该可以以标准格式访问这些数据。

默认的安全设置非常重要，因为很少有用户会费心去检查它们。从严格的安全策略开始并根据需要放松它总是比试图从宽松的策略转向严格的策略要好。在实施之前，应考虑所有新功能和收集的数据的安全和隐私影响。

***用户讨厌单调的外观***

单调的界面很无聊。人类希望在我们所属的群体中脱颖而出，希望将我们的身份洒在我们使用的东西上；我们喜欢我们的物品来反映我们的个性。背景、皮肤和声音是让用户能够改变他们使用的应用程序外观的几种方式。设置背景和声音的简单偏好将满足大多数用户。

但是，要小心！虽然用户喜欢新的外观，但他们不喜欢学习新的交互，这就是为什么最好保持界面元素在视觉上与交互一致的原因——这样，如果用户向您的应用程序应用新的皮肤，他们就不必重新学习界面。

***用户讨厌平台限制***

我们讨厌那些阻止我们做我们想做的事的人。法律的制定是为了防止现实世界中的限制。“让任何人、任何地方都可以访问它！”应该是座右铭。应用程序不应局限于特定的操作系统、外形或用户能力。

用户喜欢学习使用一次应用程序，然后能够在任何地方使用它:在电脑上，在手持设备上，或者在平板电脑上。不同能力的用户也应该能够使用专门的软件访问应用程序。

在选择开发平台时，在不同的操作系统上跨不同的可访问性软件部署应用程序的能力是一个重要的标准。

***用户讨厌死板的功能***

一种尺寸不适合所有人。如果一个工具不完全适合用户，它的功能就会大打折扣。如果只有一个鞋码，世界会怎样？

应该为用户提供添加、删除和修改不同功能的选项。用户不太可能为了让应用程序适合自己而改变所有的偏好。他们希望应用程序是为他们定制的，或者通过跟随他们的动作自动知道他们想要什么。

根据您确定的可能使用该产品的不同角色，可以为用户提供各种预设。如果控制面板链接到用户可能想要进行更改的上下文，则首选项设置会变得更容易。增加字体大小这样的功能对许多用户来说非常有价值。

***用户讨厌错误***

我们讨厌犯错误，但往往没有办法阻止它们。拥有撤销我们行为的能力可以让我们从错误中恢复并从中吸取教训。如果撤销功能出现在现实世界中，那该有多好！

您的应用程序应该跟踪所有用户操作，并使他们能够撤销这些操作。如果应用程序被错误关闭，自动保存所有操作可以防止数据丢失。撤销功能收集的数据还可以帮助用户跟踪他们所遵循的导航路径。跟踪用户动作允许从用户与内容的交互中出现有趣的元数据；通过这一过程，个人利用元数据改进工作方式的方式也变得显而易见。

分离单个用户操作也有助于开发用于重复用户操作的宏。如果某个动作是不可逆的，一条消息说“你真的，真的确定吗？”需要在用户的请求生效之前呈现给用户。最好备份数据，让用户能够撤销他们的操作，而不是用这些信息淹没用户。

在记录操作的过程中，跟踪所有用户操作有巨大的好处。Web 浏览器会通知站点所有者有关引用页面的信息，但是当用户离开站点时，会留下跟踪信息的空白。富互联网应用程序应该能够收集所有用户的动作，这对设计者来说具有巨大的价值，并且有能力决定应用程序的未来道路。

“尽早发布，经常发布”是一个能让用户反馈迅速融入软件的原则。通过在真实用户样本上测试新特性，可以获得更好的反馈。当用户试图学习界面时出现的错误和测量本身的问题可以通过直接测试来最小化。

##### 可用的建筑

尽管我们在软件开发方面有几十年的经验，但是我们还没有解决软件复杂性的秘密。为了开发可用的富互联网应用程序，开发过程必须尽可能简单，同时不忽略任何上述可用性限制。

模式为架构师提供了一个存储他们所学到的经验的系统，从而使他们变得更加容易。大多数软件模式帮助设计者从认知上理解其他人遇到的问题，并避开它们。一个既能降低程序员的复杂性，又不会让他们失去对用户的了解的模式将会给应用程序设计者带来好处。

要设计一个复杂的结构，一个强有力的技术是找到可行的方法，将它分解成与许多功能部分相对应的半独立组件。然后，每个部分的设计可以在一定程度上独立进行，因为每个部分都将通过其功能在很大程度上影响其他部分，并且独立于完成该功能的机制的细节。

当应用程序被分割成相互依赖程度较低的部分时，它的复杂性就会降低。这种分成可管理部分的方法也将使应用程序能够进化，而不是革命，因为这种方法允许进行简单的小的改变。

现在的问题是:我们如何拆分富互联网应用程序，以便在上述可用性约束下进行设计变得更简单？将应用程序分成页面和导航会有所帮助。每个页面都由数据、组件、逻辑和表示组成。页面被封装在导航外壳中，导航外壳也有数据、组件、逻辑和表示。

***页面***

页面是数据、组件、逻辑和表现的集合。一起执行的所有交互被分组以形成一个任务。该页面封装了所有这些交互。用户通过不同的视图可视化和修改页面上的各种对象来进行交互。在一个视图中修改对象可以立即反映在页面的其他视图中，因为富互联网应用程序处理的是对象的数据模型，而不是对象的数据视图。

本地反映数据模型也将支持离线交互。例如，表格数据对象可以在不同的视图中显示。包含所有列的表格、包含列的最大值和最小值的滑块以及包含列的值的勾选框是用户可能与之交互的表格数据对象的一些视图。调整滑块视图应该会实时更新表格视图。当用户连接到网络时，可以将对数据所做的修改发送到服务器。

将导航与页面分离并不意味着页面不应该包含任何到其他页面的链接。如果页面符合默认流程，导航应该集成到其中，以帮助新手用户。简洁明了的页面很容易学习。简单性可以通过减少界面中元素的数量和将它们分成组来实现。交互架构师负责基于任务的页面设计。

***导航***

导航提供从给定页面到各种其他页面的链接，这取决于用户原始位置的上下文。它是页面中导航数据、导航逻辑、导航表示和导航组件的集合。导航数据将一个页面和其他页面之间的关系组织成不同的方面。

像 Xfml(可交换刻面元数据语言)这样的标准格式可以用来存储导航数据。其他数据，例如用户遵循的导航路径，也可以在导航中使用。

拥有一致的导航布局有助于更轻松地浏览网站页面。对相关链接进行分类或减少链接的数量可以简化导航。给页面一个 URL，并使它们可以从网上访问，这将使应用程序可被找到。

信息架构师负责各种页面的导航布局和上下文链接。

***数据***

在富互联网应用程序中，数据是从服务器分离出来的，或者是在客户端单独收集的。一些不同类型的数据包括应用数据、偏好数据、日志数据和导航数据。

有些数据只存在于客户端，但是在服务器上反映所有数据为用户提供了跨各种设备的一致体验。服务器端的 Web 服务根据收到的请求处理发送应用程序数据的复杂性。限制应用程序之间的共享数据访问并有效地管理它将有助于网站所有者维护安全性。此外，使用标准格式存储数据将实现应用程序之间的互操作性。

***逻辑*T3】**

逻辑是描述应用程序行为的代码。它连接组件、表示和数据。面向对象的原则可以用来降低逻辑的复杂性。应用程序程序员负责进入应用程序的逻辑。

***组件***

组件是界面的基本构件。一些类型的组件是交互组件、数据加载组件、导航组件和代理/警报组件。

交互组件提供可视化，并从用户处获取特定数据类型的交互。将组件从逻辑、数据表示和结构中分离出来将有助于快速开发，并为应用程序设计人员提供实现可访问用户界面的灵活性。组件应该接受文本，并向所有用户提供可访问的可视化。

提供各种输入相同信息的方法有助于用户与应用程序进行交互，而无需改变他们的用户模型。支持自动完成的组件也减少了用户的工作量。

确保组件的可学性和可扫描性使得新手和高级用户都能轻松使用它们。组件应该能够记录用户操作，并接受操作以启用撤销功能。

真实世界的隐喻和启示使用户能够与软件互动，而无需学习新的东西。可以使用各种隐喻来简化与组件的交互。

Web 使用了旅行的比喻，用户点击对象进入新的页面执行操作。桌面应用程序使用车间隐喻，在这里我们有许多工具来处理不同的可选对象。在形态图形系统中，任务在选择对象后出现在对象本身上，这属于检查隐喻。遵循一致隐喻的一组行业范围的标准组件将使用户有可能学习一次并在任何地方使用该知识。

数据加载组件管理客户端和服务器端的数据。它们向页面提供进度更新，向交互组件发送部分数据下载，并提供无缝的在线和离线数据访问。导航组件封装了应用程序中的页面和导航信息。警报或代理组件在后台运行，并向用户发出不同类型的警报。

***简报***

表示处理组件的可视显示。组件的位置、大小、颜色和背景由演示文稿定义。

通过将表示从应用程序中分离出来，鼓励相似类型元素的一致位置。在网页中使用 CSS 带来的大部分好处在这里也适用。视觉设计师负责应用程序的表现部分。

##### 该模式的优势

总之，该模式为应用程序设计人员和开发人员提供了一系列好处，帮助他们开发可用的富互联网应用程序。模式…

1.  根据专家领域分解任务，允许半并行应用程序开发
2.  允许开发人员在可用性限制内工作
3.  将组件从软件中分离出来，实现更快的开发
4.  创建一个页面独立于导航的环境，它支持实时流程
5.  使演示独立于页面，鼓励一致的页面布局
6.  由于降低了复杂性，为应用程序提供了更大的可伸缩性
7.  通过将应用程序分成不同的部分，使维护变得更加容易

即使当前的工具不支持这种模式，以这种方式考虑应用程序也是有利的，因为它提供了一些好处:它提供了灵活性，给领域专家以独立性，并且简化了问题而不会失去可用性约束。

多种选择的可用性有时会让新设计师不知所措，并增加复杂性。通过设计模式引入的限制孕育了设计约束下的创造力。

模式有助于降低复杂性，使给定问题更容易理解。然而，设计模式不是解决任何问题的灵丹妙药或灵丹妙药，只有在真正需要它所提供的灵活性时，才应该应用它。
