# 六个月后:新的 HTML 工作组

> 原文：<https://www.sitepoint.com/six-months-later-the-new-html-working-group/>

以下内容转载自《科技时报》第 164 期。

因为我只是没有收到足够的电子邮件(哈！)，上个月我加入了 W3C 新的 [HTML 工作组](https://www.w3.org/html/wg/)。

大约六个月前，蒂姆·伯纳斯·李[宣布](https://www.sitepoint.com/htmls-uncertain-future/)W3C 将成立一个新的工作组来开发 HTML 规范的下一个版本，同时[再次努力](https://www.w3.org/MarkUp/)完成 XHTML 2.0。

随着新工作组的顺利开展，这篇博客文章将着眼于已经取得的进展，已经出现的问题，以及我们对下一版本 HTML 的期望。

## 新的工作组

与大多数 W3C 工作组不同，新的 HTML 工作组的[章程](https://www.w3.org/2007/03/HTML-WG-charter.html)欢迎公众的监督和参与。任何人都可以[加入工作组](https://www.w3.org/2004/01/pp-impl/40318/instructions)，张贴到[邮件列表](http://lists.w3.org/Archives/Public/public-html/)，参加电话会议，并投票决定最终规范的内容。

更好的是，没有最低水平的参与，所以如果你有时间做的只是监视邮件列表和对你感兴趣的问题投票，那也没问题。

工作组目前由共同主席克里斯·威尔逊(微软/Internet Explorer)和丹·康诺利(W3C)幽默地领导着，他们尽最大努力从工作组成员每周发布的大约 1000 封电子邮件中挤出共识。

参与该工作组的还有来自 Mozilla 基金会、苹果 Safari 团队、Opera 浏览器团队的代表，以及来自 Web 超文本应用技术(WHAT)工作组的熟悉面孔，该工作组在新的 W3C 工作组成立之前承担了自行更新 HTML 的任务。

按照官方说法，该小组的第一个目标是在 6 月底之前产生一个某种描述的工作草案(即使只是进一步开发的路线图)。这是在 2010 年底完成规范的第一步。为此，该小组需要一名或多名编辑来编辑和维护该小组制作的文档。

**一个惊喜的提议**

然而，在搜索开始之前，Mozilla、Apple 和 Opera 的代表提出了一个提案[来采用 WHAT 工作组的 HTML5 草案规范，作为在 W3C 内部进一步开发 HTML 的起点。](http://lists.w3.org/Archives/Public/public-html/2007Apr/0429.html)

经过大量的讨论后，W3C 的 HTML WG 今天[投票接受了提案](http://lists.w3.org/Archives/Public/public-html/2007May/0909.html)，结果如下:

*   什么工作组的 HTML5 ( [Web 应用 1.0](http://www.whatwg.org/specs/web-apps/current-work/) 和 [Web 表单 2.0](http://www.whatwg.org/specs/web-forms/current-work/) )将成为当前的工作草案，新工作组现在将进行广泛的审查。
*   最终的 W3C 规范将被命名为“HTML 5”。
*   W3C 规范将由 WHAT-WG 的 HTML5 编辑伊恩·希克森(谷歌)和大卫·凯悦(苹果/Safari)编辑。

我们看到了:主要浏览器供应商和 W3C 之间有害的分歧似乎已经成为过去！到目前为止一切顺利，对吧？

当然，未来仍有巨大的挑战，尤其是让这个庞大而开放的工作组就似乎无穷无尽的技术细节达成一致。

## 摆在桌面上的 HTML 5 问题

正如你对公共团体的期望一样，W3C 的新 HTML 工作组是一个嘈杂的地方，它对任何对 web 标准有热情的人开放。尽管世卫组织工作组的 HTML5 相对容易地获得通过，但已经有一些有分歧的问题摆在桌面上。

正如受人尊敬的 web 标准博客作者 Roger Johansson [在本周早些时候](http://www.456bereastreet.com/archive/200705/help_keep_accessibility_and_semantics_in_html/)所写的那样，提案经常被发布在工作组的邮件列表上，这会让一个了解标准的开发人员吓出一身冷汗。最近的一个例子是一个非常严肃的关于`<indent>`标签的提议，其论点是[你不必学习 CSS 来缩进 HTML](http://lists.w3.org/Archives/Public/public-html/2007Apr/1620.html) 中的东西。

以下是工作组目前正在讨论的几个问题…

**破网**

我希望本博客的大多数读者会同意 Internet Explorer 7 是对 IE6 的一个明显的改进，并且使我们的网站与新浏览器兼容所需的微小变化是为这一改进付出的小小代价。克里斯·威尔逊在微软的老板不一定同意。

在给工作组的一篇冗长的帖子中，Wilson 解释了开发一个上亿用户依赖的浏览器的现实情况:

> 事实是，当一个主要的浏览器发布时，这是网络上的一个奇点；它有能力，但不能被允许，造成大范围的破坏。[……]IE7 确实造成了广泛的破坏，这是一个恰当的例子。我支持进行这些广泛的变革，以提高我们的标准合规性。严肃地说，我设法保住了我的工作，但有时我觉得只是勉强维持。我不能对我的团队说“嘿，我们要再次破坏网络(一次又一次)，但这没关系，因为这是一个好的原因。”世界不是那样运转的。我不会负责任地做我的工作——五亿网络用户依赖我的团队不兼容他们的银行网站，即使他们的银行不知道如何正确使用 CSS“浮动”。"

因此，他接着说，Internet Explorer 8 对 HTML 内容的默认呈现方式几乎没有什么改变。为了利用新功能和标准兼容修复，开发人员必须在他们的代码中包含一个“开关”，告诉浏览器使用其新的、不向后兼容的呈现模式。

这种情况以前在 Internet Explorer 5.5 中发生过一次，当它在 HTML 文件的顶部看到一个更新的 DOCTYPE 声明时，它会切换到“标准遵从模式”。然而，由于微软并不期望在一个版本中添加对所有 HTML 5 的支持，IE8 中的新开关有望更加精细。

这种类型的开关应该在 HTML 规范中定义吗？工作组中的辩论激烈进行着。

**表象元素的命运**

`<b>`和`<i>`似乎是不死的表象标签。虽然它们的同类`<tt>`、`<u>`和其他标签在 HTML 4 中被弃用，取而代之的是语义上有意义的标签，如`<strong>`和`<em>`、`<b>`和`<i>`仍然是“信誉良好”的标签，直到并包括当前的 HTML 5 草案。

根据现在参与 W3C 工作组的几个 WHAT 工作组成员的说法，保留这些标签的理由是，如果它们被删除，天真的内容作者将开始误用语义上有意义的标签。例如，寻找“斜体标签”的人在发现 HTML 5 中不再允许使用`<i>`时，可能会开始使用`<em>`将斜体应用于文本，这将破坏该标签目前享有的相对较强的语义。

同样的逻辑导致了在 WHAT 工作组的 HTML 5 草案规范中[重新包含了`<font>`标签](http://www.whatwg.org/specs/web-apps/current-work/#the-font)。是的，真的！当然，标签的描述清楚地表明，使用它通常是一个坏主意:

> `font`元素不代表任何东西。除了 WYSIWYG 编辑器之外，不能使用它，因为它们可能会使用它来实现演示效果。然而，即使是 WYSIWYG 编辑器也应该尽一切努力使用适当的语义标记，避免使用特定于媒体的表示标记。

在这个阶段，我期望，如果它们*确实*保留在最终的 HTML 5 规范中，所有的表示元素都将以类似的术语描述。

**如何处理损坏/废弃的标记**

大多数 web 开发人员没有意识到的一点是，W3C 的 HTML 4 规范没有很好地描述浏览器在遇到错误代码时应该做什么。这是一个典型的例子:

```
<strong>This line <em>contains</strong> some words</em>
```

在 XHTML 中，这种类型的重叠标记结构是完全非法的，当前支持 XHTML 的浏览器只会显示一条错误消息，而不会试图理解这段代码。

与此同时，HTML 采取了更温和的立场，仅仅声明对上述结构的解释取决于个人浏览器。

在现实世界中，像这样的变幻莫测意味着市场上的每一个新浏览器都必须通过对 Internet Explorer 等主要竞争对手进行艰苦的逆向工程来弄清楚预期的行为。

除了扩展和增强 HTML 的功能，HTML 5 规范还旨在填补这些空白，以便第一次有一个完整的规范来解释今天的 web 内容。

从这个意义上说，HTML 5 规范服务于两个主人——想要完整描述今天的浏览器特性的浏览器供应商，以及想要更好的语言来构建未来网站的开发者。

有些人会说，为网络的未来打下坚实基础的最佳方式是抛弃混乱的 HTML，从头开始构建一种新的标记语言。这正是 XHTML2 工作组正在做的事情，但是由于浏览器供应商的支持少得多，许多人认为它的工作在未来许多年都没有实用价值。

**扩展 HTML 语义**

目前，`class`属性通常用于扩展 HTML 文档中可能表示的语义范围。例如，HTML 没有一本书的标题标签，但是你可以在你所有的文档中使用`<span class="booktitle">`,并用 CSS 对它们进行适当的样式化。

为了标准化少数常见的类名，以便浏览器和辅助技术可以使用它们来从 HTML 文档中推断出更大的含义，WHAT 工作组提出了以下预定义的类:`copyright`、`error`、`example`、`issue`、`note`、`search`和`warning`。

自然，这些类名中的许多已经在今天的 Web 上使用了。WHAT 工作组的希望是，在使用它们*的地方，它们的使用将与 HTML 5 规范中定义的含义一致。*

新的 HTML 工作组中并不是每个人都同意这一点。一些人认为新规范不应该突然为一个属性定义保留值，这个属性以前是开放给内容作者任意使用的。[提议的替代方案](http://lists.w3.org/Archives/Public/public-html/2007May/0873.html)包括以曾经为`target`属性(例如`class="_copyright"`)所做的相同方式为预定义的类名添加前缀，或者采用 XHTML 2.0 的`role`属性的变体来与`class`并列。

如您所见，构建下一个版本的“简单”标记语言并不简单。但是随着 HTML 工作组向所有人开放，将您的想法呈现给重要的人变得前所未有的简单。

如果我上面讨论的任何问题对你很重要，我鼓励你[加入](https://www.w3.org/2004/01/pp-impl/40318/instructions)并说出你的想法！
