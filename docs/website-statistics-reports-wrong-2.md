# 为什么你的网站统计报告是错误的，第二部分

> 原文：<https://www.sitepoint.com/website-statistics-reports-wrong-2/>

在我的第一篇文章中，我们分析了[服务器端数据收集和分析](https://www.sitepoint.com/website-statistics-reports-wrong-1/)的优缺点。尽管生成的报告很有用，但它们也有缺陷。今天，我们来看看另一种选择…

## 客户端数据收集和分析

客户端数据收集需要在想要分析的每个页面上插入图像或 JavaScript 代码。图像计数器的受欢迎程度已经下降，因为它们只能提供基本的页面点击信息。JavaScript 代码可以实现更多，并且被最流行的统计报告系统之一 Google Analytics 所使用。

每当页面加载到您的浏览器中时，JavaScript 就会运行，因此它可以克服[服务器端数据收集](https://www.sitepoint.com/website-statistics-reports-wrong-1/)遇到的许多缓存问题。该代码还可以收集更详细的客户端信息，例如在页面上花费的时间、鼠标活动、点击的链接、屏幕分辨率、颜色深度、浏览器窗口大小、安装的插件等。Cookies 还可用于识别唯一用户和显示导航路径。最终，JavaScript 代码将数据发送到后端服务器进行处理。

不幸的是，有几个缺点:

**阻止**
可以禁用或阻止 JavaScript 和 cookies。一般来说，大约有 5%的人不运行 JavaScript，但这将因网站而异。同样，搜索引擎机器人不运行 JavaScript，所以报告永远不会告诉你你的网站是否或何时被索引。这份报告可能会考虑到遗漏的用户，但这只是猜测。

JavaScript 是脆弱的:如果页面上的另一个脚本导致致命错误，它可能会阻止数据整理。也许更糟糕的是，您可能有一个脚本在一个浏览器中失败；如果您的报告显示没有 Internet Explorer 用户，是因为他们从未被记录吗？

**页面问题**
JavaScript 几乎可以分析用户页面交互的任何方面，但代码可能会导致其他脚本中断或运行缓慢。因此，大多数系统采取保守的方法，只在页面加载后不久记录基本细节。

**仅 HTML 页面**
只能分析返回 HTML 的网页——系统不能记录 CSS、图像、MP3 或 PDF 文件访问。Google Analytics 允许你在文件下载链接中添加点击处理程序，但不能保证用户会点击它们(他们可以通过电子邮件收到文件的 URL)。

**人为错误**
脚本代码必须手动添加到你想要监控的每一个页面。如果你不小心漏掉了一页，它将永远不会被记录在你的报告中。

**无历史数据**
只能从添加脚本时开始记录统计数据:不能生成历史报告。

**安全风险**
链接第三方 JavaScript 代码有风险。在收集私人数据的安全页面上使用这样的系统当然是不明智的。

尽管许多客户端分析器生成了外观精美、信息丰富的报告，但数据收集过程本质上比服务器端方法更不稳定。
