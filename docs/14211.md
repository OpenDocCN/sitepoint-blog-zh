# Linux 系统管理员的时间表

> 原文:[https://www . site point . com/Linux-system-administrator s-schedule/](https://www.sitepoint.com/linux-system-administrators-schedule/)

我最近关于 Linux 系统安全性基准测试的文章引发了一个关于我维护 Linux 系统健康的过程的问题。除了基于一些客户的特殊需求的一些具体程序，我还记下了一些奇怪的任务，然后将我的日常事务分解成每日、每周和每月的时间表。

我偶尔会做一些事情，比如季度用户审计、管理存储目录清理(下载、补丁、零碎东西)和季度备份恢复测试。对于后者，每个管理员每年至少应该执行两次，也就是测试从磁带、磁盘或在线存储进行恢复，以确保您的备份过程正常运行。

最后一项季度任务是从备份服务器中清除旧的 Apache 访问日志，如果适用的话，因为其中一些日志占用了一个又一个千兆字节的空间。

作为一些背景，我管理的大多数 Linux 服务器并不都在我自己的数据中心，也不是来自一个客户机，而是分散在不同的环境中。我仍然设法维护有限数量的操作系统构建和配置，这样我就不会像系统管理员一样制造混乱。我还花了大部分时间通过 iptables 加固 Linux 机器，设置详细的日志记录，并在部署之初收紧 Apache、SSH、邮件服务器、ftp 等应用程序。

我的日常工作非常简单，包括对 iptables、系统日志、邮件、web、ftp 和 ssh 的粗略日志审查。我通常会查询所有邮件服务器的队列(通常是垃圾邮件，但偶尔也会有合法的丢失邮件)。在 qmail 中，通过命令行上的 qmailctl 和一个显示队列中邮件头的方便的 Perl 脚本，这是非常容易的。

除了在大多数生产 Linux 服务器上的 cron 中运行的预期日常任务之外，我还检查了一些定制的 cron 作业，这些作业通过 rsync 处理热在线备份、一些邮件任务、特定于客户端的脚本、带宽使用测量和 Tripwire 日志。

每周我都会抽出一些时间再次检查一些系统和客户端日志，以确保一切运行顺利，大量的自动数据传输和浏览我自己的 SSL 证书、域名和托管合同日期数据库。这是每周的窗口，我也花时间深入调查任何片状日志条目，如渗透企图和其他可能的恶意攻击企图。如果有任何实质性的发现，我会报告给相应的 IP 来源的 ISP。

我还每周留出一段时间来查看收到的各种安全和漏洞电子邮件，以及我监控的 RSS 源或网站(当然除了我根据需要回复的红色警报电子邮件)。

最后，我每个月都会确保日志在我管理的机器上循环，并仔细查看备份日志。最后，我通常会运行一些实用程序来监控磁盘空间，检查上个月不同时间的“top”和“mytop”日志记录，可能还会运行 rootkit 检查。

当然，这并不完全包括所有的任务，因为有专有的客户端特定的任务，但反映了保持 Linux 服务器的健康需要正确地构建它，了解正在发生的事情并保持系统最新。

如果有灵活性，最理想的是至少有一个开发 Linux 服务器，这样您就可以为您的环境运行补丁、更新等快速测试(我将一个生产机器复制到运行 10.5.x.x 非公共 IP 的内部 Linux 服务器上，并根据自己的需要进行测试)。这使您能够对紧急更新做出快速反应，并且仍然进行测试，因此不会中断任何生产服务。

也许最重要的是，我每月的最后一项活动是发送发票！；>)
