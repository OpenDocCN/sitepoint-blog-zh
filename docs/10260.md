# 网络应该为声音连线吗？

> 原文:[https://www.sitepoint.com/website-sound-effects/](https://www.sitepoint.com/website-sound-effects/)

网络和音频的关系有点尴尬。如果你(不)足够幸运，还记得 20 世纪 90 年代末的网络，当页面加载时，其他所有网站都播放一段糟糕透顶的 midi 音乐。这是令人震惊的——尤其是在配有破烂声卡的商用电脑上。

网站声音最终被关闭，因为:

*   Internet Explorer 的垄断结束了。其他浏览器并不总是支持。WAV 还是。MID 文件，因此闪存成为最可靠的替代方案，这也带来了一系列问题。
*   开发者意识到声音不一定能改善他们的网站。这些效果连同框架、字幕、闪烁标签、闪屏和愚蠢的动画“页面正在构建”gif 一起被丢弃。

今天，开发人员很少考虑声音，除非他们正在创建一个音乐或游戏网站。但我们都在使用基于云的应用程序，声音长期以来一直是桌面的可用性特征。当日历事件发生时，用户会喜欢快速的“奉”声吗？当新推文到达时，用户会喜欢“唧唧”声吗？当用户在 Google Docs 中完成重要的电子表格时，用户会喜欢互联网连接在关键时刻中断时的“警报声”吗？

我认为声音将会在网络上卷土重来——可能就在今年。然而，要让它发生:

1.  HTML5 < `audio` >标签需要被广泛采用。很少有开发人员会考虑应用程序的音效，除非它们在大多数浏览器中容易实现并且可靠——包括 Internet Explorer。
2.  声音需要不引人注目并有助于使用。开发人员应该谨慎使用它:偶尔的电子邮件“tah-da”提醒是好的，但我们不要回到页面加载协奏曲或每次点击都发出“咯咯声”的时代。
3.  Web 应用程序开发人员应该提供一个声音开/关选项——可能默认为“关”。
4.  事件发生时，声音必须立即播放。在必要的地方，应用程序应该预先缓存效果，以便它们可以按需播放。Remy Sharp 已经调查了这个问题:参见[音频精灵](http://remysharp.com/2010/12/23/audio-sprites/) —一种在单个音频文件中加载多种效果的技术。
5.  供应商应该在浏览器中实现智能声音控制。音量控制或“禁用声音”选项将是一个良好的开端。它可以走得更远，例如，不活动的标签可以根据页面最近被查看的时间播放模糊或较低的音量效果。

我承认我不是预测网络声音未来的最佳人选。我关闭了所有的桌面和应用程序效果，因为当你一天使用电脑 10 个小时时，它们很快就会变得令人讨厌。但是，根据我的经验，大多数用户似乎喜欢音效*(或者不知道如何禁用它们！)*

Web 应用程序的效果甚至可能比桌面应用程序更有用，更少干扰。与我的电子邮件客户端、IDE 或 Photoshop 不同，我很少整天开着一个网络应用。
