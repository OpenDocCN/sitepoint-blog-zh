# 为什么更大的团队不会导致更快的开发

> 原文：<https://www.sitepoint.com/larger-teams-do-not-develop-faster/>

软件公司可能犯的最大错误之一是假设更多的开发人员将导致更快的产品发布。并没有。这起初似乎不合逻辑，然而，收益递减规律适用于大多数行业。有句老话说得对，厨师太多了…

软件开发是一个不寻常的复杂过程。与其他形式的制造不同，有无限多种多样的结果，过程的每个部分都是集成的。看似困难的过程可以快速简单地实现，而简单的功能可能隐藏着复杂性。

然而，一个主要原因是更多的开发商！=更快的开发是更大的团队增加了沟通的复杂度。创建或修改组件时，它会影响项目的其他部分。例如，假设开发人员 A 正在编写一个数据库连接类，并决定做一个简单的更改——比如交换两个参数的顺序。

*   在一个团队中，开发人员可以不与任何人交流就做出改变。
*   在两人小组中，开发人员必须通知另一个人。
*   在一个 3 人小组中，有两个人需要通知。

因此，在一个有 N 个成员的团队中，一个开发人员必须联系(N-1)个其他人来讨论变更。由于任何团队成员都可以在任何时候做出改变，因此必须有 N * (N-1)条沟通渠道。然而，由于开发人员 A 与开发人员 B 之间的交流等同于开发人员 B 与 A 之间的交流，因此我们可以将总数一分为二。我们最后的等式:

c = N(N–1)/2

其中 N 是团队成员的数量，C 是交流的总次数。

不幸的是，每增加一个成员，这些交流渠道就会成倍增加。一个 5 人团队，有 10 条线。10 人团队有 45 条线。一个 20 人的团队有 190 行。一个 100 人的团队有 4950 行。

你如何运作你的沟通渠道并不重要。每个开发人员必须决定一个变化是否影响他们的代码，并相应地实现更新。软件开发的复杂性又引发了其他问题:

*   每个人都收到并理解变化了吗？
*   一些团队成员需要进一步的帮助吗？
*   开发人员 A 需要向其他团队成员证明更新的合理性吗？
*   变更是否已经在所有系统中测试过？
*   这种变化会导致其他组件的级联变化吗？

不仅仅是代码变化。任何项目决策都必须分散到整个团队中。

复杂的项目需要时间，在超负荷的时间表上投入资源不一定有帮助。你可能需要接受这个时间表从第一天开始就不现实。
