# 从来没遇到过喜欢的盒装 CMS

> 原文：<https://www.sitepoint.com/ive-never-met-a-boxed-cms-i-like/>

好了，伙计们，是时候从最近的 Sql Server 事件中休息一下了。目前，我正在帮助我们的一个子组织推出他们的新网站。一个建立在精心设计的成熟开源 CMS 上的网站。他们对它的重要部分深恶痛绝，甚至对许多功能都不屑一顾，因为它们在应用程序中完全没有必要。当开发人员在他们自己的定制上合并项目主干的变更时，不要介意一系列有趣的初期问题。他们的苦难让我想起了几年前我想通的一件事:盒装 CMS 应用程序是噩梦，最好避开，而不是拥抱。

首先，为了清楚起见，让我定义一下“盒装 CMS”。它是一个功能齐全、功能丰富的内容管理系统，可能是从 sourceforge 下载的 zip 文件。通常包括菜单、文章、会员区、安全系统、RSS 源、图片库和论坛。95%的时候，你只需要上传文件到服务器，运行一个数据库脚本，然后添加内容就可以得到一个功能齐全的网站。实际来源——许可的商业应用程序或下载的开源应用程序并不重要。无论如何，它可能比你需要的功能多了半打。

现在，你可能会说“哇，怀亚特！当我可以下载并安装它，然后使用一些配置设置和模板进行 futz 时，我为什么要构建所有这些管道呢？”—一个非常合理的问题，值得一些非常合理的答案。

第一个问题是，CMS 的本质是不容易装箱的，除非创建一个试图为每个人做所有事情的应用程序，并且在大多数事情上做得特别好。内容管理所需的任务是一般性的，但是每个组织在应该如何管理该内容以及如何看待该内容方面都有不同的侧重点。我花了几天时间开会，试图帮助主题专家理解，根据这个系统，一篇文章实际上是一页。试图为所有人开发一个通用的应用程序来处理这个问题是一件非常非常棘手的事情。

因此，最终结果是我还没有看到一个最终用户真正喜欢使用现成的/sourceforge CMS。一种方法是不变地实现一些变通方法，比如说，让一个博客发布系统像一个文章发布系统一样工作。或者使用横幅广告管理系统来处理侧栏内容。几乎不变的是，人们必须在良好的信息架构和系统想要处理事情的方式之间做出妥协。损害您的信息体系结构从来都不是一件好事。

另一个主要问题是，我从未见过生产实现中的“盒装 CMS”仅限于更改配置设置和/或模板。总有一些或许多需求推动着重要的代码扩展和/或修改。一旦你承诺对系统进行重大修改，你就会陷入真正的噩梦。

首先，当一个人钻研未知领域时，搞乱核心组件很容易破坏 CMS 本身的其他东西。但是这些编码问题在大多数情况下是可以克服的。这与大多数潜入他人代码库的经历非常相似——很少有趣，但通常可行。不幸的是，事情还没完。

对我来说，最可怕的部分是，在对这个应用程序(众所周知的方钉)进行重大修改以适应需求(众所周知的圆孔)之后，您有了一个奇怪的混合应用程序。大部分是遗留代码，带有一些关键任务自定义位。因为那些自定义位存在于遗留系统中，所以它们对所述系统有很大的依赖性。然后，盒装 CMS 的控制方——无论是供应商还是开源项目——发布一个更新。而且，这个更新经常会破坏您的更改，使您的应用程序实现变得无用。

对于我们这些负责我们所控制的机器的安全的人来说，加剧这一噩梦的是，许多这些更新包含关键的安全补丁，通常带有恶意攻击。这意味着您的定制版盒装 CMS 现在对组织来说是一个重大的安全风险，因为您为了适应应用程序的要求而打破了更新周期。这是一个棘手的问题。你现在被迫要么破坏你的网站，做一个匆忙的修补工作，使你的改变与新制度兼容，要么只是愉快地希望没有人注意到这个安全漏洞。

在处理了相当多的盒装 CMS 实现后，我的网络生活经验是避免它们——除非你清楚自己实际上会呆在盒子里。
