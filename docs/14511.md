# Gmail 是怎么回事？

> 原文:[https://www.sitepoint.com/whats-the-deal-with-gmail/](https://www.sitepoint.com/whats-the-deal-with-gmail/)

如果你在过去的几个月里曾经连接过网络，你一定不会错过围绕谷歌新推出的 1 GB 网络邮件服务 Gmail T1 的大肆宣传。Gmail 的邀请制政策创造了一种大多数营销人员只能梦想的轰动效应，这比让他们的搜索引擎一炮而红的口碑营销活动更进一步。但是除了存储空间的噱头和它是谷歌的产品这一事实，你为什么要关心 Gmail 呢？

答案是，它颠覆了许多关于 web 应用程序接口的先入之见。

Gmail 光滑、流畅，速度非常非常快。它显示消息和在不同视图之间切换的速度可以与桌面邮件客户端相媲美，并且许多功能(创新的线程和出色的搜索功能)都超越了常规应用程序。这是一个网络应用程序，在很大程度上比它的桌面版要好。

从技术角度来看，性能提升是通过使用一个特别聪明的 JavaScript 技巧实现的。第一次访问该网站时，Gmail 应用程序的大部分内容会以隐藏框架的形式加载到内存中。从那时起，电子邮件、线程列表和其他视图作为超轻量 JavaScript 数据结构从服务器加载。从用户的角度来看，带宽使用率极低，宽带上的响应时间几乎可以忽略不计。甚至电子邮件地址自动补全(Gmail 难题中特别巧妙的一块)在每一次击键时都会回呼服务器！

可悲的是，这种性能的提高是以沉重的代价为代价的。Gmail 是我见过的最不像 web 的 web 应用:所有的东西都通过同一个 URL 访问，在新窗口中打开链接等大量常见的浏览器操作根本不起作用。由此得出的一个推论是，Gmail 遭遇了糟糕的可访问性，这让我想到了几周前,[问,](https://www.sitepoint.com/blog/)这类应用是否是另类“可访问”版本网站是个坏主意这一规则的例外。

无论如何，超丰富的互联网应用领域正在升温，雅虎！上周五刚刚买断了 Oddpost 的股份。由于 OddPost 只是一个 IE/Windows 事件，所以我自己并没有看过多少 OddPost，但它是一个用 DHTML 实现的几乎像素级完美的 Outlook 克隆，它似乎是 Gmail 的先行者，只需加载一次应用程序脚本，然后使用尽可能少的开销与服务器通信。

这些应用程序对网络的发展有好处吗？我不确定——但是他们肯定会把 Joel Spolsky 的关于 HTML 作为下一个主要应用平台的愿景提前实现。
