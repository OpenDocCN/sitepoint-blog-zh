# OSCON 2007:管理技术债务

> 原文：<https://www.sitepoint.com/oscon-2007-managing-technical-debt/>

Andy Lester 在 Perl 基金会工作，在 petdance.com 开了一个博客。

Andy 的演讲给出了一些实用的建议，帮助你补上所有你推迟到以后某个日期(不可避免地永远不会到来)的任务——比如推迟的文档、修复损坏的测试(或者干脆写测试)、备份制度、代码中的待办事项等等。

他有一个非常简单明了的公式:

*   确认你的债务
*   确定成本
*   支付利润最高的
*   停止招致新的债务
*   必要时重复

对我来说，最重要的一点是，不要试图一次解决所有的问题。你要弄清楚什么样的解决方案会给你带来最大的收益——这不是最容易解决的问题，也不是最大的问题。你解决所有最有利可图的问题。

在这个演讲中也有很多很好的讨论，关于如何说服经理允许你开始通过重构、增加测试覆盖率、添加文档或实际测试你的灾难恢复计划来减少技术债务。最好的建议是用经理能理解的术语表达(通常是用金钱或时间)。

例如，如果管理人员知道投资两天测试灾难恢复可以节省数周的时间来重建关键基础设施，或者花一周的时间重构糟糕的旧代码可以使实现新功能的速度加快 20%，他们就更容易证明成本的合理性。
