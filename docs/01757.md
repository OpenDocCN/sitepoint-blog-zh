# Scrum 工件:完成的定义

> 原文：<https://www.sitepoint.com/scrum-artifacts-definition-of-done/>

![scrumthumb](img/4d90f39acd0923a5e8ec5b5675aeed0a.png)

以下是摘自我们的书《Scrum:新手到忍者，作者是 M. David Green。世界各地的商店都有出售，或者你可以在这里买到电子书。

我们已经多次提到过 done 的定义。完成 的 **定义适用于 sprint backlog 中的每个故事。宣布完成一个故事是一种验证其所有关键方面已经基于每个团队的工作方式完成的方法。**

对于一个团队来说，对所做的事情对自己的实际意义有一个共同的想法是至关重要的。团队应该创建一个完成的书面定义，作为他们工作的每个故事的清单。这个定义应该独立于特定故事或项目的文档，因为 done 的定义应该适用于团队的工作方式，独立于他们正在进行的工作。

完成的定义是团队应该一起想出来的。它可能在第一个 sprint 计划期间创建，并在 sprint 回顾期间迭代。团队对完成的定义可能会随着时间的推移而彻底改变，因为团队在回顾过程中意识到过程的哪些方面需要改进。

![ch5-05](img/94e98f025ec8ed9db5baa6f27f307ba4.png)

把 done 的定义公布在显要位置也未尝不可。这使得团队内外的人都能理解完成一个故事需要付出多少努力。这份文件提供了一个参考点，可以用来询问一个被提议的故事是否已经准备好了。如果 story 不可能满足基于验收标准的 done 的定义，发布 done 的定义可以帮助团队明确这一点。

通常“完成”的定义包括许多熟悉的期望。例如，代码应该是完整的，构建不应该失败，测试套件不应该中断，产品应该按预期运行。done 定义中通常包含的其他要点包括同行评审代码、编写新的单元测试、代码注释和更新产品文档。不同的团队会有不同的需求，创建 done 定义的练习对于帮助团队认识到每个故事的重要性是有价值的，而不仅仅是某些故事。

### 注意:完成的定义必须是实际的

虽然对“完成”的定义要彻底，但它的实用性也很重要。对每个故事来说，完成的定义必须是可实现的。为了代码库，团队可能认为应该遵循一些实践，但是考虑到市场的限制，这些实践是不切实际的。通常工程师会主张更彻底的范围，而产品负责人可能会主张更精简的方法，这取决于产品负责人的远见和市场的稳定性。在开始工作之前，每个团队都要提出自己的案例，并给出一个大家都同意的“完成”的定义。
