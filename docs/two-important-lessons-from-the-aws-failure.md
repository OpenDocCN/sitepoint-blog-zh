# 自动气象站失败的两个重要教训

> 原文：<https://www.sitepoint.com/two-important-lessons-from-the-aws-failure/>

上周，亚马逊网络服务经历了云计算服务有史以来最大的一次中断。这一故障引起了连锁反应，许多网站如 [Quora](http://www.quora.com) 和 [HootSuite](http://www.hootsuite.com) 在停电期间完全瘫痪。其他网站，如 Reddit，至少部分受到影响。

失败时有发生，尽管云计算行业还很年轻，但我确信这不会是最后一次，也不会是最大的一次宕机。

但是这一事件(双方)有几个问题可以教会我们如何计划和处理危机。

### 如何在危机中沟通至关重要

亚马逊因 AWS 服务中断而面临批评，这是理所应当的。但我所见过的最严厉的批评不是关于停电时间的长短，也不是一开始就有停电的事实。我读过的最糟糕的批评是关于亚马逊在中断开始后的 40 多分钟内没有任何回应——当你的网站停机时，这是一段永恒的时间。

随着停机的继续，许多客户感到不安的是，这些更新似乎是由法律部门编写的(从获得更新的延迟来看，可能是这样)，而不是由正在解决问题的真实人员编写的。

这里的教训很清楚——当你遇到任何类型的危机时，与受影响的人沟通是极其重要的。在紧急模式下，可能无法拿起电话与客户或顾客通话，但更新您的网站或更改语音邮件消息会产生重大影响。

当你和你的客户就一个问题进行交流时，要诚实和真诚。令人惊讶的是，一点点真诚就能安抚一个心烦意乱的顾客。37 Signals 就是一个很好的例子。当他们的 Basecamp 服务中断时(这是极其罕见的)，[他们用一个详细的解释和富有同情心的道歉来回应](http://productblog.37signals.com/products/2009/11/the-basecamp-outage-slowdown-today.html)，这我们还没有在亚马逊上看到过。

### 有应急计划

当我听说一些非常大的网站由于 AWS 中断而完全瘫痪时，我不禁想知道为什么他们在没有任何冗余或备份计划的情况下建立他们的系统。云计算是一个相对年轻的行业，尽管亚马逊网络服务非常可靠，但失败时有发生。你不会仅仅因为从来没有过硬盘崩溃就停止备份电脑——这只是迟早会发生的事情。

云计算最大的优势之一就是快速的可扩展性。完全有可能建立两个完全独立的云环境，例如一个在 AWS，一个在 Rackspace，并且简单地让一个作为备份，以便在发生故障时(手动或自动)扩展到生产环境。

现在，这对你我意味着什么？

在我的电子商务业务中，我们严重依赖搜索引擎来驱动流量(和收入)。2003 年，谷歌的一次更新导致我们的网站从第一页的顶端下降到第 50 页左右，几乎每一个主要的关键词都出现了。我们的流量(和收入)一夜之间消失了。我们争先恐后地通过搜索营销和其他途径增加流量，但那时已经太晚了。这是我们的旺季，没有顾客。

任何时候你都有可能因为一个单点故障而导致一个项目或服务完全失败，你只是在自找麻烦。我一次又一次地看到这种情况，当公司雇佣一个承包商在紧张的期限内编写应用程序，或者依赖一个客户获得几乎所有的收入。

我们电子商务业务的解决方案是使我们的营销策略多样化。我们仍然从搜索引擎获得大量的流量，但也通过搜索营销、电子邮件营销、社交媒体和其他形式的广告来增加收入。如果另一个更新导致下降，我们肯定会受到影响，但不会是毁灭性的。

无论你是否有几个承包商可以在紧急情况下介入项目，或者努力使你的客户名册多样化，以便你最大的客户不会带来你的大部分收入，你都应该研究如何限制任何一个事件对你的业务的影响。
