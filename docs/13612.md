# OSCON 2006:网络异端:海边框架

> 原文：<https://www.sitepoint.com/oscon-2006-web-heresies-the-seaside-framework/>

本周，凯文·杨克从俄勒冈州波特兰市的 [OSCON 2006](http://conferences.oreillynet.com/os2006/) 发回报道。

Avi Bryant 是少数对 Smalltalk 有浓厚兴趣的网络极客之一，他创造了一个适合他的特殊爱好的 web 应用程序开发框架。那个框架就是[海边](http://www.seaside.st/)。事实证明，Seaside 也打破了许多其他 web 框架所接受的许多传统智慧，而那些“异端邪说”正是 Bryant 在这次演讲中所关注的。

Seaside 打破的第一条规则是，网页设计师负责制作 HTML 代码。许多其他框架不遗余力地提供复杂的模板系统，允许开发人员和设计人员在应用程序的 HTML 上协同工作，而不会踩到彼此的脚趾(理论上)。Seaside 放弃了这种方法，在应用程序中从 Smalltalk 代码生成所有 HTML 代码，其理论基础是，如果生成的 HTML 代码在语义上有意义，那么设计者真正需要做的就是编写样式化 HTML 的 CSS 代码。

这种方法的一个好的副作用是，您不需要为给表单字段分配名称这样的事情而失眠，因为这些可以由框架自动生成，就像大多数其他框架生成会话 id 一样。

说到会话，Seaside 并不麻烦序列化它们，这是大多数其他框架要费很大劲去做的事情。Bryant 认为，序列化会话旨在解决的问题——需要在负载平衡的服务器之间共享会话，以及需要在服务器崩溃时进行故障转移而不会丢失任何活动会话——都不是问题。因此，Seaside 将所有会话存储在内存中。

这又一次带来了自由的结果。会话可以包含一些疯狂的东西，比如闭包，甚至延续——这意味着响应请求的代码可以有效地“暂停自己”，将其状态(执行上下文)存储在会话中，并向浏览器发送响应，提示用户输入一些信息。当下一个带有所需信息的请求到来时，将从会话中检索执行上下文，代码将从它停止的地方继续执行。突然间，多页表单工作流变得微不足道，不仅要实现，而且要修改(比如说，通过重新排列步骤)。

坦率地说，对于像我这样大部分时间都停留在 PHP/Java/随便什么思维模式中的人来说，看到这种东西的运行令人震惊，自然倾向于认为这是前沿的、实验性的东西。但是，在这一点上，Seaside 是已经投入生产使用长达五年的应用程序的核心！

当然，Seaside 基于 Smalltalk，这是一个与主流平台截然不同的开发环境。因此，这不是你可以在几个小时内投入并让它运行起来的事情。但是，从 Seaside 拒绝接受的普通假设中肯定可以学到一些东西。
