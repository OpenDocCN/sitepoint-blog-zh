当业务经理、团队领导和其他人计划业务连续性时，他们是前瞻性的思考者。制定计划是为了确保企业在各种压力情况下继续运营，如自然灾害、死亡、恶意行为等。企业为网站、建筑、人力资源、金融交易、资产搬迁、设备更换等任何事情制定灾难计划。

当考虑网站的灾难计划时，你必须考虑很多这些事情，但你也必须考虑黑客攻击、用户错误和依赖关系失败(想了解如何更好地管理依赖关系？看看 SitePoint Premium 上的[管理依赖关系](https://www.sitepoint.com/premium/screencasts/managing-dependencies)课程。

这些业务连续性和灾难计划的关键要素通常是*流程*。这些详细说明了公司的所有员工，无论是只有一名员工、几十名员工，还是几千名员工，在这些情况下将采取的步骤，以确保业务仍能尽可能接近正常运行。如果企业不能经受住这样的风暴，当危机结束时，他们可能没有公司可以回归。

对于网站和 web 应用程序的维护者来说，灾难计划可能是至关重要的，但由于与业务连续性计划整体相同的原因，它们经常被忽视——它们只在发生不好的事情时有用，而在正常的日常操作发生时，它们似乎是一种浪费。

## 网站的灾难计划

在考虑如何为您维护的网站进行灾难规划时，您应该首先问自己(或您的团队)的一个问题是:

> 您的网站或 web 应用程序可以停机多长时间？

没有顾客参观、互动和购买，你能坚持多久？你的网站每分钟或每小时会损失多少钱，或者会失去多少商机？您是否与您的客户签订了服务水平协议，如果是，是否保证他们一定比例的正常运行时间？你什么时候会达到那个极限？

### 灾难规划

对你的企业来说，有些灾难是史诗般的，区域性的，甚至是全球性的，而有些灾难甚至不会成为当地的新闻。以下是一些可能需要计划和预先考虑的情况示例:

*   地震、龙卷风、飓风或火灾等自然灾害。如果您的数据中心遭到损坏或破坏，会发生什么情况？
*   关闭对您的业务不可或缺的关键服务或产品，如客户关系管理软件、会计软件、社交媒体、版本控制库、托管服务等。
*   撤销对资源的访问。如果某种安全问题，或者某个工作在您的网站或应用程序所使用的服务上的人的人为错误阻止了您的网站访问该服务，该怎么办？如果您使用邮件发送服务，当 API 请求突然失败时，您是否有后备方案？更换服务或进行故障诊断并解决问题需要多长时间？
*   黑客攻击。是的，你希望有适当的安全措施，但是当这些措施失败时，你有计划吗？要恢复的备份，或者检测和删除入侵者、快速修补漏洞、撤销受损用户凭据的方法？
*   关键人员流失([总线测试](https://medium.com/swlh/the-bus-test-8c5e2a2d7507))。失去一名关键工程师、开发人员、设计师或支持专家，你的网站还能生存吗？

### 进行练习

一旦你对上述场景和你能想到的其他场景进行了头脑风暴，你可能还想把你不同团队的关键人员(如果你有他们的话)召集起来，进行一些心理练习。把你自己放入情境中。如果 X 发生了，然后 Y 发生了，我们怎么办？头脑风暴。你可能会在你的计划中发现更多的不足，需要新流程的想法。

### 备份系统

备份是一个关键的考虑因素。不仅备份数据，如文件、数据库、媒体等，还备份凭证(它们是否存储在单个用户意识之外的某个地方？).服务是另一个考虑因素。您是否有各种用途的备份 cdn、邮件服务、NPM 软件包？他们有实施计划吗？任何付费软件或服务的许可证？

在你启动网站之前，最好考虑一下网站的备份策略。检查你的托管公司的备份政策。一些提供额外的备份解决方案以及他们的托管服务。我们的托管合作伙伴 [SiteGround](https://www.siteground.com/go/article-sp) 为日常备份和快速数据恢复提供了强大的内部工具。

### 开发和共享流程和访问

开发流程并共享它们也是灾难规划的一个重要特征。创建从备份中恢复数据的流程、应对中断的故障排除程序以及其他类似情况当然很重要，但是记录这些流程并与相关人员分享它们是另一半工作。快速高效地完成故障排除或灾难响应流程可能意味着一项业务的结束，也可能意味着一次小小的公关挫折。

### 共享灾难计划

需要参与的非 IT 人员应该了解情况和计划。例如，人力资源可能不得不参与雇用关键的替代人员，或者在公司进行替代时，顾问可能需要介入协助。会计部门可能需要了解为恢复目的而付费的服务，等等。
