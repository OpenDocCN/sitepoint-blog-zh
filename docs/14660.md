# Javascript 和安全性

> 原文:[https://www.sitepoint.com/javascript-and-security/](https://www.sitepoint.com/javascript-and-security/)

Dori Smith 在她的 Javascript 博客上有一篇关于使用 Javascript 保护网站页面的精彩文章。她认为 Javascript 真的不适合这份工作，我完全同意她的观点。然而，可以使用 Javascript 来进一步提高服务器端登录系统的安全性。

除非你使用 SSL，否则你网站的登录系统很可能会明文发送密码。它们可能被嵌入在 POST 请求中，但对于嗅探器程序来说仍然是公平的游戏，例如 [Ettercap](http://ettercap.sourceforge.net/) 。虽然还不能完全解决问题，但是您可以通过使用 Javascript 在将用户密码传输到应用程序之前对其进行加密来减少嗅探攻击的影响。evesdropper 可能能够使用他们嗅探到的数据访问您的应用程序，但他们无法访问使用相同密码的其他应用程序(大多数人在任何地方都使用相同的密码)。

但是，如果加密是由 Javascript 执行的，攻击者肯定能够对其进行逆向工程并自己解密密码吗？一点也不，多亏了单向加密算法的魔力——特别是 MD5。

MD5 是一种单向散列算法。你给它一个输入，它吐出一个输出。诀窍在于，除了强力攻击之外，没有从输出中检索原始输入的系统方法。Javascript 不提供对 MD5 的本地支持，但是这个免费库提供了一个 MD5 Javascript 实现，专门用于防止密码被明文传输。

下面是对其工作原理的简单解释:

*   web 服务器提供一个带有隐藏字段的表单，该字段包含一个随机的“挑战”字符串，还可以选择提供表单时的时间戳。
*   用户输入密码并提交表单。
*   Javascript 将它们的密码粘贴到质询上，MD5 对它们进行哈希运算。只有散列被发送回服务器。
*   服务器知道用户的密码和发送的质询，因此它对它们进行哈希处理，并将结果与用户发送的数据进行比较。

自然，当实现这样的东西时，即使禁用了 Javascript，系统也能正常工作是很重要的。当使用 Javascript 加密方法时，您可以通过在隐藏的表单字段中设置变量来实现这一点；服务器可以检查该字段是否已经设置，如果没有设置，则认为密码是明文发送的。

如果您的 web 应用程序存储加密的密码(正如一个表现良好的应用程序应该做的那样),这种技术仍然可以使用——您只需在客户端对密码进行两次 MD5，一次获得加密版本，另一次将加密版本附加到质询中，以获得应该发送到 web 服务器的响应。

这种技术的示例服务器端代码可以在 Javascript MD5 站点的这个页面上找到。
