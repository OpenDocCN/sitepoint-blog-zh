# 混合应用是废话

> 原文：<https://www.sitepoint.com/hybrid-apps-are-redacted/>

到底什么是原生应用？

> “我今天不打算进一步界定我所理解的属于‘硬核色情’这一简称的材料种类；也许我永远无法明白地做到这一点。但是当我看到它的时候我就知道了”
> 
> > 美国最高法院波特·斯图尔特法官，Jacobellis 诉俄亥俄州

最近，关于本地、网络和所谓的“混合”应用程序的相对优点有很多争论，这些应用程序使用网络技术加上 PhoneGap 等技术来创建可以安装在 iOS 等平台上的应用程序，“就像本地应用程序一样”。

所有这些都没有回答这个看似无害的问题。究竟什么是原生 app？它与混合应用的区别到底在哪里？或许我们可以像大法官波特(Justice Potter)对色情作品的著名评论那样回应，当我们看到一个原生应用时，我们会认出它。

但是除此之外，还有什么经验法则呢？

### 外观和感觉

像约翰·格鲁伯这样的人，这位被广泛阅读的苹果技术专家，认为原生应用在本质上更有吸引力。通过哪些逻辑使用原生 UI 的 app 是原生的。那么，这让很多游戏，比如说《愤怒的小鸟》，看不到本地控制。是本土的吗？谁会说不是呢？

如果一个用网络技术构建的应用程序“愚弄”了用户，让他们相信他们正在交互的列表是一个“原生”列表对象，就像网飞的 iOS 应用程序和许多其他应用程序一样，为什么会不那么原生呢？

### 表演

用 Objective-C 这样的“真正的”编程语言编写的原生应用程序意味着这样的应用程序具有更好的性能，这当然总是很重要的。除非事实并非如此。

你可以用任何一种编程语言编写效率非常低的代码(我已经做了 25 年了！)，在其他条件相同的情况下，像 Objective-C 这样的编译语言比解释语言的性能更好，这是事实。虽然这通常是对的，但并不总是相关的。事实上，它并不总是完全正确的。

性能只有在重要的时候才重要，也就是说，更高的性能只有在影响用户时才重要，而不是绝对重要。在游戏等图形密集型应用中，性能当然是一个大问题。断断续续、每秒帧数低的游戏不太可能取得巨大成功。但即使在这里，也不是那么清晰。在 iOS 上，CSS 动画和过渡是硬件加速的。开发人员可以毫不费力地将动画任务交给 GPU，因此 JavaScript 代码的性能对整体应用程序性能的实际影响可能很小，如果有的话。

随着 WebGL 进入移动领域，编译语言的性能优势变得更加无关紧要。许多最成功的 iOS 游戏都是在 OpenGL 3D 渲染引擎上使用 [Lua](http://www.lua.org/about.html) 编写的。在移动浏览器中使用 WebGL，JavaScripted WebGL 游戏的性能很可能会与当前的本地游戏相媲美，即使没有像 JIT JavaScript 引擎这样的进步，这些引擎已经在移动 Safari、Playbook 和黑莓浏览器中出现。

除了图形密集型应用程序之外，对于使用 web 技术构建的应用程序来说，性能是否真的是一个很大的问题还不完全清楚。当然，这种限制被广泛吹捧，但这种坚定信念的证据很少。诚然，Android 的 CSS 过渡和动画表现相当糟糕。

就像本机的外观和感觉一样，性能也是因人而异。如果用户没有注意到，这是一个问题吗？

### 小心缝隙

因此，如果在许多情况下性能并不完全是它所声称的问题，并且原生外观和感觉有点偏离主题，那么还有什么可以使某些东西成为原生的呢？显然，在浏览器中运行的应用程序不具备原生应用程序的某些方面。

1.  原生应用可以离线运行。

    嗯，等等，大多数平台上基于 web 技术的应用程序也是如此。

2.  本机应用程序可以安装在设备上。

    事实上，iOS 和其他移动平台上基于网络的应用程序也可以做到这一点，而且比原生应用程序更省力；不需要通过应用程序商店，用户可以在应用程序内部完成。

3.  本机应用程序可以访问所有设备，如摄像头、加速度计等。

    现在，这种*是*一种明显的优势，尽管正在减少。现在，你不能直接从浏览器访问很多设备 API。例如，如果你需要在 iOS 上访问相机或地址簿，你需要超越浏览器。就目前而言，情况确实如此。

然而，你可以使用 [PhoneGap](http://www.phonegap.com/home) 、 [Appcelerator](http://www.appcelerator.com/) 、 [Rhomobile](http://rhomobile.com/) 和其他类似的解决方案，使用 HTML、CSS 和 JavaScript 开发你的应用，并通过 DOM 访问 JavaScript 中的这些底层 API。

此外，随着来自 W3C 的新兴 [DAP 标准](https://www.w3.org/2009/dap/)(它以标准化的方式提供对大量设备 API 的访问，并且已经部分实现了 Android 3.0，其中浏览器可以通过 DOM API 访问设备摄像头)，以及浏览器中广泛的地理位置支持和越来越多的加速度计支持，在浏览器中访问许多这些设备 API 的日子已经不远了。

### 房间里的大象

所以，我错过了什么？啊，客厅里的大象，应用商店。要进入苹果的应用商店，无数的安卓应用商店等等，你确实需要打包一个用网络技术构建的应用。可悲的是，你不能简单地为他们提供一个 URL，即使是为 Chrome web 应用商店构建的应用程序也需要打包。

这是否是一个优势，而不仅仅是一个与众不同的特性(或者它是一个缺陷？)的本地应用程序，但即使如此，您的应用程序的逻辑和表示也可以 100%用 web 技术构建，用 PhoneGap 打包，并部署到应用程序商店中。

### 混合非推论

那么，在所有这些中，是什么让一个应用程序混合呢？为什么用网络技术构建的应用程序，以及用 PhoneGap 打包的应用程序，会比用 unity 3d T1 构建的游戏更缺乏本地性呢？或者就此而言，比用 CocoaTouch 和 Objective-C 构建的应用程序更好？在 WebOS 上，HTML、CSS 和 JavaScript *是开发的本地语言，本地应用程序几乎都是用 web 技术开发的。*

最重要的是，用户在乎吗？我对此表示怀疑，因为很少有人能说得清。如果它运行良好，看起来不错，他们会使用它，如果不是，他们不会。用户可能不太在乎，就像他们几十年来对大多数平台的态度一样。

有 web 应用，也有原生应用。没有混合应用这种东西。天知道这个迷因从何而来，但现在是时候让它退休了。使用任何你想要的，或者你觉得舒服的技术来为你的用户构建优秀的应用。
