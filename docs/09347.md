# 9 个流行的云使用案例

> 原文：<https://www.sitepoint.com/nine-cloud-use-cases/>

你可能想知道人们是如何使用云的。我收集了一些(但肯定不是全部)最常见的用例，从简单的开始，逐步发展到更复杂的。

## 托管静态网站和复杂的 Web 应用程序

云可以轻松托管由静态 HTML 页面、CSS 样式表和图像构建的静态网站。事实上，最简单的此类网站可以仅使用云存储来托管，或许可以借助内容分发系统。

更复杂的网站，通常有大量的服务器端处理和对关系数据库的访问，也可以托管在云中。这些站点利用云存储和处理，并且通常需要大量的处理和存储资源来达到所需的规模。

## 软件开发生命周期支持

云非常适合软件开发生命周期每个阶段的资源需求。

在开发过程中，使用云可以确保开发人员有足够的资源来开展工作。假设一个开发团队正在构建一个经典的三层 web 应用程序，包括 web 层、应用程序层和数据库层，在部署时每个层都位于一个单独的物理服务器上。如果没有 AWS，每个开发人员将被提供三台完整的服务器，每台服务器将在一天的大部分时间里处于闲置状态。当新的开发人员加入到项目中时，成本会快速增长。迁移到云意味着每个开发人员可以在早上启动服务器，开发和测试一整天，然后在工作日结束时将服务器返回到云。

云在软件测试中也很有价值。开发人员可以启动测试服务器，并在其上运行单元测试，而不会增加开发服务器的负担。如果有许多单元测试，可以使用多个并行服务器来分散负载。

云可以用来支持持续集成环境。在这样的环境中，每个源代码提交操作都会启动一个多步骤的重建、单元测试和功能测试过程。如果代码是为多个目标环境(Linux 的几个不同版本或变体)或平台(Windows 和 Linux)编写的，那么云可能是拥有自己的基础设施的一个非常划算的替代方案。

负载和性能测试可以在整个开发周期中使用云计算资源来完成。如果应用程序本身将在云上运行，测试将确保它在重负载下运行良好，随着负载的增长增加额外的资源，并随着负载的消散删除它们。

当云能够提供在代表预期负载的规模上进行测试所需的资源时，测试面向公共或企业部署的 web 应用程序的性能变得更加容易。一些公司使用云资源来产生相当于几十万用户同时使用的负载。

一旦应用程序被部署(可能也部署到云)，云就可以在应用程序中间件层或公共组件被更新时提供执行兼容性测试所需的资源。彻底的测试有助于树立信心，在没有停机风险的情况下对生产系统进行实质性升级。

## 培养

云可以提供支持各种培训项目所需的短期资源。

如果学生正在学习如何安装、运行和监控上一节中描述的三层应用程序，他们可以在课堂上使用自己的笔记本电脑访问分配的云资源。当一天的课程结束时，资源被返回到云中。学生可以从单个“主”机器映像开始，避免浪费时间(他们的或教师的)安装和配置所需的软件包和应用程序。

传统的培训课程必须根据他们可用的有限的物理硬件数量来限制班级的规模。领先的公司现在正在举办在线培训研讨会，由每个学生基于云的资源提供支持，在每个新学生加入班级时，都会启动一台额外的服务器。应用程序和数据库软件供应商已经使用了这种技术，并取得了令人印象深刻的结果。

## 民众

从云中提取的资源可用于托管和交付打包软件应用程序的演示和试用版本。供应商可以在销售线索“热”的时候将演示交给潜在的客户，而不是在准备好合适的测试环境之后，通常的长时间延迟。应用程序供应商可以在短时间内以低成本创建并提供对云中托管的服务器的访问。销售周期缩短，客户对基于云的资源的使用体验良好。在某些情况下，基于云的演示实际上会导致基于云的部署。

## 数据存储

云是存储私有或公共数据的好地方。可伸缩性、长期耐用性和规模经济对于这个用例来说至关重要。存储的数据可能简单而紧凑，就像备份几个个人文件一样，也可能庞大而复杂，就像备份一家公司的全部数字资产一样，或者介于两者之间。

通常，在云中使用存储被证明是很好的第一步，这一步会激发信心，并很快导致考虑将云用于其他更复杂的用例。

## 灾难恢复和业务连续性

对 IT 资源具有关键任务依赖性的企业必须有一个适当的计划来应对任何挫折，无论是暂时还是永久的资源损失或对资源的访问。该计划必须考虑到火灾、洪水、地震和恐怖主义行为破坏公司运营的可能性。许多企业保留整个数据中心作为备用；数据有时会被复制到备份中心，整个组合系统随时准备就绪，可以随时启动。不用说，建造和运行一个复制设施的成本是相当大的。

云计算再次提供了一种不同的方式来确保业务连续性。与其将资金浪费在正常情况下永远不会投入使用的硬件上，不如将整个企业网络建模为一组云资源，以模板形式捕获，然后在出现问题时实例化。在这个特定的用例中，您需要与您的云提供商合作，以确保必要的资源在您需要时可用。

一旦出于业务连续性的目的对公司网络进行了建模，就会想到其他有趣的用途。传统上，中间件和共享应用程序组件的更新版本的广泛部署需要大量的兼容性和性能测试。这项任务充满了危险！许多公司发现自己慢慢落后了:由于在部署前进行全面测试的能力有限，他们无法部署最新的代码，并且不愿意冒险面对部署失败的后果。

想象一下，构建一个完整的企业网络副本(或一个有代表性的、缩小的子集)，以及要测试的应用程序组件的指定版本，然后在其上运行兼容性和负载测试，所有这些都在云中进行，并且成本非常合理。

## 媒体处理和渲染

许多流行的网站支持上传媒体文件:音乐、静止图像或视频。上传后，文件将经历一系列处理步骤，这些步骤可能是计算密集型、I/O 密集型或两者兼有。对所有类型的文件进行病毒和其他形式的恶意软件扫描。音乐被打上指纹(以检查是否侵犯版权)，然后进行代码转换，以允许以不同的比特率播放。图像被缩放、加水印、检查复制，并以不同的格式呈现。视频也被转码和缩放，有时被分成更短的块。最后，完成的对象被存储并可用于在线查看或下载。

渲染使用场景描述来生成动画电影的帧。每个帧都可以独立于其他帧进行渲染。输入数据相对较少，但输出数据很多。这个过程是计算密集型的，因为必须计算每帧的每个像素，考虑光、阴影、颜色和运动。

由于存储、处理和互联网带宽的数量，云计算是处理和呈现用例的理想选择。

## 商业和科学数据处理

科学和商业数据处理通常涉及非常大规模的数据集，并且会消耗大量的 CPU 能力。分析通常是按需进行的，导致有限的内部资源被过度使用。事实上，有人告诉我，许多内部科学计算网格经常在 0%使用率(绝对没有工作要做)和 100%使用率(每个可能的处理器都在使用)之间转换。在大学校园里，这是一个特别尖锐的问题，因为在学期结束前和重大会议前，校园里的使用率会上升。

业务数据处理可以是临时的(不定期的)或更常规的；每月工资处理和每日网络日志处理是云计算非常明显的用例。一个大型、繁忙的网站能够在每 24 小时内生成数百亿字节的日志文件数据。由于可以从日志文件中挖掘出大量的商业智能，分析是一项至关重要的功能。更及时地访问使用数据可以实现更好的站点优化，并对变化和趋势做出更快的响应。每天的分析过程开始变得越来越长，在某一点上开始花费几乎 24 小时。一旦发生这种情况，高度并行的解决方案就会对问题产生影响，在更短的时间内消耗更多的资源——这是云计算的完美案例。

## 溢出处理

随着公司开始了解云计算带来的好处，他们开始寻找允许他们使用现有 IT 资源进行日常工作的解决方案，同时将额外的工作推到云上。这就像雇佣临时工来应对假日高峰。

溢出处理使公司能够适应云计算。随着信心水平的提高，以及云中已经存在的重要企业数据量的增长，他们找到了越来越多的使用云的方式。
