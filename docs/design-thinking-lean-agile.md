# 用精益和敏捷的设计思维工作

> 原文：<https://www.sitepoint.com/design-thinking-lean-agile/>

设计思维是商业和技术领域的最新热门词汇。似乎这个短语在几乎每一个语境中都会出现。前几年，**精益 UX** 风靡一时，之后几年重点关注了**精益创业 。在那之前的几年，我所知道的每一家科技公司都在争先恐后地实施敏捷开发流程。像 Lou Rosenfeld 这样的专家已经在预测接下来会有什么新方法。**

随着时间的推移，这些方法并没有变得不那么有用，但人们正在尝试新的方法来构建产品和成功的技术，并将其作为“下一件大事”来对待，这将被证明是每个人的神奇解决方案。问题是，在讨论新事物的兴奋中，我们并不总是将我们现有方法的点连接起来，人们可能会对如何最好地一起实现事物感到困惑。

请继续阅读，以更好地理解设计思维、精益 UX 和敏捷，以及如何为您的团队实现每个元素。

在我们走得太远之前，让我们后退一步来理解每种方法。

## 敏捷

让我们明确一点:敏捷是一种软件开发方法。它的诞生源于对传统“瀑布”软件实践的失望，需要长时间的前期需求收集和设计工作，然后是实现所述设计的长时间开发阶段，但没有能力理解或响应不断变化的需求。结果是，团队花了很长时间来构建人们并不真正想要或需要的东西，公司也在苦苦挣扎。

软件开发人员开始尝试新的构建方式，并提出了一套共同的价值观和原则来指导团队更好地工作。

*   官方的敏捷宣言于 2001 年发布，呼吁重视:
*   过程和工具之上的个人和交互
*   综合文档之上的工作软件
*   合同谈判中的客户协作
*   响应变化而不是遵循计划

敏捷联盟也定义了 [12 个详细的原则来遵循](http://agilemanifesto.org/principles.html)，但是没有规定任何特定的过程，所以开发团队经常最终使用特定的框架，比如 Scrum 或看板，来帮助他们弄清楚如何组织、计划和执行他们的工作。非常强调团队的独立性和自组织性，所以没有两个敏捷团队看起来是一样的，即使在同一个部门或组织中。

理论上，敏捷方法不仅能很好地适应 UX 实践，而且积极地要求正在进行的 UX 研究不断地了解客户不断变化的需求。然而，在实践中，团队可能会试图更快地发布更多的工作代码，并且很难投入任何时间来进行研究或专注于设计决策。敏捷团队经常纠结于如何最好地将 UX 团队成员和他们的工作整合到他们的实践中。

## 精益 UX

**精益 UX** 诞生于许多团队在调整他们的开发过程以适应敏捷方法并试图加快从想法到实现的时间时，合并 UX 最佳实践的斗争中。精益 UX 是改变传统 UX 方法以适应更快的时间框架的总称，这通常意味着将焦点从详细的可交付成果转移开。

但是要小心:你可能也听说过精益和精益创业，这两个词经常被混为一谈，但是确实有特定的含义和不同的元素。**精益**源自制造最佳实践，专注于一般业务和管理实践，以减少浪费和实现价值最大化。**精益创业**是一种更广泛的业务和产品开发方法，它建议整合试验期，以减少浪费和风险。这些术语并不相互排斥，但也不能互换。

回到精益 UX:核心思想是改变传统的 UX 设计方法，以变得更快。团队不需要花费大量时间来彻底设计和记录每个元素，而是要快速地、协作地将想法可视化，并尽快从其他团队成员、利益相关者和用户那里收集反馈。

[Jeff Gothelf](https://www.jeffgothelf.com/lean-ux-book/) 展示了以下精益 UX 流程:[概念、原型、内部验证、外部验证、学习、迭代和重复](https://cloud.netlifyusercontent.com/assets/344dbf88-fdf9-42bb-adb4-46f01eedd629/a09b5b34-d63d-473d-b934-0c3fd2ddc4a8/just-the-ux-process-large.jpg)。这一过程反映了“常规”UX 过程，但每一步都缩短了。

假设一个团队正在集成一个新特性。团队可能会先召开一个快速白板会议来充实核心工作流程。一旦小组在方向上达成一致，他们就可以向用户展示一个低保真度的设计，并整合在联合草图会议期间发现的反馈，在那里他们整理出更多的交互细节。

你会注意到这个例子没有任何功能完整的原型或详细的测试报告，但是精益 UX 不是跳过步骤的借口。相反，它是一个邀请，要求你做足够多的事情来建立一个共同的愿景并获得反馈，扩大和支持不同的工具或方法，因为它对你的特定环境最有意义。

精益 UX 也不建议你完全放弃文档，也不建议你从 UX 专业人士那里拿走经验决策。相反，这表明整个团队都参与了设计过程，因此不会出现意外或不可预见的技术挑战。我们会尽早并经常收集反馈，如果需要进行更改，可以在投入大量时间进行最终设计之前快速轻松地完成。

## 设计思维

设计思维是一种解决创造性问题的通用方法，由斯坦福大学 T2 分校和 IDEO 公司推广，最近被许多科技公司采用。这种方法有五个步骤，被表述为一种框架和解决各种问题的方式。

第一步是**移情**，这看起来非常像老派的研究阶段。你收集尽可能多的信息来了解相关人员、他们的问题和动机以及他们的背景。这可能意味着你要做一些事情，比如进行采访，观察别人的空间，与专家交谈，或者查看过去的行为模式。

接下来，你分析你收集到的信息，综合各种想法来**定义**你所服务的观众的核心问题。查看您对该团队的了解、他们的需求以及可能存在的差距，以创建一个简洁的问题陈述来规划您的后续步骤。

第三步是**构思**所述问题的解决方案，结合创造性的技术从多个角度看待问题，并尽可能多地提出不同的想法。这个想法是先扩大范围，在缩小范围到几个可能的解决方案进行测试之前，产生大量的备选方案。

在你收集任何反馈之前，团队需要**原型**，或者创建解决方案的低成本表示。原型可以是任何东西，从一系列草图到一个提议流程的像素级完美前端模拟。目标是取得平衡，使解决方案看起来足够真实以获得真实的反馈，但成本相对较低且易于生产。

最后，你想测试原型，把它们放在真实的或有代表性的用户面前，观察他们做什么和他们如何反应，然后利用这些信息形成下一轮。

你应该注意到，虽然这个过程是线性呈现的，但在现实中，各个阶段可能会重叠，或者你在这个过程中学到的一些东西会促使你重新审视问题的同理心或定义。这种方法是结构化的，但也是灵活的。

所有这些听起来应该很熟悉。设计思维是一种帮助团队协作理解和解决问题的公式，通过围绕相关人员设计问题，并尽早和经常地整合反馈，这也是良好 UX 实践的核心。

## 把这一切联系在一起

好消息是，我提到的每种方法对几乎每个团队都有一些有价值的组成部分。敏捷对协作、迭代、反馈和速度的关注意味着团队不太可能浪费时间来构建没人想要的东西，精益 UX 帮助调整 UX 的传统实践，以适应更快的开发团队的步伐。设计思维给了各种学科的人一种探索问题和关注用户的方法。

坏消息是，在实践中，这三种方法有时看起来相互矛盾，因为它们是解决不同问题的方法。敏捷是关于软件的，精益 UX 是关于设计和研究过程的，而设计思维是一种想法探索工具。通常，公司内不同的人或小组将负责这些不同的领域，他们各自的时间表和目标可能不一致。

例如，一个策略团队可能在一个计划开始时使用设计思维方法来探索想法，然后开始与 UX 设计师合作来充实概念——与此同时，开发团队可能使用敏捷迭代周期来构建他们所知道的东西。这不一定是一个直接的线性过程，利益相关者和关键参与者的交接和差异会导致紧张或返工。Lithespeed 的图很好地展示了这些过程是如何成功地交织在一起的。

当然，最适合每个团队和项目的一套独特的实现细节将取决于一大堆因素，如您所在的组织类型、您试图构建的内容以及您在该过程中所处的位置。

以下步骤应该对每个团队都有帮助。

### 共同愿景

我要做的第一步是让三个主要领域的每个人都同意一个共同的愿景，并在团队的所有成员中明确地陈述和分享它。这听起来像是显而易见的第一步，但是每个团队或个人对组织的最终目标都有自己的版本是很常见的。试试看:去问问一起工作的开发人员、UX 设计师和产品策略师他们的愿景是什么。如果他们能回答的话，很少三个人会以同样的方式回答。

共同愿景应该概述公司的最终目标，而不是团队应该如何实现。例如，[亚马逊的愿景](https://www.amazon.jobs/en/working/working-amazon)是

> 成为地球上最以客户为中心的公司，在这里客户可以找到并发现他们可能想要在线购买的任何东西，并努力为客户提供尽可能低的价格。

这个想法不是规定，而是确保每个人都朝着同一个方向前进，你可以检查你自己的工作以确保它是一致的。

### 清除测量周期

团队还应该确定他们将如何根据愿景来衡量成功。这些方法中的每一种都建议频繁的实验和反馈周期，但是你需要清楚地定义你的目标是什么结果，什么指标可以帮助你理解这些结果，你将如何收集这些信息，以及什么参数将决定你下一步做什么。

我特别喜欢这个[假设框架](http://uxmastery.com/wp-content/uploads/2017/05/img-template.png):

> **如果我们** *做、建造、或提供 x 件东西* ****那么** *这些人*
> **会做** *一些令人想要的结果* **。**
> **当** *可操作的指标*出现时我们就知道这是真的了。**

 **目标是精心设计实验，尽快回答你的问题，给你一个明确的决策点，这样你就可以放心地放弃那些不能让你达到目标的努力。

记住，你可能会从那些“否定”了他们假设的努力中学到最多的东西。学习什么有效什么无效有内在的价值，即使你不得不放弃一些工作。

### 保持用户焦点

这些方法的基础是彻底了解你要解决的人，并确保他们在整个过程中处于反馈循环和决策点的中心。在实践中，这意味着不断地与你现有的和潜在的用户交谈和学习。尝试新的解决方案很重要，但也要认识到用户及其目标或环境可能会随着时间的推移而改变。无论你处于产品开发的哪个阶段，都要继续采取措施去了解你的用户和他们的需求。

### 为成功组建团队

跨职能协作是每种方法中的另一个基本元素，对于团队来说，让多个学科的专门成员一起工作是最理想的。为你的用户提供价值应该是每个人的首要考虑，而最好的解决方案来自不同的个人贡献他们的才能和知识。

同样重要的是，要不断地检查你的团队，以评估哪些工作得好，哪些可以在过程中改进。对一个团队有效的不一定对另一个团队有效，给自己时间去思考和尝试新事物是很重要的。与其专注于一个特定的过程，不如专注于实验和不断学习。

在一天结束的时候，没有单一的“正确的方法”让每个团队在你的实践中实施敏捷、精益 UX 或者设计思维。每种方法都是为了解决一个特定的问题，每种方法的元素都可能是有用的，所以你必须找到适合你的团队的方法。享受实验吧！**
