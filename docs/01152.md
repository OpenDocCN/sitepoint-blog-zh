**问题*“我的网站/应用程序应该支持哪些浏览器？”*经常被客户和开发者提起。简单的答案是前 N 名的主流应用列表。但这一政策已经变得无关紧要了吗？**

世界范围的统计数据并不能说明全部情况:

*   不同地区的模式差异很大。例如，Yandex 是使用第二多的俄罗斯浏览器(12.7%的份额)。搜狗是中国第三大用户浏览器(6.5%)。Opera Mobile/Mini 在非洲拥有 28%的份额。
*   新的浏览器版本定期出现。Chrome、Firefox 和 Opera 每六周接收一次更新；检查几个月前的版本是不切实际的。
*   相同的浏览器可以在不同的设备和操作系统上以不同的方式工作。Chrome 可用于 Windows、macOS、Linux、Android、iOS 和 ChromeOS 的各种版本，但它并不是到处都是相同的应用程序。
*   在包括游戏机、电子书阅读器和智能电视在内的一系列设备上，新旧、怪异和奇妙的浏览器的尾巴非常长。
*   你的网站的分析永远不会匹配全球统计数据。

## 浏览器差别这么大吗？

尽管有各种各样的应用程序，所有的浏览器都有相同的目标:*呈现网页*。他们通过渲染引擎实现这一点，并且有一些交叉授粉:

1.  Webkit 用于 macOS 和 iOS 上的 Safari。
2.  Blink 是 Webkit 的一个分支，现在用于 Chrome、Opera、Vivaldi 和 Brave。
3.  火狐用 Gecko。
4.  三叉戟用于 Internet Explorer。
5.  EdgeHTML 是 Edge 中使用的 Trident 的更新。

大多数浏览器使用这些引擎中的一个。他们是不同团队的不同项目，但公司(主要)通过 W3C 合作，以确保每个人都以相同的方式采用新技术。浏览器比以往任何时候都更接近，现代智能手机应用程序可以与桌面应用程序相媲美。

然而，没有两种浏览器以完全相同的方式呈现。大多数差异都是细微的，但是随着你走向尖端技术，它们会变得更加明显。某个特定功能可能在一个浏览器中完全实现，在另一个浏览器中部分实现，而在其他地方不存在。

## 我的网站可以在所有浏览器中运行吗？

是的。渐进式增强(PE)等技术建立了一个基线(可能只有 HTML)，然后在支持可用时用 CSS 和 JavaScript 进行增强。最近的浏览器有了现代的布局、动画效果和互动部件。古老的浏览器可能只能得到无样式的 HTML。其他一切都介于两者之间。

PE 适用于内容网站和具有基本表单功能的应用程序。随着您转向具有丰富自定义接口的应用程序，这变得不太实际。你的新协作视频编辑应用不太可能在十年前的 IE7 上运行。它可能无法在 3G 网络上的小屏幕设备上工作。也许有可能提供一个替代的界面，但结果可能是一个很少有人愿意使用的单独的、笨重的应用程序。考虑到传统浏览器用户群的规模，成本会高得惊人。

## 网站所有者建议

网站所有者应该理解以下网站的基本原则和限制。

**网络不是印刷品！**
你的网站/应用不会到处看起来都一样。每个设备都有不同的操作系统、浏览器、屏幕大小、功能等。

**功能可能不同**
你的网站可以为每个人服务，但体验和设施会有所不同。即使像日期输入字段这样基本的东西也可能有各种各样的可能性，但理想情况下，核心应用程序将保持可操作性。

评估你的项目
现实一点。这是一个内容网站，一个简单的应用程序，一个类似桌面的应用程序，一个快速动作游戏等。建立浏览器兼容性的基本级别。例如，它必须在大多数两年前的浏览器上工作，屏幕宽度为 600 像素，通过快速 Wi-Fi 连接。

**评估你的受众**
不要依赖全球浏览器的统计数据。谁是主要用户？他们是 IT 新手还是技术含量高的？是个人、小公司还是政府机构？他们是坐在办公桌前还是在移动中？没有适用于所有人的应用程序——首先关注核心用户。

尽可能检查现有系统的分析，但要重视底层数据。如果你的应用不能在 Opera Mini 上运行，你就不太可能拥有 Opera Mini 的用户。你是否封锁了你的市场的很大一部分？

**变化发生**
一个二十年前编码的网页，今天还能工作，太神奇了。它不一定漂亮或可用，但浏览器保持向后兼容。*(大多是。`<blink>`标签可以保持不动！)*然而，技术在进化。您的站点或应用程序越复杂，就越有可能需要持续的维护。

## Web 开发人员建议

只要稍加小心，支持大量不同的浏览器是可能的。

拥抱网络！
网络是一个与设备无关的平台。内容和更简单的界面可以在任何地方工作:现代笔记本电脑、功能手机、游戏机、IE6 等等。学习渐进增强的基础知识。即使您选择不在您的整个应用程序中采用它，也会有一些功能变得非常有价值。

**采用防御性开发技术**
在到达最近的预写模块、库或框架之前考虑问题。在开始之前，要了解这项技术的后果。框架应该提供一个浏览器支持列表，因为它们已经在有限数量的应用程序中进行了测试。

了解浏览器的限制和怪癖。例如，如果您正在考虑一个 SVG 图表，请注意它在 IE9 到 11 中可能看起来很奇怪，在 IE8 和更低版本中可能会失败。这并不意味着拒绝 SVG 或放弃 IE 支持是一个二元选择。总有一些妥协不会导致重大的发展。例如:

*   接受 SVG 渲染很奇怪，但仍然可用
*   仅在 IE 中显示数据表，或
*   提供一个 SVG 下载，IE 用户可以在其他地方打开。

尽早测试，经常测试
你不可能测试每一款设备，但是为一款浏览器开发是徒劳的。

在各种应用程序中不断测试您的项目。将测试留到最后会产生灾难性的后果。我们很容易指责工具和浏览器的不足，但是如果能及早发现，大多数问题都可以在开发过程中得到解决。

这并不是说所有的东西每次都必须在所有的浏览器中同样地工作。功能退化是不可避免的。例如:

*   渐进式网络应用无法在 iPhones 和 iPads 上离线运行——但可以在线运行。
*   IE 不支持 CSS 网格，但 float、flexbox 或全角块回退应该是可以接受的。
*   Firefox 的桌面版没有为日期字段显示日历，但用户仍然可以输入日历。

在您的开发 PC 上安装一系列浏览器。Mac 和 Linux 用户可以在[developer.microsoft.com/microsoft-edge/](https://developer.microsoft.com/en-us/microsoft-edge/)获得微软 Edge 和 IE 测试工具。Windows 和 Linux 用户测试 Safari 的难度更大；像[浏览器栈](https://www.browserstack.com/)这样的在线测试服务是最简单的选择。

现代浏览器拥有出色的移动仿真设施，但使用一些真实设备来欣赏较慢的硬件和网络上的触摸控制和性能。

**在你那端使用 HTTPS**

网络逐渐将 HTTPS 作为首选协议，这一趋势还将继续。谷歌浏览器甚至开始指出非 HTTPS 网站是不安全的，这是你配置你的网站使用 HTTPS 的一个很好的理由。例如，我们的虚拟主机合作伙伴 [SiteGround](https://www.siteground.com/go/article-sp) 帮助他们的客户轻松地搬到了 HTTPS。为了做到这一点，他们为所有新的 WordPress 帐户自动安装了 Let's Encrypt SSL 证书，对于现有的帐户，他们只需点击一下就可以切换到 HTTPS。

## 你还没有回答问题！

问题*“你应该支持哪些浏览器？”*变得过于严格。假设你的答案只是“Chrome”:

*   它运行在哪些设备和操作系统上？
*   将支持多大范围的屏幕尺寸？
*   你指的是哪个版本？最新的？Chrome 10 及以上？
*   *新版本的 Chrome 发布后会发生什么？*
*   当 Chrome 有效地成为你的应用的运行时，其他浏览器会发生什么？

为面向客户的项目提供浏览器支持列表已经变得不切实际。也许最好的答案是:*“我们将根据假定的人口统计数据开发您的项目，然后根据预算和时间限制，在尽可能多的设备、操作系统、浏览器和版本中进行测试”*。即使那样，你也会怀念首席执行官坚持使用的老款黑莓手机。

为网络开发——而不是浏览器。
