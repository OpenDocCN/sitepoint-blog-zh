# 2005 年:大教堂年

> 原文：<https://www.sitepoint.com/2005-the-year-of-the-dom/>

Jeffrey Zeldman 在他的书《用网络标准设计》中推广了网页设计的“三脚凳”概念。这个三条腿的凳子由用于内容和结构的 XHTML、用于表示的 CSS、用于“行为”的 JavaScript 和 DOM 组成。CSS 和 XHTML 在 2003 年和 2004 年表现出色，但是 JavaScript 和 DOM 呢？

我认为 2005 年将是正确使用 DOM 真正腾飞的一年。网络上的大多数 JavaScript 仍然来自 90 年代末，但变化正在酝酿之中。Netscape 4 实际上已经死了，当前的浏览器都提供了优秀的 DOM 支持(甚至 Opera 也赶上来了)。没有什么理由不认真对待 DOM，人们投入了大量精力来探索利用客户端脚本技术的新的、有效的方法。

当然，挑战在于可访问性:尽管 JavaScript 技术大大提高了交互性，但它们对相当一部分用户代理来说仍然是不可用的。解决方案是将 JavaScript 作为一种工具，为那些可以从中受益的人增强用户体验，同时避免切断其他人的访问。一个经典的例子是[Google Suggest](http://www.google.com/webhp?complete=1)——如果你打开了 JavaScript，这很好，但你仍然可以在没有它的情况下使用 Google 的主要搜索功能。

Google Suggest 还展示了我认为将是 2005 年的一个关键主题: [XMLHttpRequest](http://developer.apple.com/internet/webcontent/xmlhttpreq.html) 。多年来，从服务器交互加载额外数据一直是客户端开发人员的梦想，XMLHttpRequest 最终提供了一个“官方”方法来实现这一点(以前的远程脚本工作一直围绕着[巧妙的技巧](http://developer.apple.com/internet/webcontent/iframe.html))。GMail 和 Google Suggest 引发了人们对这项技术的极大兴趣，可以肯定的是，在未来的 12 个月里，我们将会看到更多这样的技术。 [ObjectGraph 字典](http://www.objectgraph.com/dictionary/)是另一个例子。

乔尔·斯波尔斯基总结了[微软是如何输掉 API 战争的](http://www.joelonsoftware.com/articles/APIWar.html)，他说*“新的 API 是 HTML，应用程序开发市场的新赢家将是那些能让 HTML 歌唱的人。”*。如果这在 2005 年是正确的，那么 JavaScript 和 DOM 将迎来非常好的一年。
