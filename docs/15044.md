# 采访 eZ systems 的巴德·法尔斯塔德

> 原文:[https://www.sitepoint.com/bard-farstad-ez-systems/](https://www.sitepoint.com/bard-farstad-ez-systems/)

rd Farstad 是 [eZ systems](http://ez.no) 的首席开发者，该公司是 [eZ publish](http://develeper.ez.no) 背后的公司。作为一名 C++背景的 PHP 开发人员，他已经完成了与 [PHP](http://www.php.net) 相关的所有工作。对于这位来自挪威 Skien 的 25 岁程序员来说，协调一个国际开发团队并将一个开源软件项目转化为企业级的商业成功只是调整他的步伐的问题。除了电脑键盘，他还是吉他指板的大师。友好地问他，他可能会把他乐队的最新单曲放在 mp3 里发给你……

继 [SitePoint 的 eZ publish 系列](http://www.webmasterbase.com/article/917)之后，我与博德取得了联系，以了解更多关于 eZ systems 的生活，他对 eZ publish 的计划，以及他对使用 PHP 进行在线开发的见解。

##### 博德和 eZ 系统

你能给我们介绍一下你自己吗？是什么让你第一次涉足软件开发，你的编程背景是什么？是什么让你开始开发基于 Web 的应用程序？

我第一次接触编程是在我大约 15 岁的时候。当时有一个活跃的演示现场，我是其中的一员。演示场景是关于用音乐编程图形演示，很像 MTV 视频。我们使用的编程语言是带有内联汇编程序的 Pascal。过了一段时间，当编译器在优化代码方面变得更好时，我开始使用 C++，并成为一个名为 Sorrox 的小组的一员。与此同时，我在学习电子学，后来发展到计算机工程，并于 2000 年 5 月毕业。

在学习期间，我对 Linux 产生了兴趣。当时挪威还没有多少 Linux 公司，所以我的兄弟、两个朋友和我决定开一家公司:eZ systems。我们一开始是一家咨询公司，专注于 Linux 和开源解决方案。后来，我们扩展到使用 C++和 Qt 开发图形应用程序。1999 年，我们有一个客户希望我们开发一个电子商务解决方案，所以我在一次休假期间开始写一个网店。这是 eZ publish 的起点，我们最终在 2000 年 11 月在 GPL 下发布了 eZ publish 的第一个版本。

你第一次接触 PHP 是什么时候，是什么让你选择了它，而不是其他技术，比如 ASP 和 JSP，甚至 Perl/CGI，来开发 eZ publish？

我第一次接触 PHP 是在 1998 年，当时我正在学习，为 Linux 用户组创建了一个网站。用 PHP 开发应用程序真的很快，性能也很好。当时，在你提到的语言中，唯一一种专门为动态网站设计的替代语言是 ASP。事实上，这是一个封闭源代码的微软产品，使我的选择非常简单。

你当时有没有想过 eZ publish 会变成什么样，会有多成功？

我们没有想到 eZ publish 会如此受欢迎。来自用户的反馈势不可挡。我怎么也想不到它会如此受欢迎，尤其是在这么短的时间内。

是什么让你决定创建 eZ systems？早期在 eZ 系统公司的生活是怎样的？

我们创建 eZ systems 的基本想法是，我们希望有一个有趣的工作场所。我们的任务很简单，享受生活，努力谋生。所以当我们开始的时候，主要是为了好玩。

随着 eZ publish 的出现，eZ systems 似乎找到了开源和商业成功的完美秘诀。你觉得保持这种平衡很难吗？对于考虑采用类似策略发布软件的开发者和小公司，你有什么建议吗？

很难平衡。一方面你想尽你所能做出最好的软件，另一方面你需要赚钱。除了远离风险投资家，我真的没有什么建议；他们会试图让你赚太多的钱，赚得太快——这样开源精神就会消亡。

SP:你的客户如何看待 eZ publish 开源？

非常肯定！许多客户要求他们的解决方案是开源的，当然价格标签非常有吸引力。缺点是一些客户期望一切都是免费的，因为 eZ publish 本身就是免费的(GPL)。

查看 [eZ systems 合作伙伴名单](http://developer.ez.no/partner/list/) **，名单令人印象深刻，包括西门子和 IBM。你把这次成功归功于什么？**

有很多公司从事 Web 开发，eZ publish 是一个很好的工具包，他们可以用它来创建最动态的网站。因此，他们希望与我们合作，提供基于 eZ publish 的解决方案。网络泡沫破灭后，客户不想为这类解决方案支付太多费用，因此提供这些解决方案的公司不得不寻找替代方案。因此，越来越多的公司选择 eZ publish。

鉴于 eZ systems 的商业“自我改变”,你是否发现获得开源开发者的支持更加困难了？你希望 eZ publish 收到哪种形式的投稿？您有什么计划或希望看到的东西来鼓励开源社区进一步参与到产品中吗？

eZ publish 也许不是一个传统的开源程序，因为它是由 eZ systems 单独编写的。我们仍然从用户那里得到很多反馈；从业余爱好者到西门子和 IBM 这样的大公司。我们确实有一些外部开发人员向我们发送他们的贡献——典型的是 eZ publish 用户，他们已经修补了他们发现的一个 bug。开源反馈最大的好处之一就是它能帮助我们快速追踪 bug。

我们现在正在完成 eZ publish 3 的代码，它将更加模块化和基于插件。当我们发布它时，我希望它会鼓励开发者编写插件，我们可以从我们的网站上提供这些插件——这将提供一个第三方下载区，用户可以在那里上传他们的插件以供 eZ publish 使用。我们现在正在开发我们的新网站，它将在最终的 eZ publish 3 发布之前发布，以进一步刺激社区的参与。

谁在 eZ Systems 工作，他们做什么？你如何描述 ezSystems 的工作场所？

现在总共有 20 个人在 eZ 系统公司工作。该团队主要由来自世界各地的开发人员组成。我们有来自挪威、乌克兰、中国和德国的人。

eZ systems 当然是一个有趣的工作场所，但同时也很有挑战性。我们的员工与来自世界各地的人们一起工作，从事一系列不同的项目。

作为一家用 PHP 开发 Web 应用的挪威公司，你和其他挪威同行有什么联系吗——比如 PHP 团队的成员 [fast.no](http://www.fast.no) **或者**[Stig s ther Bakken](http://www.zend.com/comm_person.php?id=23)**？**

我们和 Stig 没有联系，但是 Fast 是 eZ 系统的客户。他们使用 eZ publish 创建了他们的内部网，并且也在从事一些其他的 eZ publish 项目。我们与挪威和其他国家的不同公司有很多联系——许多公司使用 eZ publish 来构建他们的内部网。我们经常与公司进行初步接触，因为他们已经使用 eZ publish 建立了他们的内部网。

尽管网络泡沫破裂，eZ systems 似乎越来越强大。你认为 eZ systems 作为一家企业的未来是什么样的？你如何看待互联网给 eZ systems 这样的公司带来的机遇？

未来充满机遇，我们有很多计划。我们的第一个也是最近的目标是发布 eZ publish 3，这将是对当前应用程序的巨大改进。有了这个版本，需要版本控制和工作流功能的企业站点就有了他们所需要的。

开发时，你喜欢怎样工作？长时间的工作被长时间的休息打破，每天稳定的进步，还是其他？你能给那些感觉“一天 24 小时都不够用”的 PHP 开发人员提供一些建议吗？

我喜欢紧张的工作，当工作有趣时，我不认为这是工作。同时，休息和进行体育锻炼也很重要。在过去的三年里，我想我有大约两个星期的假期，所以很明显我和其他开发人员有同样的时间问题。帮助自己最好的方法是建立一个开发团队；你不能一个人做所有的事情。

你最喜欢的开发工具是什么(例如文本编辑器、项目管理、源代码维护)？你最常用的操作系统是什么，为什么？

我的编辑器是 [emacs](http://www.gnu.org/software/emacs/emacs.html) ，当然也用于项目管理。对于代码，我们现在使用 svn， [subversion](http://subversion.tigris.org/) 。我们大多使用 Linux，因为它拥有我们需要的一切，这也是我们最初的出发点。

当你不工作的时候，你喜欢怎么打发时间？

我喜欢和我的朋友在一起。我也在一个乐队里弹吉他——我们曾经制作了一张 cd(所以想要一些 mp3 的人可以给我发电子邮件)。实际上，大部分时间我都是和我的女朋友 Rikke 在一起。但在周末，我不会拒绝派对。

##### eZ 发布

**SP:虽然 eZ publish 在产品文献中被描述为内容管理系统，但你更倾向于如何看待它；作为 CMS 还是应用开发框架？**

我喜欢把它看作一个工具箱。除了设计变更之外，在启动网站之前，通常还需要做一些工作，比如构建定制模块。有了 eZ publish 3，我们让这一点变得更加清楚；我们有一个通用 PHP 类的代码库，如 XML 解析器、模板引擎和 SOAP 库。我们还有一个为您的网站提供基本功能的内核，包括发布、版本控制、图像处理和工作流程。但是 eZ publish 的基本思想是把内容和设计分开，所以从这方面来说，它也是一个内容管理系统。

鉴于 Web 应用程序开发仍然是一个相当年轻的学科，当你设计 ezPublish 时，是什么因素影响了你？有没有什么现有的应用或设计方法启发了你？

我们所有的开发都是从编写需求规范开始的，在这里我们决定应用程序应该做什么。当然，我们已经查看了许多其他程序来寻找灵感，但是不要试图复制任何功能。对于 eZ publish 3，我们在可用的时间框架内，花了大部分开发时间来改进设计，以创建最灵活的系统。当我们设计应用程序时，我们使用设计模式。

eZ publish 似乎遇到的最大问题之一是它很难安装。你认为这是将来可以改进的地方，还是像这样的大型复杂应用程序中固有的问题？

它肯定会在 eZ publish 3 中得到改进。这些问题大多发生在人们希望在共享主机服务器上运行它的时候，因为这不是当前版本 eZ publish 的目标主机环境——这给一些用户带来了很多麻烦。在 eZ publish 3 中，我们已经使它变得更加平滑。此外，像大多数其他功能一样，图像转换现在是通过插件系统实现的，因此用户将能够选择适合他们特定目的的设置。

在 SitePoint 的 eZ publish 系列文章中，eZ publish 对模板的使用受到了一些批评。你使用模板来交付 eZ publish 用户界面的原因是什么？为什么不利用某种可视化组件库，比如 Java 的 AWT(抽象窗口工具包),或者使用 XSLT 的 XML 替代品呢？这是我们未来可能看到的变化吗？

我们将 eZ publish 基于第三方模板引擎，这在当时看起来是个好主意。现在我们已经开发了自己强大的面向对象的模板引擎，它被很好地集成到 eZ publish 中。这个模板引擎是您将在该产品的下一版本中看到的巨大改进之一。XSLT 可能会被用作导出插件，但不会被用作主模板引擎。

有了 eZ publish，你已经为从数据库抽象到 DOM XML 解析器的一切构建了一个坚实的 PHP 类库。是什么让你选择自己编写，而不是使用原生 PHP 扩展或 PEAR 库？你还写了自己的类文档工具， [eZ PHP DOC](http://developer.ez.no/article/articlestatic/35/1/42/) **。你为什么这样做，而不是用** [PHP 比如说 DOC](http://www.phpdoc.de/) **？**

嗯，我们喜欢写自己的代码。这就是我们的工作方式。然而，我们现在已经开始使用 [doxygen](http://www.doxygen.org) 来生成类文档，因为这不是我们的重点。我们没有使用你上面提到的工具的原因是，基本上，我们想使用我们在 C++项目中使用的相同语法。

在 PHP 4.2 中，PHP 团队决定关闭“注册全局变量”？你同意这一举措吗？eZ publish 会在不久的将来效仿吗？

我完全同意。是的，你肯定会看到的。eZ publish 3 对新的 PHP 配置没有问题。

考虑到 eZ forum 这样的模块与 [vBulletin](http://www.vbulletin.com/) **和** [phpBB2](http://www.phpbb.com) **这样的应用程序“竞争”，你是愿意继续开发这个模块，还是将这些外部应用程序集成到 eZ publish 中？**

我们用 eZ publish 3 所做的是创建一个更低级的工具包。您现在可以定义自己的内容类，例如，您可以定义自己的文章、产品、消息等。然后，所有这些对象可以组合成一个基于树的结构，并由基于角色的权限系统来控制。因此，eZ publish 3 可以用来创建各种各样的应用程序。一个简单的例子是，对于一个房地产网站，您可以创建一个定义房子的类。然后，您可以立即开始在您的站点上发布“house”类的对象，无需编程！

目前，eZ publish 适合在一个单独的 Web 服务器上使用，可能还有一个远程数据库。你已经开始在 [eZ publish 桌面版](http://shop.ez.no/trade/productview/25/2/) **内使用 XML-RPC。有没有计划在 eZ publish 的扩展中进一步利用 XML Web 服务协议？我们是否有可能看到跨多个服务器分布的“N 层”版本的 ezPublish，或许利用像** Mohawk 的 Msession **这样的项目，或者效仿** XML-RPC 类服务器**？**

我们已经编写了一个名为 eZ soap 的库，它将在 eZ publish 3 中用于通信和集成。与此同时，我们正在努力让 eZ publish 在集群上平稳运行。在不久的将来，您将会看到这个方向的许多新功能。

最近 ezPublish 的发行版和你关于 [PHP/Java 集成](http://www.zez.org/article/articleview/26/) **的 Zez 文章让我们想知道 eZ publish 中是否有利用这一点的计划？ezPublish 能在 J2EE 环境中充当 Web“前端”吗？**

目前还没有计划，但是我们以前已经以这种方式使用过 eZ publish。我们通常根据具体情况处理客户的要求。您基本上需要编写自己的模块来与 Java 应用程序集成，您已经可以这样做了。

eZ publish 能和一些像**和** [Atomz Publish](http://www.atomz.com/) **这样的“大”CMS 竞争吗？eZ publish 与一些“高端”CMS 有什么不同？ezPublish 需要克服哪些障碍才能实现这一目标？**

这就是 eZ publish 3 的计划。绝对可以匹配功能性。与那些系统相比，当前的 eZ publish 缺少的主要特性是版本控制、工作流和协作，所有这些在 eZ publish 3 中都有。

有没有计划将 eZ publish 的元素作为原生 PHP 扩展(例如类库)发布，作为提高性能的一种手段？

目前还没有具体的计划。这将使 eZ publish 更难安装，现在我们已经找到了用 eZ publish 3 提高性能的其他方法。

关于 eZ publish [SDK](http://sdk.ez.no/) **的工作已经开始。你在这里有什么想法？**

这是 eZ publish 3 库的一个预览——它们将是与 eZ publish 3 捆绑在一起的默认文档和库。然而，当它完成时，将会有更多的文档。eZ publish 内核文档和许多教程也将成为 sdk 的一部分。

总的来说，你认为 eZ publish 的主要优势是什么，你希望在哪些方面进行改进？

eZ publish 的优势在于它使用简单，并且提供了很多功能。你可以用这个软件创建从内部网到大型电子商务网站的任何东西。我们想在 eZ publish 3 中改进的主要方面是它的灵活性和性能。

**SP:** [eZ 发布 3.0 版本](http://developer.ez.no/article/articleview/383/) **即将面世。你为新版本准备了什么？有惊喜吗？**

我已经提到了一些事情——eZ publish 用户会有很多惊喜，我希望大部分是积极的！你基本上会有更多的灵活性，并发现这个版本可以更快地定制。您还可以重用模板和代码，并在同一个代码库上托管尽可能多的站点。

##### PHP 和互联网

总的来说，你最喜欢和不喜欢 PHP 的什么？

我喜欢 PHP，因为用它开发软件非常快。当你在动态网站上工作时，你也有几乎任何你需要的库。我不喜欢 PHP 的地方是面向对象，但这将在版本 5 中得到改进。

你认为 PHP 的成功归功于什么？PHP 有什么其他地方找不到的东西吗？

PHP 很容易上手。做一些 PHP 脚本不需要很懂 PHP 编程。PHP 仍然提供更高级的功能，所以它比 Java 适用于更广泛的受众。

作为一个拥有其他“严肃的”面向对象语言经验的开发者，你对 PHP 支持面向对象有什么看法？你对 Zend 2.0 引擎即将到来的变化有什么感觉？你对那些喜欢在这个领域批评 PHP 的 Java 类型(或任何其他类型)有什么看法？

面向对象是 PHP 最差的部分，但是如果你小心使用它，它是没问题的。我真的很期待 Zend 2.0 引擎——那些批评 PHP 的人是完全正确的。然而，我认为 PHP 是最适合开发动态网站的语言。记住语言只是一种工具也很重要；重要的是你的软件的用户是怎么想的——他们通常不关心软件是用什么语言写的。

PHP 是一种吸引具有网页设计背景而非软件开发背景的开发者的语言。你是否觉得这些 PHP 开发人员更喜欢编写过程化的而不是面向对象的代码，这阻碍了语言的进步？你有什么建议可以提供给希望组织成功的 PHP 项目的 PHP 开发人员？

我认为越来越多的人开始编写面向对象的 PHP，但如果你现在看看用 PHP 编写的开源项目，大多数项目仍然没有这样做。如果更多的人会编写面向对象的代码，这肯定会促进 PHP 在这个方向上的发展。所以给其他开发者的提示一定是:编写面向对象的库和可重用的代码。

尽管 PHP 是互联网上最流行的服务器端语言，你认为它在主流 IT 界被忽视的根本原因是什么？你认为 PHP 作为一种可行的 Web 开发解决方案能在企业中得到广泛接受吗？

我认为这其中的主要原因是营销。但是 PHP 继续发展，被越来越多的公司使用。我认为 PHP 在未来的主流 IT 圈会被更多的使用。许多大公司已经在使用它了。

鉴于你在为 PHP 寻找一个可靠的 XML 解析器时遇到的问题(到了你自己写解析器的地步)，你对 PHP 的 XML 支持有什么看法？你看过 PHP 的 DOM 扩展的最新版本吗？你觉得它现在能够为使用 DOM API 解析和生成 XML 提供一个稳定的库了吗？您如何看待 PHP 对 XML-RPC 和 SOAP 等技术的支持(鉴于您已经为这些技术编写了自己的实现)？

PHP 的 XML 支持相当好，然而，它并不适合我们当时的需要，这就是为什么我们写了自己的实现。此外，还有一些内存泄漏的问题，DOM 解析器会占用所有的内存。我认为 PHP 最重要的是拥有良好的底层特性；您不需要在编程语言中本地实现每个协议。目前我认为 PHP 需要在这些扩展上做更多的工作，但是这些不应该转移对 PHP 中更重要的其他开发的关注。

除了 XML 支持，你认为 PHP 未来面临的主要技术挑战是什么？

面向对象。

你如何比较 PHP 与目前来自 Sun (J2EE)和微软(。网)？鉴于实验性的**和****。NET extensions/#enl#/，你认为 PHP 在这两大框架中的未来角色是什么？**
