许多人误以为网络统计是绝对不可辩驳的:这些数字是由独立的计算机生成的，不可能有错。那么，为什么两个或更多来源的统计数据很少匹配呢？为了理解这个问题，我们需要检查用于整理和分析统计数据的两种方法。今天，我们来看看服务器端的方法…

## 服务器端数据收集和分析

每次您访问网站时，web 服务器都会记录有关每个文件请求的信息，例如 HTML 文件、CSS 文件、JavaScript 文件、图形文件、Flash 电影、PDF 文档、MP3 音乐等。实现有所不同，但是大多数服务器在日志文件的一行中记录每个请求。数据通常包括:

*   请求类型—通常是 GET 或 POST
*   请求文件的完整路径名
*   日期和时间
*   请求者的 IP 地址，以及
*   请求者的用户代理；标识设备的字符串，即特定的操作系统和浏览器或搜索引擎机器人。

可以理解的是，即使在相对安静的网站上，日志文件也会增长到数百兆字节。

基于服务器的数据收集的主要好处是，不管使用什么技术，它都会记录每个文件请求。很容易评估下载的受欢迎程度或发现性能瓶颈。默认情况下，大多数服务器都会生成日志文件，因此可以访问有关流量增长的历史信息。

不幸的是，有许多缺点:

*   很少能确定用户的浏览设备。用户代理字符串提供的信息很少，而且可以伪造(Opera 曾经伪装成 IE，以确保网站不会屏蔽浏览器)。您通常无法评估用户的屏幕分辨率设置或他们是否启用了 JavaScript 和 Flash。
*   大型组织通常通过单一网关传递所有互联网请求。当两个或更多用户共享相同的 IP 地址和用户代理字符串时，用户识别变得困难。
*   The server logs cannot record cached files. Caching is essential and the Internet would grind to a halt without it. Your browser will cache files so, when you return to a page, it will show the files that were downloaded previously.

    此外，许多<acronym title="Internet Service Provider">ISP</acronym>在代理服务器上缓存流行的网站文件。当您输入网址时，您可能会看到从该代理返回的文件，而不是原始网站。随着你的网站越来越受欢迎，你甚至会经历文件访问的下降，因为更多的代理服务器缓存你的网站。

AWstats 之类的应用程序可以分析服务器日志文件，得出有意义的数字，比如独立用户或访问量。但是，这些应用程序在解释数据时必须做出假设。

例如，应用程序可以将单个“访问者会话”定义为在相同的 15 分钟内来自相同 IP/用户代理的访问。用户访问一个页面，然后等待 16 分钟，然后点击其他地方的链接，将被记录为两个单独的访问者会话。但是假设 20 分钟不活动期的应用程序将只记录一个访问者会话。
