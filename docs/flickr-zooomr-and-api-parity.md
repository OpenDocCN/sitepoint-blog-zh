# Flickr、Zooomr 和 API 奇偶校验

> 原文：<https://www.sitepoint.com/flickr-zooomr-and-api-parity/>

以下内容转载自《科技时报》第 141 期。

我在关于[Web 2.0 connectivity](https://www.sitepoint.com/web-20-connectedness/)的社论中提到的一个问题本周浮出水面，有[新闻](http://www.techcrunch.com/2006/06/16/why-is-flickr-afraid-of-zoomr/)称 [Flickr](http://flickr.com/) 拒绝了竞争对手 [Zooomr](http://zooomr.com/) 访问 Flickr API 的请求，这样 Zooomr 就可以为他们导入用户的 Flickr 照片和元数据(例如标签)。

现在，在你发飙之前，我应该指出，Flickr *已经*批准了来自其他竞争对手的 API 访问请求，如 [Riya](http://riya.com/) 和 [Tabblo](http://www.tabblo.com/) 。然而，Zooomr 代表的 Flickr 竞争的直接性打破了平衡，Flickr 做出了决定。通过电子邮件发送至 Zooomr:

> …对于直接替代 Flickr 的网站，我们选择不支持使用 API。

创始人斯图尔特·巴特菲尔德甚至在 FlickrCentral 论坛上分享了他的推理:

> 关于将商业 API 许可授予直接竞争对手:我们可能不会。[……]如果有真正的*直接竞争对手(到目前为止，我们只有极少数)，我们可能不会这么做。我不认为这是我们的恶意:为什么我们要浪费带宽和 CPU 周期直接发送东西给他们的服务器？*

可以理解，这引发了一些严肃的讨论。当像雅虎这样负责任的公司。声称他们的服务是开放的，允许用户检索他们放入他们的任何数据，当你想用你的数据做的事情是把它转移到一个竞争的服务时，他们有权利使它不那么方便吗？

在对这个问题进行了一些内部讨论后，巴特菲尔德[重新考虑了他的立场](http://www.flickr.com/groups/central/discuss/72157594165399644/#comment72157594167782546):

> 实际上，我改变了想法，并被 Eric 的立场所说服，即我们肯定应该批准直接竞争对手的请求，只要他们也这样做。这意味着(a)他们需要一个完整的 API ,( b)愿意让我们访问。
> 
> 这里的理由部分是“公平的公平”，更微妙的是，像 GPL 许可证一样，它强制用户在链上自由。我觉得我们会采取这种方式(内部还在讨论)。

O'Reilly 的 Marc Hedlund 对竞争服务之间的 API 开放应该是双向的这一提议表示欢迎，并将其称为“API 对等”。

虽然这种安排的对称性和内在的公平性乍看起来确实很有吸引力，并且在 Flickr/Zooomr 的情况下应该可以很好地解决这个问题，使用户受益，但我不认为这是这个问题的最终解决方案。

我并不是说所有基于网络的服务都应该被强制开放其用户创建的内容以便于输出。我当然相信这样做是正确的，并且最终会增强用户对服务的信心，但是正如[尼古拉斯·卡尔指出的](http://www.roughtype.com/archives/2006/06/why_do_you_thin.php)，厂商锁定是一种被证明的商业策略，不太可能很快消亡。但是*也会有*部分用户坚持公开他们的数据，像 Flickr 这样的服务会迎合这些用户。

我反对的是任何这样的开放服务故意为用户数据的某些用途设置障碍。回到 Flickr/Zooomr 的案例，Flickr 以“消耗带宽和 CPU 周期”为由反对向竞争对手提供单边 API 访问，这是站不住脚的。无论是用户下载数据，然后手动上传给竞争对手，还是通过 Flickr 的 API 直接传输给竞争对手，Flickr 的服务器都会运行得很好。Flickr 拒绝其竞争对手的 API 访问所带来的唯一不同是，当用户想要将他们的数据带到别处时，它任意降低了服务的便利性。

因此，虽然所谓的 API 平价肯定是要争取的，但它不应该成为像 Flickr 这样的开放服务提供 API 访问的条件。Flickr 应该关心的不是对竞争对手的公平:而是对用户的公平。除非服务的开放性是统一的、没有偏见的，否则它就不是开放的。
