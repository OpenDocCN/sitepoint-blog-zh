# 语义标记如何帮助服务器端开发人员编写可重用的代码

> 原文：<https://www.sitepoint.com/how-semantic-markup-helps-server-side-developers-write-reusable-code/>

如果你把大部分时间花在直接使用浏览器上，你可能已经知道当你的标记具有良好的结构和语义时，维护、测试和呈现一致性是多么容易。你不会忘记将内容和表现分开的重要性。此外，你知道如何很好地完成分离。

如果你能在下一个项目中写出每一行代码，你肯定它会是互联网上最干净、最易维护的网站！但是网站很少是前端开发人员的唯一产品，在任何涉及大量编程繁重工作的项目中，即使是最有质量意识的服务器端程序员也很难意识到高质量前端代码库的好处。很多时候，在大多数项目中，初看起来是 HTML 语义的一个小实例，实际上包含了许多对大多数人(包括管理和运营部门)来说显而易见的含义。

最近，我的一个同事(一个非常有天赋的 PHP 开发人员)就这种情况问了一个很棒的问题。在我们的项目中，我们希望在 notes 中出现的链接的锚文本之前创建一个换行符，这在整个站点中会以不同的方式呈现。我的解决方案是看起来像这样的 HTML:

```
<div class="help-note"><p>
    <span>Can you add a picture to the page about</span>
    <a href="..." title="Help us improve the article about 'Altocumulus clouds'">Altocumulus clouds</a>?
</p></div>
```

CSS 是这样的:

```
.help-note span { display: block; }
```

我的同事想知道为什么我选择使用带有额外 CSS 规则的`span`来完成我们艺术家设计中的换行符。他想，如果我们只使用一个`div`不是更自然吗？这样，我们可以完全放弃添加 CSS。另外，`div`元素和`span`元素不是一样的吗，除了默认应用了`display: block`？

这是一个很好的问题，可以用来说明标记在网站代码库的许多部分中的相互联系。让我们来看看其中的一些含义。

虽然实际上来说`div`和`span`元素都是无意义的，但还是有细微的区别。这不仅是因为浏览器对这些元素的默认样式不同，而且更重要的是，基于上下文的逻辑分组的含义也不同。

W3C 称:

> `div`和`span`元素，与`id`和`class`属性一起，提供了一种通用的机制来为文档添加结构。这些元素将内容定义为内联(`span`)或块级(`div`)，但没有对内容强加其他表示习惯用法。因此，作者可以结合样式表、`lang`属性等使用这些元素。，根据自己的需求和口味来定制 HTML。

乍一看，这些似乎是一样的，但是仔细看看措辞:<q cite="https://www.w3.org/TR/html401/struct/global.html#h-7.5.4">这些元素定义内容是内联的(`span`)还是块级的(`div` )</q> 。

换句话说，内容本身暗示了适当的元素。由于整个帮助注释在逻辑上是一个句子，并且只打算在应用样式时拆分成多行(假设*而不是*在内容的纯文本表示中，因为这对于句子结构来说很奇怪)，那么`span`元素是这两个元素中唯一合适的一个。[电影下载](http://moviedl.net/ "movie downloads")

使用一个`span`也有技术上的好处。例如，假设我们的用户非常喜欢 HelpNotes，以至于我们决定在更多的地方使用它们。我们创建了一个类`HelpNote`，它只有一个输出方法，签名如下:

```
/**
  * Produces appropriate output of a note for help regardless of display context.
  */
public static function renderHelpNote( Object $title )
```

这个函数是后端开发人员可以用来输出一个`HelpNote`的，不管他们从哪里调用它。因此，不管我们是否在多部分的电子邮件或网页中使用帮助注释(例如)，在 PHP 中我们只需说`renderHelpNote( $title )`就可以完成调用。这在循环或模板系统中是有用的。

现在问题变成了，帮助笔记句子应该用一个`span`还是一个`div`？如果我们使用一个`div`，那么我们知道要创建一个帮助注释的列表项版本，我们必须向`renderHelpNote`函数添加逻辑，以便省略`div`，或者将`div`重新样式化为一个内联级元素。然而，如果我们使用一个`span`，那么我们不仅坚持句子结构，而且通过坚持渐进增强的概念，通过只对最具体的情况进行样式化，并让更一般的、低保真度的设计不进行样式化，我们还保持了灵活性。

不可否认，这种精确性和过分的正确性对于一个没有意识到微妙之处的人来说可能是令人讨厌的，甚至是毫无意义的。然而，我认为这更有理由促进关注点的适当分离。

作为一名前端开发人员，帮助笔记列表如何排序对我来说并不重要；我只是相信会有一个列表，并且我的同事会按照他想要的顺序对它进行排序。同样，我也不想强迫我的同事去关心换行符看起来是否正确；我只想让他们输出代码，让我来处理显示。使用`span`的结果不仅仅是更干净的前端代码，也是更简单、更可重用的服务器端输出逻辑。

对我来说，后端开发人员和前端开发人员之间经常会有巨大的文化冲突。一个群体做出的选择在另一个群体看来可能是疯狂的。然而，更多的时候，仅仅一点点跨职能的教育就可以对编写对我们所有人都更好的代码大有帮助。所以要问问题，因为帮助你同事的技巧也可能帮助你。
